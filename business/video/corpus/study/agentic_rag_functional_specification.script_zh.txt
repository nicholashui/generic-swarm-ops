想象一下，你的硬盘里放着一座知识的大山。几万份文件，半个 gigabyte 的文字，而里面就收藏着几乎任何问题的答案。问题是，你怎样才能用一种聪明、可靠，而且不会乱编的方式，真正拿到那些答案呢？这就是这个项目想要解决的挑战，而答案就是一个叫做 Agentic Retrieval-Augmented Generation，也就是代理式检索增强生成的系统。在这条视频里，我们会一步一步带你看清楚怎样构建一个出来，而且这个目标相当有野心。

首先讲一下大局。这里的目标，是交付一个完整、可投入生产、可观测、可度量、可扩展，又可以做基准测试的系统。它不是玩具。它的设计，是要忠实地实现四个核心代理式设计模式和七个架构元素，这些都是直接来自一份关于 Agentic RAG 的研究综述论文。整件事预计要用十到十四天去构建，大约六天有一个可运行的核心，接着剩下的时间就用来扩大规模、整合那些比较进阶的部分，再加入那些让它值得信赖的观测和基准测试功能。

那么这个系统实际上要做什么呢？第一，它要原生地读取和索引大约六万五千份 Markdown 文件，一个大约五百 megabyte 的语料库。而且它要用一种叫做分层分块的技术去做，做的时候要节省内存、要逐步处理，还要在中途断了也能重新接下去。第二，它要支持混合式的知识表示。也就是说，要把一个向量数据库，叫做 Chroma，跟另一个叫做 LightRAG 的系统结合起来。LightRAG 会建立一个实体和关系的图谱，允许一种更聪明的、双层次的检索。第三，还有一个可选功能，叫做持续知识复利，灵感来自 Andrej Karpathy 的一个想法，系统可以写出它自己不断增长的概念维基，包括一个索引、一份日志，以及互相连接的笔记。最后，整件事都应该是本地优先、用 Docker 打包、完全可追踪，并且为生产环境而加固。

这份规格是经过超过十轮反复打磨的成果。它一开始只是一个简单的 Agentic RAG 系统请求，接着加入了跟维基想法的比较，再加入 LightRAG 的强化，再为六万五千份文件做扩展，还有反复要求越来越深入的设计细节。所以你现在听到的，是一个提炼过、最终定稿的版本。

我们讲一下核心概念，因为这些是没得商量的。有四个核心代理式设计模式，每一个都要在系统的行为里清楚地显示出来。第一个是反思。也就是说，那些代理会评估自己的输出，检查相关性、忠实度，以及有没有编造出任何东西，接着再反复改进。第二个是规划，系统会自主地把一个复杂的问题拆解成更小的子任务，或者一个多步骤的计划。第三个是工具使用，代理会以一种交错的方式动态地调用工具，遵循一个想、做、观察的节奏。第四个是多代理协作，那些专门化的代理会一起合作，共享状态，有时会有一个主管监督它们，有时就以平等的同伴身份运作。

在这四个模式之上，还有七个架构元素。第一个是路由，一个单一代理会决定把这个查询送到哪里，也可以委派给其他代理。第二个是分层的、基于图的控制流程，掌管整个流程怎样执行。第三个是自适应检索，查询的复杂程度决定了用哪种策略。第四个是有状态的记忆，会记住对话、长期索引，以及沿途的检查点。第五个是混合知识，结合向量数据库、一个轻量级的知识图谱，以及持续的 Markdown 维基。第六个是反复精炼，配合质量关卡和一个合理的循环次数上限。第七个是具备评估意识，内置了指标、追踪以及健康检查。

那么究竟是什么让这个做法这么突出呢？其实它比起朴素的 RAG 更好，因为它加入了真正的代理能力，也就是推理和行动的能力。它又比起一个纯粹的维基更好，因为它在你提问的那一刻就带来代理式的推理，还可以选择把发现写回去。而 LightRAG 就给了它快速的关系处理能力，不用付出从零开始重建一个完整知识图谱的沉重成本。

现在我们来看一下这个系统实际上是怎样连接起来的。一个用户问一个问题，可以在命令行或者网页界面里问。那个问题首先会去到一个路由器，一个查询分析器，挑选正确的策略。从那里去到一个规划代理，把这个查询拆解，再建立一个多跳的计划。这个计划会流向一个工具路由器，它会做一个结构化的决定，挑选用哪些工具，是向量检索器、LightRAG、网络搜索，还是维基。向量检索器会在 Chroma 里抽取，用一种平衡相关性和多样性的技术，再加上分层分块和重新排序。LightRAG 检索器就同时在实体和关系之间运作。而网络搜索工具就在有需要的时候连到实时的互联网。

那些检索器找到的所有东西，接着就会去到一个研究员兼评分代理那里。反思就是在这里发生。这个代理会根据一套准则为每份文件评分，如果找回来的结果不够好，具体来讲也就是这个分数跌破百分之八十五，而又还没循环过三次，它就会把这件事送回给规划代理再试一次。一旦结果足够强，就会交给生成代理，它会综合出一个附带引用的答案。接着一个评论代理会独立地检查那个答案的忠实度和有没有编造。如果不合格，这件事会送回给研究员。如果合格，你就会得到附带引用的最终答案。而可选地，一个维基综合代理可以拿那个结果，做出一个持续的、Karpathy 风格的记录到维基库里，更新索引、日志和概念笔记。在所有这些东西底下运作的，是状态和记忆层，它会保存检查点、对话摘要和长期索引，再加上混合知识层，里面同时放着 Chroma 向量数据库和 LightRAG 知识图谱。

这个系统依赖一些仔细定义的数据模型。每一份检索回来的文件都有一个结构，记录着文件本身、一个介于零和一之间的相关性分数、它来自哪里、它是父分块还是子分块、它在哪些标题之下，以及 LightRAG 从它里面抽取出来的任何实体和关系。还有一个更宽广的代理状态，会追踪那些消息、原本的查询、计划、检索回来的文件、任何评论、一个置信度分数、循环的次数、最终答案、引用、任何维基输出的路径、LightRAG 的上下文，以及各种元数据，例如时间戳和追踪信息。循环的次数被刻意限制在三次，这样系统就不会无限地循环下去。

图里每一个节点都有清晰的工作。查询分析器会分类一个问题有多复杂，是简单而事实性的，还是多跳而关系性的，再决定路由。规划器会输出一个有编号的计划或者一组子查询，用例子来处理多跳问题，例如把"比较 X 和 Y"变成分开的子查询加上一个综合步骤。工具路由器会产生结构化的输出，挑选工具和它们的参数，而且在安全的情况下可以并行地运行多个工具。研究员兼评分者会执行选中的检索器，根据相关性、完整性、时效性和权威性为每份文件评分，筛掉任何低于阈值的东西，如果结果薄弱就会反思。生成器会用筛选后的文件、计划和任何评论，去产生一个有依据、附带正文引用的答案。评论者会独立地为忠实度评分，如果分数低就可以触发另一次循环。而维基综合器就会产生干净、兼容于 Obsidian 的笔记，配以恰当的前置数据和维基链接，一路更新索引和日志。这些节点每一个都有自己的提示，有严格的角色、一套详细的准则或输出格式、一些正面和负面的例子，并且在涉及反思和规划的时候鼓励它一步一步地思考。

要把这样东西扩展到六万五千份文件，是最难的部分之一，所以这个摄取流程特别受重视。一个加载器会用多个工作进程去读取所有 Markdown 文件，一次分批处理几千份。接着来到分层分块，这个被视为 Markdown 的最佳实践。首先，那些文字会按它的标题切分，造出较大的父分块，每个有几千个 token，连同完整的标题路径一起存储。接着那些父分块会再切成较小的子分块，每个有几百个 token，并且有少许重叠，两者再用一个共享的标识符连接起来。向量索引会把所有东西加载到 Chroma，接着是 LightRAG 索引，会以异步的方式抽取实体和关系，并且用一个 OpenSearch 后端去应付这个规模。整个过程都可以重新接续，用那些文件的哈希值去追踪它已经处理过哪些，并且在每一批之后清理内存。它的目标很惊人：在一台有三十二 gigabyte 内存的机器上，少于四十五分钟完成一次完整摄取，而对于细微改动的增量更新就少于一分钟。

讲到检索，向量搜索会抽出一组候选然后重新排序。LightRAG 会以一种混合模式运作，结合低层次的实体和高层次的关系。而自适应路由器就会在一个查询涉及比较、问某样东西怎样运作，或者问事物之间的关系和联系的时候，倾向选用 LightRAG。

这个系统配备了一套既动态又可扩展的工具：一个混合检索功能、一个通过 Tavily 的网络搜索、一个维基写入器，还有作为附加的一个计算器和一个 arXiv 抓取器。这个图本身是围绕代理状态而构建的，配有条件边去驱动反思循环，依据置信度和循环次数，在可能的情况下并行执行工具，并且在每一个节点上都做完整的追踪。

在表面上，用户通过几个渠道去交互。有一个 Streamlit 网页应用程序，有聊天界面、一个可展开的推理过程视图，一步一步显示分数、文件和评论，还有一个把结果保存到维基的按钮。也有一个命令行界面，让你做支持重新接续的摄取、用可选的混合模式、维基写入和追踪去执行查询、检查语料库、运行评估，以及建立维基。讲到评估，有一个专门的工具去度量忠实度、答案相关性和上下文精确度，再加上一个自定义的反思分数，针对一个超过五十条查询的黄金测试集去测试，配以自动化运行和 JSON 报告。观测能力就通过为每次运行而设的追踪项目来实现，配以为代理指标而设的自定义元数据。所有东西都打包在一个多容器的 Docker 设置里，而错误处理就包括优雅的回退方案、重试逻辑和速率限制。

整个构建过程遵循一个严格的、分阶段的顺序。第零阶段奠定项目的骨架、需求、配置、数据模型、提示模板和 Docker 设置。第一阶段构建摄取流程，并且针对完整的六万五千份文件做基准测试，包括增量模式和 LightRAG 索引。第二阶段实现混合检索器和那些工具。第三阶段构建核心图，连同它的状态、节点、边，以及反思和规划循环，再加上记忆。第四阶段加入多代理协作、评论者和维基综合器。第五阶段引入用户界面、命令行工具、追踪可视化和评估工具。第六阶段处理 Docker、测试、日志记录、安全性和文档，包括图表、比较表和基准测试。而第七阶段就是用一百条复杂查询做端到端压力测试、延迟和质量的基准测试，再加最后的打磨。

那么我们怎样知道自己成功了呢？那些标准都是具体而可度量的。完整的语料库必须可以增量地摄取，不会耗尽内存又不会崩溃，而且要记录下这个基准。每一条复杂查询都应该清楚地展示出全部四个模式和全部七个元素。反思循环应该在至少百分之三十的查询上触发，并且可度量地改善质量。LightRAG 的混合模式应该在关系性和多跳的问题上，明显胜过纯粹的向量搜索。维基综合应该产生一个干净、即用的维基库。在评估方面，忠实度应该达到至少百分之九十二，答案相关性至少百分之九十，而平均延迟应该在普通消费级硬件上维持在四秒之内。而那些代码应该干净、完全有类型标注、有文档，并且在每一个阶段都提交到版本控制。

这份蓝图指向它的来源：原本的综述论文、配套的视频、带有 OpenSearch 支持的 LightRAG 项目、Karpathy 用来做输出风格的维基 gist，以及最新的 LangGraph 例子。即时要做的行动，是立刻开始第零和第一阶段，首先专注于稳健、可重新接续地摄取那个庞大的语料库。一旦摄取完成和做完基准测试，下一步就是详细审视那些提示，以及一节接驳这个图的工作。这就是一个 Agentic RAG 系统的典范、极度详尽、生产级的计划，结合混合检索和持续、复利的知识。最后的指示很简单：把它交付得干净、可观测、又有性能。
