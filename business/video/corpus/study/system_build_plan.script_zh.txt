今天我们会一起走过一些很有野心的东西。它叫做 VA Agent Swarm 系统建造计划，是一份单一、权威、一步一步的蓝图，用来建造一个完整的影片制作平台，由一百一十四个专门的 AI agent 去推动。你可以把它想成是交给一个 AI 编码 agent 的总指令集，明确地告诉它要建造什么、用什么顺序、设什么质量检查点，以及怎样用它自己的工具去可靠地完成这份工作。

让我先把背景讲清楚。这份文件并不是要重新发明这个系统的设计。那个设计已经存在于一整个规格档案库里面。相反，这份计划是建造手册。它从一个空白的 repository 开始，一路带到一个经过加固、可观察、达到生产级别的平台。这个系统本身是一个层级式的多 agent 系统。一百一十四个 agent，组织成十个类别，每一个都是一个更大的工作流程图里面的一个节点，全部一起协作，把一份客户简报，变成跨多个渠道交付的成品影片。

计划的第一部分，描述这个编码 agent 实际上应该怎样运作。它每完成一个里程碑和每一个任务，都会遵循一个清晰的循环。首先，它进入一个规划模式，在那里它会阅读相关的规格、重新陈述目标、列出它打算建立或者改动的档案，再提出任何不清楚的地方。最重要的是，在这个规划阶段它不会写任何 code。接着，它会对照那个里程碑的验收标准去确认这个计划。如果有什么东西含糊，它会问一个合并起来的问题，而不是靠猜。然后，它会先写一个会失败的测试，遵循测试驱动开发的做法。只有在那之后，它才会实现让这个测试通过所需的最小的那一段 code。它会跑一个本地验证关卡，把 linting、类型检查和单元测试结合在一起，而如果那个关卡是红色的，它绝对不会往前走。之后，它会做自我审查，用一个清晰、有结构的 commit 信息去提交这份工作，更新一份持续更新的进度档案，并且在不相关的任务之间清空它的工作上下文，让它的思路保持敏锐。

这里有一个不断出现的主题，作者把它叫做「重新想一百次」的指令。委托这份计划的用户要求它全力以赴地重新想一百次。计划没有把这句话当成一句口号，而是把那种强度直接嵌入到它的结构里面。在接近尾段有一份实实在在的一百点加固检查清单，组织成十个主题，每个主题十项检查，整个系统要全部一百项都通过，才算做完成。每一个里程碑都会在几个维度上被重新挑战，从可追溯性一路到用词。它的理念是，这里的质量是递归的。一个评判影片的系统，本身也必须用同样的标准去被评判。

计划列出七条黄金规则，而违反其中任何一条都算是缺陷，就算那些测试技术上是通过了也一样。规则一，合约先于 code。共用的数据合约在写任何 agent 之前就建立和冻结，这样一百一十四个 agent 不会漂移成一百一十四种不同的消息格式。规则二，先做垂直切片，再求广度。一个完整的工作流程在其他 agent 建造之前，就要通过真实的基础设施从头跑到尾。规则三，每一个 agent 都是同一个共用基类的实例。没有自定义、一次性的 agent 循环。新的 agent 只是配置、一份评分标准和工具。规则四，没有任何 agent 直接和用户界面沟通。Agent 把事件发布到一个共用的 event bus，再由一个 gateway 把这些事件分发出去。规则五，尽可能保持确定性。固定好随机种子、模型版本和 prompt 版本，并且记录下来做审计。规则六，成本和安全是关卡，不是事后才想的东西。每一个 token 从很早期就开始计量，而合规 agent 一旦存在，就可以立刻封锁工作。最后规则七，从第一天起就把昂贵的外部生成模型藏在一个接口后面去模拟，让整个系统在测试期间可以靠廉价的替身去运行。

在整个建造过程中，这个编码 agent 会维护几份持续更新的文件。有一份进度检查清单、一份架构决策日志（每一个不显而易见的选择都有一条带日期的记录）、一份持久的项目记忆档案，还有一个配置文件夹，装着它的 sub agent、自定义命令、设置和自动化 hook。

那么究竟在建造什么呢？用一段话来讲，它是一个层级式的多 agent 系统，把专业影片制作从一份客户简报，一路自动化和强化到多渠道交付。一百一十四个 agent 在一个工作流程图里面作为节点运行，由一个工作流程引擎让它们持久化，这样长时间运行的制作可以在宕机和重启之后继续存活。它们通过一个共用的交接合约彼此传递创意工作，并且通过一条专用的消息总线互相评论。所有东西都受一个分层的质量框架把关，通过追踪和仪表盘来观察，再通过一个现代的网页控制台呈现给人。

建造哲学可以用几条原则归纳。先从一个「会走路的骨架」开始，也就是那条最薄、但真的走得通的端到端路径，然后才加肌肉。先建平台再建 agent，因为大约七成的艰难工程其实是平台本身。宁愿用工厂式的做法都不要逐个手工打造，等到平台和几个参考 agent 完成之后，其余的 agent 是从它们的规格生成出来再去审查，而不是一个循环一个循环地写出来。还有，把质量当成是递归而且可度量的，把系统自己的评估哲学套用在系统本身。

有一大段是配置这个编码 agent 来确保可靠性。它描述怎样用一份项目记忆档案做一份长开的宪法，装着技术栈、monorepo 地图、黄金规则、建造命令，以及指向每一份规格在哪里的指针。它的建议是保持精简，因为那份记忆会在 agent 有限的上下文里面，和实际的任务争位置。

计划也定义了几个 sub agent，每一个都有独立的上下文和一套狭窄的工具。有一个 spec reader，会消化一份规格，再返回一份紧凑的摘要连同确切的要求。有一个 contract guardian，会检查一个改动是不是偷偷破坏了或者分叉了已经冻结的共用合约。有一个 test author，会先写那套会失败的测试。有一个 test runner，会跑相关的测试再返回一份最精简的诊断。有一个 code reviewer，会对照「完成的定义」去检查一个改动。还有一个专门的 agent factory smith，会把 agent 登记册里面的一行，变成一个具体的 agent 配置。使用规则很简单。把阅读和验证委派给 sub agent，但是把决策和整合留在主线程上面。

在这些之上，计划定义了一些可重复使用的斜杠命令，用来做常见的工作流程，例如加载一个里程碑、建立一个新 agent、跑验证、检查合约、跑某一个特定的质量关卡、记录一个架构决策，以及跑一次加固审计。它也描述了一些可选的插件服务器，用来查看数据库、工作流程历史，以及可观察性追踪，并且建议把这些东西保持到最少。它还设置了一些自动化 hook，在 agent 的动作周围触发，例如自动格式化改过的档案、封锁对受保护合约档案的编辑、在一个回合结束之前跑验证关卡、扫描密钥数据，以及提醒这个 agent 更新它的进度档案。

权限处理得很小心。有一份允许清单，给安全、频繁的命令用，而具破坏性或者不可逆转的命令，没有人工明确确认之下绝对不会被允许。这个 agent 会实践上下文卫生，在不相关的任务之间清空它的窗口，在自然的断点压缩上下文，并且为不同的轨道（比如用户界面和 meta agent）用并行的工作树。

每一个任务都有一个严格的完成定义。行为必须由在写 code 之前就写好的测试所覆盖，而且全部都要是绿色。整套验证套件必须通过。公开合约必须维持不变，或者只能通过一份有记录的决策和 guardian 的签核去改动。Code reviewer 必须报告没有任何阻塞性或者重大的发现。里程碑的验收标准必须有证据去满足。必须做一个干净、有结构的 commit，没有密钥数据，也没有遗留的 debug 杂物。而如果公开接口有改动，文件也必须更新。

接着计划就钉死了技术栈，而且它很强调这些是决定，不是选项。在后端，是 Python 配一个快速、由 lockfile 驱动的依赖工具。在前端，是 TypeScript 配 React，再用一个采用 app router 的现代网页框架。两边都有严格的 linting、格式化和类型检查，还有一套丰富的测试设置，包括给合约用的属性式测试，和给用户界面用的浏览器式测试。

至于平台服务，agent 编排会跑在一个支持条件分支和人机协作关卡的图框架上面。一个持久的工作流程引擎保证这些制作（可以运行几分钟到几个钟头）可以在重启之后存活，连同重试和重放。一个 event bus，从一个流式存储开始，之后可以扩展到一个更强大的消息系统，处理带持久化和重放能力的发布和订阅。一个关系型数据库保存制作元数据、关卡状态、评论、配置和审计日志。对象存储保存影片、音频和图像产物，用内容寻址的 key。一个向量数据库和一个混合式、基于图的检索层去推动知识和记忆系统。有一个统一的客户端，在一个接口后面和多个大型语言模型沟通；有可观察性工具，给追踪、指标和日志用；还有一个来源证明标准，会签署每一个产物，让这个链条可以在下游被验证。所有东西都容器化，从开发期的本地组合，到生产期的一个受管理集群，连同一个专用的图形处理器池给生成任务用。

外部工具供应商，也就是实际的影片生成器、声音克隆、口型同步、音乐、空间音频和评估指标，永远都藏在共用的接口后面。Agent 永远不会直接调用它们。有一个决策被记录下来，就是每一个供应商都实现同一个协议，连同报告自己能力、估算成本、运行和检查自己健康状况的方法。一个 router agent 会根据成本、质量和延迟在供应商之间挑选，而测试环境只会用模拟供应商。

这个 repository 组织成一个 monorepo。在根目录里，坐着那份项目记忆、进度检查清单、决策日志、一个单一的命令界面、agent 配置文件夹、容器定义和工作区档案。在一个共用 packages 区域之下，住着那些可重用的构建组件，而其中最重要的，就是那个被冻结的合约 package，它是最先建造的。其他共用 package 包括 agent core（连同它的基类和自我精炼循环）、agent factory、语言模型 gateway、供应商、检索层、质量控制层、event bus、记忆存储、来源证明签署，和可观察性接线。在一个 services 区域之下，住着那些可部署的程序，包括 orchestrator、agent runtime worker pool、api gateway 和一个调度器。网页控制台住在它自己的应用文件夹里。还有一个专用的 agents 文件夹，装着那一百一十四个 agent 定义，分成生产工艺 agent、meta agent、支持 agent 和横切服务，全部由一个权威的登记册去索引。也有文件夹给工作流程图，和给评估资产用，例如黄金测试集和评分标准。规则就是每一个 package 从出生那一刻起，就附带至少一个可以轻易通过的测试，让验证关卡永远都是绿色。

现在我们到了最重要的那一段，就是横切合约。这些是最先建造和冻结的，因为所有下游的东西都依赖它们。第一个是共用产物交接合约，是一个单一的数据模型，会随着每一个产物在各个阶段之间移动时一起带着。它捕捉技术规格，比如 codec、宽高比、时长、帧率、色彩空间、响度，以及是不是需要字幕。它捕捉权利和同意，包括授权状态、是不是给了肖像和声音同意、地域限制，以及任何禁播日期。它捕捉连贯性状态，比如一个角色的外观、道具、服装、环境，以及一个用来检测漂移的身份哈希值。它捕捉跨各层的质量状态，以及一份来源证明清单，装着签署引用、签核链、所用的确切模型版本和种子，全部都是为了可复现性。产物本身带着一个 id、一个版本号、它的媒体类型、它的位置、它的父资产、简报范围，和它的目标渠道。这个合约的测试是最先写的。它们检查这个模型能不能经过序列化的来回转换而存活、检查在没有升版之下移除或者重命名一个字段会不会让建造失败、检查父资产是不是形成一个没有循环的有效图，以及检查每一个发布的产物是不是带着一份非空的来源证明清单。

第二个合约是评论消息总线架构。这个就是任何 agent 对任何其他 agent 的作品作出评论的方式。每一条评论带着一个 id、它来自哪个 agent、它针对哪个 agent、一个对相关产物的引用、一个严重程度、一个类别、支持的证据、一个建议行动、一个可选的评分标准引用、一个截止阶段、一个可选的分数，和一个时间戳。严重程度从阻塞级（blocker），向下经过重大（major）、轻微（minor），到最小的那个小瑕疵（nit）。类别涵盖节奏、连贯性、准确性、合规、无障碍、品牌、工艺和美学。验收规则很严格而且经过彻底测试。一个阻塞级会让那个工作流程节点停下来，直到解决为止。一个重大会在接收的 agent 上面触发一个自我精炼循环，上限三次迭代，之后就升级交给一个裁判。轻微和小瑕疵的评论会记录进记忆，并且汇总成下一个训练周期的奖励信号。两个 agent 之间的争议会被路由到一个裁判那里做一场有结构的辩论，而合规评论永远当作阻塞级去处理。

第三个合约是共用的 agent 基类。那一百一十四个 agent 每一个都是这一个类的实例。它的配置会命名这个 agent、它的类别、它的系统 prompt、它的模型政策（连同偏好的模型和预算）、它被允许调用的工具、它的评分标准、它的可度量自我质量指标、它的评论关系，和它的精炼上限。基类本身定义了生命周期。它起草作品、对照评分标准去评论自己的草稿，再修订，遵循 self refine 的模式。失败的时候，它会保存口头反馈再重试，遵循 reflexion 的模式。基类也接驳了那个有计量功能的语言模型 gateway、检索客户端、记忆存储、event bus、来源证明签署和追踪。没有任何子类会重新实现这些东西的任何一部分。Agent 之间的唯一区别，就只是它们的配置。

第四个合约是那套供应商接口，全部都可以模拟。第五个是质量网（quality mesh）本身，连同三个评估层，再加一个六道交付检查。第一层用确定性验证器去检查规格合规性，而且必须完全通过。第二层是由一个扮演裁判的语言模型，用那个角色的宪法去给出的评分标准分数，而且必须达到至少一百分之中的八十五。第三层是一个偏好比较，对着一个人类参照做两两比较，再加上一个至少二百个人物角色的模拟观众，和至少二十个人机协作样本，而且必须达到对等或者超越一个门槛。那六道交付检查涵盖规格、产物完整性、音频同步、连贯性、感知质量和出街准备，而六道全部都必须通过。那些确定性检查是最先建造的，因为它们完全可以测试。那些基于裁判的检查是用冻结了、钉死了的裁判和黄金测试集去建造的，让那些分数保持稳定。计划很坚决，就是一个裁判模型永远都不应该被允许浮动而没有钉死。

合约 package 也会生成前端类型，让网页控制台和后端保持同步，并且定义 event bus 的主题。每一个制作都有给 agent 事件、评论、关卡和产物用的主题，再加一条系统范围的警示频道。每一个事件都必须是其中一个有类型的事件模型，比如一个 agent 状态变更、一个产物已建立、一条评论消息、一个关卡变成准备好或者已解决、一个预算更新、一个指标更新、一个记忆条目、一个工具调用、一个制作阶段变更，或者一个错误。这些有类型的形状，是总线上面唯一被允许的。

合约就位之后，计划就铺陈出一个分阶段的路线图，有十三个里程碑，从零编号到十二。排序原则是从基础到智能、到生产、再到强化，但是那个垂直切片会在很早期就打穿过去，让架构在扩展到广度之前已经被证明。每一个里程碑都会指明它的目标、它的依赖、要建造的档案、编码 agent 的工作流程、测试，和一个硬性的验收关卡。在当前的关卡变成绿色和记录下来之前，你不会开始下一个里程碑。

里程碑零是引导启动。目标是一个可以编译、绿色、工具齐全但是空白的 monorepo，连同编码 agent 的所有配置都就位。暂时没有任何东西做到什么，但是验证关卡通过，而且启动那个容器堆栈会带起每一个后端服务。这里就是建立脚手架、命令界面、容器定义、依赖设置、持续集成管线，和所有编码 agent 配置的地方。这个关卡就是：从一个干净的 clone 开始，引导启动、带起服务和验证全部都通过绿色。

里程碑一是要建立基础知识服务，也就是每一个 agent 都会调用的 agentic 检索系统。它包含一条摄取管道，会把文档切块、做 embedding，再索引到一个 vector store 以及一个 graph 层。它提供混合式检索，结合 vector、graph 和关键字搜索并加上重新排序，还可以规划多步骤的检索。它的 client 提供检索、多跳复合查询、摄取，以及附带来源评级的引用。知识会分隔成不同的 namespace，这样一个项目的世界就不会泄漏到另一个项目那里。这一关要求在一个 golden set 上达到精确度目标，并且返回带评级的引用。

里程碑二是最大的平台里程碑，也就是编排运行环境，是整个系统跳动的心脏。它会建立带类型的 event bus，具备至少一次的交付保证以及幂等性；也会建立可观测性层，在那里每一个节点的运行都是一条 trace span；还有编排服务，结合 graph 运行环境以及人在回路的中断点，以及一个持久化的 workflow engine，在那里每一个 agent 任务都是一个可重试的 activity。它还会建立平台 agent 的骨架版本，包括 orchestrator、planner、router、judge、gatekeeper 和 memory agent，再加上 memory store 以及资产骨干，后者具备不可变的 id、写时复制的版本以及来源签署。这是一个要「想到爆」的里程碑。计划明确要求把 graph framework 和 workflow engine 之间的边界，记录成一个有据可查的决策。它的验收关卡，也就是所谓的 walking skeleton，是一个写死的三节点 graph，可以在真实基建上从头跑到尾，扛得住中途杀掉一个 worker，发出正确的类型事件，并且为它的产物签署。

里程碑三是要建立语言模型 gateway、router 以及成本仪表盘，这样从第一天开始，每一个 token 都会被计量并且路由。这个 gateway 包住一个统一的 client，提供 provider 抽象、自动重试以及退路、每次调用的 token 和成本计量并发送到 bus、把版本标记写入来源记录、响应缓存，还有一个冻结的 judge 模式。Router agent 会按成本、质量和延迟挑选最好的 agent 和模型，而成本仪表盘则按每个制作、每个 agent、每个 provider 汇总开支，并设有警报阈值。预算护栏会在一个制作超出它的额度时，强制硬性停止以及升级处理。这一关要求任何 agent 调用都被计量、被路由、被版本标记、在仪表盘可见，并且在预算违规时触发一次真正的停止。

里程碑四是要建立 research agent 和 coding agent 的框架。Research agent 会做查询规划、多来源检索、综合，并且产出一份带来源评级和引用的档案，再写入知识 namespace 里面。有趣的是，计划指出 coding agent 的规格，其实正好描述的就是 coding agent 在这次建构过程中扮演的角色，所以它的惯例会被抽取进项目记忆，再由 code reviewer 强制执行。这一关是要产出并存储一份带评级和引用的档案，同时 coding 惯例得到强制执行。

里程碑五是要建立智能层，也就是每一个制作 agent 都会用到的共享推理服务。其中有深度意图分析，会把一份简报解析成结构化的意图、目标、受众、隐藏议程以及约束，它是每一个制作的入口点。也有通用创意 agent，是一个创意构思引擎。还有一个流程优化 agent，会根据遥测数据改善工作流程。也有一个策略目标达成框架、一套复杂问题解决方法论，以及一个 aesthetics agent，会提供感知评分、新颖性信号以及美学评论。Creative agent 和 aesthetics agent 会形成一个生成以及评估的回路。这一关要求全部六个推理服务都通过它们的 golden 行为测试，并且要展示出那个回路。

里程碑六是关键的那一个，也就是 agent factory 加上垂直切片。目标是在建立剩下那一百零九个 agent 之前，先用最便宜的真实工作流程去验证整个架构。这个 factory 会把一份 agent 配置变成一个可运行的 agent，验证它的引用并且注册它。然后就会经这个 factory 建立刚好够用的工艺 agent，去从头跑一个工作流程到尾，也就是病毒式 hook 短片，在真实基建上用 mock 生成器运行。整条完整路径会从简报开始，经过意图分析，经过一个会建构工作流程 graph 的 planner，经过在 mock 上执行的各个 agent，产物会经由交接契约流动，critique bus 处于启动状态，质量网把关，产出一份已签署的交付物，并且在 bus 上有事件。这是一个要「超级深思」的里程碑，因为这个 factory 最终一定要产出全部一百一十四个 agent，所以它的配置 schema 现在就要做到完整。它的验收关卡是关键性的。从头跑到尾运行整个工作流程，一定要由一份简报产出一份已签署的交付物，每一份交接契约都填好，每一个关卡都有强制执行，所有事件都发出，有完整的来源记录，并且在预算之内，全部用 mock provider。计划讲得很强硬。这一关是用来验证整个平台的，而在它还不够稳之前，你是不应该走到铺开广度那一步的。

里程碑七是要建立制作工艺 agent，也就是头五十二个，做法是用 factory 产出的配置、评分准则以及提示。对每一个 agent，都会用一套可重复的脚本去产出它的配置、它的版本化提示、它的评分准则、它的可量度自我质量指标、它的工具许可清单，以及它的 critique 关系。它们会按类别一个一个地建立，共用评分准则的脚手架，而且在类别之间会清空上下文。这一关要求全部五十二个都已注册，并且在它们的规格测试、评分准则测试和 critique 测试上都是绿灯，再加上至少三个额外的工作流程原型在 mock 上从头跑到尾。

里程碑八会把平台 agent 的骨架升级成完整实现，并且加入 meta agent，也就是第五十三到第八十个。那些涵盖具备完整争议解决的编排 agent、主要是委派给 creative 和 aesthetics 服务的创意 meta agent、建立在共享 research 核心之上的 research meta agent，以及涵盖提示、成本、延迟、留存、广告支出回报以及无障碍的优化 agent，再加一个评估框架以及一队安全红队。完整的质量网会在这里完成，包括模拟受众层以及全部六个交付验证器，由 gatekeeper 强制执行零泄漏缺陷。计划建议在这个里程碑早段就建立评估框架以及安全红队，这样它们就可以持续测试其他所有东西。这一关要求全部八十个 agent 上线，完整的质量框架在每一条发布路径上都有强制执行，并且红队和框架每晚运行。

里程碑九是要建立工作流程支持 agent，也就是第八十一到第一百一十四个，以及交付织体。那些包括资产管理、版本控制以及渲染派发，配一个会跨图形处理器分批处理以及自动扩展的 render farm。也包括质量关卡、交付打包以及合规，会打包成影院、流媒体、广播、存档、预告片以及社交格式，附带各个渠道专属的规格、字幕、元数据、版权管理以及来源负载。还包括分析、反馈以及重新训练 agent，会把那些细小的 critique 汇总成一个奖励信号。交付织体会运行一条分支管道，这样影院、流媒体、广播和存档，再加上营销衍生品，就会并行产出，并且逐个渠道做验证。这一关要求一个制作要产出全部四条交付分支再加营销衍生品，每一个都有效，并且发布后的遥测数据会回流到一张重新训练的工单。

里程碑十是要建立用户界面、控制台、api gateway 以及 websocket 层，也就是人类操作员的界面。值得留意的是，这个在里程碑三之后就可以并行开始，在一个独立的工作树里面，先对着一个假的事件发射器开发，之后再整合。这个 gateway 会完全按照契约暴露 rest 端点以及 websocket gateway，具备身份验证、基于角色的访问控制、速率限制、验证，以及在关卡批准时的签署。它会订阅 event bus、按制作过滤，再经 websocket 扇出去。网页控制台会提供一个简报工作室、一个显示节点状态的实时 graph 画布、一个产物画廊、一个 critique 动态、一个关卡批准对话框、一个预算追踪器、一个质量仪表盘、一个 agent 查看器、一个记忆面板，以及一个交付中心，全部都由类型事件以及生成的类型驱动。这一关是要一个人类可以完全通过浏览器，去为第一个工作流程启动、实时监控、做 critique、批准关卡，并且下载交付物，而且实时更新要很快，也没有任何 agent 直接调用界面。

里程碑十一是增强层，加入个性化以及以音频为先的变体。其中有创作者的心理剖析、预测受众偏好的心理推荐、一个可以在严格隐私以及同意审计之下更换姓名、面孔和声音的个性化引擎，还有一个用来做以音频为先工作流程的 podcast agent。这一关要求在同意关卡之下产出个性化以及按受众调节的变体，并且要 podcast 原型从头跑到尾。

里程碑十二是加固、规模、安全以及发布，而这个里程碑就是一个主题一个主题地执行那张一百点清单。它涵盖负载测试以及自动扩展、安全加固（包括提示注入防御以及一次完整的红队扫荡）、通过混沌测试以及灾难恢复演练达到的可靠性、每一件可发布产物都有来源记录的合规、经过验证的成本仪表盘、更新的文档以及操作手册，以及一个由内部到有限再到普遍可用的分阶段发布。最后那一关要求全部一百项检查都通过，要有一次完整剧情片的演练在 mock 上操练全部一百一十四个 agent，并且要在真实 provider 上做一次实时冒烟测试，在预算之内成功。

计划停下来解释为什么垂直切片要放在这个位置。在验证一个工作流程从头到尾之前就去建立一百一十四个 agent，会是分布式系统那个典型错误，也就是在大部分代码已经假设了某个架构之后，才发现那个架构有缺陷。所以计划刻意先建立平台，把一条薄切片从头打通到尾，然后在架构已经身经百战的情况下，再经 factory 去铺开广度，之后再加界面以及增强，最后再加固。如果切片揭示出问题，你是在只有九个 agent 进行中的时候去修，而不是一百一十四个。它称这个为整个计划里面最重要的那一个排序决策。

核心位置有一个可重复的模式。十个工作流程原型每一个都变成一个 graph。它们共用一个共通骨架，也就是 greenlight、前期制作、制作、后期、审查和发布、分发，以及发布后，而它们唯一的区别就只是哪个 agent 领导每一个阶段，以及哪些 critic 为交接把关。一个工作流程是在它可以在 mock provider 上从头跑到尾、每一个阶段关卡都强制执行它那套 critic、并且最终产物带着一条完整的来源记录链的时候，才算是完成。

Agent 实现脚本被详细写明，是对全部一百一十四个 agent 都会运行的确切配方，没有任何一个 agent 是在脚本之外用手工砌出来的。那些步骤是：通过 spec reader 读取那个 agent 的规格、把它的自我质量准则映射成带阈值的具体指标、引用规格里面点名那个工艺权威去撰写它的评分准则或者章程、定义它被允许的工具、接驳它的 critique 关系、写好它的配置并注册它、撰写它的版本化提示、先写好它的失败测试、通过 factory 把它实例化而不是写定制逻辑、审查它，再在用到它的工作流程里面注册它。计划也点名了一些要拒绝的反模式，比如没有评分准则的 agent、直接调用 provider 而不是经工具界面的 agent、改动另一个 agent 产物而不是发出 critique 的 agent，以及自我质量是无法量度的空泛文字的 agent。

讲到测试以及评估，计划坚持，因为这个系统本身就是一部评估引擎，它自己的测试套件就一定要做得堪称典范。一共有五层。单元测试是纯逻辑、完全 mock、确定性而且快速，包括为那些契约做的属性测试。契约测试会为每一个共享模型的 schema 拍下快照，而一个移除或者重新命名字段的改动，除非经过适当的版本控制以及批准，否则会让构建失败。集成测试会对着真实的后端服务配 mock 生成器运行，涵盖 graph 执行、杀死和恢复的持久性、事件重播、关卡中断、契约传递、critique 路由，以及预算强制执行。行为以及 golden set 评估会把三层网应用在系统自己身上，用冻结以及钉死的 judge、一个模拟受众，以及一个每晚以及在每次相关改动时运行的框架。对抗以及安全测试会持续发动攻击，比如深伪滥用、提示注入、越狱、诽谤和偏见，目标是攻击成功率在百分之一或者以下。而端到端测试就涵盖浏览器里面那些关键用户旅程。持续集成金字塔会在每次改动时运行那些快速检查，而在每晚就运行那些繁重检查。

讲到可观测性、成本、安全以及合规，每一个 agent 运行、工具调用、模型调用以及关卡决策都会变成一条 trace span，所以一个制作会由简报到交付形成一棵 trace 树。指标会流到仪表盘，事件溯源的 bus 以及工作流程历史让任何制作都可以被重播，而日志都是结构化并且以标识符互相关联。成本会逐个调用计量进预算事件以及仪表盘，在违规时硬性停止以及升级处理，并且有一个成本优化器把路由保持在最佳成本和质量的前沿。安全以及合规会由一个 compliance agent 强制执行，它在每一条发布路径上都可以拦截，在任何肖像或者声音生成之前都要求一条同意链，为百分之一百的可发布产物签署，并且具备完整的来源记录以及审计。计划定义了一个不可以讨价还价的发布谓词。一件产物只有在通过规格层、在评分准则上至少拿到八十五分、达到偏好阈值、通过全部六项交付检查、过了合规、已经签署，并且在预算之内的情况下，才算是可以发布。整个条件会包裹在一个函数里面，而 gatekeeper 就只会调用那一个。

讲到部署，一共有三个环境，也就是用 mock provider 的开发环境、用合成负载的预发布环境，以及用真实 provider、完整密钥，以及一个自动扩展图形处理器池的生产环境。那些管道会在每次改动时运行验证、契约和 golden 检查，在主分支运行完整集成，并且运行那套繁重的每晚套件。惯例包括按里程碑划分范围的结构化提交、短命的分支、不可以直接推送到主分支，以及每一个改动都要经过一个有审查的请求。

讲到数据、模型以及提示管理，每一个提示都会被版本化并且记录进来源记录，而提示的改动在晋升之前要经评估把关。模型会按每个 agent 的策略钉死，而升级要经评估把关，不容许任何退步。种子、风格参考，以及参考帧库都会被版本化以保证可重现性。Golden 固定装置会被冻结并且审查，而更改一个预期输出就需要理据，因为这可能代表评分准则漂移了。美学档案受同意管治并且签署，而评估数据集就包裹在一个共通界面后面，并且长期追踪基线。

计划包含一份风险登记册，点名那些风险以及它们的缓解措施。大的那些包括在广泛建立 agent 之后才发现架构缺陷，由垂直切片缓解。所有 agent 之间的契约漂移，由冻结契约以及快照测试缓解。失控的成本，由计量、预算关卡、测试里面用 mock，以及一个实时冒烟上限缓解。在 graph framework 和 workflow engine 之间那个边界产生的混乱，由一个有据可查的决策以及杀死及恢复测试缓解。嘈杂的 judge 评分让关卡不稳定，由冻结钉死的 judge 缓解。奖励黑客产出好看但是空洞的输出，由 aesthetics agent 的反黑客层以及在低置信度时的人类审查缓解。Provider 中断、同意以及知识产权违规、上下文膨胀让 coding agent 退步、经摄取内容而来的提示注入、扩展瓶颈、不稳定的测试，以及新 agent 其实没有填补任何真实缺口那种范围蔓延，每一个都有它自己的缓解措施。

接下来就到那张一百点加固清单本身，也就是「想一百次」这个指令的字面以及结构化形态，组织成十个主题、每个主题十项检查。主题一是契约以及 schema 完整性，确保每一条消息都有类型、每一份交接都有填好、快照守护住那些模型，并且没有任何套件在本地重新定义一个共享契约。主题二是编排以及状态，涵盖杀死和恢复、扇出和扇入的正确性、死锁检测、超时、有据可查的框架边界、真正会阻塞的关卡、幂等性、确定性重播、背压，以及优雅降级。主题三是 agent 正确性，确保所有 agent 都是来自 factory、通过它们的规格和评分准则检查、和它们的 critique 边完全吻合、强制执行它们的工具许可清单、为它们的精炼回路封顶、保存它们的教训、永远不改动另一个的产物、用可量度的自我质量，并且尊重合规拦截。主题四是质量网，确保只用那一个发布谓词、那些确定性验证器是完整的、那些 judge 是钉死和稳定的、模拟受众的规模合适、全部六项交付通过都把关、检测到连续性漂移、美学评分被分解、奖励黑客防御处于启动状态、无障碍有把关，并且一个植入的缺陷会被可靠地捕捉到。

主题五是成本以及性能，涵盖计量准确性、预算强制执行、最佳路由、缓存、延迟目标、自动扩展、存储分层、工作负载分离、实时冒烟的预算上限，以及并行负载。主题六是安全、保安以及合规，涵盖红队成功上限、注入防御、同意链、完整签署、各项监管清单、密钥保护、基于角色的访问、依赖扫描、披露，以及对个人信息的小心处理。主题七是可观测性以及可操作性，涵盖每个制作一棵 trace 树、那些仪表盘、结构化日志、可重播性、警报、快速退步检测、操作手册、备份和恢复、灾难恢复演练，以及为安全推出而设的功能旗标。主题八是前端以及人类体验，涵盖完整的浏览器工作流程、快速准确的实时更新、没有直接调用界面、重新连接处理、批准时签署、实时仪表盘、浏览器测试旅程、访问控制、用户界面改善待办清单，以及按规模自适应的工作流程。主题九是构建流程以及 coding agent 的卫生，涵盖当下精简的记忆档案、有定义的子 agent 和指令、有强制执行的 hook、一份完整的决策日志、一份准确的进度档案、干净且划分范围的提交、每次提交都有一个绿灯的验证关卡、有实践的上下文卫生，以及有遵守的测试先行纪律。而主题十是端到端系统验证，涵盖全部十个原型在 mock 上运行、剧情片演练操练全部一百一十四个 agent、可验证的来源记录、有效的多渠道交付、重新训练回路的闭合、可展示的优化改善、creative 和 aesthetics 回路产出更好的候选、来源评级的 research、一次受预算约束的成功实时冒烟，最后就是，一个全新的工程师可以单凭这个计划以及那些规格建构出来，不需要任何口耳相传的知识。完成的意思是一百项里面有一百项打了勾，每一项旁边都有证据。

计划以一段排序总结作结，把每一个里程碑对应到它的验收关卡以及它的主要规格，并且点名那条由启动开始、经过编排、gateway、智能层、垂直切片、广度、meta agent、支持，再到加固的关键路径。界面工作可以在早段并行，而增强层就在接近尾声那里插入。杠杆最高的那个检查点是垂直切片关卡，因为它把架构风险转化成一个经过验证的基础。在这里全力以赴，意思是开头时深度先于广度，然后经 factory 做机械式的广度扩展，接着做递归式的质量，再做那张字面上的一百点扫荡。整个计划的设计，是要让一个缺陷在它最有可能被发现那一刻，刚好也是最便宜的去修。

还有一些附录，附有可以复制粘贴的起步材料、一个项目记忆档案的模板、子 agent 的定义、斜杠指令的定义、一个关于权限和 hook 的配置文件、一个要钉在每一个改动请求里面的完成定义，以及一份关键术语的词汇表。

而最后那句话，用一句就捕捉到整份信念。在铺开广度之前，先建立平台以及一片完美的切片，然后就让一个 factory 以及一个递归式质量网去做扩展。契约最先被冻结，这样那些 agent 就不可以分歧。垂直切片把系统最大的那个风险，也就是太迟才发现架构缺陷，转化成一个便宜、早，而且可以证明的检查点。质量是递归的，所以那个审判影片的系统，在自己身上也要通过同一套标准。而「想一百次」这个指令不是修辞。它就是那个字面上的一百点关卡，站在「在我机器上跑得起来」以及「生产环境」之间。给 coding agent 的指示很简单。从里程碑零开始，然后开工。
