Imagine you could hand a single document to a coding agent and say, "Rebuild this entire project from scratch, no questions asked." That's exactly what this specification is designed to do. It's a complete redevelopment handoff for a tool called the Grok Research Agent, and today we're going to walk through how it works, what it does, and the rules that govern building it. So let's dig in.

First, let's talk about what this document actually is. It's a contract. Its whole purpose is to let a coding agent rebuild the project without needing any extra clarification. It lays out the project context and the intended outcomes. It defines the required technical stack and the exact implementation parameters. It spells out the architecture and where each module begins and ends. It describes how every feature should behave and the acceptance criteria that prove it works. And it covers everything from user experience and accessibility to quality, testing, performance, and security. It even defines the milestones that have to be hit before the rebuild can be called finished. The key idea here is that wherever the original code was vague or unclear, this document steps in and removes the ambiguity. It's not a loose suggestion. It's the target state.

So what is the Grok Research Agent? At its heart, it's a local-first command-line tool written in Python. It automates an eight-phase research workflow, and it uses Grok, the AI model from xAI, through an OpenAI-compatible interface. You start with a topic and an optional area of focus. From there, the system refines the scope, analyzes sources, and pulls evidence-preserving notes from carefully chosen material. It builds what's called a master notebook, then synthesizes a detailed research report. And on top of that, it can optionally produce a structured knowledge base, a study drill pack, image prompts for generating visuals, a YouTube narration script, and full offline copies of the sources it used.

The whole thing is built around a few core principles that the rebuild absolutely must preserve. It's local-first, meaning all your session state and your artifacts live right there on your own machine. It's human-governed, so the workflow deliberately pauses at key checkpoints to keep the operator in control. It's artifact-transparent, which means every meaningful step writes out files you can actually open and inspect, not just a final result. It's resumable, so you can stop and pick a session back up across multiple runs without losing your place. And it's assisted by AI without being a black box. Every output from the model gets saved as a file you can review and reuse.

Who's this for? The primary user is the research operator, a technical person running the tool locally to produce deep research outputs. There are also secondary users, like reviewers who consume the reports, and study users who use the drill packs and the knowledge graphs to learn.

Now, the redevelopment has clear objectives. The coding agent needs to build a production-quality equivalent of the current system. That means preserving all the workflow commands and the output contracts, keeping the session-based local storage, maintaining the eight-phase lifecycle and the helper commands, supporting external document preprocessing, and keeping the knowledge base, drill pack, image prompt, and YouTube script generation intact. On top of all that, the rebuild needs to be hardened for maintainability, testing, and deterministic file outputs, with every edge case documented and validated.

Let's be clear about what's in scope and what's not. In scope: a Python command-line app, full session handling, the eight-phase workflow engine, an optional unattended automatic mode, local document preprocessing, web fetching and text extraction, prompt orchestration, knowledge compilation, drill pack generation, diagram export, image prompts, YouTube scripts, a full suite of automated tests, and packaging for local install. Out of scope: there's no browser app required, no user accounts or authentication, no team collaboration, no remote database, no background job queues, no cloud infrastructure, no vector databases, and no graphical dashboards unless someone explicitly approves them later.

On the delivery model, this is a local command-line application. It runs as a single process with synchronous orchestration and a bounded thread pool for concurrency. Persistence is filesystem only. It ships as an installable Python package with a command-line entry point. And the primary target for acceptance is Windows PowerShell first, though the code should stay portable to macOS and Linux wherever that's practical.

Now let's talk about the technical stack, because this spec is very precise about versions. The language is Python 3.11.9, with specific versions pinned for the build tools. For runtime libraries, it relies on the OpenAI client for talking to Grok, a library for loading environment files, a rich terminal rendering library for nice console output, a validation library for the session schema, a token counting tool reserved for future work, plus libraries for extracting text from PDFs and Word documents, for fetching web pages, for cleaning HTML, for pulling readable article content, and for handling character encodings. The point is that each of these has an exact required version, and they all need to be pinned down. For development and quality assurance, there's a testing framework, a coverage reporting tool, a linter for code hygiene, and a static type checker. When it comes to packaging, the package is named grok-research-agent, it exposes a single command-line script, and the prompt files it uses have to ship as packaged data inside the project.

The repository has a defined structure too. There's a main package folder containing the command-line interface, the Grok client, the session manager, the workflow phases, the external docs processor, and a folder of text prompt files. Alongside that there's a tests folder and the usual project metadata files. The coding agent is free to add more test files and even more modules if it improves how concerns are separated, as long as the public behavior stays consistent with the spec.

Let's look at the architecture. At the top, the research operator interacts with the command-line entry point. That entry point talks to two main pieces: the session manager and the workflow runner. The workflow runner is the busy one. It coordinates the Grok client, the external docs processor, and the web fetching logic. The session manager and the workflow both read and write to the local filesystem. The Grok client reaches out to the xAI Grok API, and the web fetching layer reaches out to remote source hosts.

Each core component has a clear job. The command-line layer parses arguments, validates required flags, sets up the session manager and workflow runner, and translates known errors into clean exit codes. The session manager creates unique session IDs, builds the directory structure, reads and writes the session file, creates unique run directories, and provides the canonical paths. The workflow runner orchestrates the state machine, builds the prompts, calls Grok, fetches and preprocesses content, writes out artifacts, and manages both interactive and automatic behavior. The Grok client loads the API configuration, configures the connection to use the xAI base address, runs the chat completions, classifies failures into meaningful errors, and supports tracing requests and responses. And the external docs processor recursively finds supported local documents, extracts their text, classifies them, generates summaries and constraints and relevance signals, and writes out the aggregated external context.

The data flow follows a clear sequence. The user issues a command. The command-line layer loads or creates a session, then calls the workflow runner. The workflow creates a run directory and starts reading and writing artifacts. During the research phases, it asks Grok to generate scopes, discovery, extractions, drafts, and final outputs, and it fetches the chosen sources from the web. During the knowledge commands, it asks Grok to compile and update knowledge outputs and writes those to disk. Finally, status flows back to the user along with console output and an exit code. For integrations, only three things matter: the xAI Grok API over secure connections for all the AI work, arbitrary source hosts over standard web protocols for fetching sources, and the local filesystem for everything that gets saved. Nothing else is required.

Now let's get into how sessions and persistence work, because this is where the local-first philosophy really shows. Each session gets its own directory, and inside it you'll find a session file plus a whole series of numbered artifacts. There's a confirmed scope file, a discovery table, curated sources, folders for extracted content and source snapshots and chunks, a master notebook, section evidence and drafts, full source copies, the final report, an image generation file, a YouTube script, a folder for external documents, a knowledge base folder, and a runs folder that holds run-scoped copies of everything.

The session state itself tracks a long list of fields: the session ID, the topic, the optional focus, the mode, details about external documents including their status and summary and file counts and completion rate and relevance score and any last error, plus creation and update timestamps, the Grok model, the current phase, and the history of runs.

Session IDs follow specific rules. The ID is built from a simplified, slugified version of the topic combined with the current date. The slug is lowercase, with non-alphanumeric runs replaced by hyphens, repeated hyphens collapsed, and leading and trailing hyphens trimmed. If the topic prefix gets too long, the system appends a short eight-character hash. And if a directory with that name already exists, it adds a numeric suffix, counting up until the name is unique. Run directories follow their own pattern too. Every command that invokes the workflow runner creates a new run directory named with a date, time, and microseconds, and if there's a collision it retries until it finds a unique name or gives up after a thousand attempts.

Configuration comes down to environment variables. There's one mandatory variable: the Grok API key, which has to be present and non-empty. Then there are several optional ones: the model name, which defaults to Grok 3; a maximum output token setting that defaults to fifty thousand and falls back to that default if you give it something invalid, with a minimum of one; a request timeout that defaults to three hundred seconds with the same fallback and minimum behavior; and an editor variable used only during the manual editing step in the first phase. The system tries to load an environment file from the project root, and if there isn't one, it relies on the actual environment variables. If the API key is missing, it has to give a clear, actionable error.

Let's walk through the command set. There are twelve commands in total: start, resume, list sessions, list types, update, synthesize, compile, drill, feed, show, generate images, and youtube script. There are also shared flags available across commands, things like specifying the sessions directory, enabling automatic mode, enabling full automatic source collection, and turning on tracing of the AI calls.

Each command has a defined contract. Start takes a required topic, an optional focus, an optional external docs directory, and an optional mode that can be report, compiler, or drill. It creates the session, prints the session ID, and drops you into the first phase. Resume takes a session ID and picks up from the saved phase. List sessions prints all the valid session IDs it finds, or tells you none were found. List types simply prints the one supported type, which is auto-hypergraph. Update reruns discovery, flagging it as a follow-up since the last run, and resets to the curation phase. Synthesize forces the synthesis phase using the current notebook. Compile takes a session ID and a type, and produces hypergraph and core concept files in the knowledge base. Drill takes a session ID and a backward mode and produces a drill pack and a questions file. Feed takes a session ID and a new document, saving a timestamped copy and updating the hypergraph. Show produces the diagram file. Generate images produces the image file. And youtube script produces the YouTube script file.

Now, the heart of the system is the workflow state machine, and it runs through nine numbered phases, from zero to eight. Phase zero is scope confirmation, and it involves the human. Phase one is discovery, which runs on its own. Phase two is curation and gap analysis, with the human back in the loop. Phase three is extraction, automatic. Phase four is notebook assembly, automatic. Phase five is synthesis and review, with human involvement. Phase six is full offline collection, with the human deciding. Phase seven is final polish, automatic. And phase eight simply means complete.

There are several runtime constants that need to stay configurable but default to specific values. Source text gets split into chunks of forty-five thousand characters with a five thousand character overlap. Notebook chunks are seventy thousand characters. Web fetching uses up to four parallel workers, while extraction and evidence generation each use two. And the final report targets between nine thousand and ten thousand words, aiming for ninety-five hundred. The standard report also has six fixed section names that must be preserved exactly: core definitions and scope; architecture and technical mechanisms; workflows, processes, and operational patterns; evidence, examples, and case studies; limitations, trade-offs, and failure modes; and open questions and future directions.

Let's go feature by feature through what each phase actually does. In phase zero, scope confirmation, the system generates a concise scope summary from your topic and focus, saves it locally, and prints it to the console. You can respond with yes, edit, or cancel. Saying yes confirms the scope and moves to discovery. Saying edit creates an editable file, optionally opens your editor, reloads what you changed, and keeps prompting. Saying cancel leaves the phase as it was. In automatic mode, it just confirms and moves on. Importantly, a blank or malformed focus must not crash anything, and a missing editor must not block the edit flow.

Phase one, discovery, renders a discovery prompt with your topic, focus, and whether this is a follow-up. It saves the discovery output and moves to curation. It doesn't block on format validation before saving, so even imperfect output gets stored. The discovery table file should exist afterward, and the update command can rerun this as a follow-up.

Phase two is curation and gap analysis. It requires the discovery table, shows you a preview of discovered sources, and accepts your selection as free-form instruction. It tries up to three times to get valid curated source data, then normalizes each source into a consistent shape with a title, URL, type, reason it's relevant, credibility, and priority. If it can't parse structured data, it recovers URLs heuristically straight from the discovery text. It saves the curated sources, generates a gap report, and requires you to type the exact word "approve" to advance. The system has to handle fenced data blocks, URLs wrapped in punctuation or backticks, and even empty discovery files without crashing.

Phase three is extraction. It needs valid curated sources, creates the extraction directories, and requests an extraction plan. It fetches the source bundles concurrently, preserves both raw snapshots and normalized text snapshots, splits text into overlapping chunks, and extracts evidence-preserving notes from each chunk in parallel. It saves the individual chunk outputs, assembles them into source dossiers, and crucially, skips any failed source without aborting the whole phase. A timeout on one chunk or one source only skips that piece. Invalid URLs are skipped safely, and PDFs are extracted and stored just like everything else.

Phase four is notebook assembly. It takes the extracted dossiers and builds a master notebook containing the title, topic, the notebook's purpose, optional external document context, a catalog of sources, an optional knowledge base outline, and the dossiers themselves. The notebook file gets written, and the session moves to synthesis.

Phase five is synthesis and review. It splits the notebook into chunks, generates section-specific evidence packets in parallel, and then generates section drafts from those packets and the source catalog. It assembles a full draft, versioned by number, and accepts your review feedback. Typing the exact word "approve" moves you to the next phase. Anything else is treated as revision input, and the revised draft is saved as a new version. The drafted sections use those exact canonical section names, and earlier drafts are always preserved. Sections with no evidence are skipped with a warning rather than failing.

Phase six is full offline collection. It shows you the available sources and lets you choose all, none, or a comma-separated list of numbers. It prefetches the selected sources and saves full offline copies as Markdown. If there are no curated sources, it tries to recover them from discovery, and if that fails, it skips ahead to final polish. Invalid or out-of-range numbers are simply ignored, and a failed fetch only skips that one source.

Phase seven is final polish. It requires the notebook and at least one draft, and uses the latest draft version. It generates an executive summary and a glossary, builds a table of contents from the section headings, and enforces the target word count using retargeting logic. Then it saves the final report, generates image prompts, and generates the YouTube script. The final report contains the title, table of contents, executive summary, the report body, the source catalog, an optional knowledge base overview, and the glossary. If the word count retargeting times out, it still saves the best available version. If the executive summary or glossary times out, it inserts fallback text. And if the body starts with a duplicate title, that duplicate gets removed before assembly.

Beyond the main workflow, there are the knowledge commands. Compile builds from the notebook if it exists, otherwise from the extracted dossiers. It generates a hypergraph in a structured format and saves it to both canonical locations, then generates and saves the core concepts. If the model returns invalid data, it still saves it in a safe wrapped form rather than failing. Drill supports only the backward mode and automatically runs compile first if the core concepts don't exist yet, then generates the drill pack and the structured questions. If core concepts still aren't there after compiling, it exits with a clear message. Feed validates that the supplied file exists and is a regular file, copies it into a timestamped feed folder, and either initializes the hypergraph through compile if it's missing or updates the existing one. Unreadable content is loaded with replacement characters rather than crashing. And show requires the hypergraph and renders a diagram file that starts with a standard graph declaration, drawing nodes and edges from the available data.

External documentation preprocessing deserves special mention. It discovers supported files recursively, processes PDFs, Word documents, plain text, and Markdown, and classifies each document as a guideline, background, steering, or general material. It extracts key concepts, constraints, requirements, and algorithm insights, then aggregates everything into a manifest, an extracted summary, and a context file. It stores processing metrics in the session state and injects that external context into the relevant prompts. Partial file failures must never abort the whole workflow.

Then there's image prompt generation, which requires the final report and produces between five and ten prompts emphasizing concrete mechanisms, workflows, architectures, and evidence. And YouTube script generation, which also requires the final report but prefers the section drafts as source material when available. It deliberately excludes the table of contents, source catalog, glossary, references, knowledge base overview, and executive summary from the narration. It generates an introduction, section headings, and a conclusion, and expands short outputs to meet minimum length thresholds.

The spec also defines exact output contracts. Curated sources are stored as a list of structured records, each with a title, URL, type, reason for relevance, a numeric credibility rating, and a priority. The hypergraph is stored with nodes and hyperedges, where each edge connects multiple nodes with a named relationship and supporting evidence. And if valid structured output can't be produced, the system saves a wrapped fallback that simply holds the original raw output. Core concepts each carry a name, a definition, and an explanation of why they're foundational. And drill packs combine the Markdown content with structured questions, each tied to a concept and including a question, an answer, and common pitfalls.

Error handling is carefully specified. The Grok layer exposes three error types: a general error, a timeout error, and a quota error. Quota-like failures map to the quota error, timeout-like failures to the timeout error, and everything else to the general error. For retries, ordinary failures that aren't timeouts or quota issues retry up to five times using exponential backoff capped at thirty seconds. The system is also designed to degrade gracefully rather than crash. When things like gap analysis, extraction plans, evidence packets, drafts, revisions, summaries, glossaries, image prompts, or word-count retargeting fail, the system continues wherever it can. But it stops cleanly and clearly in genuinely fatal situations: when the API key is missing for a required path, when prerequisite files are absent, when command arguments are invalid, or when a session can't be loaded.

On performance, the spec sets concrete benchmarks for a standard developer workstation with broadband. Creating a session and writing its state should take under a second. Listing sessions should take under a second even for up to five hundred sessions. Creating a run directory should take under a quarter second. And metadata writes for the knowledge commands should take under a second, not counting AI or network time. For concurrency, web fetching supports up to four parallel requests, extraction supports two parallel AI calls, and evidence generation supports two parallel calls per section. For scale, it should handle at least twenty curated sources, at least twenty dossiers in the notebook, at least forty external documents, and diagrams with at least two hundred nodes and four hundred edges. The guiding philosophy is that correctness and resilience matter more than aggressive parallelism, and timeouts should always preserve the progress already saved.

Security is taken seriously. API keys must never be written into tracked files or session artifacts. Secrets load only from the environment. Fetched pages and processed documents are treated as potentially sensitive local data. The xAI API is accessed only over secure connections, and any non-secure URLs are rejected during fetching. Traced AI output has its control characters stripped before being shown. The system never executes remote content, never shells out except for the optional editor launch, and makes sure failure paths never leak secrets into logs or the console.

For user experience and accessibility, the deliverable is command-line first, and a browser interface must never replace that workflow. The focus is on a clean terminal experience, readable Markdown artifacts, and usable text outputs. Prompts should be short, explicit, and action-oriented, always telling the user what to do next. The accessibility target adapts web accessibility principles to the command line: color must never be the only way meaning is conveyed, prompts and errors must make sense in plain text, generated headings must be properly ordered, tables must have clear headers, and errors must be descriptive and point toward a fix. Since the product is command-line only, responsive browser layout isn't required, but generated reports must still read well in narrow panes, and any optional preview interface someone adds must be mobile responsive and meet the same accessibility standards without replacing the command line.

Testing requirements are extensive. The agent must implement unit tests, integration tests, and end-to-end workflow tests. Unit tests need to cover session ID generation, state round-trips, configuration parsing, error mapping, URL normalization, relaxed data parsing, source canonicalization, discovery recovery, text chunking, HTML cleaning, diagram generation, and external document categorization. Integration tests cover command dispatch, the start and resume flow using a fake AI client, the compile and drill workflows, feed and show, external doc preprocessing, and timeout degradation. There must be at least one full happy-path end-to-end test that walks through session creation, every phase from zero to eight, and the generation of all the final artifacts. And there must be an automatic-mode test proving that no interactive input ever happens, that the workflow reaches completion, and that offline copies are saved when full collection is enabled. The spec lists required edge-case tests too, like a missing API key, missing files, fenced data input, various timeouts, and missing prerequisites. The coverage targets are at least eighty-five percent line coverage and at least seventy-five percent branch coverage in the core orchestration modules, and the suite must pass on Windows PowerShell.

Quality standards round things out. Public functions in core modules should be typed, orchestration logic shouldn't be duplicated, and HTTP and AI clients should be injected for testability. Source text should stay ASCII where possible, and comments should appear only where behavior is genuinely non-obvious. The agent must run and pass the linter, the type checker, and the test suite, and modified files must be free of editor diagnostics before handoff.

The spec then defines acceptance criteria for each deliverable. The core command-line tool is accepted when all commands exist, the help text is accurate, exit codes follow the spec, and local installation works. The workflow is accepted when the interactive path reaches completion through repeated resumes, automatic mode finishes without input, and all outputs land in the right directories. The knowledge features are accepted when compile, drill, feed, and show all produce their expected artifacts. External docs are accepted when files are discovered recursively, the supported formats are processed, the aggregated files are produced, and partial failures don't stop the workflow. And the final content is accepted when the report has its required sections, image prompts and the YouTube script exist when generation succeeds, and file naming is deterministic.

The work is organized into milestones, in order. First, the project skeleton and packaging, ending with an editable install and passing basic tests. Second, the session and state foundation, ending with passing session tests and a verified state round-trip. Third, the core workflow phases zero through five, ending with happy-path tests passing through draft generation. Fourth, finalization and output expansion, covering phases six and seven plus the final report, images, and script, ending when the full workflow reaches completion. Fifth, the knowledge features, ending when those artifact tests pass. And sixth, external docs and hardening, covering preprocessing, error recovery, timeout handling, and cleanup of coverage, linting, types, and diagnostics, ending when everything passes and the quality checklist is complete.

Before handing the work back, the coding agent has to provide updated source code and tests, an updated readme if behavior changed, updated installation instructions if dependencies changed, the exact list of validation commands run, a short change summary, and an explicit statement that all acceptance criteria were checked. If any criterion isn't met, the agent has to identify it, explain the blocker, and lay out the exact remaining work. The required validation commands include installing the package in editable mode with development dependencies, running the linter, running the type checker, and running the tests, plus a coverage run if that tooling is included.

There's also a detailed quality assurance checklist covering functional items, reliability items, code quality items, and documentation items, all of which must be verified before finalization. And the definition of done is strict: the project installs and runs locally, the full feature set is implemented, all artifacts have the correct names and structures, all acceptance criteria are satisfied, all the test, lint, and type gates pass, no unresolved diagnostics remain, and the agent can hand off the work without needing any follow-up clarification.

Finally, the spec lays down non-negotiable constraints. Don't replace the command-line model with a graphical-only approach. Don't move persistence to a database. Don't remove the human checkpoints. Don't remove the local run-scoped artifacts. Don't require third-party services beyond xAI Grok and normal source hosting. Don't silently swallow fatal errors. And don't break the output file names or the core data contracts. It recommends a specific implementation sequence, starting with the package and parser, then the session manager, the Grok client, the workflow engine, the early phases, fetching and extraction, the notebook and synthesis, the final outputs, the knowledge commands, the external docs processor, and finally the hardening and full test pass.

The closing handoff note makes the philosophy crystal clear. If the existing code ever conflicts with this document, the document wins, unless the owner explicitly overrides it. The goal isn't a loose approximation of the current project. The goal is a faithful, testable rebuild with stronger engineering discipline and zero ambiguity about what the outcome should be. And that's the whole picture. A local-first research agent, fully specified, ready to be rebuilt with confidence. Thanks for watching.
