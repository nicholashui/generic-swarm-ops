我们今天来聊一聊，怎样建立一个可靠的、达到生产级别的 agent loop。这是经过优化的第三版设计，它在之前所有版本的基础上，加入了一样新东西：人类真正的思考方式。它的核心理念说起来很简单，但实际用起来非常强大。与其让语言模型在一个失控的连锁反应里乱跑，不如给它一个受控的循环。也就是要有明确的状态、结构化的输出、在每个关卡都有质量检查，还有一个清晰的层级结构，由规划者把工作分派给专家。这是有管理的协调，而不是失控的反应。结果会逐层向上汇集，经过整合，再变成一个深思熟虑的综合成果。

那么这些理念是从哪里来的呢？它们建立在扎实的研究之上。有二零二三年一篇著名论文提出的 ReAct 范式，有研究多智能体系统为什么会失败的分析，有关于反思和自我改进的著作，有记忆方面的研究，还有开发者实际遇到无限循环和上下文膨胀的报告。在这一切之上，第三版还加入了对四十多种人类认知框架的系统性回顾，并且按照它们对 agent loop 的实用程度做了排序。

我们从基础讲起，就是 ReAct 范式。核心理念是把两样东西交织在一起。第一样是用文字表达的推理，我们叫它 thought，也就是思考。第二样是行动，比如调用工具、与环境交互，或者把工作交给另一个 agent。每次行动之后，你都会得到一个观察结果，而这个观察会把推理锚定下来并加以更新。为什么这个组合这么有效呢？如果你只推理，没有外部的依据，模型就会倾向于产生幻觉，错误也会像滚雪球一样越滚越大。如果你只行动，没有高层次的规划，你就会得到笨拙、低效的行为，也处理不好突发情况。但把它们结合起来，神奇的效果就出现了。思考会分解目标、追踪进度、处理异常情况。行动则带回真实的观察结果，去修正推理。研究显示，在交互式任务上这种做法带来了有意义的两位数提升，同时减少了幻觉。

一个完整的循环是这样的。模型先有一个思考，琢磨一下当前的状态和下一步。然后它采取行动，输出一个可执行的步骤，无论是调用工具、分派工作，还是发出已完成的信号。接着就是观察，一个结构化的结果，附带有用的元数据，比如状态、置信度、摘要和任何问题。这个结果会加入到正在进行的历史记录里，然后循环再次重复。当有东西失败了，思考那一步就会察觉到，比如说"没有任何有用的东西返回"，然后在下一次循环就会重新规划。

这不只是理论。Grok 系列的模型在服务器端就运行着这个循环的生产级版本。模型自己决定用哪些工具，在后台执行，比如网页搜索和代码执行，然后不断迭代，直到它能够产生最终答案。客户端只看到完成的输出，加上可选的推理标记。

现在我们扩展到生产级的多智能体协调。想象一个系统可以启动可配置的 agent 团队，可能用四个去做快速集中的工作，用十六个去做深入全面的研究。这些专门化的 agent 并行运作，每个都实时贡献推理、工具调用和发现。然后有一个领导 agent 综合整个讨论，相互比对所有资料，再交出最终的结构化答案。各个子智能体的内部状态默认是隐藏的，目的是为了控制和安全，所以只有领导的输出会展示出来。这里的强项就是深入的多步骤研究、像表格和比较这样的结构化输出，还有自动使用工具，而客户端完全不需要介入循环。相关的工具还采取规划优先的做法，先生成一个明确的计划，然后才分散给在隔离环境里工作的并行子智能体。

下一层就是层级结构和自我演化。最顶层坐着一个中央规划者，或者叫协调者、监督者。它把目标拆解成子任务，再分派给专门化的子智能体：一个深度研究员、一个分析员、浏览器和工具 agent、一个报告员等等。每个子智能体都运行自己的循环。路由遵循一个树状结构，结果就逐层向上汇集。这里有个启发，就是把工具、环境和 agent 都当成一等的、有版本管理、有生命周期管理的实体，全部都遵循标准化的协议。还有一个用来自我演化的闭环反馈：系统会反思自己的执行轨迹，归因错误，提出修改，在留出的样本上验证，再注册改进后的版本，而且可以回退。一个追踪器会记录完整的执行轨迹，这对审计和优化都有用。最后，规划者会整合这些子结果，协调证据，解决冲突，再更新全局计划。用这种方式建立的系统，在艰难的基准测试上已经达到大约九成的成绩，而自我演化还带来了更多提升。

所以整个优化后的模型是这样的：从 ReAct 核心循环开始，在上面为复杂的工作加一层层级分派，然后再加入一个明确的规划阶段、反思和批判的关卡，以及带版本管理的结构化状态，去达到生产级的可靠性。

现在讲一下第三版真正全新的东西：来自那些排了序的人类思考模型的认知架构强化。这些不是事后补上去的附加品，而是一等的机制，它们带来四样东西：通过情境感知路由带来的适应性智能、通过在行动前思考风险带来的前瞻性稳健、通过快慢思考路径带来的高效认知，以及通过结构化反思带来的更深层学习。我来给你逐个讲一下最重要的那几个。

第一个是 Cynefin 框架。这个排在第一位。在定好规格之后，以及每次决定要不要重新规划的时候，系统都会把任务的情境分类。是不是简单，因果关系清晰？是不是繁杂，需要专家分析？是不是复杂，事情会无法预料地涌现出来？还是混乱，是一个真正的危机？根据这个分类，它就会调校这个循环。简单和繁杂的任务走快速路径，关卡较轻。复杂和混乱的任务就会得到全面的深思熟虑处理，重度反思、多个批判者，以及更深入的诊断。这个单一理念，在平衡效率和可靠性方面，是最高杠杆的新增功能。

第二个是事前验尸分析。在计划生成之后但还没有确定之前，系统会做一个强制性的练习：想象这个计划在六到十二个月之后彻底失败了，然后逆向推演找出最主要的成因，再在规格、成功标准或者 agent 角色里明确地去化解它们。这个可以由协调者执行，或者由一个专门的红队批判者执行。它用接近零的成本，通过前瞻性的风险模拟去强化规划。

第三个是事后复盘，在每个里程碑和终止的时候使用。它遵循一个简单的四个问题的模板。原本应该发生什么？实际上发生了什么？为什么？以及下一步怎么做？最后那一步把教训变成具体的演化行动，直接反馈给自我改进。

第四个是双环学习。在事后复盘提出了战术性的修正之后，系统会问一个更深层的问题：是哪些主导的假设把我们带到这一步？那些提示模板、成功的定义、agent 角色的边界、验证的阈值，甚至是拆解任务的方式。这些在更高的层次上应不应该改变呢？要到这个时候，它才确认带版本的更新。这样才能让自我演化真正强大，而不只是修补表面症状。

第五个是识别启动的决策，再搭配双重历程理论，也就是快思和慢想的概念。这引入了一条快速识别路径。在投入冗长的推理之前，系统会查询一个模式库，也就是一个存储了成功轨迹的长期记忆。如果找到一个强有力的匹配，而情境又允许，它就会做一个轻量级的心理模拟，类似"这个跟过去一个成功的轨迹很相似，那就采取那个行动"，然后用极少的标记继续下去。这就是系统一，快速而直觉，用来做例行的专家工作。系统二就是缓慢而审慎，也就是完整的 ReAct 循环，用来应对新奇或者高风险的情况。元认知会决定用哪一个，并且在信心不足的时候回退到完整循环。

说到这里，第六个就是元认知循环，一个在每次迭代旁边运行的轻量级并行进程。它会规划、监测偏差和情境是否吻合、评估严谨程度，再实时做出调整，可以切换模式、提前重新规划，或者升级某个关卡。

接着是根本原因分析：五个为什么、石川鱼骨图，以及故障树。在持续失败的时候，系统会用反复的为什么去深入挖掘，再在多个维度把根本原因分类，比如人和提示、流程和方法、模型和工具、数据、环境和指标。还有红队思维，一个专门的批判者去对抗性地攻击一个计划或者输出，从而揭示隐藏的弱点。还有 Paul-Elder 批判性思维框架，它用思考元素和智识标准，比如清晰、准确、精密、相关、深度和公平，去丰富验证器和思考那一步。还有约束理论，搭配 TRIZ 的创新解决问题原则，在整合阶段用来解决相互冲突的目标。最后，像六顶思考帽和 SCAMPER 检查清单这样的可选多角度审视，可以帮助创意型的子智能体减少盲点。

为了支持这一切，记忆架构会升级，加入一个模式库，存住按情境类型打标签的过往轨迹的向量和元数据。验证器现在可以在不同的批判模式下运行，标准、红队、Paul-Elder 或者六顶帽，甚至可以做集成。任务规格也多了一个认知档案，带有启用快速路径、选择反思风格和批判模式的标志。最关键的是，所有这些新步骤都是有边界、有版本、有记录的，并且可以按任务开关或者限制深度。它们把 agent 从一个能干的引擎，转化成一个会思考自己思考方式、会预料失败、会按情况调整自己审慎程度的系统。

那么，这一切为什么这么重要呢？因为研究很清楚地指出了 agent 系统为什么会失败。分析人员研究了超过一百五十条跨越热门框架的轨迹，发现大部分失败都源于设计和规格的问题，其中超过四成是这样，再加上协调崩溃和薄弱的验证。通常都不是模型本身智能的问题。我们来逐个讲一下主要的问题类别和它们的解决方法。

最大的类别是规格和设计上的含糊。Agent 会违反或者误解含糊的规格，角色不清晰，也没有成功标准，于是很早就走错路，错误就不断累积。解决方法就是在一开始有一份强制性的结构化任务规格，列明成功标准、约束、输出格式和质量阈值，并且把它当成一份可以更新的活文档。在循环开始之前加入自动的规格验证，并且在协调者和子智能体之间定下清晰的角色和信息契约。

接着是无限循环和空转，也就是 agent 不停重复同样的行动但毫无进展。这会浪费标记、导致超时，也让人人都很沮丧。解决方法就是循环检测，把最近的行动和观察做哈希，如果相似度超过某个阈值就强制重新规划。设定明确的步骤和反思轮次上限，用基于进度的提前退出，再要求一个必须通过验证才会被接受的完成行动。在层级结构里，协调者会监测子智能体，可以终止或者重新分派卡住的分支。

第三是上下文窗口爆炸，也就是过长的轨迹让早期的信息被丢失，导致前后不一和目标漂移。解决方法就是积极的层级记忆，短期工作记忆搭配一个有向量搜索的长期持久存储。在里程碑做摘要，保持结构化的状态，只留下关键事实，而不是每一轮都把整份历史倒出来，再只给子智能体相关的上下文片段。

第四是幻觉和错误累积，也就是捏造的事实不断传播，这在多智能体的设置里更加严重。解决方法就是把验证器和批判者 agent 作为强制的质量关卡，一个带有状态、置信度和问题清单的结构化观察架构，跨来源的交叉验证，以及轨迹排序去在多次尝试里挑最好的那个。而在自我演化方面，只确认那些在留出轨迹上验证过的改动。

第五是智能体之间的错位，比如越权扮演角色、目标冲突，以及过时的共享状态。解决方法就是一个有明确拆解和路由的强力中央协调者、清晰的信息契约、带版本的共享状态，以及熔断器去检测不一致并暂停，从而调和或者升级。

第六是终止和目标漂移，也就是太早停止，或者认不出工作已经完成。解决方法就是明确的成功标准、一个必须通过验证器的专门完成行动，以及定期对照原本目标做对齐检查。

还有其他问题：状态过时，用混合记忆去处理；像提示注入这样的安全威胁，用沙箱、输入清理和最小权限工具访问去处理；成本和可扩展性，也就是只在好处大过协调成本的时候才用多智能体；以及可调试性，在这方面，一个完整的追踪器和结构化日志是绝对不可缺少的。这一切研究的关键启示就是：修正规格质量、验证层和明确的终止控制，带来最大的可靠性提升。只是多加一些 agent 或者更多原始算力而没有这些，往往得到的是递减甚至负面的回报。

现在我们逐个阶段走一遍整个 agent loop。

第零阶段是初始化，也就是规格驱动的设置。目标是在任何迭代开始之前，先确立一个清晰的契约。你把人类的指令解析成一份结构化的任务规格，包含目标、成功标准、约束、输出格式、预算和质量阈值。你建立初始状态：一个有当前计划和待办清单的结构化便签、一个有短期和长期部分的记忆层、一个追踪器，以及一个版本登记册。对于复杂的任务，你可以选择生成一个高层次的计划，再对照规格验证。接着就是第三版的认知强化：用来为任务做路由的 Cynefin 分类，以及事前验尸分析，这个现在对除了最简单以外的所有任务都是一个建议的强制关卡。最后，你决定架构，是扁平、层级还是混合，再设定初始的认知档案标志。

第一阶段是核心迭代循环。只要任务没终止，你就会一直走下去。在每次循环开始的时候，系统会挑一个模式。如果情境简单、有强有力的模式匹配，而信心又高，它就会走快速识别路径，做一个用极少标记的轻量级思考。如果不是，它就会进入全面的深思熟虑模式，做详细推理和全套关卡，同时元认知会运行一个并行的监测器，如果不确定性突然飙升就可以强制切换模式。在每次迭代里，你观察当前状态，如果上下文太长就积极做摘要。你推理，同时用元认知作为覆盖层去检查偏差和进度。你决定下一个行动，是调用工具、分派给子智能体、在内部更新状态，还是完成。你执行并收集一个结构化的观察，带有状态、数据、摘要、置信度、问题和建议。每隔几步你就做一次轻量级反思，快速检查一下轨迹是不是还对齐。

这里有个很实用的生产模式，就是熔断器。它在一个工具或者子智能体不断失败的时候，防止连锁故障。它有三个状态。闭合就是正常运作，请求可以通过。断开就是已经失败了太多次，所以它立刻快速失败去保护系统。半开就是在超时之后，它允许几个测试请求去看看是不是已经恢复。如果测试成功，它就重新闭合；如果失败，它就重新断开。你每个工具类型或者每个子智能体角色用一个熔断器，再搭配用指数退避的重试包装。这个实现也包含把最近历史做哈希的循环检测，以及一个支持那些不同批判模式的验证器。终止条件包括达到成功标准、用尽预算、一个明确而经过验证的完成行动、一个无法挽救的失败，或者在结果已经满足目标的时候提前退出。

第二阶段是层级分派和子循环。当协调者决定要分派的时候，它首先做拆解和路由，挑选合适的专家子智能体，再建立一份只带相关上下文片段的狭窄子任务规格。子智能体接着运行自己的独立循环，维持自己的本地状态，也可以进一步分派或者调用工具。它返回一个结构化的结果，逐层向上汇集，其中包含来源，也就是哪个 agent 做的和一个轨迹摘要。父层把它记录下来，验证并整合，合并分支和解决冲突，更新全局计划，再决定下一步。而在依赖关系允许的地方，独立的分支可以并行运行。

第三阶段是整合和综合。在这些子结果返回之后或者在重大里程碑的时候，你汇集所有相关的观察。然后你做协调，由一个语言模型或者一个专门的报告员 agent 去合并、去重、相互比对和解决矛盾，整合成一个统一的视图。你把它重组成目标的输出形态，并执行规格里的格式。接着就是质量关卡：一个批判者对照成功标准为成果评分，并检查有没有幻觉和缺口。如果不合格，你就触发一个优化循环。如果合格，你就继续下去。然后你把整合后的知识持久化到长期记忆。

第四阶段是反思、批判和自我演化，而这里就是第三版真正出色的地方。现在每次反思都遵循一个明确的多层协议。它从强制的事后复盘模板开始，也就是那四个问题：原本应该发生什么、实际发生了什么、为什么，以及下一步。对于"为什么"，会有第一轮的标准归因，而对于更难的问题，就会有深入的一轮，运用五个为什么，再用鱼骨图把成因分类，接着用 Paul-Elder 的视角去交叉检查。之后就是双环学习那一层，问一下是哪些主导假设造成了这个结果的条件，只有经得起那个元问题的改动才会被提出。然后自我演化循环就启动：收集轨迹、在那个框架里归因错误、提出针对性的改动、在留出的任务上重新执行去验证，如果有改善而又通过双环批准，就注册新版本，连同它的谱系和理据，并更新模式库供未来的快速匹配用。批判者 agent 支持所有那些模式，红队模式在高风险工作上特别受推荐。结果就是第四阶段变成一个真正让智能复利增长的引擎，而不只是渐进式的修补。一个好建议：把事后复盘的输出存储成连接着版本的结构化产物，建立一个可审计的学习历史，供未来的 agent 检索。

第五阶段是终止和输出。当这些关卡通过，你就做最后一次综合，产生符合规格的结构化最终输出，可以选择加一个反思摘要，再把完整的轨迹和版本持久化，以供审计和回放。而在整个过程里，在质量关卡失败、高风险行动或者预算耗尽的时候，都有人工介入的钩子。

讲几句关于基础设施的东西。你的状态架构结合了任务规格、计划和待办、思考-行动-观察三元组的历史、记忆、版本和追踪器。记忆应该是层级式和会话隔离的，在上下文压力下做摘要。追踪捕捉完整的执行图，用来调试和优化。版本管理涵盖提示、工具、agent 角色和产物，全部都有谱系和回退。而生产级的加固，意思就是严格的输出架构、超时和重试、成本监测、可观测性，以及沙箱化的执行。

那么什么时候该用哪一样呢？对于简单的事实查询，一个扁平的单一 ReAct 循环就够了。对于多步骤研究，就用多智能体或者层级结构，有并行的 agent 和一个负责综合的领导。对于编程或者长期项目，就用规划优先，搭配在隔离工作树里的层级子智能体和一个待办文件。对于开放式的创意工作，就把 ReAct 和反思和自我演化结合在一起。而对于高风险、需要可靠的自动化，就用以上全部，再加强力的质量关卡。

总结一下常见的陷阱和它们的解决方法：用积极的摘要和层级状态去处理上下文爆炸；用硬性的迭代上限和进度追踪去制止无限循环；用结构化的子结果和一个专门报告员去修正糟糕的整合；把每一个重大论断都落实有依据，去避免计划里的幻觉；用明确的子任务规格去防止脆弱的分派；以及确保有完整的追踪去提供可见性。

那么实际上怎样开始呢？从你选的框架里一个最简的 ReAct 框架开始。加入一个结构化的观察架构和待办和状态管理。再加一层层级分派和整合。装置追踪和质量关卡。在重复的任务类型上试验反思和自我演化。再为研究密集的子任务整合一个多智能体模式。整个设计都是要做可执行的指引。复制这些模式、调整这些伪代码，再不断迭代。最大的领悟就是：可靠性不是单靠原始的模型算力得来的。它来自受控的循环、清晰的规格、强力的验证、深思熟虑的终止，以及一个学会去思考自己思考方式的系统。
