Imagine you've signed up for a whole pile of different AI services. You've got an account with x.ai for the Grok models, another with Poe, one with MiniMax, one with Kimi from Moonshot AI, one with OpenRouter, and probably a handful of others on top of that. Every one of them charges you separately, tracks your spending differently, and shows your remaining balance in its own little corner of its own website. Keeping tabs on all of it quickly turns into a headache. So the idea here is to build one web application that pulls everything together into a single central view, a place where you can track your usage, your costs, your balances, your spending, and how many tokens you're burning through across every single one of those API accounts at once.

The way it works is simple. You add your API keys one time, and from then on the app shows you everything aggregated into one beautiful dashboard. Your total spend for the month, the credits you have left, a breakdown for each provider, charts, trends, alerts, the whole picture in one place.

For inspiration, this project takes cues from a popular desktop tool called cc-switch, which people use to manage their LLM providers for things like Claude Code, Codex, and the Gemini command line. But this web app has a narrower, sharper focus. It's purely about usage and cost analytics across your own personal API keys. Think of it as a usage-focused companion to that tool, except it lives in your browser.

Let's talk about the core goals. First, one unified place to monitor all of your spending and usage. Second, secure storage of your API keys that stays entirely on your own machine and never gets sent to any server. Third, the ability to fetch usage and billing data either automatically or whenever you ask for it. Fourth, historical tracking with proper visualizations so you can see how things change over time. Fifth, the system should be extremely extensible, meaning it's easy to add new providers as they come along. And finally, the interface should be beautiful and modern, in the same spirit as cc-switch.

Now for the features that absolutely have to be there. Start with provider management. You should be able to add, edit, and remove accounts, and for each one you record a name, the type of provider chosen from a preset, the API key, a base URL for custom endpoints, and any notes you want. The app ships with pre-built presets for as many providers as possible, and it supports having multiple accounts under the same provider. There's a one-click refresh-all button, plus individual refresh buttons for each account.

Next comes the actual fetching of usage and balance data. Wherever a provider offers an official API, the app prefers to use it, things like a usage endpoint, a billing endpoint, a balance endpoint, or a remaining token plan endpoint. When no official option exists, there are fallbacks. You can manually enter your current usage or balance, and as a last resort the app can scrape the provider's web dashboard using a browser automation tool. The data refreshes automatically in the background on an interval you can configure, and you can always trigger a manual refresh. Every snapshot of history gets stored in a local database.

Then there's the dashboard itself. The overview page shows your total estimated spend in US dollars for today, for this month, and for all time. It shows your total remaining credits or balance, normalized across providers wherever that's possible. And it shows how many providers are active along with a quick status check. Below that you get provider cards, laid out as a grid or a list, each showing the name and logo if one's available, the current balance or remaining credits, the spend this month with a trend indicator, and a timestamp for when it was last updated.

The dashboard is rich with charts too. There's a line chart showing your spending trend by day or by week. There's a pie chart breaking down cost by provider. There's a bar chart showing token usage by model. And there's a heatmap or calendar view of your usage patterns. Alongside the charts sit detailed tables, one showing usage history for each provider, another breaking things down to the level of individual models. The app also raises alerts, warning you when a balance runs low based on thresholds you set, and notifying you when your daily spend climbs unusually high.

Cost calculation is its own important piece. The app comes with built-in pricing tables for the major models, translating input and output tokens into dollar amounts. You can override the pricing for any model yourself, and even when a provider only reports raw token counts, the app still shows you an estimated dollar figure.

For data persistence and export, everything historical lives in a local SQLite database, and you can export the full dataset as either CSV or JSON whenever you like.

Security and user experience matter a great deal here. API keys are stored encrypted on your machine using symmetric encryption. There's a dark and light theme, with dark as the default to match the look of modern AI tools. The interface runs in your browser, accessible from localhost, and it's fully offline-first once the initial setup is done. The whole thing should feel responsive, clean, and professional.

There are also some nice-to-have ideas for a later phase. AI-powered insights, for example, that might tell you something like, you spent sixty-eight percent of your budget on Kimi this month, so consider shifting your heavy tasks to Groq. Importing and exporting your configuration, possibly even importing from cc-switch directly. Per-model cost forecasting. And an optional proxy or router mode, similar to LiteLLM or cc-switch, so the app could log usage straight from your actual API calls.

On the question of which providers to support, the app should ship with pre-built presets covering the fetch logic and pricing for as many as possible. The high priority ones are the user's current providers, xAI with the Grok API, Poe with its usage and balance endpoints, OpenRouter with its account usage API, MiniMax with its token plan and usage endpoints, and Kimi from Moonshot AI with its usage and balance API. Beyond those, the app should include full presets for a long list of other major players, including OpenAI, Anthropic's Claude, Google's Gemini and Vertex AI, Groq, Mistral, Together, Fireworks, DeepSeek, SiliconFlow, Zhipu's GLM models, Baichuan, StepFun, Alibaba's DashScope and Qwen, Baidu's ERNIE, Tencent's Hunyuan, iFlytek's Spark, 01.AI, Cohere, Perplexity, Replicate, Hugging Face inference endpoints, Novita, Lepton, Azure OpenAI, and AWS Bedrock where that's feasible. On top of all that, it should handle any custom OpenAI-compatible endpoint, where the user simply supplies a base URL and a key. For providers that don't expose a public usage API, the presets still help by offering manual balance entry along with notes on how to copy and paste the numbers from their dashboards.

Now for the technical stack. On the backend, it's built in Python, version three point eleven or newer, using the FastAPI framework. The API follows the OpenAPI specification, automatically generated by FastAPI and browsable through built-in docs. Data lives in SQLite, accessed through the SQLAlchemy object-relational mapper. HTTP requests go out through an async client called httpx. API keys are encrypted at rest using the cryptography library's symmetric encryption. And background tasks run through FastAPI's own background task system combined with a scheduler.

The API design is RESTful, with endpoints for all the create, read, update, and delete operations. The schema is generated automatically, and there are interactive docs available through both Swagger UI and a ReDoc alternative.

On the frontend, it's React eighteen with TypeScript, built using Vite. Styling comes from TailwindCSS together with the shadcn component library. Charts are drawn with Recharts. State is managed with Zustand. And HTTP calls go through Axios or the native fetch API.

For the overall architecture, remember this is a web app, not a desktop app, and it runs locally in your browser. The backend runs as a local server on port eight thousand. The frontend is served either through FastAPI's static files or through a separate Vite development server. And it's a hundred percent local, with no cloud sync unless that gets explicitly added down the road.

The suggested development path moves through clear phases. First, set up the project, the FastAPI backend, the React frontend, and SQLite. Then build provider management and secure key storage. Next, create the core usage fetcher system as an abstract interface. After that, implement five or six of the high-priority providers like xAI, Poe, OpenRouter, MiniMax, Kimi, and OpenAI. Then build out the dashboard interface and its charts. Then add the remaining providers and their pricing tables. Then layer in background refresh, alerts, export, and general polish. And finally, testing and documentation.

As for what gets delivered, the project includes complete source code with excellent comments and a thorough readme. There are clear instructions on how to add a new provider, which comes down to a new Python module plus a pricing configuration. There are setup scripts for running everything locally with FastAPI and React. There's sample data and a test mode. And the license is MIT, or whatever the user prefers.

All together, this specification gives the coding agent everything it needs to build a production-ready, beautiful, and genuinely useful central usage dashboard. And with that, it's ready to code.
