想象一下，你已经注册订阅了一大堆不同的人工智能服务。你在 x.ai 有个账户用来玩 Grok 模型，在 Poe 又有一个，MiniMax 有一个，Moonshot AI 旗下的 Kimi 又有一个，OpenRouter 也有一个，而且很可能除了这些之外，还有好几个。每一个服务都是分开向你收费，各自用不同的方式追踪你的开支，又各自在自己网站的一个角落显示你还剩多少余额。要一个一个地全部跟住，很快就会变成一件特别头疼的事。所以这个项目的想法很简单，就是做一个网页应用程序，把所有东西集中到一个中央视图里，让你可以在同一个地方，一次性追踪你在每一个 API 账户上的用量、成本、余额、开支，以及你烧掉了多少 token。

它的运作方式其实很简单。你只需要把你的 API 密钥添加一次，从那一刻开始，这个应用程序就会把所有东西汇总到一个漂亮的仪表盘里显示给你看。你这个月的总开支、还剩多少额度、每一个服务商的明细、各种图表、趋势和提醒，整个全貌都集中在一个地方。

说到灵感来源，这个项目参考了一个很受欢迎的桌面工具，叫 cc-switch，很多人用它来管理 Claude Code、Codex 以及 Gemini 命令行等等的 LLM 服务商。不过这个网页应用程序的焦点更窄、更集中。它纯粹专注于你个人 API 密钥的用量和成本分析。你可以把它想成是那个工具的用量分析伙伴，只不过它是在你的浏览器里运行。

我们来聊聊核心目标。第一，一个统一的地方，让你监控所有开支和用量。第二，安全地存储你的 API 密钥，完全留在你自己的机器上，永远不会发送到任何服务器。第三，可以自动或者随时按需去抓取用量和账单数据。第四，有历史追踪和恰当的可视化呈现，让你看得到这些数据怎样随时间变化。第五，这个系统要极其容易扩展，意思是随时有新服务商出现，都很容易加进来。最后，界面要漂亮、有现代感，跟 cc-switch 一脉相承。

现在说说那些一定要有的功能。先从服务商管理开始。你应该可以新增、编辑和移除账户，而每一个账户你都会记录一个名称、从预设列表里选择的服务商类型、API 密钥、用来连接自定义端点的基础网址，还有你想写下的任何备注。这个应用程序会内置尽可能多的服务商预设，也支持在同一个服务商下开多个账户。它有一个一键全部刷新的按钮，再加上每个账户各自的刷新按钮。

接下来就是真正去抓取用量和余额数据。凡是服务商提供了官方 API，应用程序都会优先用它，比如用量端点、账单端点、余额端点，或者剩余 token 套餐的端点。当没有官方途径的时候，也有后备方案。你可以手动输入你当前的用量或余额，而作为最后的手段，应用程序可以用浏览器自动化工具去抓取服务商网页仪表盘上的数据。数据会按你可以设置的间隔在后台自动刷新，而你也随时可以手动触发刷新。每一个历史快照都会存储在本地数据库里。

接着就是仪表盘本身。概览页面会显示你今天、这个月以及历来的美元估算总开支。它会显示你剩下的总额度或余额，尽可能在不同服务商之间统一标准化。它还会显示有多少个服务商正在启用，连同一个快速状态检查。再往下一点，你会看到服务商卡片，可以排成网格或者列表，每张卡显示名称以及在有的情况下显示标志、当前余额或剩余额度、这个月开支连同趋势指标，还有最后更新的时间戳。

这个仪表盘也有很多图表。有一个折线图显示你按天或按周的开支趋势。有一个饼图按服务商拆解成本。有一个柱状图显示按模型划分的 token 用量。还有一个热力图或者日历视图，呈现你的用量模式。在图表旁边就有详细表格，一个显示每个服务商的用量历史，另一个细分到单个模型的层面。应用程序也会发出提醒，当余额跌到你设置的阈值以下时提醒你，又会在你每天开支异常攀升的时候通知你。

成本计算本身也是一个重要的部分。应用程序内置了主流模型的定价表，把输入和输出 token 换算成美元金额。你可以自己覆盖任何一个模型的定价，而即使服务商只报告原始 token 数量，应用程序依然会给你显示一个估算的美元金额。

至于数据存储和导出，所有历史数据都存放在本地的 SQLite 数据库里，而你随时都可以把完整的数据集导出成 CSV 或者 JSON。

在这里，安全和用户体验都很重要。API 密钥会用对称加密存储在你的机器上。有深色和浅色主题，默认是深色，配合现代人工智能工具的风格。界面在你的浏览器里运行，从 localhost 访问，而初始设置完成之后，它就完全以离线优先。整个东西应该让人感觉响应快、干净又专业。

还有一些属于后期阶段的锦上添花的想法。比如由人工智能驱动的洞察，可能会告诉你，你这个月在 Kimi 上花了百分之六十八的预算，不如考虑把重型任务转到 Groq。又或者导入和导出你的配置，甚至可能直接从 cc-switch 导入。再有按模型的成本预测。还有一个可选的代理或路由模式，类似 LiteLLM 或者 cc-switch，这样应用程序就可以直接从你实际的 API 调用里记录用量。

说到要支持哪些服务商，应用程序应该内置预设，涵盖尽可能多服务商的抓取逻辑和定价。高优先级的那些就是用户当前正在用的服务商，包括用 Grok API 的 xAI、有用量和余额端点的 Poe、有账户用量 API 的 OpenRouter、有 token 套餐和用量端点的 MiniMax，以及有用量和余额 API、来自 Moonshot AI 的 Kimi。除了这些之外，应用程序还应该为一长串其他主要参与者纳入完整预设，包括 OpenAI、Anthropic 的 Claude、Google 的 Gemini 和 Vertex AI、Groq、Mistral、Together、Fireworks、DeepSeek、SiliconFlow、智谱的 GLM 模型、百川、阶跃星辰、阿里巴巴的 DashScope 和 Qwen、百度的文心一言、腾讯的混元、科大讯飞的星火、零一万物、Cohere、Perplexity、Replicate、Hugging Face 的推理端点、Novita、Lepton、Azure OpenAI，以及在可行情况下的 AWS Bedrock。在这些之上，它还应该能处理任何兼容 OpenAI 的自定义端点，用户只需要提供一个基础网址和一个密钥就行。至于那些没有公开用量 API 的服务商，这些预设也仍然能帮上忙，会提供手动输入余额的功能，再加上怎样从它们的仪表盘复制粘贴这些数字的说明。

现在说说技术架构。在后端，它用 Python 构建，版本要三点一一或者更新，采用 FastAPI 框架。这个 API 遵循 OpenAPI 规范，由 FastAPI 自动生成，可以通过内置文档浏览。数据存放在 SQLite，通过 SQLAlchemy 对象关系映射工具访问。HTTP 请求则经由一个叫 httpx 的异步客户端发送出去。API 密钥用 cryptography 库的对称加密在静态时加密存储。而后台任务则通过 FastAPI 自带的后台任务系统，配合一个调度器来运行。

API 设计是 RESTful 风格，为所有新增、读取、更新和删除操作提供端点。这个结构描述是自动生成的，而交互式文档则通过 Swagger UI 以及一个 ReDoc 替代方案提供。

在前端，它用 React 十八配合 TypeScript，并用 Vite 构建。样式来自 TailwindCSS，再加上 shadcn 组件库。图表用 Recharts 绘制。状态则用 Zustand 管理。而 HTTP 调用则经由 Axios 或者原生的 fetch API。

说到整体架构，要记住这是一个网页应用程序，不是桌面应用程序，它在你的浏览器里本地运行。后端会作为一个本地服务器，在八千号端口运行。前端则由 FastAPI 的静态文件，或者由一个独立的 Vite 开发服务器来提供。而它是百分之百本地运行，除非以后明确加入，否则不会有任何云端同步。

建议的开发路线会经过几个清晰的阶段。第一，把项目搭建好，包括 FastAPI 后端、React 前端和 SQLite。接着建立服务商管理和安全的密钥存储。再来，把核心用量抓取系统做成一个抽象接口。之后，实现五六个高优先级的服务商，比如 xAI、Poe、OpenRouter、MiniMax、Kimi 和 OpenAI。然后构建仪表盘界面和它的图表。再加入其余的服务商和它们的定价表。然后逐层加入后台刷新、提醒、导出和整体打磨。最后就是测试和文档撰写。

至于交付的成果，项目包括有出色注释和详尽说明文档的完整源代码。会有清晰的说明，讲解怎样加入一个新服务商，其实归根结底就是一个新的 Python 模块加上一份定价配置。会有安装脚本，让你用 FastAPI 和 React 在本地运行所有东西。会有示例数据和一个测试模式。而授权则用 MIT，或者用户喜欢的任何授权。

全部加在一起，这份规格给了编码代理所需的一切，去构建一个可以投入生产、漂亮而又真正实用的中央用量仪表盘。说到这里，已经准备好开始写代码了。
