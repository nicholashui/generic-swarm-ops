想象一下，你可以把一份文档交给一个编码代理，然后跟它说：「把这个整个项目从零开始重新做一遍，别问问题。」这份规格文档，正是为了实现这件事而设计的。它是一份完整的重新开发交接文档，对象是一个叫做 Grok Research Agent 的工具。今天我们就一起走一遍，看看它怎么运作、做些什么，以及构建它的时候要遵守的规则。那我们就开始吧。

首先，我们说说这份文档究竟是什么。它是一份合约。它的整个目的，就是要让一个编码代理能够重新做一遍这个项目，而不需要任何额外的澄清。它写明了项目的背景和预期的成果。它界定了所需的技术栈，以及精确的实现参数。它讲清楚了架构，以及每个模块从哪里开始、在哪里结束。它描述了每一个功能应该怎么运作，以及证明它有效的验收标准。它还涵盖了从用户体验、无障碍设计，到质量、测试、性能和安全的所有内容。它甚至界定了在重建可以被称为完成之前，必须达到的里程碑。这里的核心想法，就是无论原本的代码在哪里模糊或者不清晰，这份文档都会介入，消除这些含糊不清。它不是一个松散的建议。它是目标状态。

那 Grok Research Agent 究竟是什么呢？它的核心，是一个本地优先的命令行工具，用 Python 写成。它把一个八阶段的研究工作流程自动化，并且通过一个 OpenAI 兼容的接口，使用 xAI 的 AI 模型 Grok。你从一个主题和一个可选的聚焦范围开始。从那里开始，系统就会收窄范围、分析来源，并且从精心挑选的材料里抽取保留证据的笔记。它会建立一个叫做主笔记本的东西，然后综合出一份详细的研究报告。除此之外，它还可以选择性地生成一个结构化的知识库、一个学习训练包、用来生成视觉图像的图像提示、一个 YouTube 旁白脚本，以及它所用过的来源的完整离线副本。

整件事是围绕着几个核心原则来构建的，而这些原则在重建的时候绝对必须保留。它是本地优先的，意思是你所有的会话状态和你的产物，都放在你自己的机器上。它是由人类主导的，所以这个工作流程会刻意在几个关键检查点暂停，让操作员可以保持掌控。它是产物透明的，意思是每一个有意义的步骤都会写出你真正可以打开和检查的文件，而不只是一个最终结果。它是可以续做的，所以你可以停下来，然后跨越多次执行再接着做下去，不会迷失你的位置。而它是由 AI 辅助但又不是一个黑盒。模型的每一个产出都会保存成一个你可以审阅和重用的文件。

这是给谁用的呢？主要用户是研究操作员，也就是一个在本地运行这个工具、用来产出深度研究成果的技术人员。还有一些次要用户，比如说会看报告的审阅者，以及会用训练包和知识图谱来学习的学习用户。

那么，重新开发有清晰的目标。这个编码代理需要构建一个生产级别、等同于现有系统的东西。也就是说要保留所有工作流程指令和产物契约，保持以会话为本的本地存储，维持八阶段的生命周期以及辅助指令，支持外部文档的预处理，并且保持知识库、训练包、图像提示和 YouTube 脚本生成功能完好无缺。在所有这些之上，重建还要在可维护性、测试和确定性的文件输出方面加固，并且要把每一个边界情况记录和验证好。

我们要讲清楚什么在范围之内，什么不在范围之内。范围之内的有：一个 Python 命令行应用程序、完整的会话处理、八阶段的工作流程引擎、一个可选的无人值守自动模式、本地文档预处理、网页抓取和文字抽取、提示编排、知识编译、训练包生成、图表导出、图像提示、YouTube 脚本、一整套自动化测试，以及供本地安装的打包。范围之外的有：不需要浏览器应用程序、不需要用户账户或者认证、不需要团队协作、不需要远程数据库、不需要后台任务队列、不需要云端基础设施、不需要向量数据库，也不需要图形化仪表盘，除非日后有人明确批准。

说到交付模式，这是一个本地命令行应用程序。它以单一进程运行，采用同步编排，并且用一个有上限的线程池来做并行处理。持久化只用文件系统。它以一个可安装的 Python 包的形式发布，附带一个命令行入口点。而验收的主要目标是 Windows PowerShell 优先，不过代码在合理的地方应该保持可移植到 macOS 和 Linux。

现在我们说说技术栈，因为这份规格在版本方面非常精确。语言是 Python 3.11.9，并且为构建工具锁定了特定版本。至于运行时的库，它依赖 OpenAI 客户端来和 Grok 沟通、一个用来加载环境文件的库、一个用来让控制台输出美观的 rich 终端渲染库、一个用来做会话结构描述的校验库、一个保留给将来用的 token 计数工具，再加上从 PDF 和 Word 文档抽取文字、抓取网页、清理 HTML、抽取可读文章内容，以及处理字符编码的库。重点是，每一个都有一个确切的所需版本，而它们全部都要锁定。至于开发和质量保证方面，有一个测试框架、一个覆盖率报告工具、一个维持代码整洁的 linter，以及一个静态类型检查器。说到打包，这个包名叫 grok-research-agent，它提供一个单一的命令行脚本，而它所用的提示文件必须以打包数据的形式，随项目一起发布。

代码仓库也有一个界定好的结构。有一个主包文件夹，里面包含命令行接口、Grok 客户端、会话管理器、工作流程阶段、外部文档处理器，以及一个放文本提示文件的文件夹。在它旁边，还有一个测试文件夹以及惯常的项目元数据文件。这个编码代理可以自由地多加一些测试文件，甚至多加一些模块，只要是改善了关注点的分离，并且公开行为和规格保持一致就行。

我们看看这个架构。在最顶层，研究操作员和命令行入口点交互。那个入口点和两个主要部分沟通：会话管理器和工作流程执行器。工作流程执行器是最忙的那个。它协调 Grok 客户端、外部文档处理器和网页抓取逻辑。会话管理器和工作流程都会读写本地文件系统。Grok 客户端会向外连接到 xAI Grok API，而网页抓取层会向外连接到远程来源主机。

每一个核心组件都有一个清晰的职责。命令行层负责解析参数、校验所需的标志、设置会话管理器和工作流程执行器，并且把已知错误转换成干净的退出代码。会话管理器负责创建唯一的会话 ID、建立目录结构、读写会话文件、创建唯一的执行目录，并且提供标准路径。工作流程执行器负责编排状态机、构建提示、调用 Grok、抓取和预处理内容、写出产物，并且管理交互和自动两种行为。Grok 客户端负责加载 API 配置、设置连接到用 xAI 的基础地址、执行聊天补全、把失败分类成有意义的错误，并且支持追踪请求和响应。而外部文档处理器负责递归地找出支持的本地文档、抽取它们的文字、把它们分类、生成摘要、约束条件和相关性信号，并且写出汇总好的外部背景。

数据流程遵循一个清晰的顺序。用户发出一个指令。命令行层加载或者创建一个会话，然后调用工作流程执行器。工作流程创建一个执行目录，并且开始读写产物。在研究阶段期间，它会让 Grok 生成范围、发现、抽取、草稿和最终输出，并且从网上抓取所选的来源。在知识指令期间，它会让 Grok 编译和更新知识输出，并且把那些写入磁盘。最后，状态连同控制台输出和一个退出代码，一起流回给用户。至于集成方面，只有三样东西要紧：通过安全连接的 xAI Grok API 来做所有 AI 的工作、通过标准网页协议的任意来源主机来抓取来源，以及本地文件系统来存储所有要保存的东西。其他都不需要。

现在我们说说会话和持久化怎么运作，因为这里正是本地优先哲学最明显体现的地方。每一个会话都有它自己的目录，在里面你会找到一个会话文件，加上一整系列有编号的产物。有一个已确认范围文件、一个发现表、经筛选的来源、放抽取内容、来源快照和分块的文件夹、一个主笔记本、章节证据和草稿、完整来源副本、最终报告、一个图像生成文件、一个 YouTube 脚本、一个放外部文档的文件夹、一个知识库文件夹，以及一个 runs 文件夹，里面放着所有东西的执行范围副本。

会话状态本身追踪一长串的字段：会话 ID、主题、可选的聚焦范围、模式、有关外部文档的详情（包括它们的状态、摘要、文件数目、完成率、相关性分数和任何最近的错误），加上创建和更新的时间戳、Grok 模型、当前的阶段，以及执行的历史。

会话 ID 遵循特定的规则。这个 ID 是由主题的一个简化、slug 化版本，结合当前日期来生成。这个 slug 是小写，把非字母数字字符的连续段落用连字符替换，把重复的连字符合并，并且把前后的连字符剪掉。如果主题前缀太长，系统会加上一个短的八字符哈希值。而如果一个用那个名字的目录已经存在，它会加一个数字后缀，一路往上数，直到这个名字是唯一的为止。执行目录也遵循它自己的模式。每一个调用工作流程执行器的指令，都会创建一个新的执行目录，名字用日期、时间和微秒来命名，而如果有冲突，它会重试，直到找到一个唯一的名字，或者试了一千次之后放弃。

配置就归结为环境变量。有一个强制的变量：Grok API 密钥，它必须存在而且不能为空。然后有几个可选的：模型名称，默认是 Grok 3；一个最大输出 token 的配置，默认是五万，如果你给它一个无效的值，就会回退到这个默认值，最小值是一；一个请求超时，默认是三百秒，有同样的回退和最小值行为；以及一个编辑器变量，只在第一个阶段的手动编辑步骤才用到。系统会尝试从项目根目录加载一个环境文件，如果没有，它就依赖实际的环境变量。如果 API 密钥找不到，它必须给出一个清晰、可执行的错误。

我们走一遍整套指令。一共有十二个指令：start、resume、列出会话、列出类型、update、synthesize、compile、drill、feed、show、生成图像，以及 YouTube 脚本。还有一些跨指令共用的标志，比如指定会话目录、启用自动模式、启用全自动来源收集，以及打开对 AI 调用的追踪。

每一个指令都有一个界定好的契约。Start 需要一个必填的主题、一个可选的聚焦范围、一个可选的外部文档目录，以及一个可选的模式，可以是 report、compiler 或者 drill。它创建会话、打印出会话 ID，然后把你带入第一个阶段。Resume 需要一个会话 ID，然后从保存的阶段接着做下去。列出会话会打印出它找到的所有有效会话 ID，或者告诉你找不到。列出类型就只打印出那一个支持的类型，就是 auto-hypergraph。Update 重新执行发现，并且把它标记为自上次执行以来的后续跟进，然后重置回筛选阶段。Synthesize 用当前的笔记本，强制执行综合阶段。Compile 需要一个会话 ID 和一个类型，并且在知识库里生成超图和核心概念文件。Drill 需要一个会话 ID 和一个 backward 模式，并且生成一个训练包和一个问题文件。Feed 需要一个会话 ID 和一份新文档，保存一个有时间戳的副本，并且更新超图。Show 生成图表文件。生成图像生成图像文件。而 YouTube 脚本生成 YouTube 脚本文件。

那么，这个系统的核心是工作流程状态机，它会走九个有编号的阶段，从零到八。阶段零是范围确认，涉及人类。阶段一是发现，自己运行。阶段二是筛选和缺口分析，人类回到这个循环里。阶段三是抽取，自动。阶段四是笔记本组装，自动。阶段五是综合和审阅，有人类参与。阶段六是完整离线收集，由人类决定。阶段七是最终润色，自动。而阶段八就纯粹代表完成。

有几个执行阶段的常量，需要保持可配置，但默认为特定的值。来源文字会切分成四万五千字符的分块，加五千字符的重叠。笔记本分块是七万字符。网页抓取最多用四个并行工作者，而抽取和证据生成就各自用两个。而最终报告目标是在九千到一万字之间，目标是九千五百字。标准报告还有六个固定的章节名称，必须一字不差地保留：核心定义和范围；架构和技术机制；工作流程、过程和操作模式；证据、例子和案例研究；局限、取舍和失效模式；以及开放问题和未来方向。

我们逐个功能地走一遍，看看每一个阶段实际做些什么。在阶段零，范围确认，系统会从你的主题和聚焦范围生成一个精简的范围摘要，把它保存在本地，并且打印到控制台。你可以回应 yes、edit 或者 cancel。说 yes 就确认范围，并且转到发现。说 edit 就创建一个可编辑的文件、可选择地打开你的编辑器、重新加载你改过的内容，并且继续提示你。说 cancel 就让这个阶段保持原样。在自动模式里，它就只确认然后接着走下去。重要的是，一个空白或者格式错误的聚焦范围不能让任何东西崩溃，而一个找不到的编辑器也不能阻塞这个编辑流程。

阶段一，发现，会用你的主题、聚焦范围，以及这是不是一次后续跟进，来渲染一个发现提示。它会保存发现输出，然后转入筛选。它在保存之前不会卡在格式校验上，所以哪怕是不完美的输出也能保存下来。之后发现表文件应该存在，而 update 指令可以把这次重新执行当成一次后续跟进。

阶段二是筛选和缺口分析。它需要发现表，会给你看一份已发现来源的预览，并且接受你的选择作为自由格式的指示。它会尝试最多三次去拿到有效的已筛选来源数据，然后把每一个来源规范化成一个统一的形态，包含标题、URL、类型、它相关的原因、可信度和优先级。如果它无法解析结构化数据，就直接根据发现文字用启发式的方式恢复出 URL。它会保存已筛选的来源、生成一份缺口报告，并且要求你一字不差地打出 approve 这个词才会继续往前。整个系统必须能处理围栏数据块、被标点符号或者反引号包住的 URL，甚至空的发现文件，全都不能崩溃。

阶段三是抽取。它需要有效的已筛选来源，建立抽取目录，并且请求一个抽取计划。它会并行地抓取来源集合、同时保留原始快照和规范化文字快照两者、把文字切分成有重叠的分块，并且并行地从每一个分块抽取保留证据的笔记。它会保存单个的分块输出、把它们组装成来源文件，而且至关重要的是，它会跳过任何失败的来源，不会中止整个阶段。某个分块或者某个来源超时，就只跳过那一件事。无效的 URL 会被安全地跳过，而 PDF 则会和其他内容一样被抽取和保存。

阶段四是笔记本组装。它取出抽取出来的文件，建立一个主笔记本，里面包含标题、主题、笔记本的用途、可选的外部文档背景、一个来源目录、一个可选的知识库大纲，以及文件本身。笔记本文件会被写出，然后工作会话转入综合。

阶段五是综合和审阅。它把笔记本切分成分块、并行地生成各章节专属的证据包，然后由这些证据包和来源目录生成章节草稿。它组装出一份完整草稿，按编号做版本控制，并且接受你的审阅反馈。一字不差地打出 approve 就把你带到下一个阶段。其他任何内容都会被当成修订输入，而修订后的草稿会保存成一个新版本。草拟出来的章节使用那些一字不差的标准章节名称，而较早的草稿永远会被保留。没有证据的章节会被跳过并且发出警告，而不是直接失败。

阶段六是完整离线收集。它给你看有哪些来源可用，并且让你选择全部、不选，或者一个用逗号分隔的数字列表。它会预先抓取所选的来源，并且把完整的离线副本保存成 Markdown。如果没有已筛选的来源，它会尝试从发现里恢复出它们，而如果这样也失败了，它就会跳到最终润色。无效或者超出范围的数字会直接被忽略，而一次失败的抓取只跳过那一个来源。

阶段七是最终润色。它需要笔记本和至少一份草稿，并且使用最新的草稿版本。它会生成一份执行摘要和一个术语表、根据章节标题建立一个目录，并且用重新定向的逻辑来强制达到目标字数。然后它保存最终报告、生成图像提示，并且生成 YouTube 脚本。最终报告包含标题、目录、执行摘要、报告正文、来源目录、一个可选的知识库概览，以及术语表。如果字数重新定向超时，它仍然会保存当时最好的可用版本。如果执行摘要或者术语表超时，它会插入后备文字。而如果正文开头有一个重复的标题，那个重复会在组装之前被移除。

在主工作流程之外，还有知识指令。Compile 如果笔记本存在就从笔记本建立，否则就从抽取出来的文件建立。它以一个结构化格式生成一个超图，并且保存到两个标准位置，然后生成并保存核心概念。如果模型返回无效的数据，它仍然会以一个安全包装的形式把它保存下来，而不会失败。Drill 只支持 backward 模式，如果核心概念还不存在，它会自动先执行 compile，然后生成操练包和结构化问题。如果 compile 之后核心概念仍然不在，它会带着一条清晰的消息退出。Feed 会验证所提供的文件存在而且是一个正常文件、把它复制到一个带时间戳的 feed 文件夹，并且如果超图不见了就通过 compile 把它初始化，否则就更新现有的那个。读不出来的内容会用替换字符来加载，而不会崩溃。而 show 需要超图，并且渲染一个图表文件，这个文件以一个标准的图声明开头，根据可用的数据画出节点和边。

外部文档预处理值得特别一提。它会递归地发现支持的文件、处理 PDF、Word 文档、纯文本和 Markdown，并且把每一份文档分类为 guideline、background、steering 或者 general 材料。它抽取关键概念、约束条件、需求和算法洞见，然后把所有内容汇总成一个清单文件、一份抽取摘要和一个背景文件。它把处理指标保存在工作会话状态里，并且把那个外部背景注入回相关的提示里。部分文件失败绝对不能中止整个工作流程。

然后还有图像提示生成，它需要最终报告，并且生成五到十个提示，强调具体的机制、工作流程、架构和证据。还有 YouTube 脚本生成，它同样需要最终报告，但在章节草稿可用的时候，它会优先用它们作为来源材料。它特意把目录、来源目录、术语表、参考文献、知识库概览和执行摘要排除在旁白之外。它会生成一段引言、章节标题和一个结论，并且把过短的输出扩充到符合最低长度门槛。

这份规格也界定了确切的输出契约。已筛选的来源以一个结构化记录列表保存，每一个都有标题、URL、类型、相关的原因、一个数值的可信度评分，以及一个优先级。超图以节点和超边的形式保存，每一条边用一个有名称的关系和支撑证据连接多个节点。而如果无法生成有效的结构化输出，系统会保存一个包装后备，单纯保留原始的原本输出。核心概念每一个都带有一个名称、一个定义，以及一段解释它们为什么是基础的说明。而操练包把 Markdown 内容和结构化问题结合在一起，每一个都关联着一个概念，并且包含一道问题、一个答案，以及常见的陷阱。

错误处理是细致地界定的。Grok 层暴露三种错误类型：一个一般错误、一个超时错误和一个配额错误。类似配额的失败对应到配额错误，类似超时的失败对应到超时错误，其他所有的就对应到一般错误。至于重试，不是超时或者配额问题的普通失败，会用指数退避重试最多五次，上限是三十秒。系统也设计成优雅地降级，而不是崩溃。当像缺口分析、抽取计划、证据包、草稿、修订、摘要、术语表、图像提示或者字数重新定向这些环节失败的时候，系统会在它能继续的地方继续下去。但在真正致命的情况下，它会干净而清晰地停下来：当所需路径的 API 密钥不见了、当必要的文件缺席、当指令参数无效，或者当一个工作会话无法加载的时候。

说到性能，这份规格为一台标准的开发者工作站加上宽带，设定了具体的基准。建立一个工作会话并且写出它的状态，应该在一秒以内。列出工作会话应该在一秒以内，即使是多达五百个工作会话也是如此。建立一个执行目录应该在四分之一秒以内。而知识指令的元数据写入应该在一秒以内，不算 AI 或者网络时间。至于并行处理，网页抓取支持多达四个并行请求、抽取支持两个并行 AI 调用，而证据生成支持每个章节两个并行调用。至于规模，它应该能处理至少二十个已筛选来源、笔记本里至少二十个文件、至少四十份外部文档，以及至少二百个节点和四百条边的图表。指导哲学是，正确性和韧性比激进的并行处理更重要，而超时应该永远保留已经保存下来的进度。

安全是认真对待的。API 密钥绝对不能写入被追踪的文件或者工作会话产物。机密只从环境中加载。被抓取的页面和被处理的文档，都被当成可能敏感的本地数据。xAI API 只通过安全连接访问，而任何非安全的 URL 在抓取期间都会被拒绝。被追踪的 AI 输出，在显示之前它的控制字符会被剥除。系统永远不会执行远程内容、永远不会调用 shell，除了可选的编辑器启动之外，并且确保失败路径永远不会把机密泄漏到日志或者控制台里。

至于用户体验和无障碍设计，交付物是命令行优先的，而一个浏览器界面绝对不能取代这个工作流程。重点是一个干净的终端体验、可读的 Markdown 产物，以及可用的文字输出。提示应该是简短、明确并且以行动为导向的，永远告诉用户下一步该做什么。无障碍目标把网页无障碍原则适配到命令行：颜色绝对不能是传达含义的唯一方式、提示和错误在纯文本里必须讲得通、生成的标题必须正确地排序、表格必须有清晰的表头，而错误必须有描述性，并且指向一个修正方法。因为产品只是命令行，所以不需要响应式的浏览器布局，但生成的报告在窄窗格里仍然必须读起来舒服，而任何有人添加的可选预览界面，都必须是移动设备响应式的，并且符合同样的无障碍标准，同时又不取代命令行。

测试需求很广泛。这个代理必须实现单元测试、集成测试和端到端工作流程测试。单元测试需要覆盖工作会话 ID 生成、状态来回转换、配置解析、错误映射、URL 规范化、宽松数据解析、来源标准化、发现恢复、文字分块、HTML 清理、图表生成和外部文档分类。集成测试覆盖指令派发、用一个假 AI 客户端的 start 和 resume 流程、compile 和 drill 工作流程、feed 和 show、外部文档预处理和超时降级。必须有至少一个完整的快乐路径端到端测试，走完工作会话建立、从零到八的每一个阶段，以及所有最终产物的生成。并且必须有一个自动模式测试，证明永远不会发生交互式输入、工作流程会走到完成，以及当启用了完整收集的时候会保存离线副本。这份规格也列出了所需的边界情况测试，例如一个不见了的 API 密钥、不见了的文件、围栏数据输入、各种超时和不见了的必要条件。覆盖率目标是在核心编排模块上，至少百分之八十五的行覆盖率，和至少百分之七十五的分支覆盖率，而整套测试必须在 Windows PowerShell 上通过。

质量标准把这一切收尾。核心模块的公开函数应该有类型标注、编排逻辑不应该重复，而 HTTP 和 AI 客户端应该被注入以便测试。来源文字应该在可能的地方保持 ASCII，而注释只在行为真的不显然的地方才出现。这个代理必须运行并且通过 linter、类型检查器和整套测试，而被修改的文件在交接之前必须没有编辑器诊断信息。

这份规格接着为每一个交付物界定验收标准。当所有指令存在、说明文字准确、退出码符合规格，而且本地安装行得通的时候，核心命令行工具就被接受。当交互路径通过反复 resume 走到完成、自动模式无需输入就完成，而且所有输出都落在正确的目录里的时候，工作流程就被接受。当 compile、drill、feed 和 show 全都生成它们预期的产物的时候，知识功能就被接受。当文件被递归地发现、支持的格式被处理、汇总文件被生成，而且部分失败不会停下工作流程的时候，外部文档就被接受。而当报告有它所需的章节、图像提示和 YouTube 脚本在生成成功时存在，而且文件命名是确定性的时候，最终内容就被接受。

这些工作是按里程碑组织的，按顺序排列。第一，项目骨架和打包，以一个可编辑安装和通过基本测试收尾。第二，工作会话和状态基础，以通过工作会话测试和一个经过验证的状态来回转换收尾。第三，核心工作流程阶段零到五，以快乐路径测试通过到草稿生成收尾。第四，定稿和输出扩充，涵盖阶段六和七，加上最终报告、图像和脚本，以整个工作流程走到完成收尾。第五，知识功能，以那些产物测试通过收尾。而第六，外部文档和加固，涵盖预处理、错误恢复、超时处理，以及清理覆盖率、linting、类型和诊断信息，以所有内容通过和质量清单完成收尾。

在把工作交回之前，编码代理要提供更新后的源代码和测试、如果行为改变了就更新后的 readme、如果依赖改变了就更新后的安装说明、所运行的验证指令的确切清单、一份简短的改动摘要，以及一个明确的声明，表示所有验收标准都检查过了。如果有任何标准没达到，代理就要识别它、解释那个阻碍，并且列出确切的剩余工作。所需的验证指令包括以可编辑模式连同开发依赖一起安装这个包、运行 linter、运行类型检查器和运行测试，再加上一次覆盖率运行，如果那个工具有包含的话。

还有一份详细的质量保证清单，涵盖功能项、可靠性项、代码质量项和文档项，全都必须在定稿之前验证。而完成的定义是严格的：项目在本地安装并运行、完整的功能集都实现了、所有产物都有正确的名称和结构、所有验收标准都满足了、所有测试、lint 和类型关卡都通过了、没有剩下未解决的诊断信息，而且代理可以交接工作而无需任何后续澄清。

最后，这份规格定下了不容商量的限制。不要用一个纯图形化的方式取代命令行模式。不要把持久化搬到数据库。不要移除人工检查点。不要移除本地的、范围限定在运行内的产物。不要要求 xAI Grok 和正常来源主机以外的第三方服务。不要悄悄地吞下致命错误。还有不要破坏输出文件名或者核心数据契约。它建议一个特定的实现顺序，从包和解析器开始，然后是工作会话管理器、Grok 客户端、工作流程引擎、早期阶段、抓取和抽取、笔记本和综合、最终输出、知识指令、外部文档处理器，最后是加固和完整测试通过。

收尾的交接备注把这套理念讲得清清楚楚。如果现有的代码什么时候和这份文档有冲突，文档胜出，除非所有者明确地推翻它。目标不是对现有项目的一个松散的近似。目标是一次忠实、可测试的重建，有更强的工程纪律，并且对成果应该是什么样子完全没有含糊。这就是全貌了。一个本地优先的研究代理，完整规格化，准备好可以充满信心地重建。谢谢收看。
