階段四係筆記本組裝。佢攞抽取出嚟嘅檔案，建立一個主筆記本，入面包含標題、主題、筆記本嘅目的、可選嘅外部文件背景、一個來源目錄、一個可選嘅知識庫大綱，以及檔案本身。個筆記本檔案會被寫出，然後工作階段移去綜合。

階段五係綜合同審閱。佢將筆記本切分成分塊、並行咁產生章節專屬嘅證據包，然後由嗰啲包同來源目錄產生章節草稿。佢組裝一份完整草稿，按編號做版本控制，並且接受你嘅審閱回饋。打出一字不差嘅 approve 就移你去下一個階段。其他任何嘢都會被當成修訂輸入，而修訂後嘅草稿會儲存成一個新版本。草擬出嚟嘅章節用嗰啲一字不差嘅標準章節名稱，而較早嘅草稿永遠會被保留。冇證據嘅章節會被跳過並且發出警告，而唔係失敗。

階段六係完整離線收集。佢俾你睇有邊啲來源可用，並且俾你揀全部、冇，或者一個用逗號分隔嘅數字清單。佢會預先抓取所揀嘅來源，並且將完整離線副本儲存成 Markdown。如果冇已篩選嘅來源，佢會嘗試由發現度回復返佢哋，而如果咁都失敗，佢就會跳去最終潤飾。無效或者超出範圍嘅數字會直接被忽略，而一次失敗嘅抓取淨係跳過嗰一個來源。

階段七係最終潤飾。佢需要個筆記本同至少一份草稿，並且用最新嘅草稿版本。佢產生一個執行摘要同一個詞彙表、由章節標題建立一個目錄，並且用重新定目標嘅邏輯嚟強制執行目標字數。然後佢儲存最終報告、產生圖像提示，並且產生 YouTube 腳本。最終報告包含標題、目錄、執行摘要、報告正文、來源目錄、一個可選嘅知識庫概覽，以及詞彙表。如果字數重新定目標逾時，佢仍然會儲存最好嗰個可用版本。如果執行摘要或者詞彙表逾時，佢會插入後備文字。而如果正文開頭有一個重複嘅標題，嗰個重複會喺組裝之前被移除。

喺主工作流程之外，仲有知識指令。Compile 如果筆記本存在就由筆記本建立，否則就由抽取出嚟嘅檔案建立。佢以一個結構化格式產生一個超圖，並且儲存去兩個標準位置，然後產生同儲存核心概念。如果模型回傳無效嘅資料，佢仍然會以一個安全包裝嘅形式儲存佢，而唔會失敗。Drill 淨係支援 backward 模式，如果核心概念仲未存在，佢會自動先行 compile，然後產生操練包同結構化問題。如果 compile 之後核心概念仍然唔喺度，佢會帶住一個清晰嘅訊息退出。Feed 驗證所提供嘅檔案存在而且係一個正常檔案、將佢複製去一個有時間戳記嘅 feed 資料夾，並且如果超圖唔見咗就透過 compile 將佢初始化，否則就更新現有嗰個。讀唔到嘅內容會用取代字元嚟載入，而唔會當機。而 show 需要個超圖，並且渲染一個圖表檔案，呢個檔案以一個標準嘅圖宣告開頭，由可用嘅資料畫出節點同邊。

外部文件預先處理值得特別一提。佢遞迴咁發現支援嘅檔案、處理 PDF、Word 文件、純文字同 Markdown，並且將每一份文件分類為 guideline、background、steering 或者 general 材料。佢抽取關鍵概念、限制條件、需求同演算法洞察，然後將所有嘢彙整成一個清單檔、一個抽取摘要同一個背景檔案。佢將處理指標儲存喺工作階段狀態度，並且將嗰個外部背景注入返相關嘅提示入面。部分檔案失敗絕對唔可以中止成個工作流程。

然後仲有圖像提示生成，佢需要最終報告，並且產生五到十個提示，強調具體嘅機制、工作流程、架構同證據。同埋 YouTube 腳本生成，佢都係需要最終報告，但喺章節草稿可用嘅時候，佢會優先用佢哋做來源材料。佢刻意將目錄、來源目錄、詞彙表、參考文獻、知識庫概覽同執行摘要排除喺旁白之外。佢產生一個引言、章節標題同一個結論，並且將短嘅輸出擴充到符合最低長度門檻。

呢份規格亦都界定咗確切嘅輸出契約。已篩選嘅來源以一個結構化記錄清單儲存，每一個都有標題、URL、類型、相關嘅原因、一個數值嘅可信度評分，以及一個優先次序。超圖以節點同超邊儲存，每一條邊用一個有名嘅關係同支持證據連接多個節點。而如果無法產生有效嘅結構化輸出，個系統會儲存一個包裝後備，純粹係持住原始嘅原本輸出。核心概念每一個都帶有一個名稱、一個定義，同埋一個解釋點解佢哋係基礎嘅說明。而操練包將 Markdown 內容同結構化問題結合，每一個都繫住一個概念，並且包括一條問題、一個答案，同埋常見嘅陷阱。

錯誤處理係細心咁界定嘅。Grok 層暴露三種錯誤類型：一個一般錯誤、一個逾時錯誤同一個配額錯誤。類似配額嘅失敗對應到配額錯誤，類似逾時嘅失敗對應到逾時錯誤，其他所有嘅就對應到一般錯誤。至於重試，唔係逾時或者配額問題嘅普通失敗，會用指數退避重試最多五次，上限喺三十秒。個系統亦都設計成優雅咁降級，而唔係當機。當好似缺口分析、抽取計劃、證據包、草稿、修訂、摘要、詞彙表、圖像提示或者字數重新定目標呢啲嘢失敗嘅時候，個系統會喺佢可以嘅地方繼續落去。但喺真正致命嘅情況下，佢會乾淨而清晰咁停低：當所需路徑嘅 API 金鑰唔見咗、當必要嘅檔案缺席、當指令參數無效，或者當一個工作階段無法載入嘅時候。

講到效能，呢份規格為一部標準嘅開發者工作站連寬頻，設定咗具體嘅基準。建立一個工作階段並且寫出佢嘅狀態，應該喺一秒以內。列出工作階段應該喺一秒以內，即使係多達五百個工作階段都係。建立一個執行目錄應該喺四分之一秒以內。而知識指令嘅中繼資料寫入應該喺一秒以內，唔計 AI 或者網路時間。至於並行處理，網頁抓取支援多達四個並行請求、抽取支援兩個並行 AI 呼叫，而證據生成支援每個章節兩個並行呼叫。至於規模，佢應該處理到至少二十個已篩選來源、筆記本入面至少二十個檔案、至少四十份外部文件，以及至少二百個節點同四百條邊嘅圖表。指導哲學係，正確性同韌性比起激進嘅並行處理更加緊要，而逾時應該永遠保留已經儲存咗嘅進度。

安全係認真對待嘅。API 金鑰絕對唔可以寫入被追蹤嘅檔案或者工作階段產出物。秘密淨係由環境載入。被抓取嘅頁面同被處理嘅文件，都被當成可能敏感嘅本地資料。xAI API 淨係透過安全連線存取，而任何非安全嘅 URL 喺抓取期間都會被拒絕。被追蹤嘅 AI 輸出，喺顯示之前佢嘅控制字元會被剝走。個系統永遠唔會執行遠端內容、永遠唔會 shell out，除咗可選嘅編輯器啟動之外，並且確保失敗路徑永遠唔會將秘密洩漏入記錄或者主控台。

至於用戶體驗同無障礙設計，個交付物係命令列優先嘅，而一個瀏覽器介面絕對唔可以取代呢個工作流程。重點係一個乾淨嘅終端機體驗、可讀嘅 Markdown 產出物，以及可用嘅文字輸出。提示應該係短、明確同以行動為本，永遠話俾用戶聽下一步做乜。無障礙目標將網頁無障礙原則調適到命令列：顏色絕對唔可以係傳達意義嘅唯一方式、提示同錯誤喺純文字入面必須講得通、產生嘅標題必須正確咁排序、表格必須有清晰嘅標頭，而錯誤必須有描述性，並且指向一個修正方法。因為產品淨係命令列，所以唔需要回應式嘅瀏覽器版面，但產生嘅報告喺窄嘅窗格入面仍然必須讀得舒服，而任何有人加嘅可選預覽介面，都必須係流動裝置回應式，並且符合同樣嘅無障礙標準，而又唔取代命令列。

測試需求好廣泛。個代理必須實作單元測試、整合測試同端到端工作流程測試。單元測試需要涵蓋工作階段 ID 生成、狀態來回轉換、設定解析、錯誤對應、URL 正規化、寬鬆資料解析、來源標準化、發現回復、文字分塊、HTML 清理、圖表生成同外部文件分類。整合測試涵蓋指令派發、用一個假 AI 客戶端嘅 start 同 resume 流程、compile 同 drill 工作流程、feed 同 show、外部文件預先處理同逾時降級。必須有至少一個完整嘅快樂路徑端到端測試，行勻工作階段建立、由零去到八嘅每一個階段，以及所有最終產出物嘅生成。並且必須有一個自動模式測試，證明永遠唔會發生互動式輸入、個工作流程會去到完成，以及當啟用咗完整收集嘅時候會儲存離線副本。呢份規格亦都列出咗所需嘅邊界情況測試，例如一個唔見咗嘅 API 金鑰、唔見咗嘅檔案、圍欄資料輸入、各種逾時同唔見咗嘅必要條件。覆蓋率目標係喺核心編排模組度，至少八成五嘅行覆蓋率，同至少七成五嘅分支覆蓋率，而整套測試必須喺 Windows PowerShell 上通過。

品質標準將呢一切收尾。核心模組嘅公開函式應該有型別、編排邏輯唔應該重複，而 HTTP 同 AI 客戶端應該被注入以便測試。來源文字應該喺可能嘅地方保持 ASCII，而註解淨係喺行為真係唔顯然嘅地方先出現。個代理必須執行並且通過 linter、型別檢查器同整套測試，而被修改嘅檔案喺交接之前必須冇編輯器診斷訊息。

呢份規格跟住為每一個交付物界定驗收準則。當所有指令存在、說明文字準確、結束代碼跟住規格，而且本地安裝行得通嘅時候，核心命令列工具就被接受。當互動路徑透過反覆 resume 去到完成、自動模式無需輸入就完成，而且所有輸出都落喺正確嘅目錄度嘅時候，個工作流程就被接受。當 compile、drill、feed 同 show 全部都產生佢哋預期嘅產出物嘅時候，知識功能就被接受。當檔案被遞迴咁發現、支援嘅格式被處理、彙整檔案被產生，而且部分失敗唔會停低個工作流程嘅時候，外部文件就被接受。而當報告有佢所需嘅章節、圖像提示同 YouTube 腳本喺生成成功時存在，而且檔案命名係確定性嘅時候，最終內容就被接受。

呢啲工作係按里程碑組織，有次序咁排。第一，專案骨架同打包，以一個可編輯安裝同通過基本測試作結。第二，工作階段同狀態基礎，以通過工作階段測試同一個經驗證嘅狀態來回轉換作結。第三，核心工作流程階段零到五，以快樂路徑測試通過到草稿生成作結。第四，定案同輸出擴充，涵蓋階段六同七加埋最終報告、圖像同腳本，以成個工作流程去到完成作結。第五，知識功能，以嗰啲產出物測試通過作結。而第六，外部文件同加固，涵蓋預先處理、錯誤回復、逾時處理，同埋清理覆蓋率、linting、型別同診斷訊息，以所有嘢通過同品質清單完成作結。

喺將工作交返之前，個編碼代理要提供更新後嘅原始碼同測試、如果行為改變咗就更新後嘅 readme、如果依賴改變咗就更新後嘅安裝說明、所執行嘅驗證指令嘅確切清單、一個簡短嘅改動摘要，以及一個明確嘅聲明，表示所有驗收準則都檢查過。如果有任何準則未達到，個代理就要識別佢、解釋個阻礙，並且列出確切嘅餘下工作。所需嘅驗證指令包括以可編輯模式連同開發依賴安裝個套件、執行 linter、執行型別檢查器同執行測試，加埋一個覆蓋率執行，如果嗰個工具有包含嘅話。

仲有一份詳細嘅品質保證清單，涵蓋功能項目、可靠性項目、程式碼品質項目同文件項目，全部都必須喺定案之前驗證。而完成嘅定義係嚴格嘅：個專案喺本地安裝同運行、完整嘅功能集實作咗、所有產出物都有正確嘅名稱同結構、所有驗收準則都滿足咗、所有測試、lint 同型別關卡都通過、冇剩低未解決嘅診斷訊息，而且個代理可以交接工作而無需任何後續澄清。
