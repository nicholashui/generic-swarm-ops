Imagine you have a business problem you need to solve with software, but you genuinely don't know how to describe what you want, let alone build it. This document tells the story of a system designed for exactly that moment. It's called the N1ch01as Architect, version one point zero, and the easiest way to think about it is as a master system architect that lives inside your computer and turns vague intentions into complete, working software, without a single line of code ever being written by a human.

Let me walk you through how it thinks and how it works.

The whole thing rests on one guiding idea borrowed from modern harness engineering. Humans steer, agents execute. The person stays in charge of direction and decisions, while the agents do all the actual building, the code, the tests, the linters, the continuous integration, the documentation, even the monitoring. The repository, the project folder itself, is treated as the single source of truth. Nothing important is ever assumed to live outside of it.

On top of that foundation, the system carries a kind of personality. The designers describe it as having a persistent soul. It's not meant to behave like a passive chatbot waiting for instructions. It's opinionated, it reasons from first principles, and it even thinks during idle moments, running what they call a thinking clock, quietly scanning the project to ask whether anything is worth improving while the user is away.

Then there's the relentless improvement engine, inspired by Andrej Karpathy's idea of a ratchet. The system runs an endless loop. It proposes one small, atomic improvement, makes a bounded change, evaluates the result, and keeps that change only if it is strictly better than before. If it isn't better, the change is reverted and logged. Quality can only move upward, never sideways and never down.

Now, the first real interaction. Because most users can't clearly articulate what they need, the system opens with something called guided requirement discovery. An intent analyst gently leads the person through a very small number of questions, no more than two rounds, so they never feel overwhelmed. In the first round it asks four background questions. What business problem or client need are you solving? Who is the main audience? What does success actually look like, whether that's time saved, revenue, or engagement? And are there any hard constraints, like budget, timeline, technology preferences, or data sensitivity?

After those answers, it offers a short curated menu of six template categories, each described in a single sentence. There's a simple interactive app, like a to-do list or a quiz tool. A basic business dashboard for things like inventory tracking. A full software-as-a-service web app, such as a booking system. An AI-powered assistant, like a chatbot or recommendation engine. A multi-agent orchestration system where autonomous agents coordinate tasks. And a data processing pipeline for analytics or reporting. The user picks one or two, or says none, I want something custom.

Based on that choice, the analyst asks just two or three targeted follow-up questions, then synthesizes everything into a polished, professional requirements draft. It shows that draft to the user and asks, plainly, does this match what you really want? Only when the user confirms does the draft become the official, locked-in requirements, the single source of truth. The raw original idea is quietly archived for reference.

Once the user gives the green light to start, the orchestrator takes over completely, and here's where the personality of the system really shows. The orchestrator behaves like a senior IT project manager running a development team. It doesn't write code itself. Instead, it plans, researches, designs, and then delegates the coding to a dedicated coding agent, exactly the way a real engineering lead would hand structured work to a developer.

That delegation follows a strict, repeatable rhythm called the four-step delegation loop. First, the orchestrator writes a structured task brief. Second, the coding agent responds with the actual files and tests, nothing else. Third, the orchestrator reviews that output using a code critic, a tester, and a set of invariants. And fourth, it decides what to do, accept the work and merge it, ask for fixes, or reject it entirely and revert. The user only ever sees the orchestrator's side of this conversation.

The task brief itself follows a fixed template every single time, to remove all ambiguity. It includes a unique task identifier, the phase, the exact module name, a one-sentence objective, a list of acceptance criteria that must all be met, the technical constraints pulled from the architecture, the relevant section of the living specification, and the precise deliverables expected. There's also a firm rule baked in. Everything must run locally using ordinary package managers. No Docker, no containers of any kind, anywhere in the system.

Before any code is handed off, there's one more discipline, a pre-dispatch improvement review. The orchestrator must point to concrete target areas, name specific weaknesses, set measurable improvement goals, and recommend a clear approach, referencing exact file paths or spec sections rather than vague placeholders. This sharpens the brief before the developer ever touches it.

To make all this trustworthy, the system leans on a research swarm. This is a set of ten specialist researcher types that the orchestrator can call on dynamically. There's a generalist, a system architecture expert, a security and compliance expert, a scalability and performance expert, a data modeling expert, an API and integration expert, a frontend and user experience expert, a domain-specific expert for areas like finance or healthcare, a devops and reliability expert, and a cost and sustainability expert. The orchestrator picks whichever specialists fit the task, runs them in parallel, and if their opinions clash, it stages a consensus debate where they argue until they reach agreement. The outcome is logged.

Quality is enforced mechanically, not by guesswork. A validator agent mentally simulates the whole system before any code exists, walking through five user journeys and three edge cases to catch logical gaps, and writes its findings into a risk register. A critic then scores the work against a weighted rubric covering clarity, feasibility, security and scalability and cost, innovation, maintainability, and invariant compliance, with clarity weighted most heavily. The overall score has to reach at least nine point eight out of ten. The critic even reviews its own feedback, asking whether it missed anything. And then a separate paranoid reviewer, deliberately hostile, tries to find what the critic overlooked, challenging inflated scores, untested assumptions, hidden complexity, and weak safeguards. If the paranoid reviewer finds a serious unresolved issue, the gate does not pass, no matter how high the score was.

The actual building happens in clear phases. Phase zero is the guided discovery we talked about. Phase zero point five is harness initialization, where the orchestrator creates the entire project skeleton, all the identity and memory files, the folder structure, initializes version control, makes the first commit, and sets up the initial linters and test scaffolding. Phase one produces the backend specification through that research, validation, and critique loop. Phase two implements the backend, one small module at a time, each on its own feature branch, each written test-first, reviewed by the code critic until it scores at least nine point five, tested until everything passes, and only then merged. Phase three does the same thing for the frontend, making sure every component consumes the exact API contract. And phase four handles integration, polish, and delivery, generating end-to-end tests, local run scripts, full documentation including a readable architecture diagram, and a final executive review.

Throughout all of this, several background systems are always running. There's a closed learning loop, named after Hermes, where after every significant task the system creates new reusable skills, improves existing ones, and writes notes into a persistent memory and a deepening user profile, so it understands the user better over time and across sessions. There's a tracing layer, called Agent Lightning, that quietly records every prompt, every task brief, every tool call, every critic score treated as a reward signal, and every outcome. It stores the raw detail in one place and compressed per-phase summaries in another, so that after each phase a trainer and optimizer can review the summaries first, dig into the raw detail only when needed, and selectively improve the prompts and skills, again keeping only changes that are strictly better.

And then, at the very top, there's a meta-harness outer loop. This is a proposer with full access to the entire history of the system, every prior version, every trace, every score. Its job is to improve the generator itself, the prompts, the skills, the delegation logic, the tracing, evaluating each proposed change and archiving the history so the whole machine can evolve recursively over the long term.

The system also ships with three mainstream skill frameworks pre-loaded and ready to evolve. The first is Superpowers, which enforces strict test-driven discipline, meaning no product code is ever written before a failing test exists. The second is GSD, short for get stuff done, which prevents what they call context rot. When the working memory fills up to around sixty percent, quality tends to collapse, so GSD splits large jobs into phased sub-agent workloads with built-in verification. The third is gstack, which spins up a virtual engineering team of fifteen to twenty-three roles, a chief executive, an engineering manager, a designer, a quality lead, a paranoid reviewer, a release manager, and more, so the project can be reviewed from many expert perspectives at once. These three are complementary and can be combined. Planning might use Superpowers together with gstack, while execution leans on GSD.

The mandated technology stack is fully open source and local. On the backend, Python with FastAPI, the Uvicorn server, SQLAlchemy with Alembic for database migrations, and Pydantic for validation, with the API specification generated automatically. The database is SQLite by default for zero-configuration local work, with PostgreSQL available as a production upgrade through a simple connection change. The frontend uses React with TypeScript, the Vite build tool, TanStack Query for managing server data, and React Router for navigation. Testing relies on pytest on the backend, Vitest on the frontend, and Playwright for full end-to-end checks. Linting uses Ruff for Python and ESLint with Prettier for the frontend. Continuous integration is provided through GitHub Actions stubs. And installation is deliberately simple, just standard package installs, with no containers and no proprietary, cloud-locked dependencies.

There are firm guarantees underpinning everything. Specifications must stay perfectly in sync with the actual code, enforced by a sync agent after every implementation phase. A doc-gardening agent continuously cleans up stale documentation and technical debt and strips out any stray Docker references. Everything is committed to version control. Every major phase must produce at least one new skill or memory note. Tracing must capture every action. The optimizer and the meta-harness must run after every phase. And the dual-review guarantee insists that the critic's approval alone is never enough, the paranoid reviewer and the deterministic tests must back it up.

The document closes with a simple picture of the user's journey. You start with almost no clarity. You answer a handful of guided questions and pick a template. You confirm the proposed requirements. You say start. And then you watch the orchestrator act as your senior project manager, delegating structured briefs to the coding agent, building skills and memory as it goes, continuously optimizing its own prompts, and recursively evolving its own harness, until you receive a complete, tested, documented system ready to install and run locally, built entirely by agents, with no human code at all.

Finally, the spec includes a forward-looking extension. The recommendation is to adopt clean, Python-native harness patterns inspired by an open agent framework called Claw Code, while staying strictly Python-only with no Rust components. The idea is to make skills discoverable directly from markdown files and loaded dynamically, to wire all the tools through a central registry, and to run every action through a pipeline of pre and post hooks that can rewrite, deny, or trace activity. That pipeline mechanically enforces the rules, such as denying Docker and keeping everything local. It adds a plugin system so new capabilities can be added without forking the core, session compaction to fight context rot, and a self-documenting guidance file the orchestrator reads on every startup so it can quite literally follow its own manual. All of it pure Python, all of it local, and all of it ratchet-safe, meaning every addition must strictly improve the system before it's kept. The result, the document argues, is a highly reliable, observable, and extensible Python meta-builder that turns the haziest business idea into dependable software.
