想像一下，你只要交給電腦一個題目，然後轉身離開，回來時就收到一份精心整理、研究得相當深入的報告，當中還附有知識地圖、溫習練習、配圖構思，甚至一份可以即時開錄的影片講稿。這就是 Research Agent 背後的承諾。在這段影片裡，我們會由最初的一個念頭開始，一直講到完成整個成品，帶你了解它究竟是怎樣運作的。

先講大局。這是一個本地優先的研究自動化工具。它以命令列應用程式的形式，在你自己的機器上運行，任務就是把你提供的題目，透過一條經過細心編排的流水線，變成一份詳盡的報告。它不會只吐出一個答案就算完事，而是一步一步走過一連串步驟：定義範圍、發掘來源、篩選出優質的、抽取內容、組裝成一本主筆記、合成草稿、按需要保存來源的完整離線副本、打磨最終報告、編撰知識庫、生成溫習練習、製作配圖提示，再寫出一份 YouTube 講稿。每一步都會留下一個可供查閱的檔案，所以沒有任何東西藏在黑盒裡面。

它的設計理念建基於幾項清晰的原則。它在關鍵的決策點保留人手控制。它把所有東西都存放在本地的工作階段資料夾裡，讓你可以暫停再續做。它在所有語言生成的工作上，都透過一個與 OpenAI 相容的介面，使用 xAI 的 Grok 模型。它可以按需要引入你自己的本地文件來引導研究。而最重要的是，它產出的是你可以逐一查看的中間產物，而不是一個密不透風的單一結果。

同樣重要的，是要明白這個系統並不打算做甚麼。它沒有網頁介面、沒有 API 伺服器、沒有多用戶協作。它沒有登入、沒有權限、沒有角色。它不用資料庫。它沒有語意向量搜尋，也沒有檢索索引。除了簡單直接地抓取網頁之外，它不會操控瀏覽器，也不會運行爬蟲。它不保證模型產出的內容在事實上正確。而當你餵一份文件給它，它會盡力讀取當中的文字，但對於文字以外的二進位格式，就不會作深入處理。

講講誰會用它。其實這裡就只有一種人：研究操作員。也就是那位開啟工作階段、沿途批准或修訂輸出、揀選保留哪些來源、決定是否保存完整離線副本，以及運行各項附加指令的人。另外還有一個概念上的審閱者或學習用戶，他會去消化那份報告、那些練習、那張知識地圖等等，但技術上來講，那其實是同一個人。在系統那一邊，相關的角色有語言模型供應商，即是 Grok；有遠端內容主機，即是它抓取的網站和 PDF 端點；有存放一切的本地檔案系統；還有提供 API 金鑰和文字編輯器等東西的本地環境。由於沒有身分或權限這一層，任何能夠運行這個程式並讀取工作階段資料夾的人，都可以做齊所有事。

在架構上，這個系統由幾個模組組成。有一個命令列層，負責解析你的參數，並把任何失敗對應到結束代碼。有一個工作階段管理員，負責建立和儲存工作階段狀態，並提供標準的資料夾路徑。有一個工作流程模組，負責實作整個狀態機，處理抓取、抽取、合成、編撰等等一切。有一個 Grok 用戶端，負責載入設定、發出呼叫，並把 API 錯誤翻譯成有意義的領域錯誤。有一個外部文件模組，負責遞迴地讀取你的本地檔案，並抽出引導用的脈絡。還有一套提示範本，界定每一次語言模型呼叫應該產出甚麼。整件事是單一程序。每一條指令都會建立一個新的運行資料夾，在工作階段目錄裡的檔案上工作，並寫出運行本地的副本以便追溯。並沒有甚麼背景服務在嗡嗡作響；狀態完全活在檔案之中。

技術方面，它需要 Python 3.11 或更新版本，並倚賴一組廣為人知的函式庫，去跟模型溝通、讀取環境檔案、輸出美觀的終端機畫面、驗證資料、解析 PDF 和 Word 文件、抓取網頁，以及從凌亂的 HTML 裡抽取可讀的文字。它會安裝成一條你可以直接呼叫的指令，另外還有兩個方便用的包裝，會轉發到同一個進入點。

設定是透過環境變數來處理。最重要的是 Grok 的 API 金鑰，任何會呼叫模型的動作都需要它；如果缺少了，這些動作就會以一段清晰的訊息失敗。你可以按需要設定模型名稱，預設是 Grok 3，而空白值會悄悄退回這個預設。有一個最大輸出 token 的設定，預設為五萬，無效的值會退回預設，而低於一的值就會被抬高到一。有一個請求逾時設定，預設為三百秒，套用同樣的保護規則。還有一個編輯器變數，只在你選擇親手編輯範圍時才會用到；如果未設定，系統依然會建立那個可編輯的檔案，只是不會替你開啟編輯器。當工作流程建立它的預設用戶端時，它還會在工作階段目錄上兩層的位置尋找一個環境檔案，如果找不到，就乾脆改用程序環境。

介面本身是一個終端機體驗，用一個精緻的輸出函式庫來顯示狀態訊息、回顯 Markdown 內容，以及展示預覽表格。沿途有四個人手互動點：確認範圍、批准篩選好的來源、批准或修訂草稿，以及揀選哪些來源要離線保存。但它也有一個無人看管模式。開啟自動模式，系統就會繞過所有提示，自行推進到完成。在那個模式下，範圍會自動確認，每個來源都被選取並批准，草稿自動獲批，而完整收集的選擇預設為全部，除非你另有指示。最關鍵的是，自動模式絕不會停下來等你輸入。

讓我們講講各項指令，因為這裡正正展現出系統的能力。除了那條只負責列出類型的指令之外，每條指令都要你指定一個工作階段目錄，而那些針對既有工作階段運作的指令，還要你提供一個工作階段識別碼。程式在成功時回傳結束代碼零，在工作流程引發 Grok 錯誤或配額錯誤時回傳一，在遇到無法辨識的指令或參數驗證失敗時回傳二。而如果你開啟追蹤，它會以截短並經過淨化的形式，印出請求和回應的內容，讓你看到正在發生甚麼事。

start 指令會用你的題目、一個可選的焦點、一個可選的外部文件資料夾，以及一個儲存起來的模式，去建立一個新的工作階段。它會印出新的工作階段識別碼，並即時由第零階段開始整個工作流程。可接受的模式有報告、編譯器和操練三種，不過這裡要老實說一句：模式雖然會儲存，但在目前的實作裡並不會真正改變運行時的行為。resume 指令會載入一個工作階段，由它的目前階段接續做下去；在互動模式裡，它會在下一個人手檢查點停下，或者在某個叫你再續做的階段之後停下，而如果工作階段已經完成，它就會直接告訴你。有一條指令負責列出所有工作階段，它會掃描那些含有工作階段檔案的資料夾，跳過任何不是有效工作階段的，並在一個都沒有時告訴你。還有一條列出類型的指令，它只會印出那個唯一支援的類型，而且不需要工作階段資料夾。

接著是那些較專門的指令。update 會重新運行發掘，只尋找上次運行以來的新材料，把階段退回到篩選，並叫你續做。synthesize 會強制合成階段運行，不論你目前在哪個階段，但它的前置條件依然成立，所以沒有筆記就不會進行。compile 會建立知識庫，接受 auto-hypergraph 這個類型，從主筆記和任何已抽取的內容著手，如果兩者都不存在，就以一段有用的訊息停下。drill 會生成溫習問題，只支援後向模式；如果核心概念檔案缺失，它會嘗試自動編譯，而如果之後仍然找不到，就會叫你先運行 compile。feed 會接收一份新文件，要你指明位置，檢查它存在而且是真正的檔案，把它連同一個時間戳記複製進知識庫，而如果沒有既有的知識地圖，它就會編譯一個出來然後返回。show 會把知識地圖變成一張 Mermaid 圖，或者在沒有東西可繪時叫你先編譯。generate-images 和 youtube-script 都需要一份最終報告；如果缺失就會明說，而在成功時，它們會把輸出寫進運行資料夾和工作階段資料夾兩處。

現在講講工作階段是怎樣識別和儲存的，因為可續做這一點是核心所在。每個工作階段識別碼，都來自把你的題目轉成簡化字串，再加上當天的日期。所謂簡化字串，就是把題目轉成小寫，把任何不是字母或數字的東西換成連字號，把重複的連字號收合，再把兩端修剪乾淨。如果這串字太長，它會被修剪，並附上一個短短的雜湊後綴。而如果已經有一個同名的資料夾，系統就會加上一個編號後綴，直至找到一個未被佔用的名字為止。工作階段狀態本身會記下一組豐富的欄位：識別碼、題目、焦點和模式；一整族關於任何外部文件的細節，包括它們的狀態、摘要、檔案數量、完成率、相關度分數和最後一次錯誤；建立和更新的時間戳記；模型名稱；目前階段；以及一個運行歷史清單。狀態以 UTF-8 的 JSON 儲存，更新的時間戳記在每次儲存時刷新，資料夾也會按需要自動建立。這裡同樣要老實說一句：那個運行歷史清單雖然存在於結構之中，但目前的程式碼其實並沒有把它填上。

每一條建立工作流程脈絡的指令，都會建立一個全新的運行目錄，名字用的是精確到微秒的時間戳記。萬一兩個時間戳記真的相撞，它會重試多達一千次，而如果到頭來仍然找不到一個獨一無二的名字，它寧願引發一個錯誤，也不會覆蓋任何東西。

其中一項較為強大的功能，是外部文件的前置處理。它的構思是，在工作流程運行之前，你可以餵自己的本地參考文件給它，而它會把這些文件轉化成必須遵從的引導脈絡，去塑造範圍、發掘、篩選、抽取和規劃。這個前置處理會在大部分指令之前自動運行，不過在 generate-images、youtube-script、compile、drill、feed 和 show 這幾條之前就不會。如果你沒有替它指明一個資料夾，它就會被略過。而如果它已經成功完成，並有一份摘要在檔，它就不會多此一舉再運行一次。它支援 PDF、Word 文件、純文字和 Markdown，以遞迴方式發掘，而對於它看不懂的檔案類型，它只會略過，而不是報錯。

處理本身相當周到。每份檔案都會以適合其類型的邏輯來讀取。對於 PDF，它會逐頁處理，跳過任何抽不出文字的頁面。對於 Word 文件，它會把非空白的段落連接起來。對於文字和 Markdown，它會以 UTF-8 讀取，並把任何無效字元替換掉。每份文件都會根據檔名裡的關鍵字，歸類為指引、背景、引導或一般文件。處理器接著會用句子層面的啟發法，抽出關鍵概念、約束、要求和演算法見解，並計算一個相關度分數，依據是文件的詞彙與你的題目和焦點有多大重疊，再加上相關詞語、約束和要求所帶來的額外加分。結果會寫成三樣東西：一份載有逐檔結果和整體指標的清單、一份抽取摘要，以及一份脈絡檔案，當中匯集了關鍵概念、約束、要求、可選的演算法註記和工作流程指引。那些演算法註記，只在你的題目或焦點與演算法導向的關鍵字相符時才會出現。如果那個資料夾並不存在，狀態會標記為失敗，並儲存一段解釋，工作流程則照樣繼續。如果個別檔案失敗，它們會被標記為失敗，但其餘的照樣進行。整體狀態在全部成功時為完成，在有些成功有些失敗時為部分完成，在全無成功時為失敗。而當這份脈絡可用時，它會被附加到相關的提示裡作為必須的背景，並按每個階段的字元預算來截短，而絕不會引致失敗。

現在來到重頭戲：研究工作流程本身，這是一個有九個階段的狀態機，編號由零到八。第零階段是範圍生成和確認。第一階段是發掘。第二階段是篩選和缺口分析。第三階段是抽取。第四階段是筆記組裝。第五階段是合成和審閱。第六階段是完整離線收集。第七階段是最終打磨。而第八階段就代表完成。在互動模式裡，每一次續做的呼叫，都會處理一個階段或一個人手檢查點，而有些階段會刻意以叫你再續做來收尾，而不是一口氣繼續下去。階段的轉換，會在程式碼更新目前階段的那一刻就儲存起來。在自動模式裡，工作流程就一直循環，直至到達第八階段，跨越各階段而毋須分開的續做指令。

讓我們逐個階段走一遍。在第零階段，系統會生成一份 Markdown 的範圍摘要，把它寫進一個範圍檔案，並印到主控台。在自動模式裡，它會即時接受範圍，儲存一份已確認的副本，並推進到第一階段。在互動模式裡，你的選擇有是、編輯或取消。取消會停下而不改變階段。編輯會寫一個暫存檔案，按需要開啟你的編輯器，重新載入並印出編輯後的內容，然後繼續提示你。是會儲存已確認的範圍，推進階段，並叫你續做。而如果連 Grok 用戶端都建立不了，它會印出錯誤連同關於環境檔案的指引，然後返回而不改變任何東西。

第一階段是發掘。系統會用你的題目、焦點，以及它是否只在尋找新材料，去填好一個發掘提示，然後把發掘表格寫進運行資料夾和工作階段資料夾兩處。它在儲存之前不會驗證表格的格式。在正常流程裡，完成發掘會推進到第二階段，並叫你續做以進行篩選。

第二階段是篩選和缺口分析。它需要那張發掘表格；沒有它，就會停下並叫你由第一階段續做。它會印出發掘輸出最多頭八十行非空白內容的預覽。你的指示可以是自由格式的文字，按編號揀選來源、說全部、加入網址、移除編號，或者要求做缺口檢查，而它並不會在本地解析這些，反而會直接傳給模型。它會嘗試生成篩選來源，最多三次，而在一次失敗之後的重試裡，它會加上更嚴格的指示，要求只回傳 JSON，並把清單上限定在二十項。每個篩選來源都會被規範化成一個整齊的物件，含有標題、網址、類型、相關理由、可信度和優先次序。網址會經過正規化，修剪掉零散的引號和結尾的標點。如果模型在每一次嘗試都回傳無效或不合規的資料，系統就會從發掘的 Markdown 裡啟發式地復原網址，並改為建立後備項目。原始輸出會原封不動地存進運行資料夾，而一個規範版本則會寫進工作階段資料夾。缺口分析永遠都會嘗試並儲存，而如果它逾時，所儲存的報告仍會含有一個缺口標題和一段明確的逾時註記。工作階段只在你的批准正正是「批准」這個字時，才推進到抽取；其他任何回應都會把它留在篩選，並叫你遲些再試。

第三階段是抽取，也是最費力的一環。它需要那個篩選來源檔案，而如果那個檔案存在卻產出一份空清單，它就會叫你重新篩選。它會在兩個範圍裡，建立用來放抽取檔案、來源快照和抽取片段的資料夾。它會向模型索取一份抽取計劃並儲存，而如果這一步逾時，它會儲存一個佔位內容而不是失敗。它會用最多四個抓取工人並行預先抓取來源組合，而如果個別抓取失敗，它會印出一個警告然後繼續。對於每個成功抓取的來源，它會同時儲存原始內容和一份正規化的文字快照，標頭會保留標題、網址、主機、類型、優先次序和可信度。HTML 快照會以 HTML 檔案儲存，PDF 以 PDF 儲存，其餘一切以文字儲存。來源文字會切成最多四萬五千字元的片段，並有五千字元的重疊，而這些片段會用最多兩個工人並行抽取。每個片段提示都會要求嚴格的段落，涵蓋覆蓋摘要、術語、機制、工作流程、證據、限制、開放問題、可引用段落和抽取註記。如果一個片段逾時，它會被跳過，其餘繼續。成功的片段會寫進兩個範圍，成功的來源會組裝成編號的檔案，而如果某個來源的每一個片段都失敗，那個來源就會被跳過並附上警告。當這個階段完成時，會寫出一個帶有生成標記的索引檔案。

第四階段是筆記組裝。它需要那個抽取資料夾，而沒有它，就會叫你由第三階段續做。這本筆記包含一個頂層標題、一行題目、一個用途段落、一個可選的外部文件脈絡段落、一份來源目錄、一個可選的知識庫大綱，以及那些來源檔案，全部用分隔符接合起來。它會寫進兩個範圍，而一次成功的組裝會推進到第五階段。

第五階段是合成和審閱。它需要那本主筆記，會把它切成最多七萬字元的片段，並有五千的重疊，而如果沒有片段產出，它就會告訴你筆記是空的。對於一張固定清單上的每一節，它都會從筆記片段裡建立該節專屬的證據包。那些標準節是：核心定義與範圍；架構與技術機制；工作流程、過程與運作模式；證據、例子與案例研究；限制、取捨與失敗模式；以及開放問題與未來方向。證據生成每一節最多用兩個工人，證據包會儲存到兩個範圍；如果某一節產不出證據，就會被跳過並附上警告。每一節擬好的草稿都會儲存，而草稿報告會匯集範圍與覆蓋的文字、來源目錄、擬好的各節、一個可選的知識庫對齊，以及參考資料。草稿會以遞增的版本號儲存。審閱提示會告訴你，你可以批准、連同意見修訂某一節、按標題加入一節，或者運行缺口檢查。只有「批准」這個確切字眼才會推進到第六階段；其他任何東西都會被當作一般修訂意見，並傳給修訂提示而不在本地解析。如果一次修訂逾時，先前的草稿仍然作準，階段不會推進，而一次成功的修訂會成為下一個版本，並需要再走一輪審閱。

第六階段是完整離線收集。它會嘗試載入篩選來源，而如果它們不在，它會退而求其次，由發掘表格復原網址。如果它甚麼都復原不了，它會推進到第七階段、儲存、印出一段略過訊息，並等你續做以作收尾。否則，它會展示一張選擇清單，列出每個來源的編號、標題和網址。你可以說全部、無，或者給出以逗號分隔的數字，而非數字的標記會被忽略。說無會推進到第七階段並略過。說全部會選取一切。超出有效範圍的數字會被忽略。被選的來源會被預先抓取，而每一個成功抓取的，都會把一份完整離線副本寫進兩個範圍。如果某個被選的來源抓取不到，它會被跳過而不會中止。一旦那些嘗試過的副本寫好，系統就會推進到第七階段，即時運行最終打磨，然後在第八階段把工作階段標記為完成。

第七階段是最終打磨。它需要那本主筆記以及至少一份草稿，否則它會說它們缺失。它會生成一份行政摘要，如果這一步逾時就換上一個佔位內容，並生成一份詞彙表，逾時同樣換上佔位內容。如果最新那份草稿以一個頂層標題開頭，那個標題會在組裝之前移除。它會從正文裡所有第二層標題建立一個目錄。最終報告於是會含有主標題、目錄、行政摘要、正文、來源目錄、一個可選的知識庫概覽，以及詞彙表。系統會按需要嘗試兩次去達到字數目標，一次針對正文，一次針對組裝好的報告，目標是介乎九千到一萬字之間，理想值是九千五百，而這個校正會保留標題和核心論點，同時擴充或壓縮其餘部分。最終報告會寫進兩個範圍，然後系統會嘗試生成配圖提示和一份 YouTube 講稿。第八階段就只是代表工作階段已完成，而續做一個已完成的工作階段，就只會這樣告訴你。

關於抓取和轉換來源，有一套仔細的規範。網址在驗證之前會先正規化，而只有帶有真正網路位置的 HTTP 和 HTTPS 位址才會被接受；其他任何東西都會引發錯誤。抓取會用一個自訂的用戶代理、跟隨重新導向，並把逾時拆成連線和讀取兩部分，在需要時引發一個帶有網址脈絡的逾時錯誤。PDF 會憑其內容類型或檔案後綴來偵測，並以幾種形式回傳抽取出來的文字。非 HTML、非 PDF 的回應會當作純文字。HTML 回應會產出一段來自可讀性抽取的主文字、一段來自整頁的全文字，以及一段把兩者合併或妥善退回的分析文字。在 HTML 正規化期間，script、style、noscript 和 SVG 標籤都會被移除，而重複的行會被剔除，以削減重複的樣板內容。

接著是知識編撰，它會把你的研究化成結構化的知識。編撰會先用筆記，並在有需要時附上抽取出來的檔案。超圖只會用內容的頭二十二萬字元，而核心概念抽取則會用同一段來源內容，再加上超圖 JSON 的頭十二萬字元。輸出有一個超圖檔案、一份自動類型副本，以及一個核心概念檔案。超圖遵循一個結構，由帶有識別碼和標籤的節點，以及把多個節點以一個關係和佐證連起來的超邊組成。如果模型沒有回傳有效的 JSON，系統會把原始回應包起來，而不是失敗。核心概念的結構會要求一批概念，每個都有名稱、定義，以及一段解釋它為何舉足輕重的說明，而雖然提示要求剛好七個，程式碼卻不會獨立地強制這個數目。操練包的結構會產出一份 Markdown 溫習包，再加上按概念分組的結構化問題，每題都有答案和常見陷阱；如果那個 Markdown 欄位缺失，系統會從原始回應裡剝掉程式碼圍欄並改用那部分，而如果沒有結構化問題，它就會改為把整個解析出來的物件寫出來。

餵文件的做法，是以 UTF-8 讀取那份新文件，並把任何解碼錯誤替換掉，然後把既有超圖和新文件兩者的頭十六萬字元送給模型，而更新後的超圖會覆寫兩個標準位置。Mermaid 繪製會以一個圖形宣告開頭，用上最多頭二百個節點和頭四百條邊，而對於成員多於兩個的超邊，它就只把所列的頭兩個連起來，並按關係或標籤來為邊加上標籤。

最終的輸出界定得很清楚。報告是一份 Markdown 文件，帶有明確的行政摘要和來源目錄兩節，在有相關內容時附上一個可選的知識庫概覽，並在末尾附上一份詞彙表，即使它的生成曾經逾時。配圖提示由完整的報告生成，合約要求五到十個提示，著重具體的機制、工作流程、架構、比較和證據，而不是泛泛的概念美術，而如果這一步在打磨期間逾時，報告依然會成功。YouTube 講稿主要在有的時候由各節草稿來取材，否則就由最終報告來取材，並刻意排除目錄、來源目錄、詞彙表、參考資料、知識庫概覽和行政摘要這些節。講稿含有一個頂層標題、一段引言、每一個被選節各一個標題，以及一段結語。如果引言或結語逾時，會插入佔位旁白而不是失敗；如果某一節逾時，它可能會被省略，而其餘的照樣進行；過短的輸出會由第二次呼叫加以擴充，以達到最低限度的細節；而如果某一節欠了一個標題，系統會自動補上一個。

說一說驗證。所需的指令旗標由參數解析器強制執行，而不獲支援的類型或模式值，根本連解析器的驗證都過不了。語意上的驗證是刻意從簡的：篩選來源的檢查是結構性而盡力而為的，發掘輸出不作結構驗證，最終報告不作事實正確性的檢查，而概念的數目就只受提示約束。檔案驗證會在要緊的地方發生，餵文件的指令會檢查檔案是否存在以及類型，外部文件會驗證根資料夾和支援的後綴，而工作階段的列出則會檢查工作階段檔案。

錯誤處理正是系統展現其成熟之處。缺少 API 金鑰會引發一個清晰的錯誤。配額和帳單問題會對應到一個帶有可行文字的配額錯誤。逾時類的失敗會變成一個包含已設定逾時秒數的逾時錯誤。其他 API 失敗會以指數退避重試多達五次，上限為三十秒，而配額和逾時錯誤一旦被辨識出來，就不會重試。有若干階段會用上一個可選呼叫的輔助工具，把模型逾時轉化成警告然後繼續，涵蓋諸如缺口分析、抽取計劃、抽取片段、證據包、各節草稿、修訂、行政摘要、詞彙表、配圖提示、YouTube 的各個部分，以及字數重訂。來源抓取失敗永遠不會中止整個階段。而 JSON 解析也很穩健：系統會剝掉程式碼圍欄，先試直接解析，再試方括號切片，再試大括號切片，最後才退回把原始文字包起來。貫穿其中的主題就是優雅地降級，外部文件缺失、篩選來源缺失、知識產物缺失、報告缺失，全部都會產出有用的訊息而不是當機。

講到整合，Grok 的連接用的是在 xAI 基底網址上、與 OpenAI 相容的對話 API，以環境裡的持有人金鑰來驗證，每次呼叫送出一條系統訊息和一條用戶訊息，並讀取第一個完成選項。網頁整合是純粹的 HTTP 和 HTTPS GET，啟用重新導向、不作驗證、SSL 交由請求函式庫處理，而失敗則要麼往上冒，要麼按設計被降級為警告。本地文件整合支援 PDF、Word、文字和 Markdown，而餵文件的指令在開啟檔案的層面上更為寬鬆，但其實是針對文字內容而設。

保安和私隱以務實的方式處理。API 金鑰由環境或環境檔案讀取，絕不會寫進工作階段產物。但工作階段目錄可能會存放抓取回來的遠端內容和處理過的外部文件，這些應視為具潛在敏感性，因為系統不會對這些資料夾作任何秘密遮蔽，也不會作任何存取控制。另外還有一些老實的非功能性現實。由於它本地優先，每一件重要的產物，在每一個主要步驟之後都必須能在磁碟上查閱。可續做這一點，靠的是目前階段和哪些檔案存在，而不是交易日誌或資料庫。決定性只是局部的：檔名和轉換是可預測的，但內容由模型生成，因此帶有機率性。並行是有界限的，四個抓取工人、兩個抽取工人和兩個證據工人。而大段文字靠的是以字元為本的截短和切片，而不是精確的 token 計數。

開發團隊對目前的缺口坦白得令人耳目一新。模式設定雖然儲存了，但暫時不會改變行為。運行歷史欄位存在，但未獲填上。類型清單就只暴露那一個 auto-hypergraph 類型，儘管內部常數提到其他幾個正在沉睡。那些指引字串提到加入一節或運行缺口檢查，但並沒有本地解析器去強制執行，它們只是被當作修訂意見傳遞下去。目錄就只由第二層標題建立。Mermaid 會把多成員的超邊簡化成成對的連結。而事實正確性歸根究柢取決於模型和來源，因為並沒有自動的事實核查。

那麼，怎樣才算完成？驗收準則十分具體。你可以建立一個帶有獨一無二識別碼和已儲存狀態的新工作階段。互動式推進可以帶著預期的人手檢查點，把一個工作階段由第零階段一路移到第八階段。自動模式可以在從不要求輸入的情況下完成工作流程。發掘會建立它的表格。篩選會建立篩選來源檔案和一份缺口報告。抽取會建立快照、片段和檔案。筆記組裝會建立那本主筆記。合成會建立至少一份草稿。最終打磨會建立最終報告。打磨或明確的指令會建立配圖清單和 YouTube 講稿。編譯會建立超圖和核心概念。操練會建立操練包和問題。餵文件會存下一份帶時間戳記的副本，並能更新或初始化超圖。show 會建立那個 Mermaid 檔案。而外部文件在獲提供時，會被處理成一份清單、一份摘要和一份脈絡，而絕不會因為局部失敗就卡住工作流程。

這就是完整的藍圖。Research Agent 最令人印象深刻的，是它的紀律：每一步都是一個你可以打開的檔案，每一次失敗都有一個優雅的後備，而人就在最要緊的那些時刻留在環節之中。它接過一個題目，再一步一步小心翼翼地，把它變成一份你真正信得過、又可以據以發展的成果。
