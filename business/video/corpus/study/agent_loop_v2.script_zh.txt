今天我们一起来讲解一套经过提炼的 agent loop 设计。它不是玩具式的演示，而是那种真正可靠、达到生产级水准的系统，可以实际执行深度研究、编写代码，还能统筹一支各有专长的 agent 团队，而不会散架。你可以把它当成一份实战指南。它把学术基础、来自真实世界失败的教训，以及像 xAI 这类公司在服务器端采用的统筹模式，全都结合在一起。

我们先从背后的核心理念讲起。这个版本是在大量研究之后整理出来的，研究的重点是 agent 系统究竟因为什么而崩溃。它参考了关于已知失败模式的研究、反思和自我批评的框架、记忆体架构，以及在生产环境中已经得到验证的模式。目的很简单，就是给构建者一份可实际操作的参考，去打造既可靠又能够扩展的 agent 系统。

指导原则是这样的。我们想要受控制的循环，配上明确的状态、结构化的输出、质量检查，以及层与层之间清晰的委派。我们是刻意不去建立失控的连锁反应。相反，我们追求的是有管理的统筹，结果由底层逐层向上汇集，再在顶层小心地整合。

那我们就从根基讲起，这个范式叫做 ReAct，名称来自把推理和行动结合在一起。这个理念由 Yao 和他的团队在二零二三年提出，做法是把语言上的推理，也就是我们所说的"思考"，和具体的行动交织在一起，例如调用工具、与环境互动，或者把工作交给另一个 agent。最关键的是，从这些行动返回的观察结果，会直接反馈到推理那里，并且更新它。

为什么这么有效呢？我们想一下其他做法。如果你只靠纯粹的思维链推理，模型是静态的，容易产生幻觉，而且错误会越滚越大，因为没有任何外部的东西去制衡它。另一方面，如果你只靠行动，就会缺乏高层次的规划、处理异常情况的能力差，行进的路径也会没有效率、四处游荡。ReAct 的奥妙就在于协同效应。思考负责拆解目标、追踪进度、应对突发情况和重新规划。行动则带回真实的观察结果，去纠正推理，让 agent 可以适应。实际上，这种做法在交互式任务上带来显著的提升，在知识型任务上也减少了幻觉。

我们想象一下这个循环完整的一次循环。第一步是思考，模型在那里推理当前的情况、目标、目前的进度，以及下一步要做什么，或者有没有出现什么问题。这一步是内部的，会更新 agent 的上下文。接着是行动，模型决定并输出一个可执行的步骤。可能是带着参数的工具调用、委派给一个子 agent，又或者是表示工作完成的信号。再来就是观察，由环境、工具或者子 agent 返回一个结构化的结果。这个结果除了数据之外，还带着元数据，例如状态、置信度、摘要，以及任何问题。最后，所有这些都会加入到持续累积的历史记录当中，然后循环再来一遍。

讲到提示，少量示例是不可或缺的。对于推理量大的工作，例如问答和研究，你想要密集而丰富的思考；对于决策或者偏重行动的任务，思考就可以稀疏一些。而且自始至终，agent 都应该优雅地处理异常情况。如果一个思考步骤发现没有任何有用的东西返回，它就应该重新规划或者调整下一个行动，而不是硬着头皮继续往前冲。

那么在一个真实的生产系统里，这会是什么样子呢？看一下 xAI 的 Grok。它的服务器端工具调用，在内部就实现了一个 ReAct 风格的循环。模型决定用哪些工具，在服务器上执行它们，例如网页搜索、代码执行和搜索数据集，然后不断反复迭代，直到可以交出最终答案为止。客户端只会看到最终输出，可能是流式形式，再加上可选的推理 token。

xAI 还把这个理念推得更远，搞出了多 agent 统筹。一个多 agent 模型可以启动可配置的团队，可能用四个 agent 处理快速而聚焦的任务，又或者用十六个去处理深入而全面的工作。这个循环是这样运作的。在服务器上，多个各有专长的 agent 并行运作，实时协作。每一个都贡献它自己的推理、工具调用和发现。然后一个领导 agent 把整个讨论综合起来，相互对照所有资料，再交出一个单一的结构化答案。工具可以并行调用，系统也会根据沿途学到的东西不断迭代。为了控制和安全，子 agent 的内部状态默认是加密和隐藏的。只有领导 agent 的输出才会公开。

这里的强项，就是深入的多步研究、整齐的结构化输出例如表格和对比、实时的修正，以及自动化的工具使用，不需要客户端插手进循环里面。在某些 xAI 工具里面也有一种规划优先的味道。比如 Grok Build 命令行工具，会先生成一个明确的计划，然后并行运行子 agent，每一个都隔离在自己的代码工作空间里面。

在这一切之上，再叠加一个分层、能够自我演化的结构。最顶层坐着一个中央规划者，也就是统筹者或者监督者。它把工作拆解成子任务，再委派给各有专长的子 agent，例如一个深度研究员、一个分析员、浏览器或者工具 agent，以及一个汇报员。每一个子 agent 都运行它自己的循环，可以是 ReAct 风格，也可以是针对它专业领域而调校过的。路由形成一棵树，结果就逐层向上汇集。

这里有一个很有用的协议灵感，有时叫做工具—环境—agent 协议。它的理念是把工具、环境和 agent 都当成一等公民，每一个都有版本管理、有生命周期管理，并且受标准规则去规范它们的上下文、调用和演化。

接着就是闭环的反馈循环，这一部分就是让系统可以自我改进的关键。它包括反思，agent 用日常的语言去批评自己的执行轨迹。它也包括基于轨迹的优化，把错误归因、提出修改建议，再在保留的样本上验证这些修改，才注册成为一个新版本。一个版本管理器会追踪改良过的提示、工具和 agent，支持回滚，又或者挑选最好的选项。还有一个追踪器，会记录完整的执行轨迹，既可以用来审核，也可以做优化的信号。

规划者也负责整合。它把子结果汇集起来，协调证据、化解冲突，再更新全局的计划和状态，又或者触发另一轮的修正。很多时候会有一个专门的汇报员 agent 去处理最终的综合，连引用和去重都做好。表现的证据相当鼓舞人心。这样构建的系统在严苛的基准测试上拿到高分，而子 agent 加上自我演化的组合更带来可观的提升，至于分层路由在扩展性方面就远胜过一大堆扁平排列的 agent。

所以整个提炼过的模型看起来是这样。从 ReAct 核心循环开始。为了应付复杂度，在上面叠加分层委派。再加入一个明确的规划阶段、反思和批评的关卡，以及带版本管理的结构化状态，去达到生产级的可靠性。正如 xAI 所示范的，这一切都可以在服务器端运行，并且内建强大的统筹能力。

现在我们老实地聊一下究竟哪里会出错，因为真正的教训就在那里。最近有一些系统性的研究，特别是一份大型分类，是通过分析超过一百五十条横跨多个热门多 agent 框架的轨迹而建立出来的，他们发现了一样让人印象深刻的东西。大部分失败都不是因为模型不够聪明，而是来自设计和规格上的问题、协调上的崩溃，以及验证和终止机制薄弱。事实上，超过四成的失败仅仅追溯到设计和规格的问题。

我们逐一讲一下主要的类别。最大的那个是规格和设计上的含糊。也就是 agent 违反或者误解任务规格、角色定义模糊，又或者缺乏成功标准和输出契约。后果是 agent 很早就跑偏了，而这些错误会在下游不断累积。解决方法在循环开始之前就要开始。你要把一份结构化的任务规格定为必须，其中包括明确的成功标准、约束条件、输出格式和质量阈值。你把它当成一份可以更新的"活规格"。你在循环开始前加入对这份规格的自动验证，并且在统筹者和子 agent 之间定义清晰的角色和信息契约。

第二个类别是无限循环、重复行动和空转。也就是 agent 不断重复相同或者相似的行动，但是毫无进展。在 ReAct 循环里面，当异常处理做得差，或者信息缺失的时候，这种情况很常见，甚至可以由提示注入触发。代价就是浪费 token、超时，以及很多现实中的挫败感。缓解方法包括循环检测，把最近的行动和观察做哈希处理，一旦发现相似度太高就强制重新规划。你设定明确的步数和反思轮数上限，并且允许根据进度提前退出，例如待办清单完成的百分比。你限制 agent 可以尝试改进某样东西的次数。你还要求一个完成动作，在被接受之前必须通过验证。在分层的设定里面，统筹者会监察它的子 agent，可以中止或者重新分配一个卡住了的分支。

第三个类别是上下文窗口爆炸，有时叫做上下文腐化或者历史臃肿。在一条很长的轨迹里面，重要的早期信息或者指示会被丢失，导致前后不一致、重复，以及目标漂移。解决方法就是积极的分层记忆。你把短期工作记忆和一个带向量搜索和语义缓存的长期持久存储并用。你在里程碑或者每当上下文超过某个阈值的时候做摘要。你依赖结构化的状态，例如一个任务文件、一张待办清单和一些关键事实，而不是每一个回合都把整段历史全倒出来。你也只给每个子 agent 它相关的上下文片段，连同来源出处。

第四个类别涵盖幻觉、错误累积，以及薄弱的验证。捏造的事实和误读的工具结果会扩散开去，在多 agent 的环境下情况更糟。应对方法就是把验证者或者批评者 agent 定为必须的质量关卡，在整合期间，以及子结果返回之后，都要把关。你用一个带状态、置信度和问题清单的结构化观察格式，并且跨 agent、跨来源做交叉验证。你把论点扎根于观察结果，再在外部加以核实。你可以把多次尝试排名，让一个批评者挑出最好的那个。而在自我演化里面，你只会在一个改动通过保留轨迹验证之后才去落实它。

第五个类别是多 agent 系统独有的，就是 agent 之间的错位和协调失败。这会表现为 agent 越权、追求相互冲突的目标、共享过时的状态、沟通缺失，以及相互传播对方的错误。有时，一个单一而强大的 agent 反而会胜过一个复杂的多 agent 系统。缓解方法是要有一个强大的中央统筹者，带着明确的拆解和路由，因为分层控制胜过扁平结构。你定义信息契约和结构化的交接格式。你采用带版本的共享状态，以及像流或者发布订阅这类耐用的协调工具。你加入断路器，一检测到不一致就暂停，去调和或者上报。你也要倾向采用清晰、定义明确的专家角色。

第六个类别关乎终止和目标漂移，也就是太早停下、工作未完成，又或者认不出工作其实已经做完。解决方法是在规格里面定明确的成功标准，并且对照着它追踪进度；设一个专门的终止动作，必须通过验证者；在思考步骤里面定期对照原本的目标做对齐检查；以及当中间结果已经满足标准的时候就提前终止。

还有几个值得一提的其他问题。状态过时和记忆失效，就需要混合记忆，把快速的短期记忆和持久的长期检索结合在一起。安全方面的顾虑，例如提示注入触发循环或者误用，就需要沙箱化的工具、输入清理、最小权限访问，以及监察异常的循环。成本和扩展性的额外开销，就意味着你只应该在好处大于协调成本的时候才去用多 agent，同时要监察 token 用量，并且只在安全的地方才做并行。而可调试性，就意味着一个完整的追踪器和结构化日志，是绝对不能省略的。

那所有这些缓解方法又怎样融入循环本身呢？在初始化阶段，规格工程和验证是回报最高的那个修正。在核心循环，你加入循环检测、有边界的步数和反思、结构化观察，以及进度追踪。在委派阶段，你用狭窄的子规格配上契约，加上统筹者的监察。在整合阶段，你加入验证者和批评者关卡，再加上交叉验证。在反思和自我演化阶段，你在套用任何改动之前先验证，并且让循环保持有边界。在终止阶段，你要求有一个验证者，以及一个有证据支持的明确完成。所有研究带出的关键洞见是这样。修好规格质量、验证层，以及明确的终止控制，带来最大的可靠性提升。光是加多一些 agent 或者加大模型的原始能力，而没有这些东西，往往只会换来递减甚至负面的回报。

现在我们逐个阶段，走一遍完整的 agent 循环流程。

第零阶段是初始化，也就是由规格驱动的设定。目标是在任何迭代运行之前，先确立一份清晰的契约。你解析人类的指令，生成或者验证一份结构化的任务规格，涵盖目标、成功标准、约束条件、输出格式、步数和 token 的预算，以及质量阈值。你建立初始状态，其中包括一个任务文件或者结构化草稿纸，存着目前的计划、待办清单、进度和未解决的问题。你设置记忆，短期的用来存近期观察，长期的用来存检索到的知识和过往的版本。你开始一个追踪器或者执行日志，留待日后反思之用，如果有组件会演化，还要有一个版本登记册。对于复杂的任务，建议先生成计划。统筹者起草一个高层次的计划，以编号步骤或者依赖关系图的形式呈现，通过自我批评或者一个专门的批评者去对照规格验证，再存入状态。最后，你决定架构，简单的工作用扁平 ReAct，复杂的研究或者编写代码用分层，又或者用混合式。这个阶段的输出，本质上就是一份结构化规格，列出任务、它的目标、成功标准、约束条件例如最大步数和 token 预算、想要的输出格式、一个初始计划，以及质量关卡，例如完整度高于九成、没有幻觉、结构化输出。

第一阶段是核心迭代循环，也就是这个系统受控的 ReAct 核心。在任务未终止之前，你重复一个循环。首先你观察当前状态，加载相关的历史、任务规格、目前的计划和待办，以及最新的观察，如果上下文开始变长就积极地做摘要。然后你推理。你对照成功标准分析进度，找出缺口、风险和异常情况，再决定一个策略，究竟是直接调用工具、委派一个子任务、综合手上有的东西、反思和批评，还是完成。如果有需要就更新计划或者待办。然后你行动，产出严格而可解析的结构化输出。选项甲是带着参数调用一个工具。选项乙是委派给一个子 agent，配上一个狭窄的指令、一个上下文片段，以及它自己的成功标准。选项丙是纯内部的，更新状态或者在草稿上运行批评者。而选项丁是完成，如果质量关卡通过就输出最终答案。然后你执行和观察，运行行动，收集一个结构化观察，其中包括状态、数据、一个精简的摘要、一个置信度分数、任何问题、下一步的建议，以及一个轨迹标识符。你把它加入历史，再更新待办和状态。每隔一阵，或者在失败之后，你运行一次轻量的反思，快速地自我检查，问一下轨迹是不是还对齐，有没有明显的修正。

这里有一个特别有价值的模式，就是断路器，是从有弹性的系统工程借用过来的。断路器可以在一个工具、一个 LLM 调用，或者一个子 agent 不断失败的时候，防止连锁式的故障。它有三个状态。在闭合状态，一切正常运作，请求畅通无阻。在开启状态，太多失败之后，断路器会立即快速失败，保护系统免受一个正在出问题的组件拖累。而在半开启状态，过了一个超时之后，它允许有限数量的测试请求，去检查那个组件有没有恢复。如果那些成功，它就返回闭合状态。如果失败，就回到开启状态。实际的建议是，每一种工具类型或者每一个子 agent 角色用一个断路器，再和会做指数退避的重试包装器结合在一起。

围绕这一切，你要建立稳健的错误处理。当一个工具运行的时候，你把它包装起来，让失败会以递增的延迟重试几次，并且让错误以结构化观察的形式返回，而不是让整个循环崩溃。调用子 agent 的做法也一样。如果一个子 agent 返回任何不是成功或者部分成功的东西，你就当它失败，在限制之内重试，再以结构化的方式汇报结果，包括你最后停在哪个断路器状态。

你也想要一个轻量的验证者或者批评者 agent。想象一下一个严格而带怀疑的审核者，它接过原本的任务规格和一个候选输出，再返回一个干净的裁决，包括它是不是通过、一个分数、一张具体问题的清单、可行的建议，以及它自己的置信度。这个裁决决定了这个输出会不会被接受。

你也想要一个简单的自我演化步骤。agent 分析一条执行轨迹，诊断任何失败或者低效的根本原因，找出哪个组件要负责，再提出一个有针对性的修改。如果那个修改通过验证，无论是在保留样本上重新运行，还是重新执行任务，那个新版本就连同它的来源脉络一起注册，同时保留回滚的选项。

至于什么时候停下，终止条件是每一次迭代或者在关卡处做检查。你在成功标准达到和质量关卡通过的时候停下；在到了最大步数、token 预算或者时间上限的时候停下；在有一个带经验证输出的明确完成动作的时候停下；在出现一个需要上报给人类或者更高层统筹者的无法挽回的失败的时候停下；又或者在一个中间结果已经满足目标的时候停下。

第二阶段是分层委派和子循环。当统筹者决定委派的时候，它首先把工作拆解和路由。规划者挑选或者实例化合适的那种子 agent，连同它的专家角色、工具集和提示模板，再建立一份狭窄的子任务规格，是母目标的一个子集再加上相关上下文。然后它调用那个子 agent，可能是用同一个模型但配不同的系统提示，又或者是一个完全不同的模型。那个子 agent 对照它的子规格运行它自己的独立循环，可以是 ReAct 风格或者一个优化过的变体，维持它自己的本地状态和记忆，并且可以再向树的更深处委派，又或者自己调用工具。它做完之后，会返回一个结构化结果，逐层向上汇集，用回同样的观察格式，包括来源出处，也就是哪个子 agent 产出它，以及一份轨迹摘要。母 agent 接着把它记录进全局状态和追踪器，验证并整合它，把它和其他分支合并，通过协调去化解冲突，更新全局计划和待办，再决定下一步，究竟是再委派、直接行动、整合，还是批评。而只要依赖关系允许，例如独立的研究分支，你就并行运行多个子 agent 或者工具。

第三阶段是整合、综合和重组，会在子结果或者重大里程碑之后发生。你把所有相关的观察和计划的进度汇集起来。你做协调，由一个 LLM 或者一个专门的汇报员去合并、去重、相互对照，再把矛盾化解成一个统一的视图。你把它重组成目标的输出形态，无论是一份报告、代码、一个答案，还是一个更新了的计划，并且执行回原本规格的格式。然后你运行一个质量关卡，由一个批评者对照成功标准去评分，检查有没有幻觉和缺口，再提出修正。如果不通过，你就触发一个修正循环，重新规划、重新委派某些部分，又或者自我编辑。如果通过，你就继续，可能再做最后润色。你也把整合好的知识持久化，存入长期记忆和带版本的产物。一个好的整合提示，会要求一个综合专家接过原本的规格、目前的计划和这些子结果，再产出一份更新的进度摘要、化解了的任何冲突、一份最终输出的草稿，以及余下的缺口连同建议的下一步行动。

第四阶段是反思、批评和自我演化，这个是比较进阶的一层。每条轨迹或者在里程碑处，agent 把它的轨迹做摘要，诊断有什么失败了、有什么成功了，再提出改进，可能是提示的修改、工具的修补，又或者新的子 agent 类型。自我演化循环收集那条轨迹，把错误和机会归因，向提示、工具、agent 配置，甚至生成的代码提出有针对性的改动，通过在相似或者保留任务上重新执行并且检查指标去验证那些改动，如果有改善，就把新版本连同它的来源脉络一起注册，同时保留回滚。一个独立而轻量的批评者 agent，可以在不做完整执行的情况下审查草稿和计划，并且可以在关卡处被调用。好处就是在运行期间持续改进，所以系统在相似类型的任务上用得越多，就会变得越稳健。

第五阶段是终止和输出。一旦关卡通过或者达到终止条件，你做一次最终综合，产出符合规格的结构化最终输出，可以选择为用户或者为日志加一份反思摘要，再把完整的轨迹和版本持久化，留待审核、重放和调试。而自始至终，你都保留人类介入的挂钩，用于质量关卡失败、高风险行动，或者预算耗尽的情况。

我们聊一下状态、记忆和基础设施。你的状态结构把任务规格、目前的计划和待办、思考—行动—观察三元组的历史、记忆无论是键值还是向量、版本，以及追踪器，全都汇聚在一起。追踪应该捕捉完整的执行图，谁调用了什么、结果、时序和版本，这样可以做到调试、反思和优化。版本管理，灵感来自那个工具—环境—agent 的理念，涵盖提示、工具、agent 角色和生成的产物，全都有语义上的来源脉络和回滚。在和 xAI 整合的时候，建议就是在研究量大的顶层任务用 Grok 的多 agent 模式，在混合式里面把服务器端的 agentic 工具和客户端的自定义工具结合在一起，在编写代码的 agent 采用规划优先加上在隔离工作空间里面并行的子 agent，并且在可以的时候流式传输推理 token，以求透明。至于生产级的加固，你想要通过 JSON 模式或者受约束的解码去做严格的输出结构、超时和带退避的重试以及在失败组件上的断路器、带监察的成本和 token 预算、对每一个思考行动和观察的彻底日志和可观察性，以及对工具和代码的沙箱化执行。

现在讲一下一个简单的决策框架，什么时候用什么。对于一个简单的事实查询，用一个扁平的单一 ReAct 循环，配上基本的工具调用，就像快速搜索再回答那样。对于多步研究，用多 agent 或者分层模式，配上并行的 agent 和一个负责综合的领导，去做有来源支持的深入分析。对于编写代码或者一个长期项目，用规划优先加上在隔离工作空间里面的分层子 agent，配上一个待办文件，去生成和调试一个完整的应用程序。对于开放式或者创意的工作，用 ReAct 配上反思和自我演化，配上批评者关卡和带版本的提示，去做反复的设计修正。而对于高风险、以可靠性为重的工作，把以上所有东西结合上强大的质量关卡、结构化结果和验证，去做企业级的自动化。

我们重温一下常见的陷阱和它们的缓解方法，纯粹方便快速浏览。对于上下文爆炸，用积极的摘要和带本地子记忆的分层状态。对于无限循环和空转，设硬性的迭代上限、在待办处追踪进度，再用一个可以强制重新规划或者上报的批评者。对于差劲的整合，强制要求结构化的子结果和一个专门的协调或者汇报步骤。对于计划里面的幻觉，把每一个重要的论点扎根于观察，并且在落实一个计划之前先用批评者。对于脆弱的委派，用明确的子任务规格配上成功标准，并且验证返回的结果。而对于缺乏可见性，用完整的追踪配上可选的推理流式传输。

如果你想把整件事浓缩成一句，骨架就在这里。你从指令初始化状态，连同它的规格、计划、待办、记忆和追踪器。你拿一个统筹者。然后在你不应该终止的时候，你观察去构建上下文，你推理和决定一个行动，再根据那个决定，你或者执行一个工具、委派给一个运行自己循环的子 agent、综合并运行质量关卡，又或者完成并敲定输出。你把思考、决定和观察加入历史，更新待办和计划，再可以选择运行一次轻量反思或者一次完整的自我演化。子 agent 的调用，递归地遵循同一个模式，只是范围窄一些。

最后，讲一下来源和下一步。根基性的工作是 Yao 和团队的 ReAct 论文，把语言模型里面的推理和行动协同起来。生产级的模式来自 xAI 的开发者文档，关于多 agent 统筹和服务器端的工具调用。分层和进阶的理念，来自关于用那个工具—环境—agent 协议去统筹多 agent 智能的研究，加上几份关于新兴 agent 架构的综述，还有额外的模式，例如用于自我反思的 Reflexion、规划并执行的变体、树搜索方法，以及各种多 agent 协作框架。

至于怎样付诸实行，从一个极简的 ReAct 框架开始，用你喜欢的任何框架都行。加入一个结构化的观察格式和待办和状态管理。再叠上分层委派和整合。为追踪和质量关卡加装仪表。在重复的任务类型上实验反思和自我演化。再为研究子任务整合多 agent 模式。整件事的重点，就是它是用来做可执行的指导的。把这些模式抄走、改造结构，然后一步步迭代，去打造一个真正可靠的 agent 系统。
