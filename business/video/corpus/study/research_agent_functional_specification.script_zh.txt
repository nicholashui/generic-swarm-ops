想象一下，你只要交给电脑一个题目，然后转身离开，回来时就收到一份精心整理、研究得相当深入的报告，里面还附有知识地图、复习练习、配图构思，甚至一份可以马上开录的视频讲稿。这就是 Research Agent 背后的承诺。在这段视频里，我们会从最初的一个想法开始，一直讲到完成整个成品，带你了解它究竟是怎么运作的。

先说大局。这是一个本地优先的研究自动化工具。它以命令行应用程序的形式，在你自己的机器上运行，任务就是把你提供的题目，通过一条经过精心编排的流水线，变成一份详尽的报告。它不会只吐出一个答案就算完事，而是一步一步走过一连串步骤：定义范围、发现来源、筛选出优质的、抽取内容、组装成一本主笔记、合成草稿、按需要保存来源的完整离线副本、打磨最终报告、编撰知识库、生成复习练习、制作配图提示，再写出一份 YouTube 讲稿。每一步都会留下一个可供查阅的文件，所以没有任何东西藏在黑盒子里。

它的设计理念建立在几条清晰的原则上。它在关键的决策点保留人工控制。它把所有东西都存放在本地的会话文件夹里，让你可以暂停再接着做。它在所有语言生成的工作上，都通过一个与 OpenAI 兼容的接口，使用 xAI 的 Grok 模型。它可以按需要引入你自己的本地文档来引导研究。而最重要的是，它产出的是你可以逐一查看的中间产物，而不是一个密不透风的单一结果。

同样重要的，是要明白这个系统并不打算做什么。它没有网页界面、没有 API 服务器、没有多用户协作。它没有登录、没有权限、没有角色。它不用数据库。它没有语义向量搜索，也没有检索索引。除了简单直接地抓取网页之外，它不会操控浏览器，也不会运行爬虫。它不保证模型产出的内容在事实上正确。而当你喂一份文档给它，它会尽力读取其中的文字，但对于文字以外的二进制格式，就不会做深入处理。

说说谁会用它。其实这里就只有一种人：研究操作员。也就是那位开启会话、沿途批准或修订输出、挑选保留哪些来源、决定是否保存完整离线副本，以及运行各项附加命令的人。另外还有一个概念上的审阅者或学习用户，他会去消化那份报告、那些练习、那张知识地图等等，但从技术上讲，那其实是同一个人。在系统那一边，相关的角色有语言模型供应商，也就是 Grok；有远端内容主机，也就是它抓取的网站和 PDF 端点；有存放一切的本地文件系统；还有提供 API 密钥和文本编辑器等东西的本地环境。由于没有身份或权限这一层，任何能够运行这个程序并读取会话文件夹的人，都可以做齐所有事。

在架构上，这个系统由几个模块组成。有一个命令行层，负责解析你的参数，并把任何失败映射到退出码。有一个会话管理器，负责创建和保存会话状态，并提供规范的文件夹路径。有一个工作流模块，负责实现整个状态机，处理抓取、抽取、合成、编撰等等一切。有一个 Grok 客户端，负责加载配置、发出调用，并把 API 错误翻译成有意义的领域错误。有一个外部文档模块，负责递归地读取你的本地文件，并提取引导用的上下文。还有一套提示模板，定义每一次语言模型调用应该产出什么。整件事是单一进程。每一条命令都会创建一个新的运行文件夹，在会话目录里的文件上工作，并写出运行本地的副本以便追溯。并没有什么后台服务在持续运行；状态完全活在文件之中。

技术方面，它需要 Python 3.11 或更新版本，并依赖一组广为人知的库，去跟模型通信、读取环境文件、输出美观的终端画面、校验数据、解析 PDF 和 Word 文档、抓取网页，以及从杂乱的 HTML 里提取可读的文字。它会安装成一条你可以直接调用的命令，另外还有两个方便用的封装，会转发到同一个入口点。

配置是通过环境变量来处理的。最重要的是 Grok 的 API 密钥，任何会调用模型的动作都需要它；如果缺少了，这些动作就会以一段清晰的消息失败。你可以按需要设置模型名称，默认是 Grok 3，而空白值会悄悄回落到这个默认值。有一个最大输出 token 的设置，默认为五万，无效的值会回落到默认值，而低于一的值就会被抬升到一。有一个请求超时设置，默认为三百秒，套用同样的保护规则。还有一个编辑器变量，只在你选择手动编辑范围时才会用到；如果没有设置，系统依然会创建那个可编辑的文件，只是不会替你打开编辑器。当工作流构建它的默认客户端时，它还会在会话目录上两层的位置寻找一个环境文件，如果找不到，就干脆改用进程环境。

界面本身是一个终端体验，用一个精致的输出库来显示状态消息、回显 Markdown 内容，以及展示预览表格。沿途有四个人工交互点：确认范围、批准筛选好的来源、批准或修订草稿，以及挑选哪些来源要离线保存。但它也有一个无人值守模式。开启自动模式，系统就会绕过所有提示，自行推进到完成。在那个模式下，范围会自动确认，每个来源都被选取并批准，草稿自动获批，而完整收集的选择默认为全部，除非你另有指示。最关键的是，自动模式绝不会停下来等你输入。

我们来讲讲各项命令，因为这里正好展现出系统的能力。除了那条只负责列出类型的命令之外，每条命令都要你指定一个会话目录，而那些针对已有会话运行的命令，还要你提供一个会话标识符。程序在成功时返回退出码零，在工作流引发 Grok 错误或配额错误时返回一，在遇到无法识别的命令或参数校验失败时返回二。而如果你开启跟踪，它会以截断并经过净化的形式，打印请求和响应的内容，让你看到正在发生什么。

start 命令会用你的题目、一个可选的焦点、一个可选的外部文档文件夹，以及一个保存起来的模式，去创建一个新的会话。它会打印新的会话标识符，并立即从第零阶段开始整个工作流。可接受的模式有报告、编译器和操练三种，不过这里要老实说一句：模式虽然会保存，但在目前的实现里并不会真正改变运行时的行为。resume 命令会加载一个会话，从它的当前阶段接着做下去；在交互模式里，它会在下一个人工检查点停下，或者在某个让你再接着做的阶段之后停下，而如果会话已经完成，它就会直接告诉你。有一条命令负责列出所有会话，它会扫描那些含有会话文件的文件夹，跳过任何不是有效会话的，并在一个都没有时告诉你。还有一条列出类型的命令，它只会打印那个唯一支持的类型，而且不需要会话文件夹。

接着是那些较专门的命令。update 会重新运行发现，只寻找上次运行以来的新材料，把阶段退回到筛选，并让你接着做。synthesize 会强制合成阶段运行，无论你目前在哪个阶段，但它的前置条件依然成立，所以没有笔记就不会进行。compile 会构建知识库，接受 auto-hypergraph 这个类型，从主笔记和任何已抽取的内容入手，如果两者都不存在，就以一段有用的消息停下。drill 会生成复习问题，只支持反向模式；如果核心概念文件缺失，它会尝试自动编译，而如果之后仍然找不到，就会让你先运行 compile。feed 会接收一份新文档，要你指明位置，检查它存在而且是真正的文件，把它连同一个时间戳复制进知识库，而如果没有已有的知识地图，它就会编译一个出来然后返回。show 会把知识地图变成一张 Mermaid 图，或者在没有东西可绘时让你先编译。generate-images 和 youtube-script 都需要一份最终报告；如果缺失就会明说，而在成功时，它们会把输出写进运行文件夹和会话文件夹两处。

现在讲讲会话是怎么标识和存储的，因为可续做这一点是核心所在。每个会话标识符，都来自把你的题目转成简化字符串，再加上当天的日期。所谓简化字符串，就是把题目转成小写，把任何不是字母或数字的东西换成连字符，把重复的连字符合并，再把两端修剪干净。如果这串字太长，它会被修剪，并附上一个短短的哈希后缀。而如果已经有一个同名的文件夹，系统就会加上一个编号后缀，直到找到一个未被占用的名字为止。会话状态本身会记下一组丰富的字段：标识符、题目、焦点和模式；一整族关于任何外部文档的细节，包括它们的状态、摘要、文件数量、完成率、相关度分数和最后一次错误；创建和更新的时间戳；模型名称；当前阶段；以及一个运行历史列表。状态以 UTF-8 的 JSON 存储，更新的时间戳在每次保存时刷新，文件夹也会按需要自动创建。这里同样要老实说一句：那个运行历史列表虽然存在于结构之中，但目前的代码其实并没有把它填上。

每一条构建工作流上下文的命令，都会创建一个全新的运行目录，名字用的是精确到微秒的时间戳。万一两个时间戳真的相撞，它会重试多达一千次，而如果到头来仍然找不到一个独一无二的名字，它宁愿引发一个错误，也不会覆盖任何东西。

其中一项较为强大的功能，是外部文档的预处理。它的构思是，在工作流运行之前，你可以喂自己的本地参考文档给它，而它会把这些文档转化成必须遵从的引导上下文，去塑造范围、发现、筛选、抽取和规划。这个预处理会在大部分命令之前自动运行，不过在 generate-images、youtube-script、compile、drill、feed 和 show 这几条之前就不会。如果你没有替它指明一个文件夹，它就会被跳过。而如果它已经成功完成，并有一份摘要在档，它就不会多此一举再运行一次。它支持 PDF、Word 文档、纯文本和 Markdown，以递归方式发现，而对于它看不懂的文件类型，它只会跳过，而不是报错。

处理本身相当周到。每份文件都会以适合其类型的逻辑来读取。对于 PDF，它会逐页处理，跳过任何提取不出文字的页面。对于 Word 文档，它会把非空白的段落连接起来。对于文本和 Markdown，它会以 UTF-8 读取，并把任何无效字符替换掉。每份文档都会根据文件名里的关键字，归类为指南、背景、引导或一般文档。处理器接着会用句子层面的启发式方法，提取关键概念、约束、需求和算法洞见，并计算一个相关度分数，依据是文档的词汇与你的题目和焦点有多大重叠，再加上相关词语、约束和需求所带来的额外加分。结果会写成三样东西：一份载有逐文件结果和整体指标的清单、一份抽取摘要，以及一份上下文文件，其中汇集了关键概念、约束、需求、可选的算法注记和工作流指引。那些算法注记，只在你的题目或焦点与算法导向的关键字相符时才会出现。如果那个文件夹并不存在，状态会标记为失败，并保存一段解释，工作流则照样继续。如果个别文件失败，它们会被标记为失败，但其余的照样进行。整体状态在全部成功时为完成，在有些成功有些失败时为部分完成，在全无成功时为失败。而当这份上下文可用时，它会被附加到相关的提示里作为必须的背景，并按每个阶段的字符预算来截断，而绝不会引致失败。

现在来到重头戏：研究工作流本身，这是一个有九个阶段的状态机，编号从零到八。第零阶段是范围生成和确认。第一阶段是发现。第二阶段是筛选和缺口分析。第三阶段是抽取。第四阶段是笔记组装。第五阶段是合成和审阅。第六阶段是完整离线收集。第七阶段是最终打磨。而第八阶段就代表完成。在交互模式里，每一次续做的调用，都会处理一个阶段或一个人工检查点，而有些阶段会刻意以让你再接着做来收尾，而不是一口气继续下去。阶段的转换，会在代码更新当前阶段的那一刻就保存起来。在自动模式里，工作流就一直循环，直到到达第八阶段，跨越各阶段而无需分开的续做命令。

我们来逐个阶段走一遍。在第零阶段，系统会生成一份 Markdown 的范围摘要，把它写进一个范围文件，并打印到控制台。在自动模式里，它会立即接受范围，保存一份已确认的副本，并推进到第一阶段。在交互模式里，你的选择有是、编辑或取消。取消会停下而不改变阶段。编辑会写一个临时文件，按需要打开你的编辑器，重新加载并打印出编辑后的内容，然后继续提示你。是会保存已确认的范围，推进阶段，并让你接着做。而如果连 Grok 客户端都创建不了，它会打印错误连同关于环境文件的指引，然后返回而不改变任何东西。

第一阶段是发现。系统会用你的题目、焦点，以及它是否只在寻找新材料，去填好一个发现提示，然后把发现表格写进运行文件夹和会话文件夹两处。它在保存之前不会校验表格的格式。在正常流程里，完成发现会推进到第二阶段，并让你接着做以进行筛选。

第二阶段是筛选和缺口分析。它需要那张发现表格；没有它，就会停下并让你从第一阶段接着做。它会打印发现输出最多前八十行非空白内容的预览。你的指令可以是自由格式的文字，按编号挑选来源、说全部、添加网址、移除编号，或者要求做缺口检查，而它并不会在本地解析这些，反而会直接传给模型。它会尝试生成筛选来源，最多三次，而在一次失败之后的重试里，它会加上更严格的指令，要求只返回 JSON，并把列表上限定在二十项。每个筛选来源都会被规范化成一个整齐的对象，含有标题、网址、类型、相关理由、可信度和优先级。网址会经过规范化，修剪掉零散的引号和结尾的标点。如果模型在每一次尝试都返回无效或不合规的数据，系统就会从发现的 Markdown 里启发式地恢复网址，并改为构建后备条目。原始输出会原封不动地存进运行文件夹，而一个规范版本则会写进会话文件夹。缺口分析永远都会尝试并保存，而如果它超时，所保存的报告仍会含有一个缺口标题和一段明确的超时注记。会话只在你的批准正好是“批准”这个字时，才推进到抽取；其他任何回应都会把它留在筛选，并让你过会儿再试。

第三阶段是抽取，也是最费力的一环。它需要那个筛选来源文件，而如果那个文件存在却产出一份空列表，它就会让你重新筛选。它会在两个范围里，创建用来放抽取文件、来源快照和抽取分块的文件夹。它会向模型索取一份抽取计划并保存，而如果这一步超时，它会保存一个占位内容而不是失败。它会用最多四个抓取工作线程并行预先抓取来源捆绑，而如果个别抓取失败，它会打印一个警告然后继续。对于每个成功抓取的来源，它会同时保存原始内容和一份规范化的文本快照，标头会保留标题、网址、主机、类型、优先级和可信度。HTML 快照会以 HTML 文件保存，PDF 以 PDF 保存，其余一切以文本保存。来源文本会切成最多四万五千字符的分块，并有五千字符的重叠，而这些分块会用最多两个工作线程并行抽取。每个分块提示都会要求严格的段落，涵盖覆盖摘要、术语、机制、工作流、证据、限制、开放问题、可引用段落和抽取注记。如果一个分块超时，它会被跳过，其余继续。成功的分块会写进两个范围，成功的来源会组装成编号的文件，而如果某个来源的每一个分块都失败，那个来源就会被跳过并附上警告。当这个阶段完成时，会写出一个带有生成标记的索引文件。

第四阶段是笔记组装。它需要那个抽取文件夹，而没有它，就会让你从第三阶段接着做。这本笔记包含一个顶层标题、一行题目、一个用途段落、一个可选的外部文档上下文段落、一份来源目录、一个可选的知识库提纲，以及那些来源文件，全部用分隔符连接起来。它会写进两个范围，而一次成功的组装会推进到第五阶段。

第五阶段是合成和审阅。它需要那本主笔记，会把它切成最多七万字符的分块，并有五千的重叠，而如果没有分块产出，它就会告诉你笔记是空的。对于一张固定列表上的每一节，它都会从笔记分块里构建该节专属的证据包。那些标准节是：核心定义与范围；架构与技术机制；工作流、过程与运营模式；证据、示例与案例研究；限制、权衡与失败模式；以及开放问题与未来方向。证据生成每一节最多用两个工作线程，证据包会保存到两个范围；如果某一节产不出证据，就会被跳过并附上警告。每一节起草好的草稿都会保存，而草稿报告会汇集范围与覆盖的文字、来源目录、起草好的各节、一个可选的知识库对齐，以及参考资料。草稿会以递增的版本号保存。审阅提示会告诉你，你可以批准、连同意见修订某一节、按标题添加一节，或者运行缺口检查。只有“批准”这个确切字眼才会推进到第六阶段；其他任何东西都会被当作一般修订意见，并传给修订提示而不在本地解析。如果一次修订超时，先前的草稿仍然作准，阶段不会推进，而一次成功的修订会成为下一个版本，并需要再走一轮审阅。

第六阶段是完整离线收集。它会尝试加载筛选来源，而如果它们不在，它会退而求其次，从发现表格恢复网址。如果它什么都恢复不了，它会推进到第七阶段、保存、打印一段跳过消息，并等你接着做以作收尾。否则，它会展示一张选择列表，列出每个来源的编号、标题和网址。你可以说全部、无，或者给出以逗号分隔的数字，而非数字的标记会被忽略。说无会推进到第七阶段并跳过。说全部会选取一切。超出有效范围的数字会被忽略。被选的来源会被预先抓取，而每一个成功抓取的，都会把一份完整离线副本写进两个范围。如果某个被选的来源抓取不到，它会被跳过而不会中止。一旦那些尝试过的副本写好，系统就会推进到第七阶段，立即运行最终打磨，然后在第八阶段把会话标记为完成。

第七阶段是最终打磨。它需要那本主笔记以及至少一份草稿，否则它会说它们缺失。它会生成一份执行摘要，如果这一步超时就换上一个占位内容，并生成一份术语表，超时同样换上占位内容。如果最新那份草稿以一个顶层标题开头，那个标题会在组装之前移除。它会从正文里所有第二层标题构建一个目录。最终报告于是会含有主标题、目录、执行摘要、正文、来源目录、一个可选的知识库概览，以及术语表。系统会按需要尝试两次去达到字数目标，一次针对正文，一次针对组装好的报告，目标是介于九千到一万字之间，理想值是九千五百，而这个校正会保留标题和核心论点，同时扩充或压缩其余部分。最终报告会写进两个范围，然后系统会尝试生成配图提示和一份 YouTube 讲稿。第八阶段就只是代表会话已完成，而续做一个已完成的会话，就只会这样告诉你。

关于抓取和转换来源，有一套细致的规范。网址在校验之前会先规范化，而只有带有真正网络位置的 HTTP 和 HTTPS 地址才会被接受；其他任何东西都会引发错误。抓取会用一个自定义的用户代理、跟随重定向，并把超时拆成连接和读取两部分，在需要时引发一个带有网址上下文的超时错误。PDF 会凭其内容类型或文件后缀来检测，并以几种形式返回提取出来的文字。非 HTML、非 PDF 的响应会当作纯文本。HTML 响应会产出一段来自可读性提取的主文本、一段来自整页的全文本，以及一段把两者合并或妥善回退的分析文本。在 HTML 规范化期间，script、style、noscript 和 SVG 标签都会被移除，而重复的行会被剔除，以削减重复的样板内容。

接着是知识编撰，它会把你的研究化成结构化的知识。编撰会先用笔记，并在有需要时附上抽取出来的文件。超图只会用内容的前二十二万字符，而核心概念抽取则会用同一段来源内容，再加上超图 JSON 的前十二万字符。输出有一个超图文件、一份自动类型副本，以及一个核心概念文件。超图遵循一个结构，由带有标识符和标签的节点，以及把多个节点以一个关系和佐证连起来的超边组成。如果模型没有返回有效的 JSON，系统会把原始响应包起来，而不是失败。核心概念的结构会要求一批概念，每个都有名称、定义，以及一段解释它为何举足轻重的说明，而虽然提示要求正好七个，代码却不会独立地强制这个数目。操练包的结构会产出一份 Markdown 复习包，再加上按概念分组的结构化问题，每题都有答案和常见陷阱；如果那个 Markdown 字段缺失，系统会从原始响应里剥掉代码围栏并改用那部分，而如果没有结构化问题，它就会改为把整个解析出来的对象写出来。

喂文档的做法，是以 UTF-8 读取那份新文档，并把任何解码错误替换掉，然后把已有超图和新文档两者的前十六万字符送给模型，而更新后的超图会覆写两个规范位置。Mermaid 绘制会以一个图形声明开头，用上最多前二百个节点和前四百条边，而对于成员多于两个的超边，它就只把所列的前两个连起来，并按关系或标签来为边加上标签。

最终的输出定义得很清楚。报告是一份 Markdown 文档，带有明确的执行摘要和来源目录两节，在有相关内容时附上一个可选的知识库概览，并在末尾附上一份术语表，即使它的生成曾经超时。配图提示由完整的报告生成，约定要求五到十个提示，着重具体的机制、工作流、架构、比较和证据，而不是泛泛的概念美术，而如果这一步在打磨期间超时，报告依然会成功。YouTube 讲稿主要在有的时候由各节草稿来取材，否则就由最终报告来取材，并刻意排除目录、来源目录、术语表、参考资料、知识库概览和执行摘要这些节。讲稿含有一个顶层标题、一段引言、每一个被选节各一个标题，以及一段结语。如果引言或结语超时，会插入占位旁白而不是失败；如果某一节超时，它可能会被省略，而其余的照样进行；过短的输出会由第二次调用加以扩充，以达到最低限度的细节；而如果某一节缺了一个标题，系统会自动补上一个。

说一说校验。所需的命令标志由参数解析器强制执行，而不受支持的类型或模式值，根本连解析器的校验都过不了。语义上的校验是刻意从简的：筛选来源的检查是结构性而尽力而为的，发现输出不作结构校验，最终报告不作事实正确性的检查，而概念的数目就只受提示约束。文件校验会在要紧的地方发生，喂文档的命令会检查文件是否存在以及类型，外部文档会校验根文件夹和支持的后缀，而会话的列出则会检查会话文件。

错误处理正是系统展现其成熟之处。缺少 API 密钥会引发一个清晰的错误。配额和账单问题会映射到一个带有可行文字的配额错误。超时类的失败会变成一个包含已设置超时秒数的超时错误。其他 API 失败会以指数退避重试多达五次，上限为三十秒，而配额和超时错误一旦被识别出来，就不会重试。有若干阶段会用上一个可选调用的辅助工具，把模型超时转化成警告然后继续，涵盖诸如缺口分析、抽取计划、抽取分块、证据包、各节草稿、修订、执行摘要、术语表、配图提示、YouTube 的各个部分，以及字数重订。来源抓取失败永远不会中止整个阶段。而 JSON 解析也很稳健：系统会剥掉代码围栏，先试直接解析，再试方括号切片，再试大括号切片，最后才回退到把原始文字包起来。贯穿其中的主题就是优雅地降级，外部文档缺失、筛选来源缺失、知识产物缺失、报告缺失，全部都会产出有用的消息而不是崩溃。

讲到集成，Grok 的连接用的是在 xAI 基础网址上、与 OpenAI 兼容的对话 API，以环境里的持有者密钥来验证，每次调用送出一条系统消息和一条用户消息，并读取第一个完成选项。网页集成是纯粹的 HTTP 和 HTTPS GET，启用重定向、不作验证、SSL 交由请求库处理，而失败则要么往上冒，要么按设计被降级为警告。本地文档集成支持 PDF、Word、文本和 Markdown，而喂文档的命令在打开文件的层面上更为宽松，但其实是针对文本内容而设的。

安全和隐私以务实的方式处理。API 密钥由环境或环境文件读取，绝不会写进会话产物。但会话目录可能会存放抓取回来的远端内容和处理过的外部文档，这些应视为具有潜在敏感性，因为系统不会对这些文件夹作任何机密屏蔽，也不会作任何访问控制。另外还有一些老实的非功能性现实。由于它本地优先，每一件重要的产物，在每一个主要步骤之后都必须能在磁盘上查阅。可续做这一点，靠的是当前阶段和哪些文件存在，而不是事务日志或数据库。确定性只是局部的：文件名和转换是可预测的，但内容由模型生成，因此带有概率性。并发是有界限的，四个抓取工作线程、两个抽取工作线程和两个证据工作线程。而大段文字靠的是以字符为本的截断和分块，而不是精确的 token 计数。

开发团队对目前的缺口坦白得令人耳目一新。模式设置虽然保存了，但暂时不会改变行为。运行历史字段存在，但未被填上。类型列表就只暴露那一个 auto-hypergraph 类型，尽管内部常量提到其他几个正在休眠。那些指引字符串提到添加一节或运行缺口检查，但并没有本地解析器去强制执行，它们只是被当作修订意见传递下去。目录就只由第二层标题构建。Mermaid 会把多成员的超边简化成成对的链接。而事实正确性归根结底取决于模型和来源，因为并没有自动的事实核查。

那么，怎样才算完成？验收标准十分具体。你可以创建一个带有独一无二标识符和已保存状态的新会话。交互式推进可以带着预期的人工检查点，把一个会话从第零阶段一路移到第八阶段。自动模式可以在从不要求输入的情况下完成工作流。发现会创建它的表格。筛选会创建筛选来源文件和一份缺口报告。抽取会创建快照、分块和文件。笔记组装会创建那本主笔记。合成会创建至少一份草稿。最终打磨会创建最终报告。打磨或明确的命令会创建配图列表和 YouTube 讲稿。编译会创建超图和核心概念。操练会创建操练包和问题。喂文档会存下一份带时间戳的副本，并能更新或初始化超图。show 会创建那个 Mermaid 文件。而外部文档在被提供时，会被处理成一份清单、一份摘要和一份上下文，而绝不会因为局部失败就卡住工作流。

这就是完整的蓝图。Research Agent 最令人印象深刻的，是它的纪律：每一步都是一个你可以打开的文件，每一次失败都有一个优雅的后备，而人就在最要紧的那些时刻留在环节之中。它接过一个题目，再一步一步小心翼翼地，把它变成一份你真正信得过、又可以据以发展的成果。
