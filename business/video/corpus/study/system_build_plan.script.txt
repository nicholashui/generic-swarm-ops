Today we're walking through something ambitious. It's called the VA Agent Swarm System Build Plan, and it's the single, authoritative, step by step blueprint for building an entire video production platform powered by a hundred and fourteen specialized AI agents. Think of it as the master instruction set handed to an AI coding agent, telling it exactly what to build, in what order, with what quality checkpoints, and how to use its own tooling to do the job reliably.

Let me set the stage. This document doesn't try to reinvent the system's design. That design already exists across a whole library of specification files. Instead, this plan is the construction manual. It takes an empty repository and walks it all the way to a hardened, observable, production grade platform. The system itself is a hierarchical multi agent system. A hundred and fourteen agents, organized into ten categories, each running as a node in a larger workflow graph, all working together to take a client brief and turn it into finished video delivered across multiple channels.

The first part of the plan describes how the coding agent should actually operate. There's a clear loop it follows for every milestone and every task. First, it enters a planning mode where it reads the relevant specifications, restates the goal, lists the files it intends to create or change, and surfaces anything unclear. Crucially, it does not write code during this planning phase. Next, it confirms that plan against the acceptance criteria for the milestone. If something is ambiguous, it asks one consolidated question rather than guessing. Then it writes a failing test first, following test driven development. Only after that does it implement the smallest piece of code needed to make the test pass. It runs a local verification gate that combines linting, type checking, and unit tests, and it never moves forward if that gate is red. After that, it does a self review, commits the work with a clear, structured commit message, updates a living progress file, and clears its working context between unrelated tasks to keep its thinking sharp.

There's a recurring theme here that the author calls the rethink a hundred times mandate. The user who commissioned this plan asked for it to be rethought a hundred times with full effort. Rather than treating that as a slogan, the plan bakes that intensity into its structure. There's a literal hundred point hardening checklist near the end, organized as ten themes with ten checks each, and the system is not considered done until all hundred pass. Every milestone gets re challenged across several dimensions, from traceability all the way down to wording. The idea is that quality here is recursive. The system that judges videos must itself be judged by the same standards.

The plan lays out seven golden rules, and violating any of them counts as a defect even if the tests technically pass. Rule one, contracts before code. The shared data contracts are built and frozen before any agent is written, so that a hundred and fourteen agents can't drift into a hundred and fourteen different message formats. Rule two, a vertical slice before breadth. One complete workflow runs end to end through real infrastructure before the other agents get built. Rule three, every agent is an instance of one common base class. No custom, one off agent loops. New agents are just configuration, a rubric, and tools. Rule four, no agent talks to the user interface directly. Agents publish events to a shared event bus, and a gateway fans those out. Rule five, determinism wherever possible. Pin the random seeds, the model versions, and the prompt versions, and record them for audit. Rule six, cost and safety are gates, not afterthoughts. Every token is metered from early on, and the compliance agent can block work the moment it exists. And rule seven, mock the expensive external generation models behind an interface from day one, so the whole system can run on cheap stand ins during testing.

Throughout the build, the coding agent maintains a handful of living documents. There's a progress checklist, an architecture decision log where every non obvious choice gets a dated entry, a persistent project memory file, and a configuration folder holding its sub agents, custom commands, settings, and automation hooks.

Now, what exactly is being built? In one paragraph, it's a hierarchical multi agent system that automates and augments professional video production from a client brief all the way to multi channel delivery. The hundred and fourteen agents run as nodes in a workflow graph, made durable by a workflow engine so that long running productions survive crashes and restarts. They pass creative work to each other through a shared handoff contract, and they critique each other through a dedicated messaging bus. Everything is gated by a layered quality framework, observed through tracing and dashboards, and surfaced to humans through a modern web console.

The build philosophy is summed up in a few principles. Start with a walking skeleton, the thinnest possible end to end path that actually works, before adding muscle. Build the platform before the agents, because roughly seventy percent of the hard engineering is the platform itself. Favor a factory approach over hand crafting, so that after the platform and a few reference agents are done, the remaining agents are generated from their specifications and reviewed rather than coded one loop at a time. And treat quality as recursive and measurable, applying the system's own evaluation philosophy to the system itself.

A large section configures the coding agent for reliability. It describes how to use a project memory file as an always on constitution, holding the tech stack, the monorepo map, the golden rules, the build commands, and pointers to where each specification lives. The advice is to keep it lean, because that memory competes with the actual task in the agent's limited context.

The plan also defines several sub agents, each with an isolated context and a narrow set of tools. There's a spec reader that digests a specification and returns a tight summary with the exact requirements. There's a contract guardian that checks whether a change quietly breaks or forks the frozen shared contracts. There's a test author that writes the failing test suite first. There's a test runner that runs the relevant tests and returns a minimal diagnosis. There's a code reviewer that checks a change against the definition of done. And there's a specialized agent factory smith that turns a row in the agent registry into a concrete agent configuration. The usage rule is simple. Delegate reading and verification to sub agents, but keep decisions and integration on the main thread.

On top of those, the plan defines repeatable slash commands for common workflows, like loading a milestone, creating a new agent, running verification, checking contracts, running a specific quality gate, recording an architecture decision, and running a hardening audit. It also describes optional plug in servers for inspecting the database, the workflow histories, and the observability traces, with the advice to keep those minimal. And it sets up automation hooks that fire around the agent's actions, things like auto formatting changed files, blocking edits to protected contract files, running the verification gate before a turn ends, scanning for secrets, and reminding the agent to update its progress file.

Permissions are handled carefully. There's an allowlist for safe, frequent commands, and destructive or irreversible commands are never allowed without explicit human confirmation. The agent practices context hygiene by clearing its window between unrelated tasks, compacting at natural breakpoints, and using parallel work trees for separate tracks like the user interface and the meta agents.

Every task has a strict definition of done. Behavior must be covered by tests written before the code, and all of them must be green. The full verification suite must pass. Public contracts must be unchanged, or changed only through a documented decision and a guardian sign off. The code reviewer must return no blocking or major findings. The milestone's acceptance criteria must be met with evidence. A clean, structured commit must be made with no secrets and no leftover debug clutter. And documentation must be updated if the public surface changed.

Next the plan pins down the technology stack, and it's emphatic that these are decisions, not options. On the backend, it's Python paired with a fast, lockfile driven dependency tool. On the frontend, it's TypeScript with React and a modern web framework using the app router. There's strict linting, formatting, and type checking on both sides, and a rich testing setup including property based tests for the contracts and browser based tests for the user interface.

For the platform services, agent orchestration runs on a graph framework that supports conditional branching and human in the loop gates. A durable workflow engine guarantees that productions, which can run for minutes to hours, survive restarts with retries and replay. An event bus, starting with a streaming store and later able to scale to a more powerful messaging system, handles publish and subscribe with persistence and replay. A relational database holds production metadata, gate state, critiques, configurations, and the audit log. Object storage holds the video, audio, and image artifacts with content addressed keys. A vector database and a hybrid graph based retrieval layer power the knowledge and memory systems. There's a unified client for talking to multiple large language models behind one interface, observability tooling for traces, metrics, and logs, and a provenance standard that signs every artifact so the chain can be verified downstream. Everything is containerized, moving from local composition in development to a managed cluster in production with a dedicated pool of graphics processors for generation tasks.

External tool providers, the actual video generators, voice cloning, lip sync, music, spatial audio, and evaluation metrics, are always hidden behind common interfaces. Agents never call them directly. A decision is recorded that every provider implements the same protocol with methods to report its capabilities, estimate cost, run, and check its health. A router agent picks among providers based on cost, quality, and latency, and the testing environment uses mock providers exclusively.

The repository is organized as a monorepo. At the root sit the project memory, the progress checklist, the decision log, a single command surface, the agent configuration folder, the container definitions, and the workspace files. Under a shared packages area live the reusable building blocks, and the most important of these, the frozen contracts package, is built first. Other shared packages include the agent core with its base class and self refinement loop, the agent factory, the language model gateway, the providers, the retrieval layer, the quality control layer, the event bus, the memory store, the provenance signing, and the observability wiring. Under a services area live the deployable processes, including the orchestrator, the agent runtime worker pool, the api gateway, and a scheduler. The web console lives in its own application folder. And a dedicated agents folder holds the hundred and fourteen agent definitions, split into production craft agents, meta agents, support agents, and cross cutting services, all indexed by a canonical registry. There are also folders for the workflow graphs and for evaluation assets like golden test sets and rubrics. The rule is that every package ships from birth with at least one trivially passing test so the verification gate is always green.

Now we reach the most important section, the cross cutting contracts. These are built first and frozen, because everything downstream depends on them. The first is the shared artifact handoff contract, a single data model carried with every artifact as it moves between phases. It captures the technical specification like codec, aspect ratio, duration, frame rate, color space, loudness, and whether captions are required. It captures rights and consent, including license state, whether likeness and voice consent were given, territorial limits, and any embargo date. It captures continuity state, like a character's look, props, wardrobe, environment, and an identity hash used to detect drift. It captures quality status across the layers, and a provenance manifest holding the signing reference, the sign off chain, the exact model versions used, and the seeds, all in service of reproducibility. The artifact itself carries an id, a version number, its media type, its location, its parent assets, the brief scope, and its target channels. The contract tests for this are written first. They check that the model survives a round trip through serialization, that removing or renaming a field without a version bump fails the build, that parent assets form a valid graph with no cycles, and that every released artifact carries a non empty provenance manifest.

The second contract is the critique message bus schema. This is how any agent comments on any other agent's work. Each critique carries an id, the agent it came from, the agent it's aimed at, a reference to the artifact in question, a severity, a category, supporting evidence, a suggested action, an optional rubric reference, a deadline phase, an optional score, and a timestamp. Severity ranges from blocker, down through major, minor, and the smallest nit. The categories cover pacing, continuity, accuracy, compliance, accessibility, brand, craft, and aesthetics. The acceptance rules are strict and tested exhaustively. A blocker halts the workflow node until it's resolved. A major triggers a self refinement loop on the receiving agent, capped at three iterations, after which it escalates to a judge. Minor and nit critiques are logged to memory and aggregated as a reward signal for the next training cycle. Disputes between two agents get routed to a judge for a structured debate, and compliance critiques are always treated as blockers.

The third contract is the common agent base class. Every one of the hundred and fourteen agents is an instance of this one class. Its configuration names the agent, its category, its system prompt, its model policy with preferred models and budget, the tools it's allowed to call, its rubric, its measurable self quality metrics, its critique relationships, and its refinement limit. The base class itself defines the lifecycle. It drafts work, critiques its own draft against the rubric, and revises, following the self refine pattern. On failure it stores verbal feedback and retries, following the reflexion pattern. The base class also wires in the metered language model gateway, the retrieval client, the memory store, the event bus, provenance signing, and tracing. No subclass reimplements any of this. Agents differ only by their configuration.

The fourth contract is the set of provider interfaces, all mockable. The fifth is the quality mesh itself, with three evaluation layers plus a six pass delivery check. The first layer checks specification conformance with deterministic validators and must pass completely. The second layer is a rubric score from a language model acting as a judge using the role's constitution, and it must reach at least eighty five out of a hundred. The third layer is a preference comparison, pairwise against a human reference plus a simulated audience of at least two hundred personas and at least twenty human in the loop samples, and it must reach parity or surpass a threshold. The six delivery passes cover specification, artifact integrity, audio sync, continuity, perceptual quality, and outlet readiness, and all six must pass. The deterministic checks are built first because they're fully testable. The judge based checks are built with frozen, pinned judges and golden sets so the scores stay stable. The plan is firm that a judge model should never be allowed to float unpinned.

The contracts package also generates frontend types so the web console stays in sync with the backend, and it defines the event bus topics. Each production has topics for agent events, critiques, gates, and artifacts, plus a system wide alerts channel. Every event must be one of the typed event models, things like an agent state change, an artifact created, a critique message, a gate becoming ready or resolved, a budget update, a metric update, a memory entry, a tool call, a production phase change, or an error. These typed shapes are the only ones allowed on the bus.

With the contracts in place, the plan lays out a phased roadmap of thirteen milestones, numbered zero through twelve. The sequencing principle moves from foundation to intelligence to production to enhancement, but with that vertical slice punched through early so the architecture is proven before going wide. Each milestone specifies its goal, its dependencies, the files to build, the coding agent's workflow, the tests, and a hard acceptance gate. You don't start the next milestone until the current gate is green and logged.

Milestone zero is the bootstrap. The goal is a compiling, green, fully tooled but empty monorepo, with all the coding agent's configuration in place. Nothing does anything yet, but the verification gate passes and bringing up the container stack starts every backing service. This is where the scaffold, the command surface, the container definitions, the dependency setup, the continuous integration pipeline, and all the coding agent configuration get created. The gate is that from a clean clone, bootstrapping, bringing services up, and verifying all pass green.

Milestone one builds the foundation knowledge service, the agentic retrieval system that every agent will call. It includes an ingestion pipeline that chunks, embeds, and indexes documents into both a vector store and a graph layer. It offers hybrid retrieval combining vector, graph, and keyword search with reranking, and it can plan multi step retrieval. Its client offers retrieval, multi hop compound queries, ingestion, and source graded citations. Knowledge is separated into namespaces so one project's world cannot leak into another's. The gate requires hitting a precision target on a golden set with graded citations returned.

Milestone two is the largest platform milestone, the orchestration runtime, the beating heart of the system. It builds the typed event bus with at least once delivery and idempotency, the observability layer where every node run is a trace span, and the orchestrator service combining the graph runtime with human in the loop interrupt points and a durable workflow engine where each agent task is a retryable activity. It also builds skeleton versions of the platform agents, the orchestrator, planner, router, judge, gatekeeper, and memory agents, plus the memory store and the asset backbone with immutable ids, copy on write versions, and provenance signing. This is a think hard milestone. The plan explicitly calls for documenting the boundary between the graph framework and the workflow engine as a recorded decision. The acceptance gate, which is the walking skeleton, is a hard coded three node graph that runs end to end on real infrastructure, survives a mid run worker kill, emits correct typed events, and signs its artifacts.

Milestone three builds the language model gateway, the router, and the cost dashboard, so that every token is metered and routed from day one. The gateway wraps the unified client with provider abstraction, automatic retry and fallback, per call token and cost metering emitted to the bus, version tagging into provenance, response caching, and a frozen judge mode. The router agent picks the best agent and model by cost, quality, and latency, and a cost dashboard aggregates spend per production, agent, and provider with alert thresholds. Budget guardrails enforce a hard stop and escalation when a production exceeds its envelope. The gate is that any agent call is metered, routed, version tagged, visible in the dashboard, and that a budget breach triggers a real stop.

Milestone four builds the research agent and the coding agent harness. The research agent does query planning, multi source retrieval, synthesis, and produces a source graded, cited dossier that gets written into the knowledge namespaces. Interestingly, the plan notes that the coding agent specification actually describes the very role the coding agent is playing during this build, so its conventions get extracted into the project memory and enforced by the code reviewer. The gate is a graded, cited dossier produced and stored, with coding conventions enforced.

Milestone five builds the intelligence layer, the shared reasoning services every production agent consumes. There's deep intent analysis, which parses a brief into structured intent, goals, audience, hidden agendas, and constraints, and it's the entry point of every production. There's the general creative agent, a creative ideation engine. There's a process optimization agent that improves workflows from telemetry. There's a strategic goal achievement framework, a complex problem solving methodology, and an aesthetics agent that supplies perceptual scoring, novelty signals, and aesthetic critiques. The creative agent and the aesthetics agent form a generate and evaluate loop. The gate requires all six reasoning services to pass their golden behavioral tests with that loop demonstrated.

Milestone six is the critical one, the agent factory plus the vertical slice. The goal is to prove the entire architecture with the cheapest real workflow before building the remaining hundred and nine agents. The factory turns an agent configuration into a runnable agent, validating its references and registering it. Then just enough craft agents are built, via the factory, to run one workflow, the viral hook clip, end to end through real infrastructure on mock generators. The full path runs from brief, through intent analysis, through a planner that builds the workflow graph, through agents executing on mocks, with artifacts flowing through the handoff contract, the critique bus active, the quality mesh gating, a signed deliverable produced, and events on the bus. This is an ultrathink milestone because the factory must eventually produce all hundred and fourteen agents, so its configuration schema has to be complete now. The acceptance gate is critical. Running the workflow end to end must produce a signed deliverable from a brief, with every handoff contract populated, every gate enforced, all events emitted, full provenance, and under budget, using mock providers. The plan is emphatic. This gate proves the platform, and you do not proceed to breadth until it's rock solid.

Milestone seven builds the production craft agents, the first fifty two, as factory produced configurations, rubrics, and prompts. For each agent, a repeatable playbook produces its configuration, its versioned prompt, its rubric, its measurable self quality metrics, its tool allowlist, and its critique relationships. They're built category by category, sharing rubric scaffolding, with the context cleared between categories. The gate requires all fifty two registered and green on their specification and rubric tests and critique tests, plus at least three additional workflow archetypes running end to end on mocks.

Milestone eight promotes the platform agent skeletons to full implementations and adds the meta agents, numbers fifty three through eighty. That covers the orchestration agents with full dispute resolution, the creative meta agents that mostly delegate to the creative and aesthetics services, the research meta agents built on the shared research core, and the optimization agents covering prompt, cost, latency, retention, return on ad spend, and accessibility, plus an evaluation harness and a safety red team. The full quality mesh is completed here, including the simulated audience layer and all six delivery validators, with the gatekeeper enforcing zero leaked defects. The plan suggests building the evaluation harness and the safety red team early in this milestone so they continuously test everything else. The gate requires all eighty agents live with the full quality framework enforced on every release path, and the red team and harness running nightly.

Milestone nine builds the workflow support agents, numbers eighty one through a hundred and fourteen, and the delivery fabric. That includes asset management, versioning, and render dispatch with a render farm that batches across graphics processors and autoscales. It includes quality gates, delivery packaging, and compliance, packaging into theatrical, streaming, broadcast, archive, trailer, and social formats with outlet specific specs, captions, metadata, rights management, and provenance payloads. And it includes analytics, feedback, and retraining agents that aggregate the small critiques into a reward signal. The delivery fabric runs a branching pipeline so theatrical, streaming, broadcast, and archive, plus marketing derivatives, are produced in parallel with per outlet validation. The gate requires a production to produce all four delivery branches plus marketing derivatives, each valid, with post launch telemetry flowing back into a retraining ticket.

Milestone ten builds the user interface, the console, the api gateway, and the websocket layer, the human operator surface. Notably, this can start in parallel after milestone three, in a separate work tree, against a fake event emitter, then integrate later. The gateway exposes the rest endpoints and the websocket gateway exactly per the contract, with authentication, role based access control, rate limiting, validation, and signing on gate approval. It subscribes to the event bus, filters by production, and fans out over websockets. The web console offers a brief studio, a live graph canvas showing node states, an artifact gallery, a critique feed, a gate approval dialog, a budget tracker, a quality dashboard, an agent inspector, a memory panel, and a delivery hub, all driven by the typed events and the generated types. The gate is that a human can launch, monitor in real time, critique, approve gates, and download deliverables for the first workflow entirely through the browser, with very fast live updates and no direct agent to interface calls.

Milestone eleven is the enhancement layer, adding personalization and audio first variants. There's psychological profiling of creators, psychological recommendation predicting audience preferences, a personalization engine that can swap name, face, and voice under a strict privacy and consent audit, and a podcast agent for an audio first workflow. The gate requires personalized and audience conditioned variants generated under consent gates, with the podcast archetype running end to end.

Milestone twelve is hardening, scale, security, and launch, and this milestone is the hundred point checklist executed theme by theme. It covers load testing and autoscaling, security hardening including prompt injection defenses and a full red team sweep, reliability through chaos testing and disaster recovery drills, compliance with provenance on every releasable artifact, validated cost dashboards, refreshed documentation and runbooks, and a staged launch from internal to limited to general availability. The final gate requires all hundred checks to pass, a full feature film dry run exercising all hundred and fourteen agents on mocks, and a live smoke test on real providers succeeding within budget.

The plan pauses to explain why the vertical slice sits where it does. Building a hundred and fourteen agents before proving one workflow end to end would be the classic distributed systems mistake, discovering an architecture flaw after most of the code already assumes it. So the plan deliberately builds the platform first, punches one thin slice all the way through, then scales breadth via the factory with the architecture already battle tested, then adds the interface and enrichment, then hardens. If the slice reveals a problem, you fix it with nine agents in flight, not a hundred and fourteen. It calls this the single most important sequencing decision in the plan.

There's a repeatable pattern at the core. Each of the ten workflow archetypes becomes one graph. They share a common skeleton, greenlight, pre production, production, post, review and release, distribution, and post launch, and they differ only in which agents lead each phase and which critics gate the handoffs. A workflow is done when it runs end to end on mock providers, every phase gate enforces its critic set, and the final artifact carries a complete provenance chain.

The agent implementation playbook is spelled out as the exact recipe run for each of the hundred and fourteen agents, and no agent is hand built outside it. The steps are, read the agent's specification through the spec reader, map its self quality criteria into concrete metrics with thresholds, author its rubric or constitution citing the craft authority named in the spec, define its allowed tools, wire its critique relationships, write its configuration and register it, author its versioned prompt, write its failing tests first, instantiate it through the factory rather than writing bespoke logic, review it, and register it in the workflows that use it. The plan also names anti patterns to reject, like an agent with no rubric, an agent that calls a provider directly instead of through a tool interface, an agent that mutates another agent's artifact instead of emitting a critique, and an agent whose self quality is unmeasurable prose.

On testing and evaluation, the plan insists that because the system is itself an evaluation engine, its own test suite must be exemplary. There are five layers. Unit tests are pure logic, fully mocked, deterministic, and fast, including property tests for the contracts. Contract tests snapshot the schema of every shared model, and a change that removes or renames a field fails the build unless properly versioned and approved. Integration tests run against real backing services with mock generators, covering graph execution, kill and resume durability, event replay, gate interrupts, contract propagation, critique routing, and budget enforcement. Behavioral and golden set evaluation applies the three layer mesh to the system itself, using frozen, pinned judges, a simulated audience, and a harness that runs nightly and on every relevant change. Adversarial and safety testing runs continuous attacks like deepfake misuse, prompt injection, jailbreaks, defamation, and bias, targeting an attack success rate of one percent or less. And end to end tests cover the critical user journeys in the browser. The continuous integration pyramid runs the fast checks on every change and the heavy checks nightly.

On observability, cost, safety, and compliance, every agent run, tool call, model call, and gate decision becomes a trace span, so a production forms one trace tree from brief to delivery. Metrics flow to dashboards, the event sourced bus and workflow history allow any production to be replayed, and logs are structured and correlated by identifiers. Cost is metered per call into budget events and dashboards, with a hard stop and escalation on breach and a cost optimizer keeping routing on the best cost and quality frontier. Safety and compliance are enforced by a compliance agent that can block on every release path, a consent chain required before any likeness or voice generation, signing on a hundred percent of releasable artifacts, and full provenance and audit. The plan defines a single non negotiable release predicate. An artifact is releasable only if it passes the specification layer, scores at least eighty five on the rubric, meets the preference threshold, passes all six delivery checks, clears compliance, is signed, and is within budget. That entire condition is wrapped in one function, and the gatekeeper calls only that.

On deployment, there are three environments, development with mock providers, staging with synthetic load, and production with real providers, full secrets, and an autoscaling pool of graphics processors. The pipelines run verification and contract and golden checks on every change, full integration on the main branch, and the heavy nightly suite. Conventions include structured commits scoped to milestones, short lived branches, no direct pushes to the main branch, and every change going through a reviewed request.

On data, model, and prompt management, every prompt is versioned and recorded in provenance, with prompt changes gated by evaluation before promotion. Models are pinned per agent policy and upgrades are evaluation gated with no regressions allowed. Seeds, style references, and reference frame banks are versioned for reproducibility. Golden fixtures are frozen and reviewed, and changing an expected output requires justification because it may signal rubric drift. Aesthetic profiles are consent governed and signed, and evaluation datasets are wrapped behind a common interface with baselines tracked over time.

The plan includes a risk register, naming risks and their mitigations. The big ones include an architecture flaw discovered after broad agent build, mitigated by the vertical slice. Contract drift across all the agents, mitigated by frozen contracts and snapshot tests. Runaway cost, mitigated by metering, budget gates, mocks in testing, and a live smoke cap. Confusion at the boundary between the graph framework and the workflow engine, mitigated by a recorded decision and kill and resume tests. Noisy judge scores destabilizing gates, mitigated by frozen pinned judges. Reward hacking producing pretty but empty output, mitigated by the aesthetics agent's anti hack layer and human review on low confidence. Provider outages, consent and intellectual property violations, context bloat causing the coding agent to regress, prompt injection through ingested content, scaling bottlenecks, flaky tests, and scope creep where new agents close no real gap, each with its own mitigation.

Then comes the hundred point hardening checklist itself, the literal, structural form of the rethink a hundred times mandate, organized as ten themes of ten checks. Theme one is contracts and schema integrity, ensuring every message is typed, every handoff is populated, snapshots guard the models, and no package redefines a shared contract locally. Theme two is orchestration and state, covering kill and resume, fan out and fan in correctness, deadlock detection, timeouts, the documented framework boundary, true blocking gates, idempotency, deterministic replay, backpressure, and graceful degradation. Theme three is agent correctness, ensuring all agents come from the factory, pass their specification and rubric checks, match their critique edges exactly, enforce their tool allowlists, cap their refinement loops, persist their lessons, never mutate another's artifact, use measurable self quality, and respect compliance blocks. Theme four is the quality mesh, ensuring the single release predicate is used, the deterministic validators are complete, the judges are pinned and stable, the simulated audience is sized correctly, all six delivery passes gate, continuity drift is detected, aesthetic scoring is decomposed, reward hacking defenses are active, accessibility is gated, and a seeded defect is reliably caught.

Theme five is cost and performance, covering metering accuracy, budget enforcement, optimal routing, caching, latency targets, autoscaling, storage tiering, workload separation, the live smoke budget ceiling, and concurrent load. Theme six is safety, security, and compliance, covering the red team success ceiling, injection defenses, the consent chain, full signing, the regulatory checklists, secret protection, role based access, dependency scanning, disclosure, and careful handling of personal information. Theme seven is observability and operability, covering one trace tree per production, the dashboards, structured logs, replayability, alerts, fast regression detection, runbooks, backup and restore, disaster recovery drills, and feature flags for safe rollout. Theme eight is the frontend and human experience, covering the full browser workflow, fast accurate live updates, no direct interface calls, reconnect handling, signing on approval, real time dashboards, the browser test journeys, access control, the user interface improvement backlog, and scale adaptive workflows. Theme nine is build process and coding agent hygiene, covering current lean memory files, defined sub agents and commands, enforced hooks, a complete decision log, an accurate progress file, clean scoped commits, a green verification gate at every commit, practiced context hygiene, and honored test first discipline. And theme ten is end to end system validation, covering all ten archetypes running on mocks, the feature film dry run exercising all hundred and fourteen agents, verifiable provenance, valid multi channel delivery, the closing of the retraining loop, demonstrable optimization improvement, the creative and aesthetics loop producing better candidates, source graded research, a successful budget bound live smoke, and finally, that a fresh engineer could build from this plan and the specs without any tribal knowledge. Done means a hundred out of a hundred checked, each with evidence beside it.

The plan closes with a sequencing summary that maps each milestone to its acceptance gate and its primary specs, and it names the critical path running from bootstrap through orchestration, the gateway, the intelligence layer, the vertical slice, breadth, the meta agents, support, and hardening. The interface work can parallelize early, and the enhancement layer slots in near the end. The single highest leverage checkpoint is the vertical slice gate, because it converts architectural risk into a proven foundation. Full effort here means depth over breadth at the start, then mechanical breadth via the factory, then recursive quality, then the literal hundred point sweep. The whole plan is engineered so that a flaw is cheapest to fix exactly when it's most likely to be found.

There are appendices with copy and paste starters, a template for the project memory file, definitions for the sub agents, definitions for the slash commands, a settings file for permissions and hooks, a definition of done to pin in every change request, and a glossary of the key terms.

And the final word captures the whole conviction in a sentence. Build the platform and one perfect slice before the breadth, then let a factory and a recursive quality mesh do the scaling. Contracts are frozen first so the agents cannot diverge. The vertical slice converts the system's biggest risk, an architecture flaw discovered late, into a cheap, early, provable checkpoint. Quality is recursive, so the system that judges video must pass the same standards on itself. And the rethink a hundred times mandate is not rhetoric. It's the literal hundred point gate that stands between works on my machine and production. The instruction to the coding agent is simple. Start at milestone zero, and begin.
