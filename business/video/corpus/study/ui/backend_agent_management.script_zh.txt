让我们深入看看这台机器的内部运作。这个编排后端究竟是怎么管理、派发、监控、重试,并且和一百一十四个 AI agent 组成的大军进行通信的呢?说这个系统有这么多 agent 是一回事,真正搞明白它是怎么让它们运转起来的,又是另一回事了。

核心问题是这样的。后端手上有一百一十四个 agent 定义,但它实际上是怎么创建它们、运行它们、和它们对话,并且把它们牢牢控制住的呢?

答案会让你有点意外。每个 agent 都不是一台独立的服务器,也不是一个自己在那儿运转的小型微服务。一个 agent 其实只是一份配置。它由一段系统提示、一组工具,以及一份质量评分准则组成,全部捆绑在一起。这份配置要等到有一个 worker 进程把它拿起来,再针对某个任务去执行的时候,才真正活起来。

有一个很有帮助的比喻,就是一支管弦乐团。后端是指挥。各个 agent 就是乐谱,也就是纸上的那些指示。Worker 是真正演奏的乐手。而大型语言模型就是乐器,是产生声音的那份原始能力。

那么一个 agent 在运行时究竟是什么呢?它不是一个长期运行的进程。它更像一个无状态的函数,有需要时才会被调用。每个 agent 定义都带着几样东西:一个介于一到一百一十四之间的编号、一个名称比如 Director Agent、一段告诉它自己是谁的系统提示,例如你是一位电影导演,以及一份它可以使用的工具清单,比如视频生成接口或者记忆召回功能。它还有一个架构模式,可能是 self-refine、reflexion 或者 react。它带着一份质量准则,连同它必须达到的阈值。它知道自己接受哪些 agent 的批评,也知道自己可以评论哪些 agent。它还有一个最大迭代次数,以及一个偏好的模型。

所以当后端需要 Director Agent 去生成,比如说第五个镜头的创作意图的时候,它会走一条很清晰的流程。它先载入这个定义。再把系统提示、任务上下文,以及任何批评历史合并起来,拼出提示。接着它调用语言模型。模型就会决定要调用哪些工具。后端代表这个 agent 去执行那些工具调用。如果这个模式要求自我精炼,它就会循环。最后,它把结果连同一系列事件一起发布出去。

现在讲讲编排引擎,也就是整件事的大脑。它的核心是一个状态机,形状很像一个有向的工作图。它会经过一个个节点,从 brief 开始,到计划,再到路由,然后同时并行地制作多个部分,最后到达一道门。节点之间的那些边是有条件的,也就是说系统要先通过一道门才向前走。它会跟踪状态,比如现在是哪个阶段、哪些 agent 正在运作、有哪些任务还没完成,以及还剩多少预算。而最关键的是,它每走完一步都会保存一个检查点,所以什么都不会丢失。

围着这个核心,有几个专门的部件。有一个任务派发器,负责决定哪个 agent 运行、它处理什么任务、用哪个模型、什么时候运行,以及优先顺序怎么排。有一个任务队列,每个 agent 各有一条独立的排队线。有一个批评路由器,根据那些接受批评的关系,在各个 agent 之间传递反馈消息。还有一个门评估器,负责检查条件、触发审批,并且在人类说 OK 之后推进制作。

让我们走一遍后端实际上是怎么把任务交给一个 agent 的。设想编排器决定了 Director Agent 应该负责第五个镜头。

第一步是任务创建。编排器拼出一个任务对象。里面有一个任务编号、agent 的号码、任务的类型,比如生成一个镜头意图,以及输入。那些输入是用存储编号去引用各项素材:剧本、一格分镜、一块情绪板。它还包含任何需要处理的批评,比如用宽一点的镜头,或者第二个场景清晰度偏低。它也带着约束条件:用哪个模型、用哪个生成工具、还剩多少预算,以及这一步的最高成本。

第二步,任务进入这个 agent 的队列,然后有一个空闲的 worker 把它拿起来。可以把这个 worker 池想成一个线程池。同一时间大约有十到五十个 worker 在运行,数量是可以配置的。任何一个 worker 都可以执行任何一个 agent 的任务,因为 worker 是无状态的。它们只是按照自己挑到的任务,载入对应的配置而已。

第三步是 worker 内部的实际执行。Worker 载入 Director Agent 的定义。它从素材库取回输入素材。它用向量搜索,从记忆服务拉回相关的记忆。然后它为语言模型拼好消息。系统消息设定角色。用户消息列出任务,也就是为第二个场景第五个镜头生成一个镜头意图,连同剧本上下文、分镜描述、情绪参考,可能是带着雨水意象的忧郁新黑色风格,以及要处理的批评,比如剪辑师要求用宽一点的镜头,或者观众模拟器报告第二个场景的清晰度分数是零点六,低于零点七的目标。它要求输出是一个结构化的镜头意图,描述镜头、主体、风格和时长。

接着 worker 调用模型。模型回应一些工具调用。比如说,它可能要求视频生成工具,用一个指定的种子值去制作一个缓慢的推轨镜头。Worker 真的会执行那个调用,通过网络联系视频服务,然后取回一条视频链接加上元数据。之后,本着自我精炼的精神,模型评估这个结果。它调用一个评分工具,把视频和文字提示作比较。假设它拿到零点三四的分数,对着零点三二的阈值。这就过关了。如果分数跌到阈值以下,worker 就会循环回去,带着反馈再试。一达到标准,任务就完成。

第四步,worker 把结果发布到事件总线。它宣布有一项素材已经创建、这个 agent 的状态已经转为完成,并且贴出一个带分数的指标更新。编排器听到任务做完了,就决定下一步怎么走。

那么后端是怎么管理一个 agent 的生命周期的呢?每个 agent 都会经过一连串状态。它从空闲开始。接着变成排队中,再到运行中,然后到完成,到这一刻它就发布结果。但运行中可以分岔。它可能进入自我精炼再循环,带着反馈重新运行。它可能进入等待状态,守着等批评,等那份批评到了就恢复。又或者它可能失败,那就会落入重试逻辑,甚至回到队列。另外,一个 agent 还可以变成受阻,等着一道门的审批,或者等着它需要的某些输入。

后端会为每个制作里的每个 agent 保存一份详细的状态记录,存储在数据库里。那份记录包含制作编号、agent 号码、当前状态、当前任务、它正在进行第几次自我精炼迭代、它上一组指标、任何还没处理的批评、重试次数、这个 agent 至今花掉的总成本,以及它开始和完成的时间戳。

关于各个 agent 怎么互相沟通,这里有一点很重要。它们从来不会直接对话。后端彻底居中调停所有事情。设想 Director Agent 完成了一个镜头意图。它发布有一项素材已经创建。后端接着检查谁被配置成可以评论 Director Agent 的输出,然后把一个批评任务交给,比如说 Editor Agent。剪辑师带着反馈回来,可能说节奏太快。后端就检查 Director Agent 究竟有没有接受来自剪辑师的批评。如果有,它就递交那份批评。在下一次迭代里,导演就会把反馈融会进去。

背后那套路由逻辑,精神上很直接。当一个 agent 发布一项素材,后端就找出每一个被配置成可以批评这个生产者的 agent。对每一个批评者,它只在生产者被设定为接受来自那个特定批评者的批评时,才递交批评。当两边都对得上,它就把一个批评任务排入队列。这是一个双向握手,而后端永远都是中间人。

那么并行运行多个 agent 又怎么样呢?在制作阶段,很多 agent 会同时开工。一个 worker 可能在处理 Director Agent 的第五个镜头。另一个在优化提示。另一个在为第四个镜头做质量检查。另一个在拉取视觉参考。再有一个在为第二幕谱写一段音乐主题。它们全部都把事件发布到总线,而编排器就协调它们之间的依赖关系。比如说,剪辑师在导演完成某一场景的每一个镜头之前,都不能开始组装。

为了知道什么时候派发才稳妥,编排器靠一个依赖图。有些任务要等其他任务先完成。组装剪辑需要导演的所有镜头完成,加上作曲的配乐准备好,再加一道已经通过的制作门。调色需要粗剪完成。混音则需要音效设计和作曲的最终混音两样都齐。所以每完成一个任务,编排器就检查那些待办工作,一旦某个任务的所有前置条件都满足,它就派发那个任务。

失败是无可避免的,所以后端有一套清晰的应对方案。一个任务可能因为模型出错、工具超时,或者质量跌到阈值以下而失败。编排器里的重试逻辑会问一连串问题。重试次数是不是还低于上限,也就是默认的三次?如果是,就以指数退避重新排队,等五秒,接着十五秒,再四十五秒。它是不是一个暂时性错误,例如接口超时或者速率限制?那就用同样的参数重试。它是不是一个质量失败,也就是分数太低?那就重试,不过先让一个提示优化器去调整提示。它是不是超了预算?那就用一个更便宜的模型作后备再试。如果所有重试都用完了,这个 agent 就会被标记为失败。用户会通过一个实时连接收到通知,可视化图上的那个节点会变红,而用户可以选择重试、跳过,或者修改后再重试。

记忆值得特别讲一下。记忆 agent 不是排队里又一个普通的 worker。它是一个共享服务,所有其他 agent 都可以调用。它有两半。一半是向量数据库,存储像风格锁定、语气笔记、过往决定,以及批评历史这些东西。另一半是结构化存储,保存系列圣经条目、角色状态、连戏记录,以及预算决定。两边都汇入一个单一的记忆接口,有两个主要动作:召回,接收一个查询然后返回相关条目;以及存储,把一个新事实持久化。任何 agent 在自己模型运行的时候,都可以把这个当成工具去调用。所以当 Director Agent 在处理第五个镜头,而模型想到,我需要知道我们为第二幕锁定了什么视觉风格的时候,它就调用召回,取回类似这样的资料,风格锁定是一个指定的视频模型种子,带着忧郁新黑色的外观,然后用这个东西去让自己的提示保持一致。

让我们从头到尾追踪一次完整的控制流程。用户点击启动。请求到达 API 网关,网关再发出建立一个制作的请求。制作管理器创建一条数据库记录,载入选好的模板,并且初始化预算。然后编排引擎接手。

第一阶段是计划。计划 agent 被调用,模型把这个 brief 拆解成一个任务图,连带其中的门和依赖关系。然后路由 agent 被调用,在尊重预算的前提下,为每个任务指派一个模型和供应商。

第二阶段是执行,它会一直循环,直到每个阶段都完成。任务被派发到 worker 池,在依赖关系允许的地方并行,在顺序重要的地方按序进行。每个 worker 都通过那个熟悉的循环去执行自己的 agent 任务:载入配置、拼出提示、调用模型、执行工具、自我精炼,然后发布结果。当一个任务完成,后端就更新这个 agent 的状态,检查是不是有批评者需要运行,检查是不是刚刚有新的依赖关系得到满足,然后派发下一批合资格的任务。而当到达一道门,一个守门 agent 会评估条件,一个评审 agent 会按准则评分,如果一切自动通过,制作就推进。如果需要人类,系统就暂停,并通知界面,然后等待。一获批准,它就推进。一被拒绝,它就把工作送回给修订 agent。

第三阶段是交付,在所有门都通过之后开始。一个分发 agent 为每个渠道把输出打包,一个合规 agent 用来源凭证为内容签署,然后把作品发布到目标平台。

那么总结一下后端的角色。它通过把 agent 配置存储在数据库里来定义 agent,包括提示、工具、准则和关系。它用依赖图来决定什么时候运行,只在前置任务完成时才派发。它通过路由器和成本规则来决定用哪个模型,挑出仍然符合质量而又最便宜的那个。它通过载入配置、拼提示、调用模型和运行工具来执行 agent 逻辑。它通过检查质量指标,并在需要时带着反馈循环,来管理自我精炼循环。它通过遵从接受批评的配置来路由批评。它在数据库里管理状态,跟踪每个 agent 的状况、重试、成本和指标。它通过重试、退避、更便宜的后备和人类升级来处理失败。它通过评估条件、为人类暂停,并在批准时推进来执行门。它通过往这个池里加入更多 worker 来扩展。它还通过事件溯源来持久化一切,所以每一个决定、批评和素材都可以重放。

这里有个关键洞见值得带走。Agent 不是独立的服务。错误的心智模型,是设想每个 agent 都是自己一个全天运转的微服务,一百一十四个一起在那儿嗡嗡作响。正确的心智模型,是 agent 是配置,由 worker 按需要去实例化。有一个大约十到五十个 worker 进程的池。在任何一刻,可能有一个在执行导演的任务,另一个在执行提示工程师的任务,另一个空闲地等着,再有一个在做质量保证。每个 worker 都可以执行任何一个 agent。它载入那个 agent 的配置,运行模型,然后接着去做下一件事。就好像一个演员饰演很多角色。同一个人,不同的剧本。

这正是为什么一百一十四个 agent 不需要一百一十四台服务器。一个不大的 worker 池就能应付它们全部,只是在任务一变得可用,就从队列里拿起来做。
