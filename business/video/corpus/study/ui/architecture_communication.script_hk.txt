我哋傾下用戶介面究竟係點同各個代理溝通，而後台又係點樣喺背後協調全部一百一十四個代理。聽落好似好複雜，但只要你睇清楚佢嘅結構，就會發現其實相當簡潔。

成個系統分為三層。你可以諗成一座三層嘅大廈，每一層都有清晰嘅職責。

最頂嗰層係用戶介面，亦即係喺你瀏覽器入面運行嗰部分，用 React 同 Next.js 砌成。呢度就係你做嘢嘅地方：你撳掣、打字、批准、上載。同時呢度亦都係你即時睇住一切發生嘅地方，因為介面會持續訂閱住每個代理喺度做緊乜、有咩評論飛來飛去、邊啲成品啱啱整好。介面同後台用兩種唔同方式溝通。對於指令，即係你主動發起嘅動作，佢會發出普通嘅網絡請求。對於即時串流，即係源源不絕嘅更新，佢就會保持一條開住嘅 WebSocket 連接，等資訊可以即刻湧返嚟。

中間嗰層係 API 閘道同埋編排後台。呢度係成個運作嘅大腦，用 Node 同 Python 混合運行，再配上負責工作流編排同持久性嘅框架。呢層住咗幾個服務。有一個製作管理服務，負責基本功能，例如建立同讀取記錄、檢查身份驗證同權限。有一個編排引擎，真正將工作流當成一個任務圖去執行。有一個事件匯流排負責傳遞訊息。有一個資產庫存放完成嘅作品。仲有一個 WebSocket 閘道，佢唯一嘅工作就係將最新狀態推送去前端。喺呢層，後台會透過一條隊列將任務派落去各個代理，亦會喺有需要時連去外部嘅創作工具。

最底嗰層係代理運行環境，亦即係真正嘅大型語言模型工作者所在嘅地方。呢度住咗全部一百一十四個代理嘅定義，每一個都有自己嘅系統提示、自己嘅一套工具，同自己嘅評分準則。喺呢度你仲會搵到語言模型供應商、負責生成影片、圖像同聲音嘅生成工具，以及衡量質素嘅評估工具。

而家我哋一步一步睇下實際上會發生啲乜。

想像一個用戶啟動一個新製作。佢喺工作室表格填好簡報，描述標題、願景、類型、時長同預算，跟住撳啟動。前端會將呢一切包裝好，當成一條指令送去後台。後台會建立一個製作記錄，再排隊一個開始工作嘅任務。

之後，編排引擎就接手。一個規劃代理收到簡報，向語言模型發出呼叫，將簡報分解成一個分階段嘅計劃，返回一份任務清單、一組代理分派，同埋每個批准閘口嘅準則。跟住一個總指揮代理會將執行設定成一個狀態機，並登記佢將會用到嘅每一個代理節點。一個路由代理會為每個任務分配啱嘅模型同供應商，並檢查設定入面嘅成本同質素規則。

跟住各個代理就開始做嘢，通常係並行進行。舉個例，一個導演代理可能收到一個任務，要生成某個鏡頭嘅意圖。佢會用語言模型去思考呢個任務，呼叫一個影片生成工具，然後行一個自我修正嘅循環，自己同自己嘅成品打分，如果低過門檻就再迭代。每次有一步完成，代理就會將一個事件發佈去事件匯流排，再透過 WebSocket 閘道轉發去前端。喺畫面上，你會見到工作流畫布裏面嘅節點改變狀態，新嘅成品出現喺圖庫，評論訊息出現喺動態欄，狀態列亦會更新，反映有幾多個代理喺度運行同埋用咗幾多預算。

咁就帶到第二個主要流程：即時更新由各個代理串流返去介面。當一個代理做緊嘢，佢會不斷發出一連串事件。佢會宣告幾時改變狀態、幾時開始呼叫工具、幾時建立成品、幾時向另一個代理發出評論，以及幾時佢其中一項質素指標更新。呢啲全部都會流入事件匯流排，匯流排會將佢哋保存起嚟以便日後重播，並轉發畀訂閱者。WebSocket 閘道會按製作去過濾呢啲事件，所以每個客戶端只會見到同自己相關嘅嘢，再推送去已連接嘅瀏覽器。喺前端，本地狀態更新，介面重新渲染：節點閃動顏色，新嘅成品卡出現，評論訊息滑入動態欄，計數器跳動，預算儀表亦會做動畫。

第三個流程係人喺迴路之中，即係你介入並指揮成件事。當系統去到一個批准閘口，你會見到一個對話框，列住一份準則清單同埋相關嘅成品。你審視一下，跟住就批准或者要求修改。前端會將你嘅決定送去後台。閘道會檢查你嘅權限，簽署一份來源證明清單，記錄內容係邊度嚟，再將決定發佈去事件匯流排。一個守門代理收到佢。如果你批准咗，工作流就推進去下一個階段。如果你拒絕咗，你嘅意見就會被路由去相關嘅代理做修訂，然後循環繼續。

仲有第四個流程值得一提：直接向一個代理發送評論。想像你喺評論動態欄打咗一個提示，譬如話，第三場用闊啲嘅鏡頭啦，而家感覺太侷促，並且指明係畀導演代理嘅。前端會將佢送去後台，後台建立一個評論訊息，再將佢發佈去事件匯流排，瞄準嗰個目標代理。編排引擎將佢送到嗰個代理嘅輸入隊列。喺下一次迭代，導演代理會透過一個記憶代理讀取呢個評論，調整佢嘅鏡頭意圖，用更新後嘅提示重新生成，再發佈一個回應。呢個回應會經由事件匯流排同 WebSocket 返去前端，喺嗰度代理嘅回覆會出現喺動態欄，更新後嘅成品亦會出現喺圖庫。

我哋退一步，睇下兩邊之間嘅契約。喺指令嗰邊，有一系列網絡端點，對應用戶主動發起嘅動作：建立同啟動一個製作、列出全部製作、取得某一個製作嘅狀態、批准或拒絕一個閘口、發送評論、重試或者跳過一個失敗咗嘅代理、更新路由規則、列出全部代理定義、瀏覽成品同佢哋嘅來源證明，以及觸發最後嘅交付打包。每一樣都同介面嘅某個特定部分掛鈎，由簡報工作室到儀表板，到代理檢查器，再到交付中心。

喺串流嗰邊，有一套相對應嘅事件由系統向外推送。有事件對應代理改變狀態、成品被建立或更新、評論訊息、閘口準備就緒然後被解決、預算更新、指標更新、新嘅記憶條目、工具呼叫、製作改變階段，以及錯誤。每個事件都會更新介面入面某個特定嘅版面，唔論係工作流畫布、圖庫、評論動態欄、預算追蹤器、質素儀表板，定係一個通知。

而家我哋深入啲睇下後台本身。最前面坐住一個閘道層，處理身份驗證同基於角色嘅存取控制、流量限制同驗證。喺佢後面係服務層，製作管理服務同 WebSocket 閘道服務並肩工作，一個負責記錄、簡報解析同權限，另一個負責客戶端連接、事件路由同過濾。

佢哋兩個都連去事件匯流排，匯流排會將訊息組織成主題，每個製作嘅代理事件一條串流，佢嘅評論一條，佢嘅閘口一條，佢嘅成品一條，再加一條全系統嘅警報頻道。由事件匯流排，工作流入編排引擎，引擎持有工作流嘅狀態機同每個代理嘅任務隊列，連埋處理重試、逾時同閘口評估嘅機制。

再下面係代理工作者池。每個工作者都跟住同一個節奏：由隊列拉出一個任務，載入代理嘅設定，呼叫語言模型去思考任務，呼叫所需嘅工具去生成或評估，如果做得唔夠好就自我修正，然後將結果同佢嘅事件發佈返去匯流排。呢個池可以橫向擴展，圖像運算量大嘅工作者預留畀生成任務，較輕量嘅工作者就處理純模型嘅推理。

而喺一切之下，係數據層。一個關聯式資料庫存放製作元數據、閘口狀態、評論同設定。物件儲存存放笨重嘅成品，即影片、音訊同圖像。一個向量資料庫驅動記憶代理嘅語義回想。一個高速嘅記憶體內儲存就處理快取、工作階段，同事件串流本身。

咁點解個系統會咁樣設計？每個重大選擇都有清晰嘅理由。

點解代理對介面嘅更新要用 WebSocket 而唔係輪詢？因為有超過一百個代理每隔幾秒就產生事件，輪詢即係要每秒發出十個或以上嘅請求，好快就會變得昂貴又延遲。一條 WebSocket 幾乎即時送到更新，少過五十毫秒，每個製作只用一條連接，由伺服器幫手過濾相關性。

點解要喺後台服務之間加一個事件匯流排，而唔係畀佢哋直接互相呼叫？因為各個代理係異步同並行嘅，所以簡單嘅請求加回應模式根本唔啱用。事件溯源畀你由頭到尾重播一個製作入面任何一個決定。佢將各個代理同介面解耦，亦容許好多消費者同時收聽，包括分析、合規記錄同警報。

點解要用一個基於圖嘅編排框架而唔係自己寫嘅程式碼？因為一個帶條件分支嘅圖，可以直接對應到製作嘅組成。佢自帶狀態管理，應付又長又多步驟嘅工作流，佢將人手批准閘口當成一等公民嘅概念，仲可以做檢查點，就算後台重啟之後都可以恢復。

咁又點解要喺上面再加一個持久性框架？因為製作可以由短片嘅幾分鐘，去到劇情長片嘅幾個鐘。你需要保證交付，等就算一個工作者崩潰都唔會失去任何任務，需要喺出事時自動重試並退避，亦需要一份完整嘅工作流歷史去做除錯同審計。

要將一切串連埋一齊，想像一個請求嘅完整生命週期。你撳一個節點，前端取得嗰個代理嘅數據並渲染出嚟。你批准一個閘口，前端送出你嘅決定，閘道將佢發佈，總指揮收到綠燈，將工作派落去一個代理工作者。嗰個工作者發出一個語言模型呼叫，收到回應，呼叫一個工具，攞返一段影片。當佢完成，佢就將一個成品建立事件同一個狀態改變事件發佈去匯流排，匯流排再透過 WebSocket 推去你嘅瀏覽器，介面就用新狀態重新渲染。

咁，回答嗰條簡單問題，介面係點同代理溝通？你嘅直覺完全正確。介面透過網絡請求同 WebSocket 同後台溝通，而後台就透過一條任務隊列同各個代理溝通。

指令向一個方向流動：由介面，經網絡 API，入後台，去到任務隊列，再出去到一個代理。嗰啲就係你嘅動作，啟動、批准、評論、重試、設定。事件就向另一個方向流動：由一個代理，入事件匯流排，經 WebSocket 閘道，再返去介面。嗰啲就係即時更新，狀態改變、成品、評論同指標。

但呢度有個關鍵：後台唔係一個簡單嘅傳遞通道。佢做緊重活。佢處理編排，即工作流嘅執行、排程同重試邏輯。佢管理狀態，知道邊啲代理活躍緊、而家喺邊個階段。佢管理資產，存放成品同追蹤版本。佢運行閘口邏輯，評估準則同收集批准。佢透過身份驗證、權限同來源簽署去執行保安。佢亦透過記錄、指標同重播去提供可觀測性。

而最重要嘅一條規則：各個代理永遠唔會直接同介面對話。佢哋只會將事件發佈去事件匯流排，再由 WebSocket 閘道負責將嗰啲事件送畀啱嘅客戶端。正正係呢種分隔，先令到成個系統保持簡潔、可擴展又可信。

最後，簡單講下每個角色背後嘅技術。前端用 React 同 Next.js，配一個輕量嘅狀態庫加一個數據取得層，再加一個會自動重連、按房間過濾嘅 WebSocket 客戶端。閘道行喺一個快速、帶類型嘅 Python 或 Node 框架上。編排用一個 Python 嘅圖框架，再用一個獨立框架保證長時間運行工作流嘅持久性。事件匯流排建基於一個串流訊息系統，同時支援發佈訂閱同重播。各個代理係會呼叫工具嘅語言模型工作者，有帶類型嘅輸入同輸出，透過一個統一介面用多個模型供應商，並連去頂尖嘅生成工具去整影片、音訊同圖像。數據方面，有一個關聯式資料庫、存放媒體嘅物件儲存、畀記憶代理用嘅向量資料庫，同一個高速快取。而可觀測性方面，有追蹤同效能儀表板睇住成件事。每一件零件都有佢嘅用途，合埋一齊，就造出一個系統，等一百一十四個代理可以協作，而你只需要睇住、指引同批准。
