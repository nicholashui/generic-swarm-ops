我们来聊聊用户界面究竟是怎么和各个智能体沟通的，以及后端又是怎么在背后协调全部一百一十四个智能体的。听起来好像很复杂，但只要你看清楚它的结构，就会发现其实相当简洁。

整个系统分为三层。你可以把它想象成一座三层的大楼，每一层都有清晰的职责。

最顶层是用户界面，也就是运行在你浏览器里的那部分，用 React 和 Next.js 搭建而成。这里就是你做事的地方：你点击、打字、批准、上传。同时这里也是你实时看着一切发生的地方，因为界面会持续订阅每个智能体正在做什么、有哪些评论在来回飞、哪些成品刚刚做好。界面和后端用两种不同的方式沟通。对于命令，也就是你主动发起的动作，它会发出普通的网络请求。对于实时数据流，也就是源源不断的更新，它则会保持一条打开的 WebSocket 连接，让信息可以立刻涌回来。

中间层是 API 网关和编排后端。这里是整个运作的大脑，用 Node 和 Python 混合运行，再配上负责工作流编排和持久性的框架。这一层里住着好几个服务。有一个制作管理服务，负责基本功能，比如创建和读取记录、检查身份验证和权限。有一个编排引擎，真正把工作流当成一张任务图来执行。有一个事件总线负责传递消息。有一个资产库存放完成的作品。还有一个 WebSocket 网关，它唯一的工作就是把最新状态推送到前端。在这一层，后端会通过一条队列把任务派发给各个智能体，也会在需要时连到外部的创作工具。

最底层是智能体运行环境，也就是真正的大语言模型工作者所在的地方。这里住着全部一百一十四个智能体的定义，每一个都有自己的系统提示、自己的一套工具，以及自己的评分标准。在这里你还会找到语言模型供应商、负责生成视频、图像和声音的生成工具，以及衡量质量的评估工具。

现在我们一步一步看看实际上会发生什么。

想象一个用户启动一个新制作。他在工作室表单里填好简报，描述标题、愿景、类型、时长和预算，然后点击启动。前端会把这一切打包好，作为一条命令发送到后端。后端会创建一个制作记录，再排队一个开始工作的任务。

之后，编排引擎就接手了。一个规划智能体收到简报，向语言模型发出调用，把简报分解成一个分阶段的计划，返回一份任务清单、一组智能体分派，以及每个批准关卡的标准。接着一个总指挥智能体会把执行设置成一个状态机，并注册它将要用到的每一个智能体节点。一个路由智能体会为每个任务分配合适的模型和供应商，并检查配置里的成本和质量规则。

接着各个智能体就开始干活，通常是并行进行的。举个例子，一个导演智能体可能收到一个任务，要生成某个镜头的意图。它会用语言模型去思考这个任务，调用一个视频生成工具，然后跑一个自我修正的循环，自己给自己的成品打分，如果低于阈值就再迭代。每次有一步完成，智能体就会把一个事件发布到事件总线，再通过 WebSocket 网关转发到前端。在画面上，你会看到工作流画布里的节点改变状态，新的成品出现在画廊里，评论消息出现在动态栏，状态栏也会更新，反映有多少个智能体在运行以及花了多少预算。

这就引出了第二个主要流程：实时更新从各个智能体串流回界面。当一个智能体在干活时，它会不断发出一连串事件。它会宣告什么时候改变状态、什么时候开始调用工具、什么时候创建成品、什么时候向另一个智能体发出评论，以及什么时候它的某一项质量指标更新了。这些全都会流入事件总线，总线会把它们保存起来以便日后重放，并转发给订阅者。WebSocket 网关会按制作来过滤这些事件，所以每个客户端只会看到和自己相关的内容，再推送到已连接的浏览器。在前端，本地状态更新，界面重新渲染：节点闪动颜色，新的成品卡出现，评论消息滑入动态栏，计数器跳动，预算仪表也会播放动画。

第三个流程是人在回路之中，也就是你介入并指挥整件事。当系统走到一个批准关卡，你会看到一个对话框，列着一份标准清单和相关的成品。你审视一下，然后批准或者要求修改。前端会把你的决定发送到后端。网关会检查你的权限，签署一份来源证明清单，记录内容是从哪来的，再把决定发布到事件总线。一个守门智能体收到它。如果你批准了，工作流就推进到下一个阶段。如果你拒绝了，你的意见就会被路由到相关的智能体去做修订，然后循环继续。

还有第四个流程值得一提：直接向一个智能体发送评论。想象你在评论动态栏里打了一条提示，比如说，第三场用更宽的镜头吧，现在感觉太局促了，并且指明是给导演智能体的。前端会把它发送到后端，后端创建一条评论消息，再把它发布到事件总线，瞄准那个目标智能体。编排引擎把它送到那个智能体的输入队列。在下一次迭代时，导演智能体会通过一个记忆智能体读取这条评论，调整它的镜头意图，用更新后的提示重新生成，再发布一个回应。这个回应会经由事件总线和 WebSocket 回到前端，在那里智能体的回复会出现在动态栏，更新后的成品也会出现在画廊里。

我们退一步，看看两边之间的契约。在命令这一边，有一系列网络端点，对应用户主动发起的动作：创建并启动一个制作、列出全部制作、获取某一个制作的状态、批准或拒绝一个关卡、发送评论、重试或者跳过一个失败的智能体、更新路由规则、列出全部智能体定义、浏览成品和它们的来源证明，以及触发最后的交付打包。每一样都和界面的某个特定部分挂钩，从简报工作室到仪表盘，到智能体检查器，再到交付中心。

在串流这一边，有一套相对应的事件由系统向外推送。有事件对应智能体改变状态、成品被创建或更新、评论消息、关卡准备就绪然后被解决、预算更新、指标更新、新的记忆条目、工具调用、制作改变阶段，以及错误。每个事件都会更新界面里某个特定的版面，无论是工作流画布、画廊、评论动态栏、预算追踪器、质量仪表盘，还是一个通知。

现在我们更深入地看看后端本身。最前面坐着一个网关层，处理身份验证和基于角色的访问控制、流量限制和校验。在它后面是服务层，制作管理服务和 WebSocket 网关服务并肩工作，一个负责记录、简报解析和权限，另一个负责客户端连接、事件路由和过滤。

它们两个都连到事件总线，总线会把消息组织成主题，每个制作的智能体事件一条流，它的评论一条，它的关卡一条，它的成品一条，再加一条全系统的警报频道。从事件总线，工作流进入编排引擎，引擎持有工作流的状态机和每个智能体的任务队列，连同处理重试、超时和关卡评估的机制。

再往下是智能体工作者池。每个工作者都遵循同一个节奏：从队列里拉出一个任务，加载智能体的配置，调用语言模型去思考任务，调用所需的工具去生成或评估，如果做得不够好就自我修正，然后把结果和它的事件发布回总线。这个池可以横向扩展，图像运算量大的工作者预留给生成任务，较轻量的工作者则处理纯模型的推理。

而在一切之下，是数据层。一个关系型数据库存放制作元数据、关卡状态、评论和配置。对象存储存放笨重的成品，也就是视频、音频和图像。一个向量数据库驱动记忆智能体的语义回想。一个高速的内存存储则处理缓存、会话，以及事件流本身。

那么为什么系统会这样设计？每个重大选择都有清晰的理由。

为什么智能体对界面的更新要用 WebSocket 而不是轮询？因为有超过一百个智能体每隔几秒就产生事件，轮询就意味着每秒发出十个或更多的请求，很快就会变得既昂贵又卡顿。一条 WebSocket 几乎即时送达更新，不到五十毫秒，每个制作只用一条连接，由服务器帮忙过滤相关性。

为什么要在后端服务之间加一个事件总线，而不是让它们直接互相调用？因为各个智能体是异步和并行的，所以简单的请求加响应模式根本不合适。事件溯源让你能从头到尾重放一个制作里的任何一个决定。它把各个智能体和界面解耦，也允许很多消费者同时收听，包括分析、合规记录和告警。

为什么要用一个基于图的编排框架而不是自己写的代码？因为一张带条件分支的图，可以直接对应到制作的组成。它自带状态管理，应付又长又多步骤的工作流，它把人工批准关卡当成一等公民的概念，还可以做检查点，即使后端重启之后也能恢复。

那又为什么要在上面再加一个持久性框架？因为制作可以从短片的几分钟，到剧情长片的几个小时。你需要保证交付，让即使一个工作者崩溃也不会丢失任何任务，需要在出事时自动重试并退避，也需要一份完整的工作流历史来做调试和审计。

要把这一切串联起来，想象一个请求的完整生命周期。你点一个节点，前端获取那个智能体的数据并渲染出来。你批准一个关卡，前端发出你的决定，网关把它发布，总指挥收到绿灯，把工作派发给一个智能体工作者。那个工作者发出一个语言模型调用，收到响应，调用一个工具，取回一段视频。当它完成时，它就把一个成品创建事件和一个状态改变事件发布到总线，总线再通过 WebSocket 推到你的浏览器，界面就用新状态重新渲染。

那么，回答那个简单的问题，界面是怎么和智能体沟通的？你的直觉完全正确。界面通过网络请求和 WebSocket 和后端沟通，而后端则通过一条任务队列和各个智能体沟通。

命令朝一个方向流动：从界面，经网络 API，进后端，到任务队列，再出去到一个智能体。那些就是你的动作，启动、批准、评论、重试、配置。事件则朝另一个方向流动：从一个智能体，进事件总线，经 WebSocket 网关，再回到界面。那些就是实时更新，状态改变、成品、评论和指标。

但这里有个关键：后端不是一个简单的传递通道。它在做重活。它处理编排，也就是工作流的执行、调度和重试逻辑。它管理状态，知道哪些智能体在活跃、现在处于哪个阶段。它管理资产，存放成品并追踪版本。它运行关卡逻辑，评估标准并收集批准。它通过身份验证、权限和来源签名来执行安全。它也通过日志、指标和重放来提供可观测性。

而最重要的一条规则：各个智能体永远不会直接和界面对话。它们只会把事件发布到事件总线，再由 WebSocket 网关负责把那些事件送给正确的客户端。正是这种分隔，才让整个系统保持简洁、可扩展又可信。

最后，简单说说每个角色背后的技术。前端用 React 和 Next.js，配一个轻量的状态库加一个数据获取层，再加一个会自动重连、按房间过滤的 WebSocket 客户端。网关运行在一个快速、带类型的 Python 或 Node 框架上。编排用一个 Python 的图框架，再用一个独立框架保证长时间运行工作流的持久性。事件总线建立在一个流式消息系统之上，同时支持发布订阅和重放。各个智能体是会调用工具的语言模型工作者，有带类型的输入和输出，通过一个统一接口使用多个模型供应商，并连到顶尖的生成工具去做视频、音频和图像。数据方面，有一个关系型数据库、存放媒体的对象存储、给记忆智能体用的向量数据库，以及一个高速缓存。而可观测性方面，有追踪和性能仪表盘看护着整件事。每一个零件都有它的用途，合在一起，就造出一个系统，让一百一十四个智能体可以协作，而你只需要看着、指引和批准。
