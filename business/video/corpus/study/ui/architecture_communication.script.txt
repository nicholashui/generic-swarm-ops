Let's talk about how the user interface actually communicates with the agents, and how the backend coordinates all one hundred and fourteen of them behind the scenes. It sounds complicated, but once you see the shape of it, it's surprisingly clean.

The whole system is built in three tiers. Think of it as three floors of a building, each with a clear job.

The top floor is the user interface, the part that lives in your browser. It's built with React and Next.js. This is where you do things: you click, you type, you approve, you upload. And this is also where you watch everything unfold in real time, because the interface is constantly subscribed to live updates about what every agent is doing, what critiques are flying around, and which artifacts have just been created. The interface talks to the backend in two distinct ways. For commands, the things you initiate, it sends ordinary web requests. For live streams, the constant flow of updates, it keeps an open WebSocket connection so information can pour back instantly.

The middle floor is the API gateway and the orchestration backend. This is the brain of the operation, running on a mix of Node and Python with frameworks for workflow orchestration and durability. Several services live here together. There's a production manager that handles the basics, creating and reading records, checking authentication and permissions. There's the orchestration engine that actually executes the workflow as a graph of tasks. There's an event bus that moves messages around. There's an asset store for finished work. And there's a WebSocket gateway whose only job is pushing the latest state out to the frontend. From this floor, the backend dispatches tasks down to the agents through a queue, and it reaches out to external creative tools when needed.

The bottom floor is the agent runtime, where the actual large language model workers live. This is home to all one hundred and fourteen agent definitions, each with its own system prompt, its own set of tools, and its own scoring rubric. Down here you'll also find the language model providers, the generative tools that create video, images, and voice, and the evaluation tools that measure quality.

Now let's walk through what actually happens, step by step.

Picture a user launching a new production. They fill out a brief in the studio form, describing the title, the vision, the genre, the duration, the budget, and then they hit launch. The frontend packages all of that up and sends it to the backend as a command. The backend creates a production record and queues up a job to start the work.

From there, the orchestration engine takes over. A planner agent receives the brief and makes a language model call to decompose it into a phased plan, returning a task list, a set of agent assignments, and the criteria for each approval gate. An orchestrator agent then sets up the execution as a state machine and registers every agent node it will need. A router agent assigns the right model and provider to each task, checking the cost and quality rules from the configuration.

Then the agents go to work, often in parallel. A director agent, for example, might get a task to generate the intent for a shot. It reasons about the task with a language model, calls a video generation tool, and then runs a self-refinement loop, scoring its own output and iterating if it falls below the threshold. Every time a step completes, the agent publishes an event to the event bus, which forwards it through the WebSocket gateway to the frontend. On screen, you see the nodes in the workflow canvas change state, new artifacts appear in the gallery, critique messages show up in the feed, and the status bar updates to reflect how many agents are running and how much budget has been spent.

That brings us to the second major flow: real-time updates streaming from the agents back to the interface. As an agent works, it emits a steady series of events. It announces when its state changes, when it starts calling a tool, when it creates an artifact, when it sends a critique to another agent, and when one of its quality metrics updates. All of these flow into the event bus, which persists them for later replay and forwards them to subscribers. The WebSocket gateway filters those events by production, so each client only sees what's relevant, and pushes them to the connected browsers. On the frontend, the local state updates and the interface re-renders: nodes pulse with color, new artifact cards appear, critique messages slide into the feed, counters tick over, and the budget gauge animates.

The third flow is the human in the loop, where you step in and direct things. When the system reaches an approval gate, you see a dialog with a checklist of criteria and the relevant artifacts. You review it, then either approve or request changes. The frontend sends your decision to the backend. The gateway checks your permissions, signs a provenance manifest to record where the content came from, and publishes the decision to the event bus. A gatekeeper agent receives it. If you approved, the workflow advances to the next phase. If you rejected, your feedback is routed to the relevant agents for revision, and the cycle continues.

There's a fourth flow worth highlighting: sending a critique directly to an agent. Imagine you type a note in the critique feed, something like, use a wider lens for scene three, it feels too claustrophobic, addressed to the director agent. The frontend sends that to the backend, which creates a critique message and publishes it to the event bus aimed at the target agent. The orchestration engine delivers it to that agent's input queue. On its next iteration, the director agent reads the critique through a memory agent, adjusts its shot intent, regenerates with the updated prompt, and publishes a response. That response travels back through the event bus and the WebSocket to the frontend, where the agent's reply appears in the feed and the updated artifact shows up in the gallery.

Let's step back and look at the contract between the two sides. On the command side, there are web endpoints for the things a user initiates: creating and launching a production, listing all productions, fetching the state of one, approving or rejecting a gate, sending a critique, retrying or skipping an agent that failed, updating the routing rules, listing all the agent definitions, browsing artifacts and their provenance, and triggering the final delivery packaging. Each of these is tied to a specific part of the interface, from the brief studio to the dashboard to the agent inspector to the delivery hub.

On the streaming side, there's a matching set of events the system pushes outward. There are events for an agent changing state, for an artifact being created or updated, for a critique message, for a gate becoming ready and then being resolved, for budget updates, for metric updates, for new memory entries, for tool calls, for a production changing phase, and for errors. Each event updates a specific surface in the interface, whether that's the workflow canvas, the gallery, the critique feed, the budget tracker, the quality dashboard, or a notification.

Now let's go a little deeper into the backend itself. At the front sits a gateway layer handling authentication and role-based access, rate limiting, and validation. Behind that is the service layer, where the production manager service and the WebSocket gateway service work side by side, one handling records, brief parsing, and permissions, the other handling client connections, event routing, and filtering.

Both of them connect to the event bus, which organizes messages into topics, one stream for each production's agent events, another for its critiques, another for its gates, another for its artifacts, plus a system-wide channel for alerts. From the event bus, work flows into the orchestration engine, which holds the workflow state machine and the per-agent task queues, along with the machinery for retries, timeouts, and gate evaluation.

Below that is the agent worker pool. Each worker follows the same rhythm: it pulls a task from the queue, loads the agent's configuration, calls the language model to reason about the task, calls the necessary tools to generate or evaluate, refines its work if it falls short, and then publishes the result and its events back to the bus. The pool scales horizontally, with graphics-heavy workers reserved for generation tasks and lighter workers handling the model-only reasoning.

And underneath everything is the data layer. A relational database holds production metadata, gate state, critiques, and configurations. Object storage holds the heavy artifacts, the video, audio, and images. A vector database powers the memory agent's semantic recall. And a fast in-memory store handles caching, sessions, and the event streams themselves.

So why was the system designed this way? Each major choice has a clear reason.

Why use a WebSocket for agent-to-interface updates instead of polling? Because with over a hundred agents producing events every few seconds, polling would mean ten or more requests every second, which gets expensive and laggy fast. A WebSocket delivers updates almost instantly, in under fifty milliseconds, over a single connection per production, with the server filtering for relevance.

Why put an event bus between the backend services rather than letting them call each other directly? Because the agents are asynchronous and parallel, so a simple request-and-response pattern just doesn't fit. Event sourcing lets you replay any decision in a production from start to finish. It decouples the agents from the interface, and it lets many consumers listen at once, including analytics, compliance logging, and alerting.

Why use a graph-based orchestration framework instead of custom code? Because a graph with conditional branches maps directly onto the composition of the production. It comes with built-in state management for long, multi-step workflows, it treats human approval gates as a first-class idea, and it can checkpoint and resume even after the backend restarts.

And why add a durability framework on top? Because productions can run from minutes for a short hook to hours for a feature film. You need guaranteed delivery so no task is lost even if a worker crashes, automatic retries with backoff when something fails, and a full workflow history for debugging and audit.

To tie it all together, picture the complete lifecycle of a single request. You click a node, the frontend fetches that agent's data and renders it. You approve a gate, the frontend posts your decision, the gateway publishes it, the orchestrator gets the all-clear and dispatches work to an agent worker. That worker makes a language model call, gets a response, calls a tool, gets back a video. As it finishes, it publishes an artifact-created event and a state-change event to the bus, which pushes them over the WebSocket to your browser, and the interface re-renders with the new state.

So, to answer the simple question, how does the UI talk to the agents? Your intuition is exactly right. The interface talks to the backend over web requests and WebSockets, and the backend talks to the agents through a task queue.

Commands flow one way: from the interface, through the web API, into the backend, onto the task queue, and out to an agent. Those are your actions, launching, approving, critiquing, retrying, configuring. Events flow the other way: from an agent, into the event bus, through the WebSocket gateway, and back to the interface. Those are the real-time updates, state changes, artifacts, critiques, and metrics.

But here's the key point: the backend is not a simple pass-through. It's doing the heavy lifting. It handles orchestration, the execution of the workflow, scheduling, and retry logic. It manages state, knowing which agents are active and what phase you're in. It manages assets, storing artifacts and tracking versions. It runs the gate logic, evaluating criteria and collecting approvals. It enforces security through authentication, permissions, and provenance signing. And it provides observability through logging, metrics, and replay.

And the most important rule of all: the agents never talk directly to the interface. They only publish events to the event bus, and the WebSocket gateway is what delivers those events to the right client. That separation is what keeps the whole system clean, scalable, and trustworthy.

Finally, a quick word on the technology behind each role. The frontend uses React and Next.js, with a lightweight state library paired with a data-fetching layer, and a WebSocket client that auto-reconnects and filters by room. The gateway runs on a fast, typed Python or Node framework. Orchestration uses a graph framework in Python, with a separate framework guaranteeing durability for long-running workflows. The event bus is built on a streaming message system that supports both publish-subscribe and replay. The agents are tool-calling language model workers with typed inputs and outputs, drawing on multiple model providers through a unified interface, and calling out to the leading generative tools for video, audio, and images. For data, there's a relational database, object storage for media, a vector database for the memory agent, and a fast cache. And for observability, there's tracing and performance dashboards watching over the whole thing. Every piece has a purpose, and together they make a system where a hundred and fourteen agents can collaborate while you simply watch, guide, and approve.
