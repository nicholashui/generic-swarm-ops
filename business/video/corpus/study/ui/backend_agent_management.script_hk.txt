等我哋深入睇睇部機器嘅內部運作。究竟個編排後端係點樣管理、派發、監察、重試,同埋同一百一十四個 AI agent 嘅大軍溝通呢?話個系統有齊呢啲 agent 係一回事,要真正明白佢點樣令佢哋運作起嚟,就係另一回事喇。

核心問題係咁。後端手上有一百一十四個 agent 定義,但佢實際上係點樣創造佢哋、運行佢哋、同佢哋對話,並且將佢哋牢牢控制住呢?

答案會令你有少少意外。每個 agent 都唔係一部獨立嘅伺服器,亦都唔係一個自己喺度運轉嘅小型微服務。一個 agent 其實只係一份配置。佢由一段系統提示、一組工具,同埋一份品質評分準則所組成,全部捆綁喺一齊。呢份配置要等到有一個 worker 進程拎起佢,再針對某個任務去執行嘅時候,先至真正活起嚟。

有個好幫到手嘅比喻,就係一隊管弦樂團。後端係指揮。各個 agent 就係樂譜,即係紙上面嗰啲指示。Worker 就係真正演奏嘅樂手。而大型語言模型就係樂器,係產生聲音嗰份原始能力。

噉一個 agent 喺運行時候究竟係乜嘢呢?佢唔係一個長期運行嘅進程。佢更似一個無狀態嘅函式,有需要先至會被調用。每個 agent 定義都帶住幾樣嘢:一個介乎一同一百一十四之間嘅編號、一個名稱例如 Director Agent、一段話畀佢知自己係邊個嘅系統提示,譬如你係一位電影導演,同埋一張佢可以拎嚟用嘅工具清單,例如影片生成介面或者記憶喚回功能。佢仲有一個架構模式,可能係 self-refine、reflexion 或者 react。佢帶住一份品質準則,連同佢必須達到嘅門檻。佢知道自己接受邊啲 agent 嘅批評,亦都知道自己可以評論邊啲 agent。佢仲有一個最大迭代次數,同埋一個偏好嘅模型。

所以當後端需要 Director Agent 去生成,譬如話第五個鏡頭嘅創作意圖嘅時候,佢會行一條好清晰嘅流程。佢先載入個定義。再將系統提示、任務脈絡,同埋任何批評歷史合併起嚟,砌出提示。跟住佢調用語言模型。模型就會決定要調用邊啲工具。後端代表個 agent 去執行嗰啲工具調用。如果個模式要求自我修煉,佢就會循環。最後,佢將結果連同一系列事件一齊發布出去。

而家講下編排引擎,即係成件事嘅大腦。佢嘅核心係一個狀態機,形狀好似一個有向嘅工作圖。佢會經過一個個節點,由 brief 開始,去到計劃,再去到路由,然後同時並行咁製作多個部分,最後到達一道閘。節點之間嗰啲邊係有條件嘅,即係話系統要過咗一道閘先至向前行。佢會追蹤狀態,例如而家係邊個階段、邊啲 agent 正在運作、有邊啲任務仲未完成,以及仲剩幾多預算。而最關鍵嘅係,佢每行完一步都會儲存一個檢查點,所以乜都唔會走失。

圍住個核心,有幾個專門嘅部件。有一個任務派發器,負責決定邊個 agent 運行、佢處理乜嘢任務、用邊個模型、幾時運行,以及優先次序點排。有一個任務佇列,每個 agent 各有一條獨立嘅排隊線。有一個批評路由器,根據嗰啲接受批評嘅關係,將回饋訊息喺各個 agent 之間傳遞。仲有一個閘評估器,負責檢查條件、觸發審批,並且喺人類話 OK 之後推進製作。

等我哋行一次後端實際上點樣將任務交畀一個 agent。想像下編排器決定咗 Director Agent 應該負責第五個鏡頭。

第一步係任務創建。編排器砌出一個任務物件。入面有一個任務編號、agent 嘅號碼、任務嘅類型,譬如生成一個鏡頭意圖,同埋輸入。嗰啲輸入係用儲存編號去引用各項素材:劇本、一格分鏡、一塊情緒板。佢仲包含任何需要處理嘅批評,好似用闊一啲嘅鏡頭,或者第二場景清晰度偏低。佢亦都帶住約束條件:用邊個模型、用邊個生成工具、仲剩幾多預算,以及呢一步嘅最高成本。

第二步,任務進入該 agent 嘅佇列,然後有一個空閒嘅 worker 拎起佢。可以將個 worker 池諗成一個線程池。同一時間大約有十到五十個 worker 喺度運行,數量係可以配置嘅。任何一個 worker 都可以執行任何一個 agent 嘅任務,因為 worker 係無狀態嘅。佢哋只係按住自己揀到嘅任務,載入對應嘅配置咁解。

第三步係 worker 內部嘅實際執行。Worker 載入 Director Agent 嘅定義。佢由素材庫攞返輸入素材。佢用向量搜尋,由記憶服務拉返相關嘅記憶。然後佢為語言模型砌好啲訊息。系統訊息設定角色。用戶訊息列出任務,即係為第二場景第五個鏡頭生成一個鏡頭意圖,連同劇本脈絡、分鏡描述、情緒參考,可能係帶住雨水意象嘅憂鬱新黑色風格,以及要處理嘅批評,譬如剪接師要求用闊一啲嘅鏡頭,或者觀眾模擬器報告第二場景嘅清晰度分數係零點六,低過零點七嘅目標。佢要求輸出係一個結構化嘅鏡頭意圖,描述鏡頭、主體、風格同時長。

跟住 worker 調用模型。模型回應一啲工具調用。譬如話,佢可能要求影片生成工具,用一個指定嘅種子值去製作一個緩慢嘅推軌鏡頭。Worker 真係會執行嗰個調用,透過網絡聯絡影片服務,然後攞返一條影片連結加上元數據。之後,本住自我修煉嘅精神,模型評估個結果。佢調用一個評分工具,將影片同文字提示作比較。假設佢攞到零點三四嘅分數,對住零點三二嘅門檻。咁就過關。如果分數跌落門檻之下,worker 就會循環返去,帶住回饋再試。一達到標準,任務就完成。

第四步,worker 將結果發布去事件匯流排。佢宣布有一項素材已經創建、該 agent 嘅狀態已經轉為完成,並且貼出一個帶分數嘅指標更新。編排器聽到任務做完,就決定下一步點行。

噉後端係點樣管理一個 agent 嘅生命週期呢?每個 agent 都會經過一連串狀態。佢由閒置開始。跟住變成排隊中,再到運行中,然後到完成,到呢一刻佢就發布結果。但運行中可以分岔。佢可能進入自我修煉再循環,帶住回饋重新運行。佢可能進入等待狀態,守住等批評,等嗰份批評到咗就恢復。又或者佢可能失敗,噉就會跌入重試邏輯,甚至返返去佇列。另外,一個 agent 仲可以變成受阻,等緊一道閘嘅審批,或者等緊佢需要嘅某啲輸入。

後端會為每個製作入面嘅每個 agent 保存一份詳細嘅狀態紀錄,儲存喺資料庫。嗰份紀錄包含製作編號、agent 號碼、目前狀態、目前任務、佢行緊第幾次自我修煉迭代、佢上一組指標、任何仲未處理嘅批評、重試次數、該 agent 至今花咗嘅總成本,以及佢開始同完成嘅時間戳記。

關於各個 agent 點樣互相溝通,呢度有樣好重要嘅嘢。佢哋從來唔會直接對話。後端徹底居中調停所有嘢。想像 Director Agent 完成咗一個鏡頭意圖。佢發布有一項素材已經創建。後端跟住檢查邊個被配置成可以評論 Director Agent 嘅輸出,然後將一個批評任務交畀,譬如話 Editor Agent。剪接師帶住回饋返嚟,可能話節奏太快。後端就檢查 Director Agent 究竟有無接受嚟自剪接師嘅批評。如果有,佢就遞交嗰份批評。喺下一次迭代,導演就會將回饋融會入去。

背後嗰套路由邏輯,精神上好直接。當一個 agent 發布一項素材,後端就搵出每一個被配置成可以批評呢個生產者嘅 agent。對每一個批評者,佢只會喺生產者被設定為接受嚟自嗰個特定批評者嘅批評時,先至遞交批評。當兩邊都對得上,佢就將一個批評任務排入佇列。呢個係一個雙向握手,而後端永遠都係中間人。

噉並行運行多個 agent 又點呢?喺製作階段,好多 agent 會同時開工。一個 worker 可能喺度處理 Director Agent 嘅第五個鏡頭。另一個喺度優化提示。另一個喺度為第四個鏡頭做品質檢查。另一個喺度拉取視覺參考。再有一個喺度為第二幕譜寫一段音樂主題。佢哋全部都將事件發布去匯流排,而編排器就協調佢哋之間嘅依賴關係。譬如話,剪接師喺導演完成某一場景嘅每一個鏡頭之前,都唔可以開始組裝。

為咗知道幾時派發先至穩陣,編排器靠一個依賴圖。有啲任務要等其他任務先完成。組裝剪接需要導演嘅所有鏡頭完成,加上作曲嘅配樂準備好,再加一道已經通過嘅製作閘。調色需要粗剪完成。混音就需要音效設計同作曲嘅最終混音兩樣都齊。所以每完成一個任務,編排器就檢查啲待辦工作,一旦某個任務嘅所有前置條件都滿足,佢就派發嗰個任務。

失敗係無可避免嘅,所以後端有一套清晰嘅應對方案。一個任務可能因為模型出錯、工具逾時,或者品質跌落門檻之下而失敗。編排器入面嘅重試邏輯會問一連串問題。重試次數係咪仲低過上限,即係預設嘅三次?如果係,就以指數退避重新排隊,等五秒,跟住十五秒,再四十五秒。佢係咪一個暫時性錯誤,例如介面逾時或者速率限制?咁就用同樣嘅參數重試。佢係咪一個品質失敗,即係分數太低?咁就重試,不過先畀一個提示優化器去調整提示。佢係咪超咗預算?咁就用一個較平嘅模型作後備再試。如果所有重試都用晒,該 agent 就會被標記為失敗。用戶會透過一個即時連接收到通知,個視覺圖上面嗰個節點會變紅,而用戶可以揀重試、跳過,或者修改再重試。

記憶值得特別講下。記憶 agent 唔係排隊入面又一個普通嘅 worker。佢係一個共享服務,所有其他 agent 都可以調用。佢有兩半。一半係向量資料庫,儲存好似風格鎖定、語氣筆記、過往決定,同埋批評歷史呢啲嘢。另一半係結構化儲存,保存系列聖經條目、角色狀態、連戲紀錄,以及預算決定。兩邊都匯入去一個單一嘅記憶介面,有兩個主要動作:喚回,接收一個查詢然後返回相關條目;以及儲存,將一個新事實持久化。任何 agent 喺自己模型運行緊嘅時候,都可以將呢個當成工具去調用。所以當 Director Agent 喺度處理第五個鏡頭,而模型諗住,我需要知道我哋為第二幕鎖定咗乜嘢視覺風格嘅時候,佢就調用喚回,攞返類似咁嘅資料,風格鎖定係一個指定嘅影片模型種子,帶住憂鬱新黑色嘅外觀,然後用呢樣嘢去令自己嘅提示保持一致。

等我哋由頭到尾追蹤一次完整嘅控制流程。用戶撳啟動。請求去到 API 閘道,閘道再發出建立一個製作嘅請求。製作管理器創建一條資料庫紀錄,載入揀好嘅範本,並且初始化預算。然後編排引擎接手。

第一階段係計劃。計劃 agent 被調用,模型將個 brief 拆解成一個任務圖,連埋當中嘅閘同依賴關係。然後路由 agent 被調用,喺尊重預算嘅前提下,為每個任務指派一個模型同供應商。

第二階段係執行,佢會一直循環,直到每個階段都完成。任務被派發去 worker 池,喺依賴關係容許嘅地方並行,喺次序重要嘅地方順序進行。每個 worker 都透過嗰個熟悉嘅循環去執行自己嘅 agent 任務:載入配置、砌出提示、調用模型、執行工具、自我修煉,然後發布結果。當一個任務完成,後端就更新該 agent 嘅狀態,檢查係咪有批評者需要運行,檢查係咪啱啱有新嘅依賴關係得到滿足,然後派發下一批合資格嘅任務。而當到達一道閘,一個守閘 agent 會評估條件,一個評審 agent 會按準則評分,如果一切自動通過,製作就推進。如果需要人類,系統就暫停,並通知介面,然後等待。一獲批准,佢就推進。一被拒絕,佢就將工作送返畀修訂 agent。

第三階段係交付,喺所有閘都通過之後開始。一個分發 agent 為每個渠道將輸出打包,一個合規 agent 用來源憑證為內容簽署,然後將作品發布去目標平台。

噉總結下後端嘅角色。佢透過將 agent 配置儲存喺資料庫嚟定義 agent,包括提示、工具、準則同關係。佢用依賴圖嚟決定幾時運行,只喺前置任務完成時先派發。佢透過路由器同成本規則嚟決定用邊個模型,揀出仍然合乎品質而又最平嗰個。佢透過載入配置、砌提示、調用模型同運行工具嚟執行 agent 邏輯。佢透過檢查品質指標,並喺需要時帶住回饋循環,嚟管理自我修煉循環。佢透過遵從接受批評嘅配置嚟路由批評。佢喺資料庫管理狀態,追蹤每個 agent 嘅狀況、重試、成本同指標。佢透過重試、退避、較平嘅後備同人類升級嚟處理失敗。佢透過評估條件、為人類暫停,並喺批准時推進嚟執行閘。佢透過向個池加入更多 worker 嚟擴展。佢仲透過事件溯源嚟持久化一切,所以每一個決定、批評同素材都可以重播。

呢度有個關鍵洞見值得帶走。Agent 唔係獨立嘅服務。錯嘅心智模型,係想像每個 agent 都係自己一個全日運轉嘅微服務,一百一十四個一齊喺度嗡嗡作響。正確嘅心智模型,係 agent 係配置,由 worker 按需要去實例化。有一個大約十到五十個 worker 進程嘅池。喺任何一刻,可能有一個喺度執行導演嘅任務,另一個喺度執行提示工程師嘅任務,另一個閒置咁等緊,再有一個喺度做品質保證。每個 worker 都可以執行任何一個 agent。佢載入嗰個 agent 嘅配置,運行模型,然後行去做下一件事。就好似一個演員飾演好多角色。同一個人,唔同嘅劇本。

呢個就正正係點解一百一十四個 agent 唔需要一百一十四部伺服器。一個唔大嘅 worker 池就應付到佢哋全部,只係喺任務一變得可用,就由佇列度拎起嚟做。
