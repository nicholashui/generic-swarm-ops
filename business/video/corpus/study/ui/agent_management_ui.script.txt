Let's talk about how you actually manage the individual agents inside this system. Not the big production runs, but the agents themselves. How do you look at how one is set up? How do you test it on its own? And how do you make it smarter over time? That's what the Agent Management area is for, and it gives you three distinct ways to work.

The first is Configure. The second is the Playground. And the third is Knowledge. Think of them like this. Configure is where you view and edit how an agent is built. The Playground is where you test an agent in isolation, giving it a task and watching how it thinks. And Knowledge is where you feed it new information so it performs better. The nice thing here is that none of this requires running a full production, and nothing costs you anything unless you actually fire up the Playground and generate something.

So let's start with Configure. You get to any agent from the Agent Registry. You click on it, and you land on its configuration. Let's use the DirectorAgent as our running example. Across the top you'll see tabs for Configuration, Playground, Knowledge, History, and Metrics, so everything about a single agent lives in one place.

The configuration screen is organized into clear sections. At the top is identity. That's the agent's name, its category, and a short description of what it does. For the director, the description says it owns the creative vision, it issues shot intents, it sets the pacing, and it approves takes. In other words, it's the creative authority of the whole production.

Below that is the system prompt. This is the heart of the agent, the instructions that define who it is and how it behaves. The director's prompt tells it to act as an elite film director with deep knowledge of visual storytelling, drawn from sources like Criterion commentary tracks, Directors Guild seminars, and masterclass material from filmmakers like Scorsese, Lynch, and Gerwig. From there it lays out the job. Translate screenplay scenes into precise shot intents. Define the camera movement, the composition, and the lighting mood. Set pacing that matches what the genre expects. Review the generated shots against the creative vision. And issue creative-intent notes to the other agents. The prompt also specifies what the agent should produce, things like camera move, framing, subject, style, duration, and mood. You can see how long the prompt is, expand the editor for more room, and pull up earlier versions through version history.

Next comes the architecture pattern. This is how the agent reasons internally, and you can choose from a menu of approaches. The director here uses a self-refine pattern, where the agent loops and improves its own work before accepting it. Other options include things like Reflexion, ReAct, Constitutional AI, multi-agent debate, and a few more. Alongside that you set the practical limits, such as how many self-refine loops it's allowed before settling, the temperature that controls creativity, and the maximum length of its output.

Then there's model assignment. Every agent gets a primary model and a fallback in case the first one isn't available. For generation work, you assign a primary generation tool and a backup as well. So if the main video generator is busy or fails, the agent has somewhere to turn.

After that is tools. This is a checklist of capabilities the agent is allowed to use. The director has several video generation tools switched on, along with the ability to recall memories and store decisions, and a scorer that measures how well a clip matches the text it was meant to follow. A couple of tools are switched off because the director simply doesn't need them, like image generation and voice. And you can always add a custom tool if you need one.

The next section is the quality rubric. This is where you define what good actually means for this agent, as a set of measurable thresholds. For the director, the text-to-video alignment score has to clear a minimum bar. Every story beat has to be covered, with no exceptions. The pacing has to fit the genre well enough, and the visual style has to stay consistent from shot to shot above a set level. Each metric comes with a threshold and a short description, and you can add new metrics or adjust the bars whenever you like.

Relationships come next, and this is where the social structure of the swarm shows up. You decide which agents this one accepts critique from, and which agents it's allowed to comment on in return. The director, for example, takes feedback from the screenwriter, the editor, and the audience simulator, and it offers critiques to the editor, the director of photography, the screenwriter, and the composer. This is how the agents check and balance each other.

Then there are cost controls. You can cap the maximum spend on a single task, limit how many copies of the agent can run at the same time, set a timeout so a task can't run forever, and decide how many times it should retry if something fails.

At the bottom you've got the actions. You can save your changes, reset everything back to default, export the whole configuration, or clone the agent to make a variation. And there's an important warning here. Any changes you make apply only to future productions. Anything already running keeps the configuration it started with, so you never disrupt work that's in flight.

Now let's move to the Playground. This is one of the most useful tools in the whole system. Think of it as a sandbox where you hand an agent a single task, then watch it think and produce, completely isolated from any real production.

The screen splits into two halves. On the left is the input panel, where you set up the task. You pick a task type, like generating a shot intent, critiquing an artifact, reviewing a cut, or running a custom prompt. Then you give it scene context. In our example, the scene is an interior, a coffee shop at night, rain streaking down the window. A character named Maya sits alone, staring at her phone. The last text she received simply reads, I'm not coming. She turns the phone face down. You can drop in reference images to guide the look. You can also simulate critiques from other agents, so you can see how the director responds when, say, the editor or the audience simulator pushes back. And you can lock in a style or a memory context, like a neo-noir, melancholic mood tied to a specific generation seed.

Below that are the run settings. You can override the model just for this test. You can choose whether to actually generate video, which costs real money, or run text-only, which is basically free. You can turn self-refine on and set how many iterations it gets. And before you commit, it shows you an estimated cost. Then you simply run the agent, or run it text-only for free, or even line it up against another agent for comparison.

The right side is the output panel, and this is where it gets interesting, because you don't just see the final result. You see the agent's thinking trace, step by step. First it analyzes the scene context, noting the night setting, the emotional isolation, and the rain motif. Then it reaches into memory and recalls the established visual style for scenes with this character, which comes back as neo-noir, cool tones, shallow depth of field. Next it generates the actual shot intent. A slow push-in from a medium shot to a close-up. An off-center frame with an empty chair on the right to emphasize her solitude. Lighting from practical sources only, neon glowing through the rain. A mood of loneliness and resignation, running about eight seconds. After that it calls the video generator and you watch the status as it works. And finally it evaluates its own output, scoring the alignment between the video and the prompt. In this case the score clears the threshold, so the agent passes and accepts the result on its very first try.

Underneath the trace you get the actual output. The generated video with its scores for alignment, aesthetics, and style consistency, plus the shot intent itself, ready to copy or export. And there's a cost and performance summary that tells you the total spend, the time it took, the number of tokens used, how many iterations it ran, how many tool calls it made, and which model it used. From there you can save the whole thing as a test case, add it to the agent's knowledge, run it again, or try a different model.

The Playground has a lot of depth. Text-only mode lets you watch the reasoning without paying for video. Video mode runs the whole thing end to end for real. Mock critiques let you simulate feedback from other agents. The compare feature lets you run the same input through two different agents, or the same agent under two different model setups, so you can weigh quality against cost in an A/B style test. You can bookmark any run as a test case for regression checking later. The thinking trace shows you every reasoning step, tool call, and decision. And history saves every single run, so you can come back and re-run or compare results over time.

That brings us to the third mode, Knowledge. This is where you actually make an agent smarter, by feeding it new references, examples, corrections, and domain expertise. It's organized into five kinds of knowledge.

The first is reference documents. These are documents that get embedded into the agent's knowledge base so it can pull from them when it works. For the director, that might be Criterion commentary notes, transcripts from Directors Guild seminars, masterclass notes, composition guidelines, a file of pacing expectations by genre, and any custom style guide you upload yourself. You can add documents by uploading them, pasting text, or importing from a link. Behind the scenes, these get broken into chunks, embedded, and stored in a vector database, so the agent can retrieve just the relevant pieces while it's doing a task.

The second kind is few-shot examples. These are input and output pairs that teach the agent by showing it. For instance, a romantic comedy park scene, with a sample scene description going in and a golden, ideal shot intent coming out. Or an action car chase that should produce rapid cuts. Or a horror reveal that calls for a slow build, a static camera, and minimal cutting. You can add examples by hand, pull them in from a Playground run, or even generate them from a document. When the task type matches one of these examples, it gets slotted into the prompt, and generally, the more good examples you provide, the more consistent the agent becomes.

The third kind is corrections and feedback. This is a record of past mistakes paired with what the agent should have done instead. One correction might note that on a particular production, the agent used a handheld camera for an intimate scene, when it should have used a static tripod with a slow push, because handheld implies urgency rather than intimacy, and quiet emotional scenes call for stillness or a slow dolly. Another might note that the agent framed something in widescreen for vertical social content, when it should have gone vertical, with the lesson being to always check the platform target before choosing a frame. Corrections can come from your direct feedback or be detected automatically when a piece of work gets rejected. They're stored as a kind of episodic memory, so the agent can check what went wrong before when it faces a similar task.

The fourth kind is custom rules. These are hard constraints that the agent must follow, on top of its system prompt. Things like never use a Dutch angle unless the genre is horror or thriller. Always include at least one establishing shot per scene. Keep shots under twelve seconds for social content. Or, when the budget is running low, prefer a static camera because it's cheaper to generate. These rules are injected as firm constraints, and the agent checks itself against them during every self-refine loop.

The fifth kind is evaluation benchmarks. These are test cases that let you measure whether your knowledge improvements actually worked. You might have a suite of a dozen test cases, with a record of when it last ran and how many it passed. After you add new knowledge, you run the benchmark again to see if things got better. Each test lists its input, the expected behavior, the last result, and whether it passed or failed. So if a test for vertical TikTok framing is still coming back horizontal, you can see at a glance that the agent hasn't learned that lesson yet.

It helps to understand how all of this knowledge actually flows into the agent when it's working on a real task. The prompt the model receives is assembled from layers. First, the system prompt from the configuration. Then the custom rules, spelled out as things the agent must follow. Then the relevant knowledge retrieved from the reference documents, matched to the task at hand, so for an intimate, emotional scene it might pull a note about how certain directors use static frames for emotional weight. Then the relevant few-shot examples that match the task type. Then the corrections memory, reminding the agent of past mistakes to avoid. And finally the task context from the current production, including the scene itself and any critiques from other agents. All of those layers are combined into a single prompt, sent to the model, and the agent responds with that full picture in mind.

So how does improving an agent actually play out over time? It follows a natural loop. First you observe a problem during production, like the director repeatedly using handheld for quiet scenes, or failing the same quality criterion again and again. Then you add a correction in the Knowledge tab, recording what it did wrong, what it should do, and why. Next you test it in the Playground, giving it a similar scene to see whether it now behaves correctly. If it does, great. If not, you may need a stronger rule or more examples. So if the correction alone isn't enough, you add a few-shot example, a golden input and output for that exact situation. Then you run the benchmark to make sure you actually improved things and didn't accidentally break something else, watching the score climb. And finally it deploys automatically, so the next production uses the improved agent, while any productions already running stay untouched for stability.

On the storage side, all of this knowledge is kept per agent, and it can be either global or tied to a specific project. For each agent the system tracks its reference documents, broken into chunks and embedded for retrieval. It tracks the few-shot examples by task type. It tracks the corrections, each with the context, the wrong behavior, the correct behavior, an explanation, and where it came from. It tracks the constitutional rules with their priorities. And it tracks the benchmark test cases with their latest results.

That scope distinction matters. Global knowledge applies to an agent across every project. That's where the default system prompt lives, along with the core reference documents, the universal rules, and the base examples. It's usually managed by a platform admin or whoever maintains the agent. Project-specific knowledge, on the other hand, applies only inside one project. That's where a brand's own style guide goes, along with corrections that only make sense in that project, custom examples matching its genre and tone, and project-only rules. It's managed by the project owner or editor. At runtime the two are merged, and when they conflict, the project knowledge wins. So if a project rule says everything should be vertical, that overrides the global default.

There's one more powerful idea here, which is auto-learning. The system doesn't just wait for you to teach it. It watches production history and suggests improvements on its own. If an agent fails the same criterion three or more times across productions, it suggests adding a rule about that pattern. If you keep overriding an agent's output in the same way, it offers to create a correction from your pattern. If one agent's work keeps getting critiqued the same way, it suggests turning that into an example. And when gate rejections share the same feedback, it can automatically create correction entries and flag them as pending your review. These show up as friendly suggestions. For instance, this agent used widescreen for vertical content four times, want to add a rule? Or, the editor critiqued pacing six times, want to add an example? Or, alignment keeps coming back low on night scenes, want to add a reference guide? In each case you can accept, dismiss, or look at the specific instances before deciding.

So to pull it all together, you have three modes that work as a team. Configure, where you view and edit the prompt, tools, rubric, relationships, and model, and it costs nothing. The Playground, where you test an agent in isolation with your own input, costing anywhere from nothing to a few dollars per run. And Knowledge, where you add documents, examples, corrections, and rules, costing nothing beyond storage. Put together, the rhythm is simple. You see a problem in production, you add a correction in Knowledge, you test the fix in the Playground, you verify it with a benchmark, and from that point on, your agent is smarter for every production that follows.
