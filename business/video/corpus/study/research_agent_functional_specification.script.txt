Imagine you could hand a computer a single topic, walk away, and come back to a polished, deeply researched report, complete with a knowledge map, study drills, image ideas, and even a ready-to-record video script. That's the promise behind the Research Agent, and in this video we're going to walk through exactly how it works, from the first idea all the way to the finished package.

So let's start with the big picture. This is a local-first research automation tool. It runs as a command-line application on your own machine, and its job is to take a topic you supply and turn it into a detailed report through a carefully staged pipeline. It doesn't just spit out one answer and call it done. Instead it moves through a sequence of steps: defining the scope, discovering sources, curating the good ones, extracting their content, assembling a master notebook, synthesizing a draft, optionally saving full offline copies of sources, polishing the final report, compiling a knowledge base, generating study drills, creating image prompts, and writing a YouTube script. Every step leaves behind an inspectable file, so nothing is hidden inside a black box.

The design philosophy is built around a few clear principles. It keeps humans in control at the key decision points. It stores everything locally in session folders that you can pause and resume. It uses Grok, the model from xAI, through an OpenAI-compatible interface, for all of its language generation. It can optionally pull in your own local documents to steer the research. And above all, it produces intermediate artifacts you can examine, rather than a single opaque result.

Now, it's just as important to understand what this system does not try to be. There's no web interface, no API server, no multi-user collaboration. There's no login, no permissions, no roles. It doesn't use a database. There's no semantic vector search or retrieval index. It won't drive a browser or run a crawler beyond simple, direct fetching of web pages. It makes no guarantee that what the model produces is factually correct. And when you feed it a document, it does its best to read the text, but it won't deeply handle binary formats beyond that.

Let's talk about who uses it. Really, there's just one kind of person here: the research operator. That's whoever starts a session, approves or revises the outputs along the way, picks which sources to keep, decides whether to save full offline copies, and runs the extra commands. There's also a notional reviewer or study user who consumes the report, the drills, the knowledge map, and so on, but technically that's the same person. On the system side, the actors are the language model provider, that's Grok; the remote content hosts, meaning the websites and PDF endpoints it fetches from; the local filesystem where everything is stored; and the local environment that supplies things like the API key and the text editor. Because there's no identity or permission layer, anyone who can run the program and read the session folder can do everything.

Architecturally, the system is organized into a handful of modules. There's a command-line layer that parses your arguments and maps any failures to exit codes. There's a session manager that creates and saves session state and hands out canonical folder paths. There's a workflow module that implements the whole state machine, handling fetching, extraction, synthesis, compilation, and all the rest. There's a Grok client that loads configuration, makes the calls, and translates API errors into meaningful domain errors. There's an external-documents module that recursively reads your local files and pulls out steering context. And there's a collection of prompt templates that define what each language model call should produce. The whole thing is a single process. Each command creates a new run folder, works on files in the session directory, and writes run-local copies for traceability. There's no background service humming away; the state lives entirely in files.

On the technical side, it needs Python 3.11 or newer, and it leans on a set of well-known libraries for talking to the model, reading environment files, rendering nice terminal output, validating data, parsing PDFs and Word documents, fetching web pages, and extracting readable text from messy HTML. It installs as a command you can call directly, with a couple of convenience wrappers that forward to the same entry point.

Configuration is handled through environment variables. The most important is the Grok API key, which is required for anything that calls the model; if it's missing, those actions fail with a clear message. You can optionally set the model name, which defaults to Grok 3, and a blank value quietly falls back to that default. There's a maximum output token setting that defaults to fifty thousand, with invalid values reverting to the default and anything below one clamped up to one. There's a request timeout that defaults to three hundred seconds, with the same protective rules. And there's an editor variable, used only when you choose to edit the scope by hand; if it's not set, the system still creates the editable file, it just won't launch an editor for you. When the workflow builds its default client, it also looks for an environment file two folders above the session directory, and if it doesn't find one, it simply uses the process environment instead.

The interface itself is a terminal experience, using a rich rendering library to show status messages, echo Markdown content, and display preview tables. Along the way there are four human interaction points: confirming the scope, approving the curated sources, approving or revising the draft, and selecting which sources to save offline. But there's also an unattended mode. Switch on auto mode and the system bypasses all the prompts and drives itself to completion. In that mode the scope is auto-confirmed, every source is selected and approved, the draft is approved automatically, and the full-collection choice defaults to all unless you say otherwise. Critically, auto mode never pauses to ask for input.

Let's get into the commands, because this is where the system's capabilities really show. Every command except the one that just lists types needs you to specify a sessions directory, and commands that act on an existing session also need a session ID. The program returns a zero exit code on success, a one when the workflow raises a Grok error or a quota error, and a two for unrecognized commands or argument validation failures. And if you turn on tracing, it prints the request and response content in a truncated, sanitized form so you can see what's happening.

The start command creates a new session with your topic, an optional focus, an optional external-documents folder, and a stored mode. It prints the new session ID and immediately begins the workflow at phase zero. The accepted modes are report, compiler, and drill, though here's an honest note: the mode is saved but doesn't actually change runtime behavior in the current implementation. The resume command loads a session and picks up from its current phase; in interactive mode it stops at the next human checkpoint or after a phase that tells you to resume again, and if the session is already complete, it simply tells you so. There's a command to list all sessions, which scans for folders containing a session file and skips anything that isn't a valid session, telling you when none exist. There's a command to list types, which just prints the single supported type and needs no session folder.

Then there are the more specialized commands. Update re-runs discovery looking only for new material since the last run, sets the phase back to curation, and tells you to resume. Synthesize forces the synthesis phase to run regardless of where you are, though its prerequisites still apply, so without a notebook it won't proceed. Compile builds the knowledge base, accepting the auto-hypergraph type, working from the master notebook and any extracted content, and stopping with a helpful message if neither exists. Drill generates study questions, supporting only the backward mode; if the core concepts file is missing it tries to compile automatically, and if it still can't find them, it tells you to run compile first. Feed takes a new document, requires you to point at it, checks that it exists and is a real file, copies it into the knowledge base with a timestamp, and if there's no existing knowledge map, it compiles one and returns. Show turns the knowledge map into a Mermaid diagram, or tells you to compile first if there's nothing to render. Generate-images and youtube-script both require a final report; if it's missing they say so, and on success they write their outputs into both the run folder and the session folder.

Now let's talk about how sessions are identified and stored, because resumability is central. Each session ID comes from a slugified version of your topic plus the current date. Slugifying means lowercasing the topic, replacing anything that isn't a letter or number with a hyphen, collapsing repeated hyphens, and trimming the ends. If the slug is too long it gets trimmed and a short hash suffix is appended. And if a folder by that name already exists, the system adds a numbered suffix until it finds a free name. The session state itself captures a rich set of fields: the ID, topic, focus, and mode; a whole family of details about any external documents, including their status, summary, file counts, completion rate, relevance score, and last error; timestamps for creation and updates; the model name; the current phase; and a run history list. State is saved as UTF-8 JSON, the update timestamp refreshes on every save, and folders are created automatically as needed. One honest note here too: that run-history list exists in the schema but isn't actually filled in by the current code.

Every command that builds a workflow context also creates a fresh run directory, named with a precise timestamp down to the microsecond. If two timestamps ever collide, it retries up to a thousand times, and if it somehow still can't find a unique name, it raises an error rather than overwriting anything.

One of the more powerful features is external-document preprocessing. The idea is that before the workflow runs, you can feed it your own local reference documents, and it converts them into mandatory steering context that shapes the scope, discovery, curation, extraction, and planning. This preprocessing runs automatically before most commands, though not before the ones like generate-images, youtube-script, compile, drill, feed, and show. If you haven't pointed it at a folder, it's skipped. And if it already finished successfully with a summary on file, it won't needlessly run again. It supports PDFs, Word documents, plain text, and Markdown, discovered recursively, and it simply ignores file types it doesn't understand rather than failing.

The processing itself is thoughtful. Each file is read with logic suited to its type. For PDFs it goes page by page, skipping any page whose text can't be extracted. For Word documents it joins the non-empty paragraphs. For text and Markdown it reads as UTF-8, replacing any invalid characters. Each document gets categorized, based on keywords in its filename, as a guideline, background, steering, or general document. The processor then extracts key concepts, constraints, requirements, and algorithm insights using sentence-level heuristics, and it computes a relevance score from how much the document's vocabulary overlaps with your topic and focus, plus bonuses for relevant terms, constraints, and requirements. The results are written out as a manifest with per-file results and aggregate metrics, an extracted summary, and a context file that gathers the key concepts, constraints, requirements, optional algorithm notes, and workflow guidance. Those algorithm notes only appear if your topic or focus matches algorithm-oriented keywords. If the folder doesn't exist, the status is marked failed, an explanation is stored, and the workflow simply continues. If individual files fail, they're marked failed but the rest carry on. The overall status is completed when everything succeeds, partial when some succeed and some fail, and failed when nothing succeeds. And when this context is available, it's appended to the relevant prompts as mandatory background, truncated to fit each phase's character budget rather than ever causing a failure.

Now for the heart of it: the research workflow itself, which is a state machine with nine phases, numbered zero through eight. Phase zero is scope generation and confirmation. Phase one is discovery. Phase two is curation and gap analysis. Phase three is extraction. Phase four is notebook assembly. Phase five is synthesis and review. Phase six is full offline collection. Phase seven is final polish. And phase eight means complete. In interactive mode, each resume call processes one phase or one human checkpoint, and some phases deliberately end by telling you to resume again rather than rolling straight on. Phase transitions are saved the moment the code updates the current phase. In auto mode, the workflow just loops until it reaches phase eight, moving across phases without needing separate resume commands.

Let's walk through the phases. In phase zero, the system generates a Markdown scope summary, writes it to a scope file, and prints it to the console. In auto mode it accepts the scope immediately, saves a confirmed copy, and advances to phase one. In interactive mode your choices are yes, edit, or cancel. Cancel stops without changing the phase. Edit writes a temporary file, optionally launches your editor, reloads and prints the edited content, and keeps prompting. Yes saves the confirmed scope, advances the phase, and tells you to resume. And if the Grok client can't even be created, it prints the error along with guidance about the environment file and returns without changing anything.

Phase one is discovery. The system fills in a discovery prompt with your topic, focus, and whether it's looking only for new material, then writes the discovery table to both the run and session folders. It doesn't validate the table's format before saving. In the normal flow, finishing discovery advances to phase two and tells you to resume for curation.

Phase two is curation and gap analysis. It needs that discovery table; without it, it stops and tells you to resume from phase one. It prints a preview of up to the first eighty non-empty lines of the discovery output. Your instruction can be free-form text, choosing sources by number, saying all, adding URLs, removing indexes, or asking for a gap check, and rather than parsing those locally, it passes them straight to the model. It tries to generate curated sources up to three times, and on retries after a failure it adds stricter instructions to return only JSON and caps the list at twenty entries. Each curated source is canonicalized into a clean object with a title, URL, type, reason for relevance, credibility, and priority. URLs are normalized by trimming stray quotes and trailing punctuation. If the model returns invalid or non-canonical data on every attempt, the system recovers URLs heuristically from the discovery Markdown and builds fallback entries instead. The raw output is saved verbatim to the run folder, while a canonical version is written to the session folder. A gap analysis is always attempted and saved, and if it times out, the saved report still contains a gaps heading and an explicit timeout note. The session only advances to extraction when your approval is exactly the word approve; anything else leaves it in curation and tells you to try again later.

Phase three is extraction, and it's the most intensive. It requires the curated sources file, and if that file exists but produces an empty list, it tells you to re-curate. It creates folders for extracted dossiers, source snapshots, and extracted chunks in both scopes. It asks the model for an extraction plan and saves it, and if that times out it saves a placeholder rather than failing. It prefetches source bundles concurrently using up to four fetch workers, and if an individual fetch fails, it prints a warning and keeps going. For each source it fetches successfully, it saves both the raw content and a normalized text snapshot, with headers preserving the title, URL, host, type, priority, and credibility. HTML snapshots are saved as HTML files, PDFs as PDFs, and everything else as text. The source text is split into chunks of up to forty-five thousand characters with a five-thousand-character overlap, and those chunks are extracted in parallel using up to two workers. Each chunk prompt demands strict sections covering a coverage summary, terminology, mechanisms, workflows, evidence, limitations, open questions, quotable passages, and extraction notes. If a chunk times out, it's skipped and the rest continue. Successful chunks are written to both scopes, successful sources are assembled into numbered dossiers, and if every chunk for a source fails, that source is skipped with a warning. When the phase completes, an index file is written with a generation marker.

Phase four is notebook assembly. It requires the extracted folder, and without it, it tells you to resume from phase three. The notebook includes a top heading, a topic line, a purpose section, an optional external-documentation context section, a source catalog, an optional knowledge-base outline, and the source dossiers, all joined with separators. It's written to both scopes, and a successful build advances to phase five.

Phase five is synthesis and review. It needs the master notebook, splitting it into chunks of up to seventy thousand characters with a five-thousand overlap, and if no chunks result, it tells you the notebook is empty. For each section in a fixed list, it builds section-specific evidence packets from the notebook chunks. Those standard sections are core definitions and scope; architecture and technical mechanisms; workflows, processes, and operational patterns; evidence, examples, and case studies; limitations, trade-offs, and failure modes; and open questions and future directions. Evidence generation runs with up to two workers per section, and packets are saved to both scopes; if a section produces no evidence, it's skipped with a warning. Each drafted section is saved, and the draft report pulls together the scope and coverage text, the source catalog, the drafted sections, an optional knowledge-base alignment, and references. Drafts are saved as incrementing version numbers. The review prompt tells you that you can approve, revise a section with feedback, add a section by title, or run a gap check. Only the exact word approve advances to phase six; anything else is treated as general revision feedback and passed to the revision prompt without local parsing. If a revision times out, the prior draft stays authoritative and the phase doesn't advance, and a successful revision becomes the next version and requires another review cycle.

Phase six is the full offline collection. It tries to load the curated sources, and if they're absent, it falls back to recovering URLs from the discovery table. If it can't recover anything, it advances to phase seven, saves, prints a skip message, and waits for you to resume for finalization. Otherwise it shows a selection list with index, title, and URL for each source. You can say all, none, or give comma-separated numbers, and non-numeric tokens are ignored. Saying none advances to phase seven and skips. Saying all selects everything. Numbers outside the valid range are ignored. Selected sources are prefetched, and for each one fetched successfully, a full offline copy is written to both scopes. If a selected source can't be fetched, it's skipped without aborting. Once the attempted copies are written, the system advances to phase seven, runs the final polish immediately, and then marks the session complete at phase eight.

Phase seven is the final polish. It requires both the master notebook and at least one draft, otherwise it says they're missing. It generates an executive summary, substituting a placeholder if that times out, and a glossary, again with a placeholder on timeout. If the latest draft starts with a top-level heading, that heading is removed before assembly. It builds a table of contents from all the second-level headings in the body. The final report then contains the main heading, the table of contents, the executive summary, the body, the source catalog, an optional knowledge-base overview, and the glossary. The system tries to hit a word-count target twice if needed, once on the body and once on the assembled report, aiming for between nine thousand and ten thousand words with a target of ninety-five hundred, and that correction preserves headings and core claims while expanding or compressing the rest. The final report is written to both scopes, and then the system attempts to generate image prompts and a YouTube script. Phase eight simply means the session is complete, and resuming a completed session just says so.

There's a careful specification around fetching and transforming sources. URLs are normalized before validation, and only HTTP and HTTPS addresses with a real network location are accepted; anything else raises an error. Fetches use a custom user agent, follow redirects, and split the timeout into connect and read portions, raising a timeout error with the URL for context when needed. PDFs are detected by their content type or file suffix and returned as extracted text in several forms. Non-HTML, non-PDF responses are treated as plain text. HTML responses produce a main text from a readability extraction, a full text from the whole page, and an analysis text that merges them or falls back gracefully. During HTML normalization, script, style, noscript, and SVG tags are removed, and duplicate lines are stripped to cut down on repeated boilerplate.

Then there's knowledge compilation, which turns your research into structured knowledge. Compilation uses the notebook first and appends extracted dossiers when available. The hypergraph uses only the first two hundred twenty thousand characters of content, while core-concept extraction uses that same slice of source content plus the first hundred twenty thousand characters of the hypergraph JSON. The outputs are a hypergraph file, an auto-types copy, and a core-concepts file. The hypergraph follows a schema of nodes with IDs and labels, and hyperedges that connect several nodes with a relation and supporting evidence. If the model doesn't return valid JSON, the system wraps the raw response rather than failing. The core-concepts schema asks for concepts, each with a name, a definition, and an explanation of why it's load-bearing, and although the prompt requests exactly seven, the code doesn't independently enforce that count. The drill-pack schema produces a Markdown study pack plus structured questions grouped by concept, each with an answer and common pitfalls; if the Markdown field is missing, the system strips code fences from the raw response and uses that, and if there are no structured questions, it writes the whole parsed object out instead.

Feeding works by reading the new document as UTF-8 with replacement for any decoding errors, then sending the model the first hundred sixty thousand characters of both the existing hypergraph and the new document, and the updated hypergraph overwrites both canonical locations. Mermaid rendering begins with a graph declaration, uses up to the first two hundred nodes and the first four hundred edges, and for hyperedges with more than two members, it simply connects the first two listed, labeling edges by their relation or label.

The final outputs are clearly defined. The report is a Markdown document with explicit executive summary and source catalog sections, an optional knowledge-base overview when that content exists, and a glossary at the end even if its generation timed out. Image prompts are generated from the complete report, with the contract asking for five to ten prompts that emphasize concrete mechanisms, workflows, architectures, comparisons, and evidence rather than generic concept art, and if that generation times out during polish, the report still succeeds. The YouTube script derives its sections mainly from the section drafts when available, otherwise from the final report, deliberately excluding sections like the table of contents, source catalog, glossary, references, knowledge-base overview, and executive summary. The script contains a top heading, an introduction, one heading per selected section, and a conclusion. If the intro or outro times out, placeholder narration is inserted instead of failing; if a section times out it may be omitted while the rest proceeds; short outputs are expanded by a secondary call to hit minimum detail; and if a section lacks a heading, one is added automatically.

A word on validation. The required command flags are enforced by the argument parser, and unsupported type or mode values can't even pass parser validation. Semantic validation is deliberately light: curated-source checking is structural and best-effort, discovery output isn't structurally validated, the final report isn't checked for factual correctness, and the concept count is constrained only by the prompt. File validation does happen where it matters, the feed command checks file existence and type, external docs validate the root folder and supported suffixes, and session listing checks for the session file.

Error handling is where the system shows its maturity. A missing API key raises a clear error. Quota and billing problems are mapped to a quota error with actionable text. Timeout-like failures become a timeout error that includes the configured timeout. Other API failures are retried up to five times with exponential backoff capped at thirty seconds, while quota and timeout errors aren't retried once identified. A number of phases use an optional-call helper that turns model timeouts into warnings and keeps going, covering things like gap analysis, the extraction plan, extraction chunks, evidence packets, section drafts, revisions, the executive summary, glossary, image prompts, the YouTube pieces, and word-count retargeting. Source fetch failures never abort a whole phase. And JSON parsing is robust: the system strips code fences, tries a direct parse, then a bracket slice, then a brace slice, before finally falling back to wrapping the raw text. The overarching theme is graceful degradation, missing external docs, missing curated sources, missing knowledge artifacts, and missing reports all produce helpful messages rather than crashes.

On integrations, the Grok connection uses the OpenAI-compatible chat API at the xAI base URL, authenticating with a bearer key from the environment, sending one system and one user message per call, and reading the first completion choice. The web integration is plain HTTP and HTTPS GET with redirects enabled, no authentication, SSL delegated to the request library, and failures either bubbling up or being downgraded to warnings by design. Local document integration supports PDFs, Word, text, and Markdown, while the feed command opens files more broadly but is really intended for textual content.

Security and privacy are handled pragmatically. API keys are read from the environment or the environment file and never written into session artifacts. But session directories may store fetched remote content and processed external docs, which should be treated as potentially sensitive, since the system performs no secret redaction and no access control on those folders. There are also some honest non-functional realities. Because it's local-first, every important artifact must be inspectable on disk after each major step. Resumability depends on the current phase and which files exist, not on transaction logs or a database. Determinism is only partial: filenames and transitions are predictable, but the content is model-generated and therefore probabilistic. Concurrency is bounded, with four fetch workers, two extraction workers, and two evidence workers. And large text is handled by character-based truncation and chunking rather than precise token counting.

The team is refreshingly honest about the current gaps. The mode setting is stored but doesn't change behavior yet. The run-history field exists but isn't populated. The list of types exposes only the one auto-hypergraph type even though internal constants mention others lying dormant. The guidance strings mention adding a section or running a gap check, but no local parser enforces those, they're just passed along as revision feedback. The table of contents is built only from second-level headings. Mermaid simplifies multi-member hyperedges down to pairwise links. And factual accuracy ultimately depends on the model and the sources, because there's no automated fact verification.

So when is this considered done? The acceptance criteria are concrete. You can create a new session with a unique ID and saved state. Interactive progression can move a session from phase zero through phase eight with the expected human checkpoints. Auto mode can complete the workflow without ever asking for input. Discovery creates its table. Curation creates the curated-sources file and a gap report. Extraction creates snapshots, chunks, and dossiers. Notebook assembly creates the master notebook. Synthesis creates at least one draft. Final polish creates the final report. Polish or explicit commands create the image list and the YouTube script. Compile creates the hypergraph and core concepts. Drill creates the drill pack and questions. Feed stores a timestamped copy and can update or initialize the hypergraph. Show creates the Mermaid file. And external documents, when supplied, are processed into a manifest, summary, and context without ever blocking the workflow on partial failures.

And that's the complete blueprint. What stands out about the Research Agent is its discipline: every step is a file you can open, every failure has a graceful fallback, and the human stays in the loop at exactly the moments that matter. It takes a single topic and, step by careful step, turns it into a body of work you can actually trust and build on.
