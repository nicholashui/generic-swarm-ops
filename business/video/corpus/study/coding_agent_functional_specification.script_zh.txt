想象一下，你有一个业务问题需要用软件来解决，但你其实根本不知道该怎么描述自己想要什么，更别提怎么把它做出来了。这份文件讲的，正是为这一刻而设计的一套系统。它叫做 N1ch01as Architect，第一点零版本，最简单的理解方式，就是把它当成一位住在你电脑里的总系统架构师，它会把模糊的念头，变成一套完整、能够运行的软件，而其中没有一行代码是由人手写出来的。

让我从头给你讲讲它是怎么思考的，又是怎么运作的。

整件事建立在一个借自现代 harness engineering 的核心理念之上，就是人来掌舵，agent 来执行。使用者继续掌握方向和决策，而所有实际的建造工作，包括代码、测试、linter、持续集成、文档，甚至监控，全部都交给 agent 去做。而这个项目文件夹本身，也就是 repository，被视为唯一的事实来源，任何重要的东西都不会假设它存在于文件夹之外。

在这个基础之上，这套系统还带着一种个性。设计者形容它有一种持久的灵魂。它不是要像一个被动的聊天机器人那样，只是等你发指令。它有主见，会从第一性原理出发去推理，甚至在空闲的时候也会思考，运行他们所说的思考时钟，在使用者不在的时候，悄悄地扫描整个项目，问问自己有没有哪些地方值得改善。

接下来要说的是那台永不停止的改善引擎，灵感来自 Karpathy 提出的棘轮概念。这套系统会运行一个无限循环。它提出一个细小而单一的改善，做一个有边界的改动，评估结果，然后只有当这个改动严格优于之前的版本，才会保留下来。如果没有变得更好，这个改动就会被还原并记录下来。质量只能往上走，绝对不会横着走，也不会往下掉。

现在说说第一次真正的互动。因为大部分使用者都说不清自己需要什么，这套系统会以一个叫做引导式需求发现的步骤开始。一位意图分析员会温和地带着使用者，问极少数的问题，最多两轮，让他们不会觉得吃力。在第一轮，它会问四个背景问题。你想解决的是什么业务问题或者客户需求？主要的对象是谁？成功实际上是什么样子，无论是节省时间、收入，还是参与度？还有没有任何硬性约束，比如预算、时间表、技术偏好，或者数据敏感度？

问完这些答案之后，它会提供一份简短而精选的清单，一共六个模板类别，每个都用一句话来形容。有简单的交互式应用，比如待办清单或者测验工具。有基本的业务仪表盘，比如库存追踪。有完整的软件即服务网页应用，比如预约系统。有 AI 驱动的助手，比如聊天机器人或者推荐引擎。有多 agent 协调系统，由多个自主 agent 一起协调任务。还有数据处理流程，用来做分析或者报表。使用者选一个或者两个，或者说不用了，我想要量身定制的。

根据这个选择，分析员只会再问两三个针对性的跟进问题，然后把所有内容综合成一份精炼而专业的需求草稿。它会把这份草稿拿给使用者看，并且直接问，这份是不是你真正想要的东西？只有当使用者确认之后，这份草稿才会成为正式且锁定的需求，也就是唯一的事实来源。而最初那个原始念头，会被悄悄地存档，留待参考。

当使用者亮起绿灯说可以开始，这个 orchestrator 就会完全接手，而这里恰恰最能看出这套系统的个性。这个 orchestrator 表现得就像一位资深 IT 项目经理在带领一支开发团队。它不会自己写代码。相反，它会做规划、研究、设计，然后把写代码的工作委派给一个专职的 coding agent，就像一位真正的工程主管把结构化的工作交给开发人员一样。

这种委派会遵循一套严谨而可重复的节奏，叫做四步委派循环。第一步，orchestrator 写一份结构化的任务简报。第二步，coding agent 回应，交出真正的文件和测试，不会多说别的。第三步，orchestrator 用 code critic、tester 和一系列不变量来审查这份输出。第四步，它决定怎么做，接受并合并、要求修改，或者完全拒绝并还原。使用者自始至终都只看到 orchestrator 这一边的对话。

任务简报本身每一次都会遵循一套固定模板，力求消除一切含糊。它包括一个独一无二的任务编号、所属阶段、确切的模块名称、一句话的目标、一系列必须全部满足的验收标准、从架构中抽取出来的技术约束、即时规格当中相关的部分，以及预期的确切交付成果。里面还嵌入了一条硬性规则。所有东西都必须在本机用普通的包管理工具运行。没有 Docker，没有任何形式的容器，在整套系统里都不会有。

在任何代码交出去之前，还有多一重纪律，就是派发前的改善审查。orchestrator 必须指出具体的目标范围、说明特定的弱点、设定可衡量的改善目标，并且建议一个清晰的做法，还要引用确切的文件路径或者规格部分，而不是含糊的占位符。这样可以在开发人员动手之前，先把简报打磨锋利。

为了让整件事可信，这套系统会依靠一个研究群。这是一组十种专家研究员类型，orchestrator 可以按需要动态调用。其中有通才、系统架构专家、安全与合规专家、可扩展性与性能专家、数据建模专家、API 与集成专家、前端与用户体验专家、针对金融或医疗等领域的专门领域专家、DevOps 与可靠性专家，以及成本与可持续性专家。orchestrator 挑选适合任务的专家，并行地运行他们，如果他们意见有冲突，就会安排一场共识辩论，让他们争论到达成共识为止，最后把结果记录下来。

质量是靠机制来执行的，而不是靠猜。一个 validator agent 会在任何代码存在之前，在脑海中模拟整套系统，走完五条使用者旅程和三个边缘情况，去抓出逻辑上的缺口，再把发现写入一份风险登记册。接着一个 critic 会根据一套加权评分标准来打分，涵盖清晰度、可行性、安全与可扩展性与成本、创新、可维护性，以及不变量合规，其中以清晰度的权重最高。整体分数必须达到至少十分之九点八。这个 critic 甚至会审视自己的意见，问问自己有没有遗漏任何东西。之后，一个独立而刻意带着敌意的 paranoid reviewer，会尝试找出 critic 看漏的地方，挑战过高的分数、未经测试的假设、隐藏的复杂性，以及薄弱的防护。如果这个 paranoid reviewer 找到一个严重而未解决的问题，这道关卡就不会通过，无论之前那个分数有多高。

真正的建造工作会分成几个清晰的阶段。阶段零就是我们刚才说的引导式发现。阶段零点五是 harness 初始化，在这里 orchestrator 会建立整个项目骨架、所有身份和记忆文件、文件夹结构，初始化版本控制，做第一次提交，并且设置好初始的 linter 和测试框架。阶段一会通过那个研究、验证和评审的循环，产生后端规格。阶段二会逐个细小模块去实现后端，每一个都在自己的功能分支上面，每一个都以测试先行的方式来写，由 code critic 审查到至少打到九点五分为止，测试到全部通过，才会合并。阶段三会对前端做同样的事，确保每一个组件都用对那份确切的 API 合约。而阶段四就会处理集成、打磨和交付，产生端到端的测试、本机运行脚本、完整的文档，包括一张易读的架构图，以及最后一次管理层面的审视。

在整个过程当中，有几个后台系统会一直运行。有一个闭合学习循环，以 Hermes 命名，在每一个重要任务之后，这套系统会建立新的可重用技能、改善现有的技能，并且把笔记写入一份持久记忆和一份不断加深的使用者档案，让它随着时间和跨越多个工作会话，越来越了解这个使用者。还有一个追踪层，叫做 Agent Lightning，会悄悄地记录每一个提示、每一份任务简报、每一次工具调用、每一个被当成奖励信号的 critic 分数，以及每一个结果。它把原始细节存放在一个地方，又把每个阶段压缩后的摘要存放在另一个地方，这样在每个阶段之后，一个训练兼优化器就可以先看摘要，只在需要的时候才深入研究原始细节，再有选择地去改善那些提示和技能，同样是只保留严格更优的改动。

然后在最顶层，还有一个元 harness 外层循环。这是一个提案者，它可以完全访问整套系统的历史，包括每一个旧版本、每一条追踪、每一个分数。它的工作是去改善这个生成器本身，包括那些提示、那些技能、委派逻辑和追踪机制，评估每一个建议的改动，并且把历史存档，让整台机器可以长远地递归自我演化。

这套系统也预装了三个主流的技能框架，随时可以演化。第一个是 Superpowers，它强制执行严格的测试驱动纪律，意思是在有一个失败的测试之前，永远不会写产品代码。第二个是 GSD，也就是 get stuff done 的缩写，它用来防止他们所说的情境腐烂。当工作记忆填到大约六成的时候，质量就会倾向崩溃，所以 GSD 会把大型工作拆成一个个分阶段的子 agent 工作量，内建验证。第三个是 gstack，它会召集一支虚拟工程团队，由十五到二十三个角色组成，包括首席执行官、工程经理、设计师、质量主管、paranoid reviewer、发布经理等等，让这个项目可以同时从很多专家角度去审视。这三者是互补的，也可以合并使用。规划阶段可能会用 Superpowers 加 gstack，而执行阶段就会依靠 GSD。

被指定的技术组合完全是开源而且本机运行的。后端方面，用 Python 配 FastAPI、Uvicorn 服务器、SQLAlchemy 加 Alembic 做数据库迁移，以及 Pydantic 做验证，而 API 规格会自动生成。数据库默认用 SQLite，做到零配置的本机开发，也可以通过简单改一改连接设置，升级到 PostgreSQL 做生产用途。前端用 React 配 TypeScript、Vite 构建工具、TanStack Query 来管理服务器数据，以及 React Router 做导航。测试方面，后端靠 pytest，前端靠 Vitest，而 Playwright 就负责完整的端到端检查。Linting 方面，Python 用 Ruff，前端就用 ESLint 加 Prettier。持续集成就通过 GitHub Actions 的模板提供。而安装就刻意做到很简单，只是标准的包安装，没有容器，也没有任何专有或者被云端锁死的依赖。

整件事底下还有几项硬性保证。规格必须和实际代码保持完美同步，由一个 sync agent 在每个实现阶段之后去执行。一个 doc-gardening agent 会持续清理过时的文档和技术债务，并且铲除任何漏掉的 Docker 引用。所有东西都会提交到版本控制。每一个主要阶段都必须产生至少一个新技能或者记忆笔记。追踪必须捕捉每一个动作。优化器和元 harness 必须在每个阶段之后运行。而双重审查的保证就坚持，只靠 critic 批准是永远不够的，必须有 paranoid reviewer 和确定性的测试来佐证。

文件以一幅简单的图画来总结使用者的旅程。你从几乎没有任何头绪开始。你回答几个引导式问题，选一个模板。你确认那份建议需求。你说开始。然后你就看着这个 orchestrator 扮演你的资深项目经理，把结构化的简报委派给 coding agent，一路建立技能和记忆，持续优化自己的提示，并且递归地演化自己的 harness，直到你收到一套完整、经过测试、文档齐全的系统，可以在本机安装和运行，完全由 agent 建造，全程没有任何人手写的代码。

最后，这份规格还包含一个前瞻性的延伸。它建议采用一套干净、以 Python 为本的 harness 模式，灵感来自一个叫做 Claw Code 的开源 agent 框架，同时严格保持纯 Python，不用任何 Rust 组件。这个想法是要让技能可以直接从 markdown 文件发现并且动态加载，把所有工具通过一个中央注册表来连接，并且让每一个动作都经过一条由前置和后置钩子组成的流水线，这条流水线可以改写、拒绝或者追踪活动。它会以机制来执行那些规则，比如拒绝 Docker、让所有东西都保持本机运行。它加入一个插件系统，让新功能不用分叉核心就能加上，又加入会话压缩来对抗情境腐烂，还有一个自我说明的指引文件，orchestrator 每次启动的时候都会读，让它可以名副其实地按照自己的手册去做。全部都是纯 Python，全部都是本机运行，全部都是棘轮安全的，意思是每一项新增的东西都必须严格让系统变得更好，才会被保留。文件认为，最终成果就是一个高度可靠、可观察、可扩展的 Python 元构建器，能够把最朦胧的业务念头，变成可靠的软件。
