今天我们来聊一个叫做 project starter 的项目。它的核心，是一份用来搭建可执行起始仓库的蓝图，目标是把一整个系列的 AI 编码代理的配置来源，自动下载、审查、筛选并保持同步。你可以想想 Claude Code、Cursor、Codex、Gemini 的命令行工具、OpenCode、Grok Build、GitHub Copilot，还有跟它们一起运作的其他代理框架。我们现在看的，是它的可行动引导版本，最后一次更新是在二零二六年的夏天。

首先要明白的，是这份文件究竟是什么性质。它是一份样板规格。一个人可以把这份文件交给一个编码代理，然后告诉它，基本上就是：用这份文件去建立一个全新的项目。典型的要求大概是这样。用一个选好的名字，根据这份起始文件，建立一个新项目，并且把这份起始文件当成唯一的真理来源，也就是实现合约。把新项目放在一个选好的输出目录，定义它的用途，定义它主要的技术栈，然后执行一连串具体的步骤。

这些步骤值得逐个讲，因为它们为之后的一切定下了基调。代理必须完整读完这份起始文件。它要建立新的项目目录，再逐一建立规格要求的每一个文件夹和文件。它要把占位用的资料，换成真正的项目名和用途。它要把起始文件复制进新项目。它要建立软件包定义、来源清单、可执行脚本、规则文件，还有说明文档。它要把所有已启用的上游仓库下载到一个外部来源文件夹。它要生成一份来源锁定文件和一份来源审查文档。它要执行引导验证。它要更新一份状态文件。最后，它要汇报究竟是全部通过还是失败。

而最关键的指示是这样。不要只做总结。不要只写文档。要建立一个真正能运行的可执行项目。如果因为某些原因，网络或者命令行访问不可用，代理仍然应该建立每一个项目文件，然后在状态文件里，把来源下载标记为受阻。

这一点在规格所说的"对编码代理的关键指示"里再三强调。这份文件不只是说明文档。当一个代理收到它，并且被要求去实现它的时候，代理必须建立一个真正、可执行的项目。它不能在写完文字之后就停下来。它要建立仓库结构、软件包定义、列出来源和说明文档的清单，还有一整套脚本。其中有一个主要的 project starter 脚本、一个来源下载器、一个来源审查器、一个检查环境的 doctor 脚本、一个同步脚本，以及一个安全扫描器。代理必须下载清单里列出的每一个已启用仓库，把它们存在外部来源文件夹，把下载细节记入锁定文件，生成审查文档，执行验证指令，然后汇报成功或者确切的受阻原因。如果网络或者命令行根本不可用，规格要求代理直接讲明，说它受阻是因为网络或者命令行执行不可用，并且仍然要建立所有脚本和清单，这样人之后就可以执行引导程序。

由此我们来到所谓的不可妥协结果。只有一个指令，叫 bootstrap，而它必须执行一个特定的流程。首先它执行 doctor。接着建立所需的目录。它验证来源清单。它从网上克隆或者更新每一个已启用的来源。它写出来源锁定文件。它生成审查文档。它执行安全冒烟测试。它执行一次同步试运行。最后它执行测试。除非这些来源仓库真的已经下载到那个外部来源文件夹，否则项目根本不算完成。不能假装。

接着，规格列出所需的仓库结构，而它相当庞大。在最顶层，你有起始文件本身、一份说明文件，以及给各种工具用的代理指示文件。还有软件包定义、一份任务文件、一份状态文件，加上常见的编辑器和忽略设置。一个来源文件夹，放着清单、一份说明清单、锁定文件，还有它自己的说明文件。一个外部文件夹，放着下载回来的来源，不纳入版本控制。一个脚本文件夹，包含所有可执行的部件，其中有一套适配器，分别对应 Claude、Cursor、Codex、Gemini、OpenCode、Grok Build 和 Copilot，再加一个共用工具库，用来处理 git 操作、安全的文件处理、清单解析和报告。

除此之外，还有一个规则文件夹，放着一套有编号的规则文档，涵盖宪章、Karpathy 风格的行为、规格驱动开发、安全、测试、令牌效率，以及人类批准。有一个技能文件夹，按领域组织，比如规划、实现、测试、审查、安全、记忆和生命周期。还有文件夹给钩子、给模型上下文协议的配置、给持久记忆、给审查，以及给建议，在那里待处理、已批准和已拒绝的想法各有位置，旁边还有一份审查日志。有一个说明文档文件夹，涵盖安装、使用、架构、来源审查、安全、同步和疑难排解。有一个范例文件夹，展示几个从头到尾的工作流程。最后是一个测试文件夹，里面有测试夹具，以及给下载器、审查器、同步、清单和适配器的测试文件。

接着就到软件包定义。它为项目命名，标记为私有，并且把它设置成一个现代、以模块为基础的 Node 项目。它定义了一整套指令，也就是开发者真正会运行的脚本。bootstrap 和 init 用来做初始设置。doctor 用来检查健康状况。一组来源指令，用来下载、更新、检查和审查。同步，和一个同步检查。安全。审查。测试。和格式化。它要求 Node 版本二十或者以上。而值得留意的是，这个第一版实现，根本完全不需要任何外部依赖。一切都靠 Node 内置的能力运行。

现在我们来到最重要的数据文件，就是 GitHub 来源下载清单。代理必须一字不差地建立这份清单，并且下载每一个标记为已启用的来源。它绝对不能悄悄跳过任何一个已启用的来源，而如果一个必需的来源失败，这个指令就要以错误状态结束。清单里每一项，都描述一个仓库，有一个独一无二的标识符、一个易读的名字、它的网址、它应该存在本机哪里、它的类型、是否已启用、它的优先级、它的层级、是否应该隔离、一个导入策略，以及一句简短的用途说明。

让我用浅白的说法去描述这些来源，不逐个读出网址。主要那个叫做 Everything Claude Code，是核心的跨代理来源，提供技能、代理、指令、钩子、规则和安全扫描器参考，也是唯一一个设置为筛选导入而不是只供参考的来源。接着是一组官方的 Anthropic 仓库，包括 Claude Code 项目本身、它用来做自动化和拉取请求审查的 GitHub action、官方的代理技能范例，以及官方的插件市场。还有其他厂商的官方工具，OpenAI 的 Codex 命令行工具、Google 的 Gemini 命令行工具，和 OpenCode。有一组围绕模型上下文协议的来源，包括它的服务器范例、它的注册库，以及官方的 GitHub 服务器，用来处理问题、拉取请求、仓库和代码搜索。还有 agents 标准，是一份跨代理仓库指示的参考。

之后就到行为和社区来源。有两个版本的 Karpathy 风格行为规则，一个是通用版，一个是为 Cursor 和 VS Code 改编的版本，强调写代码之前先想清楚、简洁，以及精准微创的改动。有一份持久记忆架构的参考，而规格特别小心地讲明，这个绝对不能自动安装。有一个可组合的技能方法论项目、一个社区最佳实践集、几份筛选过的 Claude Code 生态系统发现清单、一套社区的子代理定义、由 Vercel Labs 提供的前端导向技能、一个 Cursor 规则范例集，以及一套着重安全的 Cursor 规则。还有几个额外的发现索引，标记为可选而不是必需。而有一个来源，是一个较旧协议服务器的归档集，被刻意停用，设置为永不导入，只保留作历史参考。

还有一份第二、较轻量的清单，是给说明文档来源用的。有些相关工具在网上有官方说明文档，但没有需要克隆的仓库。所以这份说明清单，指向 Claude Code 的功能说明文档、Cursor 的规则说明文档、Grok Build 的概览、GitHub Copilot 的自定义指示，以及模型上下文协议的网站。项目的第一版，可以只建立这份文件而不真的下载这些说明文档，而较后的版本就会实现一个说明文档下载步骤。

外部目录有它自己一套严格的策略。下载回来的上游仓库，是刻意不纳入版本控制的。它们不能被提交进这个项目。只有经过筛选、审查和妥善标注出处的文件，才可以被复制进项目自己的第一方文件夹，比如规则、技能、钩子、配置和说明文档。

下载行为本身也有仔细的定义。下载器脚本支持几种模式。你可以下载全部，或者按配置文件筛选，比如核心、官方或者发现。你可以做一次试运行，或者用严格模式运行。默认情况下，下载会读取清单、选出每一个已启用的来源、建立外部来源文件夹，然后就每一个来源做三件事之一。如果目标还不存在，它就做一次浅层克隆，只取最新那个快照。如果目标已经存在，而又是一个正当的 git 仓库，它就抓取并快进到最新版本。如果目标存在但不是 git 仓库，它就失败。一路上它会记录一些元数据，比如解析出来的远程网址、当前的提交、分支，以及最新提交的日期和标题。接着它会生成一份易读的摘要。

失败规则就看优先级。对于必需的来源，任何失败都是致命的，无论是克隆失败、更新失败、缺少 git 文件夹，还是网址无效。对于可选的来源，克隆失败是允许的，除非开了严格模式，但失败仍然必须记录下来。而任何停用的东西，默认就一定不能下载。

围绕下载的安全规则，既严格又清晰。下载器绝对不能运行下载回来的仓库里的安装脚本。它绝对不能在里面运行软件包安装。它绝对不能把一个远程脚本直接灌进命令行。它不能执行那些仓库里的任何钩子。它不能自动把它们的内容复制进运作中的代理配置。也绝对不能在项目根目录以外写任何东西。下载器只允许做两件事，克隆或者更新，以及检查元数据。再多都不行。

来源锁定文件记录结果。就每一个来源，它记下标识符、名字、要求的和解析出来的网址、目标位置、状态、提交、分支、最新提交的时间和标题、找到的许可证文件、找到的软件包文件、隔离标记，以及导入策略。它还会保留一份失败清单。

审查脚本会读取清单和锁定文件，然后生成一份易读的审查文档，每一个来源占一节。每一节都列出名字、网址、目标、状态、提交、分支、许可证和软件包文件、优先级、层级、隔离标记、导入策略、用途、选了哪些部件、拒绝了哪些部件，以及任何安全备注。审查必须在几种情况下，把一个来源标记为不适合自动导入：如果它没有许可证文件、如果它含有会运行远程指令的安装脚本、如果它有可疑的安装后脚本、如果它已归档、如果它需要凭证、如果它会修改全局代理配置，或者如果它附带的协议服务器要求广泛的文件系统或者网络访问。在最初那一轮，审查应该把差不多所有东西都标记为还未选取，而批量导入就被拒绝，要等人去审视才行。唯一的例外，是小巧、只含规则的仓库，比如 Karpathy 技能，在验证了许可证之后，可以选出来做小心的提取。

引导路由器把这些指令串到一起。它支持 bootstrap、init 和 format。而 bootstrap 会按顺序，执行 doctor、下载来源、审查它们、运行安全、以试运行方式运行同步，然后运行测试。如果任何一个必需的步骤失败，整个引导程序都会失败。

doctor 脚本在任何事情开始之前先检查环境。它确认 Node 版本是二十或者以上、git 可用、工作目录在哪、有没有写入权限、来源清单存不存在、外部文件夹存在还是可以建立、脚本文件夹在不在、用的是什么操作系统平台，以及如果同步想用符号链接的话，系统支不支持。接着它会打印一份干净的状态报告，每项检查都标着 OK，再加一个整体结果，或者用清晰的信息表示失败。

同步层就是为每一个代理生成配置那一部分。即使这些适配器一开始很简陋，也必须有一个能运作的同步框架。以试运行方式运行同步，会预览它会建立的文件，包括跨代理指示文件、Claude、Gemini 和 Copilot 的指示、Cursor 规则，以及给 Claude、Gemini、Codex 和 OpenCode 用的配置文件夹。每一个生成的文件都必须带有一个标头，标明它是由 project starter 自动生成、叫读者不要直接编辑它、指回它的来源文件夹，并且提醒他们运行同步来重新生成。

安全脚本是一个静态扫描器。在第一版，它会扫描下载回来的仓库，找可疑的顶层脚本。它会侦测环境文件、私钥、令牌，或者可能被意外提交进项目的证书。它会汇报任何赋予广泛文件系统访问的协议配置。它会就灌进命令行的远程脚本，和下载来源里的安装后脚本发出警告。而它绝对不会执行任何下载回来的脚本。一个简单的静态扫描器，作为开始已经足够。

接着是最初那几份规则文件，其中有几份写得很清楚。宪章说要完全遵从用户的目标、倾向简单而易维护的方案、做精准微创的改动、避免未经批准的破坏性行为、在审查之前绝对不安装、执行或者导入下载回来的第三方代码、运行相关测试或者解释为什么没有运行，并且在重大工作之后更新状态。Karpathy 规则说要写代码之前先想清楚、不要假设有隐藏需求、倾向最简单而又行得通的做法、避免投机性的抽象、只改必要的东西、让目标保持显眼，并且验证结果。而人类批准规则就列出了一些永远要先有人签核的行为：安装全局软件包、运行远程安装脚本、启用会用凭证的协议服务器、把第三方代码复制进运作中的配置、修改钩子、删除文件，以及套用任何自行生成的技能或者规则改动。

导入策略值得特别强调，因为它是整个设计的哲学核心。下载来源是强制的。导入它们就不是自动的。项目把四个不同的阶段清清楚楚地分开。有下载回来的来源，它住在外部文件夹，被当成不可信、被版本控制忽略，也永远不会被执行。有经过审查的来源，它的元数据已经被审视和记录在审查和锁定文件，但还未导入。有筛选导入，在那里一个特定文件被带入项目自己的文件夹，但要在验证了许可证、记录了来源提交、记录了校验码、任何高影响的东西都有人批准，并且没有隐藏安装行为之后才行。最后有生成出来的适配器输出，也就是每个代理的配置文件，这些必须永远都可以只靠运行同步来重现。

至于测试，代理必须最少为清单、下载器、审查器和同步建立 Node 测试文件。这些测试要验证清单可以解析、每一个已启用的来源都有齐所需字段、没有重复的标识符、每一个目标都在外部来源文件夹之下、停用和归档的来源默认不会被选、锁定文件在一次运行之后形态有效，以及生成出来的文件都包含那个自动生成标头。

说明文件提供一个快速入门。运行 bootstrap。这样就会把获批准的上游仓库下载到外部来源文件夹，然后写出锁定文件和审查。它还带有一个重要提醒：下载回来的仓库，在审查之前都是不可信的，项目不会执行下载回来的代码，也不会自动导入第三方技能。

任务文件把第一个里程碑列成一张清单：建立软件包定义、建立来源清单、建立下载器、下载已启用的来源、生成锁定文件、生成审查、运行 doctor，以及运行测试。状态文件从引导实现阶段开始，最新更新标记为未开始，也还没有任何受阻。

验收标准很具体。实现只有在 doctor、下载、审查、安全扫描、同步试运行和测试全部通过的时候，才会被接纳。下载之后，一张特定的来源目录清单必须存在，除非有一个可选来源失败，而那个失败又已经记录下来。锁定文件必须存在。审查必须存在。而下载回来的仓库，绝对不能被提交进版本控制。

规格接着给出确切的实现顺序，总共二十二步，从完整读完这份文件和建立文件夹结构开始，接着是软件包定义和忽略文件，再到清单，再到处理安全文件、git 和清单的工具库，再到下载器、审查器、doctor、安全扫描器、同步和主路由器，再到规则文件和说明文档，再到测试，再到运行整套验证，最后更新状态和汇报摘要。

还有一张清晰的清单，列出代理永远不能做的事。它不能只重写这份文件、假装已经下载了、悄悄跳过下载、执行下载回来的脚本、安装全局软件包、运行远程安装程序、把所有下载回来的技能复制进运作中的配置、在没有备份的情况下覆盖用户文件、改动任何一个代理的全局配置、把机密放进记忆、记下隐藏的推理过程，或者自动批准它自己的自我改进建议。

还有几个边缘情况收尾。如果一个仓库搬了或者会重定向，下载器应该让 git 克隆它、记录解析出来的网址、在锁定文件记下一个警告，并且在克隆成功的情况下继续。如果一个仓库已经不再存在，一个必需的就会让引导失败，而那个失败会被写下来和准确地汇报，而一个可选的就会让程序继续，并且把失败记录下来和在审查里提及。

最后，是完成的定义。项目在以下情况全部成立的时候，才算完成：仓库名为 project starter、bootstrap 指令存在、清单包含确切那套仓库、下载真的取回那些已启用的、外部来源文件夹里放着它们、锁定文件记下它们的提交、审查已经生成、下载回来的仓库被版本控制忽略、没有任何下载回来的代码被执行、同步试运行行得通、测试通过，并且状态文件以真实结果更新。而当以上全部都成立的时候，代理就汇报一份干净、面向人的摘要：多少个来源成功、失败或者被跳过、生成了什么、每一个验证步骤通过还是失败，再加一个最后的提醒，就是下载回来的仓库住在外部文件夹，在它们经过审查之前，不会被导入或者执行。

这就是整个设计的全貌，一个会把代理配置这个更广阔的世界，当成要小心收集、诚实审视、并且只有刻意才去采用的起始项目。
