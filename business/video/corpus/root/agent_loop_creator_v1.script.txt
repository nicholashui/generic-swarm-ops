Imagine you want to build an AI agent that doesn't just think out loud, but actually gets reliable work done, over and over, without spinning in circles or quietly failing. That's the heart of this document. It's a detailed, build-ready guide for creating what the authors call a refined hierarchical reasoning-and-acting agent loop. Think of it as the blueprint you'd hand to a coding agent and say, here, turn this into working software.

The version here is marked production ready, dated in mid two thousand twenty-six. Its whole purpose is to be executable. It's written so that a capable coding assistant, whether that's something like Grok Build, Claude Code, Cursor paired with one of the modern models, or a custom engineering harness, can read it section by section and generate real, working modules.

And this isn't just opinion. The design pulls from deep research. It leans on a taxonomy of why multi-agent systems fail. It borrows from a system called AgentOrchestra and its Tool, Environment, and Agent protocol. It folds in the classic reasoning-and-acting pattern along with later improvements like Reflexion, Prospector, and ReflAct. And it studies how xAI runs production agents, with server-side orchestration and plan-first command line tools. All of that gets blended together.

So what did the authors actually learn after iterating on this idea, in their words, a hundred times over? The single most important insight is striking. When multi-agent systems fail, roughly forty-two percent of those failures come down to specification and design problems. Not clever algorithms, not exotic tooling, but simply poor specs and weak design. Another twenty-one percent or so come from verification and termination problems, meaning the system either stops too early or never properly checks its own work. The lesson is blunt: just adding more agents, without a strong upfront specification, structured observations, an explicit notion of being done, layered critics, and real progress tracking, often makes things worse, not better.

There's also a clear alignment with how xAI builds things. For deep research subtasks, the design hands work off to a multi-agent model that runs its own reasoning loop on the server, coordinating several agents under a leader that synthesizes the result. Meanwhile, the local loop borrows the habits of plan-first coding tools: generate an explicit plan before acting, run isolated sub-agents in parallel, and track everything in a simple todo-style state. The argument is that this hybrid gives you the best of both, raw power plus tight control plus cost efficiency.

The authors also worked through the big architectural tradeoffs. They chose to build a custom Python core, with strict typed schemas and a full tracer, rather than leaning entirely on an existing framework. Why? Transparency, auditability, and educational value. They want a memory strategy that mixes a structured todo file, longer-term vector memory, and aggressive summarization so the context never bloats. They want self-improvement, but bounded and validated, so the system doesn't drift into nonsense. They want minimal dependencies first, with optional adapters added later. And they want it local-first, observable, sandboxed, and hardened with the kind of safety machinery you'd expect in production, things like circuit breakers, retries, and budgets. There's even a charming ambition baked in: the harness should help build and improve itself.

Now let's talk about the mission in concrete terms. The goal is a controlled, hierarchical, reasoning-and-acting agent loop that is reliable against known failure modes, that can evolve over time through versioning and self-reflection, that's hybrid in nature with full client-side control plus optional delegation to xAI's server-side agents, and that's genuinely production grade: observable, cost aware, secure, testable, and extensible. It's also meant to match the user's own style, which is spec-driven, iterative, Python-first, and built around clear contracts between components.

The document doesn't just wave its hands at success. It defines what success actually means, in measurable terms. On reliability, when you inject every known failure mode into synthetic tests, the residual failure rate after mitigation should stay under five percent, with early detection of specification and role violations, loop detection that triggers a replan or shutdown, and a verifier that refuses to accept an incomplete or incorrect finish. On performance, the base system should succeed at least seventy percent of the time on held-out research and coding tasks, and after a couple of rounds of self-evolution, that should climb to eighty-five percent or more, while using fewer steps and tokens than a plain baseline.

On observability, every single run should produce a complete, replayable trace, capturing the thoughts, actions, and observations along with versions, timings, and token counts, and you should be able to visualize it or replay it from any step. On evolvability, there should be a version manager that can register, roll back, and select the best version of prompts, tool code, and agent configurations, plus a self-evolver that proposes and validates improvements on held-out data. On the xAI side, research subtasks should hand off cleanly to the multi-agent model and come back with synthesized results and provenance woven into the main trajectory.

There's also a serious focus on production hardening: circuit breakers that move correctly between closed, open, and half-open states, exponential backoff on retries, per-phase budgets for tokens and steps, structured error reporting, sandboxed tool execution, and least-privilege access. And usability matters too, a clean Python package with a command line interface, an optional server mode, rich examples, full type hints and documentation, and a passing test suite. Finally, it should integrate smoothly with existing tools and self-hosted services.

The authors are equally clear about the boundaries. The primary stack is modern Python, with typed models, asynchronous execution, and lightweight HTTP clients. Optional pieces include vector stores for memory and a web server for hosting. The core loop is meant to stay minimal at first, under a couple thousand lines of code, with evolution and the xAI hybrid added in later phases. There are to be no uncontrolled loops, ever. Every run has a hard cap on steps, loop detection through state hashing, progress-based exits, and circuit breakers. Every model call uses a strict output schema. Security is non-negotiable: code execution is sandboxed, model-generated arguments are never trusted blindly, and the system watches for anomalies like rapid repetition. Cost is controlled through budgets, careful parallelism, summarization under pressure, and early termination once goals are met. And there are explicit non-goals for the early phases, things like fully distributed execution, a graphical dashboard, native multimodal support, and multi-tenancy, all deferred for later.

One lovely detail: the spec itself is considered a living document. The system, once built, is allowed to update its own specification through self-evolution.

Let's go deeper into the research that grounds all of this. The failure taxonomy comes from a study asking why multi-agent language model systems fail. The researchers analyzed over sixteen hundred execution traces from seven different frameworks, and they identified fourteen distinct failure modes, with strong agreement between human annotators and even a language model acting as a judge.

Those fourteen modes fall into three families. The first family, system design issues, is the largest, accounting for nearly forty-two percent of failures. It includes disobeying the task specification, disobeying an assigned role, repeating steps over and over, losing the conversation history, and being unaware of when to stop. The second family, inter-agent misalignment, makes up about thirty-seven percent. It covers things like resetting a conversation, failing to ask for clarification, drifting off task, withholding information, ignoring another agent's input, and a mismatch between what an agent reasons and what it actually does. The third family, task verification and termination, makes up about twenty-one percent. It covers stopping too soon, doing no verification or incomplete verification, and verifying incorrectly.

The key takeaway is that design and specification quality offers the highest return on investment. That's why there's a mandatory phase zero in this architecture, where a structured task specification is created, with an explicit objective, a list of success criteria, constraints like step and token limits, an output format, and quality gates. An automated validator and a critic review that specification before the loop even begins. Verification is identified as a persistent weak spot, even in runs that look successful, so the design adds a dedicated verifier or critic that outputs a strict, structured judgment, working at both a low schema level and a high objective-alignment level. There's an explicit done action that must pass verification and an evidence check before anything is accepted as finished. Context loss and repetition get tackled with aggressive summarization, structured state instead of dumping the entire history, and loop detection by hashing recent actions and observations. And inter-agent problems are addressed with a strong central orchestrator that hands off work using a clear, structured contract.

The second research pillar is AgentOrchestra and its Tool, Environment, Agent protocol. That system reportedly reaches around eighty-nine percent on a demanding benchmark, and self-evolution pushes it even higher. The protocol introduces three first-class abstractions. Tools are versioned and lifecycle-managed, registered with a name, description, schema, and implementation, and even retrievable by meaning. The environment defines the observation and action spaces and keeps state coherent, which here means the working directory, a sandboxed file system, and the code runtime. Agents carry roles, competencies, and metadata, support hierarchy and coordination contracts, and can even be transformed, an agent treated as a tool, or a tool treated as an agent, for dynamic reconfiguration later on.

Several mechanisms come straight from this. A version manager gives every prompt, tool, configuration, and generated artifact a version and a lineage, so you can register new ones, roll back, and select the best by some metric. A tracer records the full trajectory of execution for replay and debugging and as a signal for optimization. A self-evolution module, inspired by gradient-style textual optimization and self-reflection, follows a clear cycle: collect the trace, diagnose the root cause and the component to fix, apply an edit, validate that it actually improves things on held-out data, and only then register the new version, all within a bounded number of rounds. Context management is run-scoped and component-specific, giving sub-agents only a relevant slice with provenance rather than the whole history. And the hierarchical orchestration pattern has a central planner that decomposes the objective, routes work to specialists with narrow sub-specs, lets sub-agents run their own controlled loops, then bubbles results up to a consolidator that harmonizes and resolves conflicts, and finally a reporter that produces the structured output.

The third pillar is the reasoning-and-acting foundation itself. The original pattern interleaves a thought, which is a reasoning trace, with an action, which is a tool call, a delegation, or a finish, followed by an observation, which is the grounded result. That simple loop delivered meaningful gains over pure reasoning or pure acting on interactive tasks. The design then layers on enhancements. Reflexion adds verbal self-critique that turns into improvement plans. Prospector adds self-questioning and ranking of multiple candidate trajectories. ReflAct strengthens the grounding inside the reasoning step itself, re-anchoring each thought against the latest observation and the original objective. And patterns like plan-and-execute and tree search contribute an explicit planning phase and the option of exploring multiple branches.

xAI's production patterns round this out. Their server-side loop lets the model decide on tools, execute them internally, and iterate until it reaches a final answer. Their multi-agent setup runs parallel specialists with a leader synthesizing the result, where a reasoning-effort setting controls how many agents participate. Their build tooling favors planning first and running isolated parallel sub-agents. The hybrid proposed here keeps the client orchestrator in charge of global state, tracing, and verification, while delegating research subproblems to the multi-agent model and handling coding subtasks either with local specialists or with parallel isolated processes.

After all that rethinking, the final architectural decisions settle into place. The loop style is a controlled custom reasoning-and-acting loop, with typed state, loop detection through hashing, and circuit breakers, with hierarchy layered on top so an orchestrator decides whether to use a tool, delegate, synthesize, or finish. Central control, the authors argue, beats coordination chaos. The agent state bundles together the task specification, the history, the todo list, the plan, short and long term memory, the version registry, the tracer, the budgets, and the set of seen state hashes for loop detection. Memory favors structured, low-token state plus summarization, with optional vector retrieval, and sub-agents always receive only a sliced context. Model calling goes through a unified client that supports multiple providers, always with a strict output schema. Tools live in a registry with validation and a safe execution wrapper, and code execution is sandboxed. The xAI hybrid uses a dedicated wrapper that sends a narrow sub-objective with its criteria and enabled tools, then logs the leader's answer as a special observation. Self-evolution is introduced in phases, starting with prompts, then tool code, then agent configurations. And testing includes a failure simulator that replays known failure examples and asserts that the mitigations actually fire.

The rationale, in summary, is that this design goes straight after the number one failure category, specification and design, through a mandatory upfront phase and a living specification reviewed by a critic. It closes verification gaps with mandatory gates and structured observations. It prevents loops and context rot through detection, summarization, and structured state. It enables long-term robustness through self-evolution. And it leverages xAI's strengths without giving up control.

Now to the architecture itself, walked through as a flow of phases. Phase zero is initialization: parse the instruction, generate and validate the task specification, run a validator and critic to check for completeness, ambiguity, role clarity, and clear termination conditions, then create the agent state and optionally have a planner produce a high-level plan and a todo list. At the end, the system decides whether to run flat, hierarchical, or hybrid.

Phase one is the core controlled loop. While the system hasn't decided to terminate, it builds context by summarizing long history and pulling in key facts, the todo list, the task spec, and the latest observation. The model then makes a decision in a strict format: a thought that analyzes progress against the criteria and re-grounds against the objective, an action type which might be a tool, a delegation, a synthesis, a finish, or a reflection, and a payload of arguments. Before executing, it checks for loops by hashing the recent action and observation; if it's seen this before, it forces a replan or terminates. Then it executes safely, with circuit breakers, retries, and sandboxing, and records a structured observation that always includes status, data, a summary, a confidence score, any issues, suggestions for what's next, provenance, and the versions used. Each event is appended to history, the todo and progress get updated, and there's a light self-critique every few steps or whenever an error occurs. Termination happens when the criteria are met and the verifier passes, when budgets run out, on an explicitly verified finish, or when something irrecoverable forces an escalation.

Phase two adds hierarchical delegation. The orchestrator decomposes the work and selects or instantiates a specialist from a registry, things like a researcher, a coder, a verifier, a reporter, a tool generator, a browser agent, or an analyzer. It creates a narrow sub-task specification with just the relevant objective, criteria, context slice, and provenance, invokes the sub-agent, which runs its own controlled loop or delegates to the xAI multi-agent model, and the sub-agent returns a structured observation that bubbles up with a summary of its trace for auditing. The parent records it, validates and integrates it, updates the global plan, and decides what to do next.

Phase three handles consolidation and quality gates. An aggregator collects observations and progress, a harmonizer or reporter merges and deduplicates and cross-references everything, resolving conflicts by citing sources and versions, and produces a unified draft. The verifier then scores that draft against the success criteria, checks for hallucinations, gaps, and verification failures, and suggests fixes. If it fails the gate, the system triggers refinement. If it passes, it moves on toward the final output.

Phase four is reflection and self-evolution, the advanced part. At milestones or at the end, the full trace goes to the self-evolver, which diagnoses root causes using failure-mode-aware prompts, proposes targeted edits to prompts, tool code, or role definitions, validates the improvement on held-out data or by replay, and registers new versions if they're genuinely better, all within a bounded number of rounds.

Phase five is termination and output: a final synthesis in the format the spec requires, persistence of the full trace, versions, and metrics, an optional reflection summary, and human-in-the-loop hooks at high-stakes moments or when budgets are exhausted.

The document then gets concrete about the data models, the typed contracts the coding agent should generate first because they define how everything talks to everything else. There's a task specification carrying the identifier, objective, success criteria, constraints, output format, and quality gates. There's a structured observation carrying status, data, summary, confidence, issues, suggestions, provenance, trace identifier, and versions used. There's a trace event capturing each step with its timestamp, thought, action, payload, observation, and token usage. And there's the agent state tying together the task spec, the history, the todo list, the plan, short and long term memory, the seen hashes for loop detection, budgets, and versions.

From there it lists the key modules to build: the core loop, the circuit breaker with full metrics, the verifier tuned to catch those specific failure modes, the orchestrator with its planner and delegation router and consolidator, the self-evolver, the minimal protocol schemas, the xAI integration, the memory and tracing layers, and a versioned collection of prompts that explicitly reference the failure modes in their critic instructions. The self-evolution routine, in plain terms, analyzes a trace to find a root cause and a concrete proposed edit, and if that edit targets the right component and validates as an improvement, it gets registered as a new version with its lineage and metrics.

It also pins down the xAI hybrid integration points. When the action calls for deep research or the subproblem is complex enough, the system calls the multi-agent model instead of a local sub-loop. There's a configuration flag and a threshold for this. Logging always captures the source, the agent count, and the leader's confidence. If the xAI call fails or its circuit is open, the system falls back to a local sub-agent or direct tools. And optionally, it can emulate the build-tool style by exposing the plan and running parallel sub-agents on independent branches with isolated state.

Then comes the phased roadmap for the coding agent. Phase zero, roughly a day or two of effort, builds the project scaffold, the typed models, a basic model client, a schema enforcement helper, and a minimal loop that runs without crashing on mock components. Its gate: the loop runs ten steps on a toy task, always produces valid observations, detects an injected repetition, and catches obvious specification problems. Phase one delivers the full controlled core with loop detection, complete circuit breakers, safe execution, progress tracking, light reflection, an explicit finish with a verifier gate, structured state, and aggressive summarization. Its gate: the failure injection tests pass for repetition, history loss, and premature or incomplete termination, the verifier rejects bad finishes, token usage is logged, and there are no infinite loops.

Phase two delivers the orchestrator and a registry of pluggable sub-agent roles, the local delegation path, the consolidator and reporter, a multi-level verifier, optional plan generation, and context slicing. Its gate: an end-to-end run on a multi-step task with delegation, where sub-results are correctly integrated, conflicts resolved or flagged, and the overall criteria checked. Phase three adds the version manager, the minimal protocol, the self-evolver, and the xAI multi-agent integration, plus todo generation in the planner. Its gate: a self-evolution run across three similar tasks shows measurable improvement, the xAI delegation works end to end with cited results, and version history is queryable. Phase four is polish, examples, production hardening, and dogfooding: full examples including a deep research agent, a coding project agent, and a self-improving harness, a sandboxed code tool, the command line interface and optional server, a full test suite, observability exports, cost dashboards, and documentation. Its gate: all success criteria met or exceeded, the full test suite running cleanly, and self-evolution demonstrably improving a held-out component. Throughout, the workflow is the same: generate code, run an internal critic, fix the issues, re-validate against the gate, then move on, keeping an internal todo and logging everything.

On production hardening, security, and observability, the priorities are clear. Reliability comes from circuit breakers, retries with backoff, always-structured error observations, budget enforcement with graceful degradation, and progress-based early exit. Security means sandboxing tools, validating and sanitizing every model-generated argument before execution, least-privilege access, and anomaly detection on loop patterns. Observability makes the tracer first-class, capturing every event with optional context snapshots, versions, token counts, timings, and sub-calls, exportable for replay, with the ability to replay a trace from any step. Cost and scalability come from per-phase budgets, parallelism only on independent branches, smart summarization, and session isolation. Extensibility is everywhere: pluggable model backends, a simple validated tool registry, a factory for sub-agent roles, swappable memory backends, interchangeable evolution strategies, and adapters to other frameworks. And deployment stays local-first by default, with a minimal container and an optional secured server mode.

The testing strategy is treated as critical. There are unit tests for schema validation, hashing, the circuit breaker state machine, and verifier logic. There are integration tests running the full loop on toy tasks and the hierarchical delegation end to end. There's failure injection that forces each known failure mode and asserts early detection and correct mitigation with no silent failures. There's a mini benchmark measuring success, efficiency, and the improvement from evolution. There's property-based testing that throws random valid and invalid inputs at the system to assert its invariants always hold. And there's a dogfooding test where the system evolves its own verifier or planner prompt on real traces and proves the improved version scores higher without regressing. The mandate is firm: don't mark a phase complete until the relevant tests pass, and write the tests alongside the code.

The document closes with its references, crediting the failure taxonomy study, the AgentOrchestra and protocol paper, the original reasoning-and-acting work, the enhancement techniques, broader agent surveys, and xAI's production documentation, along with the original specification this guide expands upon.

And then a handoff. The spec, the authors say, is complete and actionable. Start implementing immediately with the phase zero scaffold and the core models, because those typed models are the contract that everything else depends on. There's even a ready-to-paste first instruction for the coding agent: read the spec fully, create the package scaffold, build the core models and a basic client, get a minimal controlled loop passing the early gates using strict schemas, include a failure injection test skeleton, and maintain a todo while applying critique to every module. After the core is solid, proceed phase by phase, and during the polish phase, use the built system to evolve its own prompts and verifier.

The expected deliverable is a fully functional, tested, and documented package, with examples and a command line interface, that a user or a higher meta-agent can run for reliable hierarchical agent workflows, with clear extension points for the xAI hybrid, custom tools, and self-evolution. As for open questions before coding begins, the authors anticipate none. The spec is self-contained, and if ambiguities arise during the build, they should be resolved through internal critique or escalated with a specific trace. That, they conclude, completes the deep research and the implementation spec. Build it production grade, iterate with critics, and make it a cornerstone of advanced agent harnesses.
