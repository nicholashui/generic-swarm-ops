Imagine you could set up the perfect coding environment for an AI agent in just a few minutes, one that comes preloaded with the best tools, the smartest rules, and a memory that actually sticks. That's the whole idea behind this project. It's called the ultimate agent harness starter, and the goal is to create a clean, production-ready starter project, a fresh repository complete with setup scripts and configuration files, that bootstraps a powerful coding agent environment you can use across many different tools.

The thinking here is to combine the best elements from the top-ranked coding agent projects out there, the ones that were leading the pack as of mid 2026. And whenever two of these projects overlap or disagree, there's a clear tiebreaker. A harness known as ECC is treated as the primary foundation. The end result should be something you can install with a single command, or just a few, and immediately get a highly productive agentic workflow that works with Claude Code, Cursor, Codex, OpenCode, Gemini's command line tool, and similar systems.

Let's talk about the principles that guide all of this. First and foremost is spec-driven development. The idea is simple but powerful, clear specifications drive everything else. When there's an overlap between features or skills or rules, you prioritize the highest-ranked source. ECC comes first, then a set of behavioral rules inspired by Andrej Karpathy, then a persistent memory tool, then a collection of best practices, and then a broader library of skills. The whole thing should stay portable, lean, local-first, and avoid heavy reliance on Docker wherever possible. Security, memory, and efficient use of tokens all matter a great deal.

The output has to be friendly to agents themselves. That means clear phases, checklists, acceptance criteria, and built-in hooks for critic and review agents. It should support an iterative rhythm, plan, implement, review, then improve, looping back around as needed. And who is this for? Solo developers, small teams, and power users who want to build their own custom AI coding workflows.

So what does success actually look like? A new project folder should initialize in under five minutes from one main script. ECC should be fully installed and configured as the core harness. The Karpathy behavioral rules should be active by default. Persistent memory should be switched on. The most valuable skills from the top libraries should be selectively merged in, with no duplication. Best-practice configurations, hooks, rules, and example workflows should all be included. A security baseline should be active. There should be clear documentation and a task file for future extension. It should work across platforms, on Mac, Linux, and Windows where possible, and with multiple agents. And it should include quality gates like linting, tests, and review prompts. Above all, every supported coding agent should share the exact same curated skills, rules, and hooks, kept in sync from a single source of truth.

That brings us to one of the central new features, the cross-agent skill and rule synchronization layer. The objective here is to make sure every coding agent knows the same high-quality skills and rules. The way you achieve that is by keeping each agent's configuration folders in sync from one central place. This solves a really common headache, where different agents end up with fragmented or outdated skills. The approach leans on ECC's cross-harness design first, then adds lightweight adapters and sync scripts to cover everything else.

The design principles for this are straightforward. There's a single source of truth, central folders for skills, rules, hooks, and MCP configurations sitting at the root of the repository. Where an agent supports it, you prefer symbolic links, because they're fast and always up to date. When an agent has a different folder structure or file format, you fall back to a smart copy with a light transformation. You lean on ECC's built-in compatibility as much as possible. And the whole sync process stays simple, scriptable, and safe, meaning you can run it repeatedly without harm, with backup and restore built in. In practical terms, the folder layout gains those central source folders alongside the agent-specific folders, like one for Claude Code and one for Cursor, plus a scripts folder holding the main sync script and per-agent helpers.

There's a clear set of tasks to make synchronization real. First, create the central folders as the authoritative source. Then build or adapt a sync script. For Claude Code, that means copying or linking the skills, commands, hooks, and rules, using ECC's plugin and marketplace patterns where you can. For Cursor, you generate or update its rules and config from the central rules. For other agents like Codex, OpenCode, and Gemini's tool, you create the appropriate root files by combining central content with ECC adapters. The script should support both a full sync and a selective one, say just the planning and review skills. It needs to be safe and repeatable, with a dry-run mode and conflict detection. You integrate with ECC's existing features first, add a simple sync command so anyone can trigger it after a skill update, document the process clearly, and even include a command that lets agents trigger the sync themselves. You also add version pinning through a manifest so the same skill versions are used everywhere, and you test the sync across at least Claude Code, Cursor, and one other agent.

How do you know it's working? When you update a skill in the central folder and run the sync script, that skill instantly becomes available to every supported agent. No manual copying between folders. The agents all behave consistently because they're drawing from the same curated content. And the sync is fast, safe, and well documented. This layer is what makes the whole starter truly portable and consistent across your stack.

Next comes a self-evaluation and critic routine, essentially a way for the agent to assess the quality of its own work. This creates a closed loop, plan, implement, self-evaluate, and refine. This section was strengthened considerably based on deep research, drawing on xAI's work with multi-agent capabilities and transparent, auditable reasoning, along with a range of high-quality research from 2025 and 2026 on reflection, self-refinement, multi-agent self-evolution, structured critic and actor loops, human-in-the-loop improvement, and memory architectures.

The research-backed design principles are worth walking through. There's a multi-agent critic pattern, using specialized roles, an actor or solver, a critic or challenger, and an optional judge, that can work in parallel, with each sub-agent showing its reasoning so everything stays auditable. There's structured self-critique that moves beyond vague feedback toward explicit verification of preconditions, state tracking, rubric scoring, and a memory of past reflections. There's a strong emphasis on human-in-the-loop safety, where every high-impact change requires human confirmation. There's careful memory and context management, supporting layered summaries and what's called context folding for long tasks. There's transparency and auditability at the core, every reasoning step, critique, and decision gets logged and can be reviewed. And again, ECC comes first, with the research layer built on top of its existing review capabilities.

The routine evaluates work across a richer set of dimensions. Correctness and functionality, with explicit verification of preconditions and state. Simplicity and alignment with the Karpathy philosophy. Faithfulness to the spec. Security and safety. Performance, efficiency, and token usage. Maintainability and clarity. Reasoning quality and auditability, which is about how transparent and verifiable the thought process is. And self-improvement potential, meaning how actionable the critique actually is.

The core flow runs like this. The actor generates the solution using ECC skills and Karpathy rules. The critic, which can itself be multiple agents, runs a structured critique using that rubric, and can spawn parallel sub-agents to dig into different dimensions, one for security, another for simplicity, producing scores, specific issues, and concrete improvement suggestions. Those critiques and the lessons learned get stored in episodic memory. The actor then uses the critique and stored reflections to improve the output, supporting multiple iterations with smart context management. High-impact suggestions pass through a human confirmation gate. And a full audit log records all of the reasoning, critiques, and decisions.

Why does this matter? It prevents what people call vibe coding drift. It enforces adherence to the spec. It catches issues early, whether that's correctness, security, complexity, wasted tokens, or maintainability. And it makes the agent more autonomous and reliable over time. In terms of design, there are central critic skills, a structured self-evaluation template that outputs an overall quality score, a breakdown across each dimension, the specific issues found, and concrete improvement suggestions framed as new tasks or diff recommendations. It can be triggered automatically through a post-completion hook, manually through a self-review or critic command, or as part of a larger multi-step workflow. The output is stored alongside the work and feeds back into the loop.

The tasks to build this include creating a core critic and self-review skill, defining a reusable self-evaluation template, adding a slash command that triggers structured evaluation, creating a post-completion hook, making the output structured so other agents or scripts can parse it, integrating with existing orchestration, adding example usage and a sample workflow, allowing the rubric to be customized with project-specific weights or extra dimensions like accessibility, and ensuring the critic routine itself can be evaluated for continuous improvement. Success means the agent can run a self-evaluation on its own recent work and produce a clear, actionable critique, triggered either manually or automatically, with scores and concrete improvement tasks that feed back into the workflow. This is what turns the agent from a one-shot generator into a genuinely self-improving system.

Building on that is the skill lifecycle, the ability to automatically suggest adding new skills, updating existing ones, or removing low-value or outdated ones, but never applying any change without explicit human confirmation. This creates a safe, controlled way for the skill set to evolve. It builds directly on the critic routine and the central source of truth.

The workflow has a human always in the loop. First, an analysis phase, triggered by the critic, by periodic review, or after finishing significant work, where the agent looks at how skills are being used, their quality scores, their relevance, any duplication, and any gaps. Then suggestion generation, producing clear, structured proposals. To add a new skill, it gives the name, purpose, source or draft content, and why it's valuable. To update one, it gives specific improvements with a before and after summary. To remove one, it gives the reason and an impact assessment. These suggestions get saved with unique identifiers. Then human review and confirmation, where a person reviews the suggestions through a file, a dashboard, or a command, and confirms, rejects, or modifies them. Only confirmed items proceed. Then safe application, where the confirmed change is applied to the central skills, and the sync layer propagates it to every agent. And finally an audit and rollback step, where all changes are logged with a timestamp, a reason, and the name of the human who approved them, with easy rollback through version control.

The guiding principles are firm. Never auto-apply, human confirmation is mandatory. Lean on ECC's continuous learning patterns first. Keep everything transparent and auditable, with clear rationale and risk for every suggestion. Stay non-blocking, so suggestions are collected and reviewed when convenient rather than interrupting work. And keep it extensible, so the same pattern can later apply to rules, hooks, or even whole projects. Success here means the agent can generate clear, actionable suggestions, no skill is ever changed without explicit confirmation, confirmed changes are safely applied and propagated, a full audit trail exists, and the whole thing feels natural and unintrusive. The agent proposes, the human decides, the system applies. Together this completes a powerful, safe self-improvement loop while keeping the human firmly in control.
