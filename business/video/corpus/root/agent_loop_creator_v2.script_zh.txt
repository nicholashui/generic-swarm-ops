我们一起来过一遍这份详细的、可以马上动手搭建的认知增强型 agent loop 蓝图。你可以把它当成第三版设计的可执行版本。早前那份文件讲的是理念，这份则是实现指南，是你交给一个 coding agent，让它真正可以开始写这套系统的东西。它以最初的第一版为基础，再加以延伸，做法是把一系列排好名次的人类思维模型当成头等的、可配置的机制。我们说的是诸如 Cynefin、premortem 事前验尸分析、after-action review 行动后复盘、double-loop 双环学习、recognition-primed decision 配合双系统思维、metacognition 元认知、五个为什么配合石川图、red team 红队，以及 Paul-Elder 标准这类框架。它们全都变成机器里真实的、可开关的部件，而不再只是幻灯片上的概念。

这份规格是预备给 coding agent 去解析的，无论它是一个专注于构建的助手、一个背后有强力模型的编辑器，还是一个自定义 harness。它综合了几个来源的深入研究：一套讲多智能体系统为什么会失败的分类法、一套把工具、环境和 agent 都当成头等实体的编排协议、对核心推理循环的各种增强、顶尖模型供应商的生产实践，以及对认知框架的系统化分析。

那么这一版真正新的东西是什么？有几样。第一，明确的 Cynefin 情境分类，让系统可以视情况在快速循环和完整循环之间做路由。第二，在一开始就有一个强制的 premortem 步骤。第三，recognition-primed decision、双系统思维和元认知，会在循环早段带动一条快速识别路径。第四，在结尾有结构化的反思，结合 after-action review、double-loop learning、五个为什么和石川图，以及多个评判者的视角。要紧的是，第一版的所有东西都被保留并延伸了，而不是被丢掉。骨架、失败模式的映射、分阶段的路线图，全都带着往下走。

让我讲讲推动整个设计的核心洞见。当研究人员研究多智能体系统为什么失败时，他们发现大约百分之四十二的失败，归根到底都是规格和设计问题。另外大约百分之二十一，来自薄弱的验证和糟糕的终止判断。教训很直接：光是加更多的 agent，但没有强有力的前期规格验证、没有结构化的观察、没有明确而且经过验证的完成条件、没有真正的进度跟踪，往往得到的是边际递减、甚至负面的回报。相反，层级式组织配合版本管理和自我演化，在漫长而复杂的任务上带来巨大收益，把基准分数推到接近百分之九十。而这里新增的一层，也就是人类认知框架，提供了杠杆率最高的缺失部件：适应性智能、主动的风险处理、快慢兼备的思考，以及多层次的学习。

在模型这一侧，这个设计与顶尖的多智能体系统对齐。它用一个服务器端的多智能体能力，由一个领导者去综合几个子 agent 的工作，来处理深入研究的子任务。而在客户端循环里，它模仿构建导向工具的做法：先生成一个明确的计划，再运行并行而隔离的子 agent，并且用一份 todo 式的文件去跟踪状态。这种混合做法同时把威力、控制和成本效率最大化，而 Cynefin 路由就帮忙决定何时向外委派，何时留在本地的快速路径。

过程中解决了几个架构上的取舍。核心是用 Python 自定义构建，采用严格的数据结构、严格的 JSON 输出，以及一个完整的追踪器，这个选择是为了透明度、可审计性和教学上的清晰度，而不选较重的框架。记忆是混合式的：一份结构化的 todo 文件、一个长期向量库、积极的摘要，以及一个专门的模式库，去喂给 recognition-primed 的决策。自我演化是刻意有界限的，会用保留的轨迹做验证，并且围绕 after-action review 和 double-loop learning 来组织，让系统不会漂移。每一份任务规格里都有一个认知档案，带着开关，去启用快速路径、选反思风格、选评判者模式，让简单任务不会被不必要的复杂度淹没。原则是先求最少依赖，之后再加可选的适配器。本地优先、可观察、有沙箱，并且经过生产级加固，有断路器、重试和预算。而且它应该可以帮忙构建和改进自己，包括它自己的认知部件。

每一个已知的失败模式都明确映射到某个阶段或部件的缓解措施，而每一个也都有一个认知上的对策。Premortem 处理规格和设计的风险。元认知加上循环检测处理重复。After-action review 和 double-loop learning 补上验证和终止上的漏洞。而 recognition-primed 的决策就通过模式匹配处理情境和历史的问题。

现在讲一下目标，以及怎样衡量成功。首要目标是一个受控的、层级式的、推理与行动兼备的 agent loop，它对已知的失败模式有可靠性，并且用人类认知最佳实践加以增强。它应该可以通过版本管理、追踪和自我反思去演化。它应该是混合式的，把完整的客户端控制和对服务器端多智能体系统的可选委派结合起来，并且能聪明地选择自己的认知模式。它应该是生产级的：可观察、有成本意识、安全、可测试、可扩展。而且它应该契合预期用户的工程风格，也就是以规格驱动，有一份活的任务规格、有迭代式的评判循环、有细致的 harness 工程、本地而最少基建、Python 优先而且有严格的数据契约，以及对现有工具有干净的集成点。

成功的准则很具体。在可靠性方面，在覆盖全部十四个已知失败模式的合成失败注入测试里，缓解之后的残余失败率应该维持在百分之五以下，并且要尽早检测规格和角色违反、循环检测要触发重新规划或终止、验证器要拒绝不完整或不正确的完成尝试。在性能方面，在保留的研究和编程任务上，基础成功率至少要百分之七十，经过两三轮自我演化之后要升到至少百分之八十五，而且步骤和 token 都要少于一个朴素基准。在可观察性方面，每一次执行都应该产生一份完整、可重放的轨迹，带着来源、版本、时序、token 计数，以及完整的思考—行动—观察三元组，而且你应该可以把它可视化、从任何一步重放。在可演化性方面，一个版本管理器应该支持登记、回滚，以及选出 prompt、工具代码和 agent 配置的最佳变体，而一个自我演化器就提出并验证改进，在三个有界轮次之后有可证实的收益。在混合集成方面，研究子任务应该可以无缝委派给服务器端的多智能体系统，并且带着综合结果和引用织回主轨迹。在生产加固方面，你想要恰当的断路器，有关闭、打开和半开三种状态、有指数退避重试、有逐阶段预算、有结构化的错误观察、有沙箱化的工具执行、有输入清洗和最小权限。在易用性方面，你想要一个干净的 Python 包，有命令行界面、有可选的服务器模式、有详尽的示例、有完整的类型提示和文档字符串、有通过的测试套件，以及好的文档。而在集成方面，它应该与常见的客户端库和可选适配器兼容，并且输出结构化的计划给构建工具去使用。

还有第二层成功准则，是专属于认知层的。在适应性路由方面，系统应该把任务情境分类得与人类基准至少有百分之八十五的一致、在至少百分之七十的重复或例行子任务上触发快速识别路径、并且相比一个永远用完整模式的基准，把 token 用量削减百分之二十到三十，而又不损失简单情况下的质量。在主动风险方面，premortem 步骤应该为每个复杂任务找出至少三到五个合理的失败模式，并且把缓解措施折回规格和质量关卡，可度量地减少下游问题。在结构化反思方面，after-action review 应该产出完整的成品，用证据回答完四个问题，double-loop learning 应该质疑主导的假设，并且在至少百分之六十的有界轮次里提出能通过验证的元层改动，而五个为什么配合石川图分类就应该用在至少百分之八十的已诊断失败上。而在多视角评判方面，验证器的红队和 Paul-Elder 模式，相比只用标准模式，应该至少多捕捉百分之十五的问题，尤其是围绕假设、偏见和边界情况那些。

还有约束和明确的非目标。技术栈是 Python 三点一一或以上，有严格的数据验证、异步调用和现代 HTTP 客户端。可选的部件包括一个向量库、一个服务器框架和一个状态机适配器，但核心循环避免重型框架的绑定。极简主义优先：核心循环、状态、可靠性和基本层级，初期应该写在两千行代码以内，之后再加演化和混合集成。永远不准有不受控的循环，所以有硬性步数上限、有靠状态哈希的循环检测、有以进度为本的退出，以及断路器，而每一次模型调用都用一个严格的输出结构。安全意味着把代码执行沙箱化、永远不盲目相信模型生成的参数，并且留意异常的模式。成本控制意味着 token 预算、只在独立分支才用并行、有压力时才做摘要，以及准则一达到就提前终止。而早期阶段的非目标包括完整的分布式执行、一个图形仪表板、原生的多模态处理和生产级的多租户。整份规格本身也是一份活的文件，所构建的系统最终可以更新它。

让我用研究综合来落地。失败分类法是主要的地图。它来自分析超过一千六百条轨迹，来自七个框架，人类标注者之间有很高的一致性，并且识别出横跨三个类别的十四个失败模式。第一个类别，系统设计问题，是最大的那个，接近百分之四十二。它包括违反任务规格、违反角色规格、步骤重复、对话历史丢失，以及对终止条件毫无察觉。第二个类别，agent 之间的错位，占大约百分之三十七。它涵盖对话重置、不去问清楚、任务脱轨、隐瞒信息、无视另一个 agent 的输入，以及推理和实际采取的行动不一致。第三个类别，验证和终止，占大约百分之二十一，捕捉到过早终止、不完整的验证和不正确的验证。

这些关键发现直接转化成缓解措施。规格质量是投资回报率第一名，所以第零阶段是强制的：一份结构化的任务规格，有明确的目标、成功准则、约束、输出格式和质量关卡，加上一个在循环还没开始之前的自动验证器和评判者，再加上委派时的角色契约。验证是弱点，即使在看上去成功的运行里也是，所以有一个专门的验证器，在多个层次输出严格的结构化判断，有一个明确的完成动作必须带着证据通过验证，以及在思考步骤里嵌入进度跟踪。情境丢失和重复很常见，所以有积极的摘要、有结构化的状态去保留关键事实而不是倾倒全部历史、有靠哈希近期动作和观察的循环检测，以及一个硬性步数上限配合以进度为本的提前退出。Agent 之间的问题就由一个强力的中央编排者去处理，它有明确的分解、路由和契约，有结构化的交接观察、每个工具和角色有断路器，以及有版本管理的共享状态。这份指引让 coding agent 在它的 prompt 和验证器里明确地引用这些失败模式，并且建立一个失败注入测试套件，去模拟每一个并且断言缓解措施确实有效。

第二条研究支柱是一套把工具、环境和 agent 当成头等抽象的编排协议。一个建基于它的系统在一个高难度基准上达到大约百分之八十九，自我演化还推得更高。工具是有版本和生命周期管理的，登记时带着名字、描述、结构、实现，以及一个用来检索的语义嵌入，而且还可以动态生成。环境定义观察和行动的空间，并且维持状态一致，在这里也就是一个工作目录、一个沙箱文件系统和代码运行时的状态。Agent 带着角色、能力和元数据，支持层级和协调契约，并且可以在做 agent 和做工具之间转换。要实现的机制包括：一个版本管理器，跟踪谱系和指标，并且支持登记、回滚和选最佳变体；一个追踪器，记录完整轨迹供重放和优化；一个自我演化模块，受梯度式文本优化启发，会收集一条轨迹、诊断根本原因、应用一个改动、在保留的示例上验证，如果更好就登记新版本；细致的情境管理，会为子 agent 把情境切片并标明来源，而不是倾倒全部；以及层级式编排，由一个中央规划者去分解目标、路由给专家、让它们运行自己的受控循环，再去综合结果。

第三条支柱是基础的推理与行动范式和它的各种增强。核心把一个思考，也就是一段推理轨迹，与一个行动，比如工具调用、委派或完成，交错在一起，接着是一个有依据的观察。在这个基础之上，这个设计折入反思式的口头自我批判、可选的生成并排名多条候选轨迹、一种在推理步骤本身就加强依据的技巧，做法是重新锚定到最新的观察和原本的目标，以及明确的计划与执行范式，配合可选的树搜索元素。它从顶尖供应商那里借来服务器端的推理循环，由模型决定工具、在内部执行、然后迭代到有最终答案，以及并行专家配一个领导者的模式。这里的混合做法，是由一个客户端编排者掌管全局状态、追踪和验证，同时把深入研究的子问题委派给服务器端的多智能体系统，而编程子任务就用本地专家，或者用带着复制状态切片的隔离并行进程来做。

经过这么多番重新思考，最终的架构决定就很清楚了。循环风格是一个受控的、自定义的推理与行动循环，以哈希为本的循环检测和断路器做基础，再在上面叠上层级，由一个编排者去决定是委派、调用工具、综合，还是完成。中央控制胜过协调的混乱。状态对象持有任务规格、历史、todo 清单、计划、短期和长期记忆、版本登记册、追踪器、预算，以及用来做循环检测的已见哈希集合。记忆依靠结构化的 todo 和关键事实做低 token 的主力，有压力时做摘要，配一个可选的向量库做语义检索，而子 agent 永远只会收到带着来源的情境切片。模型调用经过一个统一的客户端，支持多个供应商，永远带着一个系统 prompt、适当的少样本示例，以及一个严格的输出结构。工具住在一个有验证的登记册里，配一个安全执行的包装。混合集成有一个专门的包装，会送出一个狭窄的子目标连同成功准则和已启用的工具，并且把结果记录成一个特别的观察。自我演化的范围随阶段扩大，从 prompt 到工具代码，再到 agent 配置。而系统会自我实践，在开发期间用这个 harness 在保留轨迹上改进它自己的 prompt 和验证器。

现在到了系统架构和高层次的流程，它走过六个阶段，并且用认知层加固了。第零阶段是初始化，以规格驱动，连认知设定。它把指令解析成一份经过验证的任务规格，包括认知档案，接着运行一个规格验证器和评判者，去检查完整性、含糊、角色清晰度和终止条件。然后就到强制的 premortem：假设规格和计划惨烈地失败了，找出最主要的原因和缓解措施，再把它们折回活的规格、成功准则、todo 和质量关卡。然后 Cynefin 分类把任务标签为简单、繁杂、复杂或混沌，并且用这个去自动配置循环，简单和繁杂的工作偏向快速路径，复杂和混沌的工作就走完整的审慎路径连同重度反思。Agent 状态建立时带着做 recognition-primed 决策的模式库，以及一个快速或完整的当前模式。可选地，一个规划者会生成一个高层次计划和一份 todo 文件，对着规格和 premortem 风险做验证。最后，它决定架构，是扁平、层级还是混合，再设定初始模式。

第一阶段是连认知模式选择的核心受控循环。在每次迭代开始时，或者在一个重大观察之后，系统决定它的运作模式。如果 Cynefin 允许，而模式库又有一个高相似度的匹配，加上元认知的信心高，它就走快速识别路径：一个轻量的思考去引用匹配到的轨迹、最少的 token，直接到行动。否则它就进入完整审慎模式，有一个详细的思考去重新锚定到最新的观察，连同完整关卡，同时有一个轻量的并行元认知监控器，去扫描偏见、对着准则检查进度、留意情境漂移，并且把一下信心的脉，随时准备强制换模式或者提前重新规划。它从摘要过的历史、关键事实、todo、规格、最新观察和相关的模式库条目去构建情境。模型产出一个严格的决定：先有一个带元认知罩面的思考，然后一个行动类型，然后一个载荷。有一个循环检查，会哈希近期的动作和观察，如果之前见过就强制重新规划或终止。执行经过安全的工具调用或者安全的委派。每一个结果都变成一个结构化观察，现在也记录用了哪个模式。每一个事件都追加进历史，todo 和进度都更新，模式库就在高质量结果出现时更新。轻量的反思隔一段时间或者出错时做，连同一个关于模式多有效的备注。断路器逐个工具和角色地运作，而终止可以来自达到准则并通过验证、撞到步数或预算上限、一个明确而经验证的完成、一个要升级的不可挽回情况，或者在子准则上一个强劲的快速路径成功。

第二阶段是层级式委派。编排者分解工作，再从登记册选或者实例化一个专家，比如研究员、编程员、验证员、报告员、工具生成员、浏览器 agent 或分析员。它建立一个狭窄的子任务规格，带着目标的子集、成功准则、一个情境切片和来源，然后调用这个子 agent，可以是它自己的受控循环，或者一个服务器端的多智能体委派。这个子 agent 返回一个结构化观察，连同它轨迹的摘要冒上来做审计，而父层就记下它、验证并整合它、更新全局计划，再决定下一步。

第三阶段是整合和质量关卡。一个聚合器收集那些观察和计划进度。一个协调者和报告员去合并、去重、交叉引用和解决冲突，做法是引用来源和版本，产出一份统一的草稿。验证器对着成功准则打分、检查幻觉、漏洞和验证失误，并且建议修正，全部以结构化输出。如果过不了关卡，就触发精炼，重新规划一个分支、重新委派，或者自我编辑。如果过了，就到打磨或最终输出。

第四阶段是反思和自我演化，是进阶那部分，结合 after-action review、double-loop learning、五个为什么和多视角评判者。在里程碑或者结尾，完整的轨迹和模式历史会到自我演化器。强制的 after-action review 回答四个问题：本来应该发生什么、实际发生了什么（包括快速对完整的模式统计和模式匹配）、为什么会这样（连一个深入的五个为什么的一趟，和石川图分类，横跨比如 prompt 和方法、模型和工具、数据和观察、情境和环境、状态和记忆、验证，以及人类规格这些维度），最后就是下一步怎么做，化成具体的改动。然后 double-loop 那一层质疑主导的假设：质量的定义是不是太松？角色边界是不是允许了漂移？成功准则或者认知预设本身该不该演化？只有经过元批准的改动才可以推进。根本原因会带着对失败模式和认知框架的意识去诊断，会提出针对性的改动，去改 prompt、工具代码、角色定义，甚至认知预设或者新的评判者模式，在有配置的情况下用 Paul-Elder 标准或者一次红队攻击。改进会在保留的示例上或者用重放去验证，而更好的版本就带着谱系和一个 after-action 的理据去登记，模式库就为将来的识别去更新。全部都有界限在三轮，快速路径的轨迹就用一个较轻的复盘。

第五阶段是终止和输出：一个按规格要求格式的最终综合、持久化完整的轨迹、版本和指标、一个可选的事后反思摘要，以及在高风险关卡或预算耗尽时的人类介入钩子。

核心的数据模型应该最先生成，因为它们就是契约。有一个任务规格，承载目标、成功准则的清单、约束、输出格式、质量关卡、一个可选的初始计划、元数据，以及认知档案连同它的开关，去控制快速路径、反思风格、评判者模式、Cynefin 分类和最大反思轮数。有一个结构化观察，承载状态、数据、一个摘要、一个零到一之间的信心、问题、下一步建议、来源、一个轨迹 id，和用了的版本。有一个轨迹事件，承载步数、时间戳、思考、行动类型、载荷、观察、token 用量和版本。还有 agent 状态，把任务规格、历史、todo、计划、短期和长期记忆、模式库引用、已见哈希集合、预算、版本、追踪器、当前模式、Cynefin 情境，以及 after-action review 成品的历史，全部绑在一起。

要实现的关键模块包括：核心循环、一个带指标和逐个工具实例的增强断路器、一个调校去捕捉特定失败模式的验证器、一个有规划、路由、子 agent 工厂和整合器的编排者、一个从轨迹去诊断并提出和验证改动的自我演化器、一个给工具、环境和 agent 抽象的最小协议模块连版本管理器、一个给服务器端多智能体调用的集成包装、记忆和追踪模块，以及一组有版本的 prompt 文件，在评判者 prompt 里引用那些失败模式。专就混合集成来讲，当行动需要研究委派，或者子规格复杂度高的时候，系统就会调用多智能体系统而不是一个本地子循环，有一个可配置的阈值，永远捕捉来源和 agent 数目，而如果调用失败或者断路器打开，就回退到一个本地子 agent 或者直接工具。它也可以模仿构建工具的做法，输出计划，并且在独立分支上用带着隔离状态副本的并行子 agent。

分阶段的路线图给了 coding agent 一条清楚的路。第零阶段建立骨架、数据模型、一个基本的模型客户端、严格结构的辅助器，以及一个在模拟工具上能运行而不崩溃的简单循环；它的关卡是这个循环在一个玩具任务上运行十步、永远产出有效的观察、检测到注入的重复，并且捕捉到明显的规格问题。第一阶段建立完整的受控循环，连循环检测、完整的断路器、安全执行、进度跟踪、连模式匹配的快速识别路径、轻量的元认知监控器、轻量的反思、一个带验证器关卡的明确完成、结构化的状态和积极的摘要，并且在重复、历史丢失和过早或不完整终止上的失败注入测试都要通过，快速路径也要正确地触发。第二阶段加上编排者和一个可插拔的子 agent 登记册、本地委派路径、整合器和报告员，以及一个多层次的验证器，在一个带委派的多步任务上端到端地证明到。第三阶段加上版本管理器、最小协议结构、自我演化器和服务器端多智能体客户端，自我演化在三个相似任务上显示出可度量的改善，委派也端到端地行得通。第四阶段是打磨、示例、生产加固和自我实践：一个有认知路由的深入研究 agent、一个有规划和子 agent 的编程项目 agent，以及一个演化自己验证器、prompt 和认知档案的自我改进 harness，加上一个沙箱化的代码工具、一个命令行界面和可选服务器、一个包含认知功能测试的完整测试套件、带模式和复盘覆盖层的可观察性输出、成本仪表板和文档。贯穿始终的使命：每个模块之后，生成代码、运行一个内部评判者、修正问题、对着关卡重新验证，然后才前进。

讲到生产加固、安全、可观察性和可扩展性，这些主题一路贯穿。可靠性来自断路器、带退避的重试、结构化的错误观察、预算执行和以进度为本的提前退出。安全来自把工具执行沙箱化、验证和清洗所有模型生成的参数、最小权限的工具访问，以及对循环模式的异常检测。可观察性把追踪器当成头等，记录每个事件连可选的完整情境快照、版本、token 计数、时序和子调用，可以输出来从任何一步重放，并且可选地集成进现有的日志。成本和可扩展性来自逐阶段预算、只在独立分支上的并行、聪明的摘要信号和会话隔离。而可扩展性来自可插拔的模型后端、一个有结构验证的简单工具登记册、一个有角色 prompt 的子 agent 工厂、可插拔的记忆后端、可更换的演化策略，以及给状态机框架和远程编排的适配器。

测试策略很详细，并且当成关键来看。单元测试涵盖结构验证、循环哈希的正确性、断路器完整的状态机（包括半开恢复），以及验证器在边界情况上的解析和逻辑。集成测试在玩具任务上运行完整循环，并且演练层级式委派和混合路径。失败注入测试模拟每一个主要失败模式，并且断言尽早检测、正确缓解和没有无声失败。一个迷你基准衡量成功率、效率，以及三轮之前和之后的演化差距。基于属性的测试把随机的有效和无效规格和观察抛给系统，并且断言不变量，比如永远是结构化输出、合理的信心值、受尊重的预算和一致的版本。而一个自我实践的演化测试，会用真实轨迹在验证器或规划者 prompt 上运行自我演化，并且验证改进版本在保留数据上分数更高，而又没有在基础任务上退步。使命很简单：一个阶段的测试还没通过之前，不要标它为完成，而且要连同代码一起生成测试。

那些参考把这一切落实在真实的工作上：多智能体失败分类法、工具—环境—agent 的编排协议、基础的推理与行动论文、一系列反思和规划的增强、近期的 agent 综述、顶尖供应商的生产文档、它所建基的第三版规格，以及那张排好名次、列出四十个传统人类思维模型连它们采用分数和集成理据的表。

最后，交接。这份规格是完整而可行动的，建议是马上从第零阶段的骨架和核心模型开始，因为那些模型就是契约。还有一个建议给 coding agent 的起手 prompt：完整地读相关的规格，建立包骨架连核心模型（包括认知档案、模式库和当前模式）、建立一个基本的模型客户端、一个 Cynefin 分类器的雏形，以及第零阶段的一个 premortem 步骤，以及一个带快速路径和元认知模式选择的最小可运行受控循环，去通过早期的验证关卡，用严格的结构，连失败注入和认知功能测试的雏形，贯穿始终维护一份 todo 文件，并且对每个模块都施加一个评判者。核心循环稳固之后，逐阶段推进，并且在自我实践的阶段，用所构建的系统去演化它自己的 prompt 和验证器。预期不会有要澄清的问题，因为这份规格是自足的；如果出现含糊，就由内部评判者去解决，或者带着具体的轨迹输出去升级。预期的交付成果是一个功能完整、经过测试、有文档的包，连同示例和一个命令行界面，任何人都可以导入并运行，去做可靠的、认知增强的、层级式的 agent 工作流，并且对混合集成、自定义工具、自我演化和认知配置有清晰的扩展点，以及完整支持带模式和复盘覆盖层去重放轨迹。简而言之，把它做到生产级、让认知层成为一个头等而可配置的强项、在每个模块都用评判者去迭代、让系统改进自己，并且把它变成先进的、与人类对齐的 agent harness 的基石。它已经准备好执行了。
