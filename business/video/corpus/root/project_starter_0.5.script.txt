Imagine you're setting out to build a starter project, but not just any starter. This one has a very specific job. Its purpose is to download, audit, curate, and keep in sync the configuration sources for a whole family of AI coding agents. We're talking about tools like Claude Code, Cursor, Codex, the Gemini command line, OpenCode, Grok Build, GitHub Copilot, and other agent harnesses that work in a similar way. The project carries a name, project starter, a version label that marks it as a merged bootstrap release, and a recent update date. So let's walk through what it actually asks for.

First, understand what this file really is. It's a boilerplate specification. It's meant to be handed to a coding agent, and a human will typically say something like, create a new project based on this file, treat it as the source of truth, put it at this location, give it this purpose, and use this technology stack. The instructions are blunt about one thing in particular. The agent must not simply summarize what it reads, and it must not just produce more Markdown. It has to create a working, executable project. That means reading the whole file, creating the project directory, building every folder and file the spec requires, swapping in the new project's name and purpose, copying this specification into the new project, writing the package definition, the source manifests, the executable scripts, the rule files, and the docs. Then it goes further. It downloads all the approved upstream repositories, generates a lock file that records exactly what was fetched, produces an audit document, runs a validation pass, updates the status file, and reports whether everything passed or failed. And if there's no network or shell access, the agent should still create every project file and clearly mark the source download step as blocked.

There are two valid ways to run this. The first is self-bootstrap mode, where the agent implements the starter repository directly in the current location. The second is create-new-project mode, where the agent spins up a brand new downstream project somewhere else, using this file as the complete contract. In both cases, the expectation is the same. Build something that actually runs, then kick off the bootstrap process.

When generating a downstream project, the agent can be handed a handful of input variables. Things like the project name, the output path, a short description of the purpose, the primary technology stack, whether sources should be downloaded, which coding agents are supported, the package manager to use, and the license. Each of these has a sensible default, so if a value is missing, the agent just falls back to the default rather than getting stuck.

There's an important safety rule about the output location. If the target directory already exists and isn't empty, the agent has to stop and ask for permission before overwriting or updating it. The only exception is when the user explicitly says to go ahead, and even then the agent should try to preserve important files the user created.

The spec then hammers home its most critical instruction. This file is not only documentation. When an agent is asked to implement it, it must create a real executable project and must not stop after writing Markdown. It then lists everything that has to exist. The repository structure itself. A package definition. Two source manifests, one for repositories and one for documentation. A whole set of scripts, including the main router, a project creator, a downloader, an auditor, a doctor, a sync tool, and a security scanner. It must download every enabled repository, save them into an external sources area, record the download details in a lock file, generate the audit document, support both workflows, run the validation commands, and report success or the exact blockers. And again, if network or shell execution isn't available, the agent must say plainly that it's blocked because of that, while still creating all the scripts and manifests so a human can run the bootstrap later.

Now for the non-negotiable outcome. After implementation, running the bootstrap command must perform a specific flow. It runs the doctor check, creates the required directories, validates the repository manifest, clones or updates every enabled source, writes the lock file, generates the audit, runs security smoke checks, runs a sync dry run, and runs the tests. And the project simply isn't considered complete unless the source repositories are genuinely downloaded into the external sources directory.

The required repository structure is extensive. At the top sit the specification, a readme, and agent-specific instruction files for the different tools, alongside the package definition, a task file, a status file, and the usual editor and ignore configuration. Below that, a sources folder holds the manifests, the lock file, and its own readme. An external folder holds the downloaded repositories, kept out of version control. A scripts folder contains the core tools plus two subfolders, one for per-agent adapters covering Claude, Cursor, Codex, Gemini, OpenCode, Grok Build, and Copilot, and one for shared libraries handling git, safe file operations, manifest parsing, and reporting. There are folders for rules, skills, hooks, MCP configurations, memory, reviews, suggestions with pending, approved, and rejected stages, documentation, worked examples, and tests. It's a complete scaffold, not a sketch.

The package definition declares the project as private, using modern modules, with no runtime dependencies, and it wires up a set of commands. There's bootstrap, create, and init for setup, doctor for diagnostics, several commands for downloading, updating, and checking sources, an audit command, sync and sync check, a security command, a review command, a test command, and a format command. The whole thing runs on Node version twenty or higher, and the first implementation deliberately uses only built-in Node features, no external packages.

Then comes the heart of it, the GitHub source download manifest. The agent must create this manifest exactly, and must download every source marked as enabled, never silently skipping one. If a required source fails, the command has to exit with an error. The manifest lists a long roster of repositories, each with an identifier, a name, a URL, a target location, a priority, a tier, a quarantine flag, an import policy, and a stated purpose. At the core is a cross-agent harness source that brings together skills, agents, commands, hooks, rules, and security scanner references. Then there are the official repositories, the ones from Anthropic for Claude Code, its GitHub action, its agent skills, and its official plugins, OpenAI's Codex command line, Google's Gemini command line, OpenCode, the Model Context Protocol servers and registry, and the official GitHub MCP server. Most of these are flagged for quarantine and treated as reference only, meaning their contents are studied but not imported wholesale.

The roster continues with the standard that defines cross-agent repository instructions, a couple of repositories capturing Karpathy-style behavioral rules about thinking before coding and making surgical changes, a persistent memory architecture reference that must never install itself automatically, a composable skill methodology, community best-practice collections, several curated discovery lists, community subagent definitions, frontend-oriented skill patterns, a collection of Cursor rules, and a set of security-focused Cursor rules. A few discovery indexes are marked optional rather than required. And one entry, an archived collection of older protocol servers, is deliberately disabled, with an import policy of never import, because it's historical only.

Alongside the repository manifest there's a documentation manifest. Some related tools have official docs but no repository that needs cloning, so this manifest points at things like the Claude Code feature documentation, the Cursor rules documentation, the Grok Build overview, the GitHub Copilot custom instructions, and the Model Context Protocol site. The first version can simply create this file without actually fetching the docs, and a later version can implement a docs download step.

There's a clear policy for the external directory. It has its own ignore file that keeps the downloaded sources and docs out of version control entirely. Downloaded upstream repositories must never be committed into the project. Only curated, audited, and properly attributed files are ever allowed to be copied into first-party directories like rules, skills, hooks, MCP configs, and docs.

The downloader script has well-defined behavior. It supports commands to download, update, and check sources, to select a particular profile such as all, core, official, or discovery, and to run as a dry run or in strict mode. By default, downloading reads the manifest, selects every enabled source, creates the external sources directory, and for each source does the right thing. If the target doesn't exist, it performs a shallow clone. If it exists and is already a git repository, it fetches and fast-forwards. If it exists but isn't a git repository, it fails. Along the way it records metadata and produces a readable summary. For required sources, any failure, whether a failed clone, a failed update, a missing git folder, or an invalid URL, is fatal. For optional sources, a failure is non-fatal unless strict mode is on, but it still gets recorded. Disabled sources are not downloaded by default.

The security rules for the downloader are strict. It must not run install scripts from downloaded repositories, must not run an install inside them, must not pipe remote scripts into a shell, must not execute their hooks, must not copy their content into active agent configuration automatically, and must not write anything outside the project root. Its only job is to clone or update and to inspect metadata. Nothing more.

The lock file captures the results of all this. For each source it records the identifier, the name, the original and resolved URLs, the target, the status, the commit, the branch, the timestamp and subject of the last commit, any license and package files found, the quarantine flag, and the import policy, plus a list of any failures. The audit script then reads both the manifest and the lock file and produces a human-readable audit document, with one section per source covering all those same details along with selected components, rejected components, and security notes.

The audit has its own judgment rules. It must mark a source as unsafe for automatic import if there's no license file, if the repository contains install scripts that run remote commands, if it has suspicious post-install behavior, if the source is archived, if it requires credentials, if it modifies global agent configuration, or if it includes MCP servers that demand broad filesystem or network access. On the first pass, most repositories should be marked as having no components selected yet, with bulk import rejected until a human reviews them. The one exception is that small rule-only repositories, like the Karpathy skills, may be selected for careful extraction once their license is verified.

There's a bootstrap router script that ties everything together. It supports commands to bootstrap, to create a new project with a name and path, to initialize, and to format. The bootstrap command runs the doctor, the source download, the audit, the security scan, the sync dry run, and the tests, in that order, and if any required step fails, the whole bootstrap fails. The create command, which can delegate to a dedicated project creator, parses arguments for the name, path, purpose, stack, download behavior, and overwrite mode. It validates the name and path, refuses to overwrite a non-empty target unless forced or explicitly approved, builds the directory structure, copies the specification into the new project, swaps in project-specific values, generates everything required, optionally runs the download unless told not to, and prints clear next steps for the human.

The doctor script is the health check. It confirms the Node version is at least twenty, checks that git is available, looks at the working directory and write permissions, makes sure the repository manifest exists, that the external directory exists or can be created, and that the scripts folder is present. It notes the operating system and checks symlink support if sync wants to use symlinks. Then it prints a tidy report showing each check as okay, or it fails with clear messages.

The sync layer is where curated rules turn into agent-specific configuration. Even if the adapters start out minimal, there must be a working sync framework. A dry run command generates or previews the various agent files, the shared instruction files for the different tools, the Copilot instructions, the Cursor rules file, and the settings files for Claude, Gemini, Codex, and OpenCode. Every generated file carries a header announcing that it's auto-generated and shouldn't be edited directly, pointing back to the rules, skills, hooks, and configs it came from. Where practical, sync preserves any user-written content outside the managed sections, using clear begin and end markers around the blocks it controls.

The security script is a static scanner, at least to start. It scans downloaded repositories for suspicious root-level scripts, detects environment files, private keys, tokens, or certificates that might have been committed by accident, reports MCP configurations with broad filesystem access, and warns about remote install pipes and post-install scripts in the downloaded sources. Crucially, it never executes any downloaded script. A simple static scanner is enough for the first version.

The spec also defines a set of initial rule files. There's a constitution that says to follow the user's goal exactly, prefer simple maintainable solutions, make surgical changes, avoid destructive actions without approval, never install or execute or import downloaded third-party code until it's audited, run the relevant tests or explain why not, and update status after major work. There's a Karpathy-style rules file that emphasizes thinking before coding, not assuming hidden requirements, preferring the simplest working solution, avoiding speculative abstractions, changing only what's necessary, keeping the goal visible, and verifying the result. And there's a human-approval rule that lists the actions which always require sign-off first, like installing global packages, running remote install scripts, enabling MCP servers that use credentials, copying third-party code into active configs, modifying hooks, deleting files, or applying self-generated rule and skill changes.

The import policy deserves emphasis. Downloading sources is mandatory, but importing them is never automatic. The project keeps four stages clearly separate. There's the downloaded source, which lives in the external area and is treated as untrusted, ignored by version control, and never executed. There's the audited source, whose metadata has been reviewed and recorded in the audit and lock files, but which hasn't been imported. There's the curated import, where a file has had its license verified, its source commit and checksum recorded, human approval if it's high impact, and a confirmation that there's no hidden install behavior, and only then is it placed into the first-party folders. And there's the generated adapter output, the agent configuration files that must always be reproducible just by running sync.

Testing is required too. The agent must create Node test files covering at least the manifest, the downloader, the auditor, and sync. The tests verify that the manifest parses, that every enabled source has all its required fields, that there are no duplicate identifiers, that every target lives under the external sources path, that disabled archived sources aren't selected by default, that the lock file has a valid shape after a run, and that generated files include the auto-generated header.

The spec rounds out with the supporting documents. A readme with a quick start that explains how bootstrap downloads the approved repositories, writes the lock and audit files, and reminds the reader that downloaded repositories are untrusted until audited and are never executed or imported automatically. An agents document summarizing the supported agents, the shared rules, the validation commands, and that same reference-until-audited policy. A changelog with a minimal first entry. A task file listing the first milestone as a checklist. And a status file describing the current phase, the latest update, any blockers, and the command to run.

The acceptance criteria are concrete. The implementation is only accepted when the doctor, the download, the audit, the security scan, the sync dry run, and the tests all pass. After the download, every required source directory must exist, unless an optional source failed and that failure was recorded. The lock file and the audit document must both exist. And the downloaded repositories must not be committed to version control.

There's a precise implementation order to follow, starting with reading the file fully, then creating the folder structure, the package definition, the ignore files, the manifests, then the shared libraries, then each script in turn, then the rules, the docs, and the tests, and finally running the full validation sequence. If everything passes, the status file gets updated and a final summary is reported.

The spec is equally clear about what not to do. The agent must not just rewrite the Markdown, must not pretend downloads happened, must not skip the download silently, must not execute downloaded scripts, must not install global packages or run remote installers, must not copy all downloaded skills into active config, must not overwrite user files without a backup, must not overwrite a non-empty target without permission, must not mutate the global configuration of any of these tools, must not put secrets into memory, must not log hidden reasoning, and must not auto-approve its own self-improvement suggestions.

A couple of edge cases get their own handling. If a repository has moved or redirects, the downloader should let git clone it, record the resolved remote URL, note a warning in the lock file, and continue if the clone succeeded. If a repository no longer exists, then for required sources the bootstrap fails, the failure is written to the lock file, and the user is told exactly which source failed. For optional sources, it continues, records the failure, and mentions it in the audit.

Finally, the definition of done. The project is finished when the repository is correctly named, the bootstrap command exists, the manifest contains the exact set of repositories, the download actually fetches the enabled ones into the external area, the lock file records their commits, the audit is generated, the downloaded repositories are ignored by version control, no downloaded code is executed, the sync dry run works, the tests pass, and the status file is updated with the real results. There's a short list of non-goals too. This starter doesn't need a web app, a backend, a database, a frontend framework, TypeScript, Docker, Kubernetes, cloud deployment, runtime third-party dependencies, or any automatic changes to global tool configuration. Downstream projects are free to add those things later, as long as the docs and validation keep pace. And when everything is complete, the agent reports a clear, human-facing success message, summarizing how many sources succeeded, failed, or were skipped, what was generated, how each validation step fared, and a reminder that the downloaded repositories sit in the external area, not imported or executed until they've been audited.
