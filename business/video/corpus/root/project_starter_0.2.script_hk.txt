我哋一齊行一次一份改良版嘅生存任務規格，項目叫做 Project Starter。你可以當佢係一張藍圖，用嚟起一個可以直接投產嘅起始倉庫，呢個倉庫負責安裝、同步、審計，並且持續演化一套跨代理嘅編碼骨架，覆蓋一整個系列嘅 AI 編碼工具。我哋講緊嘅係 Claude Code、Cursor、Codex、OpenCode、Gemini 命令列工具、Grok Build，仲有類似嘅助手。呢個係零點二版本，亦即係重新思考嘅版本，已經準備好去做團隊所講嘅第零階段執行。佢喺二零二六年六月建立，主要目標好清晰：整一個倉庫，喺一個各自為政、配置系統各有不同嘅工具世界入面，帶嚟一致性同安全。

咁點解要重新思考呢？原本嘅計劃其實好扎實，但同時亦都太闊、有啲重複，亦有少少風險。佢假設你可以做完整安裝，又可以喺一啲根本唔會用同一種方式儲存設定嘅工具之間做到完全一致。呢個改良版令成個項目更加可執行，背後有十個重點。

第一，佢將一個基礎骨架，我哋叫做 E C C，定為一切嘅根基，但唔係盲目咁複製所有嘢，而係先安裝經過精心挑選嘅配置檔。第二，佢唔再以 shell 腳本行先，而係改用一個以 Node 寫成、跨平台嘅命令列工具。第三，佢定義一個單一可信來源，配上生成出嚟嘅轉接器，同埋自動檢查有冇偏移。第四，佢將「處處都有相同技能同規則」呢個目標重新理解為語義上嘅一致，即係行為一樣就得，唔係要求每個檔案逐個位元都完全相同。第五，佢加入正式嘅清單，記錄授權、來源、校驗碼同版本。第六，佢清晰咁分開指令、技能、記憶、掛鈎、模型上下文協議嘅配置，同埋生成出嚟嘅工具配置。第七，佢引入一個明確嘅安全威脅模型。第八，佢唔再記錄每一條推理軌跡，而係儲存可審計嘅摘要：證據、決定、差異、命令、測試結果同審查輸出。佢刻意唔會要求或者儲存隱藏嘅思維鏈。第九，自我改進喺人手批准之前只可以停留喺建議階段。第十，佢加入可量度嘅品質關卡、同步測試，同埋安裝時間嘅預算。

喺起任何嘢之前，會有一個來源核實快照。第零階段必須先重新檢查每一個來源。預定嘅優先次序係咁：E C C 骨架係主要根基，跟住係一套精簡嘅行為規則，風格同 Andrej Karpathy 相關，再跟住係一個持久記憶工具，例如 claude-mem 或者同等嘅選擇，前提係佢相容又安全。之後就係由口碑好嘅 Claude Code 倉庫挑選出嚟嘅規劃同工作流程模式，再跟住係有選擇性咁引入、而唔係大批安裝嘅精選技能庫，最後就係每個代理嘅官方文件。對每一個來源，第零階段都必須核實最新版本或者發布、授權、安裝方式、支援邊啲配置路徑、安全影響，仲有具體有咩會被納入、改造或者拒絕。

跟住係一啲不可妥協嘅原則。規格永遠驅動實作。當組件有重疊嘅時候，優先採用 E C C 嘅選擇，除非來源審計證明另一個更加適合。Karpathy 行為層嘅意思係：寫程式之前先諗清楚、偏好簡單、做精準嘅改動、以目標為導向。系統有一個單一可信來源：放技能、規則、掛鈎同配置嘅中央資料夾先係權威，其他每個工具嘅檔案都係由呢個來源生成。系統預設安全，冇人手明確批准就唔會有破壞性嘅自動化、遠端連接，或者技能改動。佢以本地為先，倚靠本地腳本、本地記憶同本地審計記錄，外部服務維持可選。核心保持精簡，較大嘅組合包維持可選同經過挑選。佢盡可能喺 macOS、Linux 同 Windows 上都運作到。佢係可審計而唔係不透明，儲存簡潔嘅理據同證據，而唔係要求隱藏嘅推理。

而家講吓實際會交付啲咩。交付物包括個新倉庫本身、一個透過 Node 運行嘅主要命令列入口，同埋一組方便嘅套件腳本，用嚟初始化、同步、檢查同步、做 doctor 診斷、掃描安全、做審查、跑測試同格式化。喺呢啲之上仲有以 E C C 為基礎嘅起始配置檔、一個跨代理嘅同步引擎、技能、規則、掛鈎同配置嘅精選清單，同埋每個主要工具嘅轉接器。亦都有為 GitHub Copilot、Zed 同 Windsurf 而設嘅可選轉接器。再有一個自我審查同評論員嘅例行程序、一個技能生命週期嘅建議同批准工作流程、一個安全基線，仲有齊全嘅文件，配上範例、測試同驗收檢查。

倉庫結構係圍繞住可信來源呢個概念嚟組織。最頂層放嘅係面向代理嘅指令檔案、說明檔、套件定義，同埋生存式嘅任務同狀態檔案。跟住有為每個工具而設、放生成配置嘅隱藏資料夾，仲有一個中央技能資料夾，按類別細分，例如規劃、實作、測試、審查、安全、記憶同生命週期。一個規則資料夾放住一啲有編號嘅檔案：一份憲章、Karpathy 行為規則、規格驅動開發嘅規則、安全、測試、權杖效率同人手批准。仲有放掛鈎嘅資料夾、放模型上下文協議配置嘅資料夾，分為最小同可選，同埋一個記憶資料夾，放住項目上下文、交接筆記同反思。亦有放審查嘅資料夾、放建議嘅資料夾，按待處理、已批准同已拒絕分類，配一份審計記錄，仲有一個腳本資料夾，入面放主路由器同每個工具嘅轉接器。最後有文件資料夾、範例資料夾，同埋一個測試資料夾，入面有測試夾具，同埋針對同步、清單同轉接器嘅測試檔案。

仲有一套嚴格嘅生成檔案政策。每一個生成或者改造出嚟嘅檔案都會帶住一個檔頭，聲明佢係由 Project Starter 自動生成、寫明佢嘅來源資料夾，並且提醒你去執行同步命令，而唔好直接編輯佢。規則好硬性：中央檔案佔上風，生成檔案只會喺衝突檢查之後先被覆寫，本地用戶檔案喺任何覆寫之前都會先備份。一個試行模式必須顯示佢會執行嘅確切寫入同刪除。一個檢查模式必須喺生成檔案過時嘅時候報錯。符號連結只會喺安全又支援嘅情況下優先採用，喺 Windows 上，除非偵測到妥當嘅符號連結支援，否則系統會退而採用複製模式。

第零階段係研究、範圍鎖定同來源審計嘅階段。目標係喺生成任何一個檔案之前，先確認最新嘅來源、安裝命令、授權同相容性。任務包括核實最新嘅 E C C 發布同佢嘅安裝器、確認 Karpathy 規則同一個 Cursor 變體、檢查持久記憶嘅候選方案、審視最佳實踐嘅工作流程倉庫但只挑選唔重複嘅模式，並且只揀高價值嘅技能組合包。一份來源審計文件會就每一個來源記錄佢嘅名稱、位置、版本、授權、安裝命令、揀咗嘅組件、拒絕咗嘅組件，同埋理據。一份決策文件會用結構化嘅格式記錄架構決定。仲會定義三個起始配置檔：一個核心配置檔，精簡、安全，可以喺五分鐘內初始化好；一個強力配置檔，安裝範圍更廣，加上記憶同精選技能；同埋一個實驗配置檔，用嚟做 Grok Build、額外連接器同多代理示範。要過關，冇一個來源可以喺冇記錄授權同版本嘅情況下使用，唔可以接受有衝突嘅重複，必須有清晰嘅優先次序，而安裝命令喺寫任何腳本之前都必須核實過。

第一階段起倉庫骨架同命令列基礎。團隊初始化 Git 倉庫、加入帶齊所有腳本嘅套件定義，並且實作主命令路由器。一個 doctor 腳本會檢查 Node 版本、Git 係咪可用、操作系統、符號連結能力、安裝咗邊啲代理，同埋必要嘅目錄係咪存在。基本配置檔、一個說明檔同起始文件都會加入，連埋有 JSON 結構描述支撐嘅空清單，同埋第一份任務同狀態檔案。呢度嘅成功標準係：doctor 命令喺一台乾淨嘅機器上行得到、佔位測試通過、骨架可以喺一分鐘內起好，而且所有嘢都一致咁用 Project Starter 呢個名。

第二階段安裝同改造 E C C 基礎，作為主要嘅骨架層。一個轉接器或者安裝器封裝會支援核心、最小、強力同實驗呢幾個配置檔。佢會安裝揀好嘅組件：核心規則、規劃、審查同安全技能、安全掃描器、權杖同上下文優化、喺安全前提下嘅記憶同本能學習、並行編排模式，同埋連接器慣例。最關鍵嘅係，佢避免盲目咁做完整複製。E C C 嘅來源版本會被記錄、衍生檔案會加上署名，之後仲會跑一次安全掃描。驗收門檻係：核心配置檔可重現咁安裝好、掃描有跑、預設唔會載入過大嘅技能堆，而當有衝突嘅時候，E C C 就係預設嘅贏家。

第三階段建立行為規則同規格驅動開發嘅憲章。呢啲係每個代理都會收到、精簡而高影響力嘅規則。行為層好簡單講得清：寫程式之前先諗、偏好簡單方案、做精準改動、以既定目標為依歸、只喺被有意義嘅含糊卡住嘅時候先發問、改動多個檔案之前先規劃、跑測試或者解釋點解冇跑、做完重大工作之後更新狀態檔，並且喺冇人手批准之前，永遠唔自動套用對技能、規則或者掛鈎嘅改動。要過關，所有面向代理嘅檔案都必須收到語義上一致嘅行為，規則必須保持精簡同唔重複，而任何衝突都必須有記錄。

第四階段建立精選嘅技能集，要夠精簡避免臃腫，但又要覆蓋高價值嘅工作流程：規劃、實作、測試驅動開發、程式碼審查、安全審查、除錯、重構、文件、記憶同交接、上下文壓縮、跨代理同步、自我審查，同埋技能建議嘅生命週期。清單入面每一個技能條目都帶住一組豐富嘅欄位：一個識別碼、名稱、版本、描述、來源、來源優先級、授權、校驗碼、相容嘅代理、副作用、係咪需要人手批准、標籤、路徑、一個負責人，同埋最後審查嘅日期。目標係核心配置檔大約有十五到三十個技能，而唔係幾百個，可選嘅組合包可以揀選，每個技能都有署名同版本，而且全部都透過同步轉接器測試過。

第五階段係跨代理同步層，而呢個其實就係整個項目嘅核心。目標係令每個受支援嘅代理，喺各自工具支援得到嘅範圍內，都收到相同嘅精選技能、規則、掛鈎同連接器配置。命令列容許你一次過同步晒所有嘢，又或者針對單一代理，例如 Claude、Cursor、Codex、Gemini、OpenCode 或者 Grok Build。你可以用試行模式預覽改動、用檢查模式核實新鮮度、喺自動、複製或者符號連結模式之間揀選，甚至只選擇一個子集嘅類別，例如規劃、審查同安全。每個轉接器都會寫入適合自己工具嘅位置：Claude Code 收到佢嘅技能、規則、設定、命令墊片同根指令檔；Cursor 收到佢嘅規則檔同連接器配置；Codex 同 OpenCode 收到佢哋嘅代理指令檔同配置，喺冇原生技能嘅地方就用一個生成出嚟嘅技能索引；Gemini 收到佢嘅指令檔同設定；Grok Build 同 Copilot 收到佢哋相容嘅引用同指令檔。同步安全係內建嘅：被修改嘅檔案會備份、人手嘅編輯會被偵測到、會產生一份同步報告、清單或者結構描述出錯會導致失敗，仲支援持續整合嘅偏移檢查。證明過程好令人滿意：改一個中央技能、跑一次同步，每個受支援嘅代理就會更新，全程冇任何地方需要人手複製。

第六階段處理記憶同交接，提供持久、注重私隱嘅記憶，但唔會令上下文臃腫。佢有幾層：狀態檔記錄當前進度、任務檔作為生存式規格、一個項目記憶檔放穩定嘅上下文、一個交接檔放精簡嘅延續摘要，同埋一個反思資料夾放經驗教訓同重複出現嘅模式。可選擇加插一個專用嘅記憶層。規則保護私隱：永遠唔儲存秘密、遮蔽憑證同個人資料、用摘要代替倒出整堆日誌、只載入相關嘅記憶、延續工作時優先用交接檔，並且令自動注入嘅上下文保持細小。一個新工作階段應該可以由任務、狀態同交接檔恢復，自我審查可以生成反思條目，而記憶亦可以完全關閉。

第七階段加入自我評估同評論員嘅例行程序，係一個規劃、實作、審查同精煉嘅結構化循環。佢有角色：一個解題者負責實作、一個評論員審查正確性、簡潔度、規格遵從、安全、效能同可維護性、一個可選嘅安全評論員、一個可選嘅測試評論員，同埋一個裁判去總結咩係阻塞性、咩唔係。一份評分表會就九個維度由一分到五分評分，涵蓋正確性、規格遵從、簡潔度、安全、測試覆蓋率、可維護性、效能同權杖效率、可審計性，同埋自我改進嘅價值。輸出係可讀同結構化兩種格式嘅審查檔案，列出阻塞性發現、非阻塞性建議、跑咗嘅測試、改咗嘅檔案，同埋建議嘅下一步。仲有命令去觸發自我審查同評論，可以係代理命令，亦可以透過命令列。呢度嘅護欄好重要：唔記錄隱藏嘅思維鏈、只儲存簡潔嘅理據同證據、預設最多兩個精煉循環，並且對高影響嘅改動要人手確認。

第八階段係技能生命週期，容許技能、規則同掛鈎只透過建議去做安全嘅自我演化。工作流程會分析使用情況、審查、失敗、缺口同重複，然後生成一個建議檔案。一個人會審視佢，然後批准、拒絕或者修改。只有獲批准嘅改動先會套用到中央來源，之後同步就會傳播佢，而審計記錄會記低呢個決定。每一個建議都包含一個識別碼、要做嘅動作即新增、更新或者移除、目標、理據、證據、提議嘅改動、風險評估、回滾計劃、信心程度，同埋一個人手批准欄位。嚴格規則係：冇明確批准就唔會有任何變動、獲批准嘅改動會被記錄、被拒絕嘅建議會保留嚟做審計，而且永遠有一條回滾路徑。

第九階段建立安全基線，令呢套骨架可以安全咁喺現實世界本地使用。威脅模型防範以下情況：透過文件、議題、網頁或者連接器輸出嚟嘅提示注入；秘密外洩；破壞性嘅 shell 命令；過於寬鬆嘅掛鈎；不安全嘅連接器伺服器；來自技能庫嘅供應鏈風險；生成檔案偏移；隱藏嘅遠端遙測；同埋意外覆寫全域配置。控制措施包括一次安全掃描、一個秘密掃描器、針對連接器同掛鈎命令嘅允許清單、對敏感檔案例如環境檔、金鑰同權杖嘅保護、對破壞性命令要人手批准、一個持續整合嘅安全檢查、依賴審計、冇預設嘅遠端憑證，同埋一條硬性規定：除非明確批准同記錄，否則禁止將下載內容直接灌入 shell。要過關，安全命令喺本地行得到、關鍵發現會令建置失敗、敏感檔案受保護，而文件亦解釋到呢個模型。

第十階段涵蓋文件同範例，目標係令呢個起始倉庫可以喺十分鐘內用得起。文件涵蓋概覽同快速入門、確切嘅安裝命令、日常工作流程、可信來源同轉接器嘅架構、同步行為同疑難排解、安全模型、決策記錄，同埋來源審計。範例涵蓋一個規格驅動嘅功能工作流程、跨代理同步、自我審查、技能建議，同埋一次安全掃描。一個新用戶應該可以喺十分鐘內初始化同同步好、文件應該解釋到啲工具有咩唔同，而範例應該真係跑得到。

第十一階段係驗證同持續整合，存在嘅目的係防止退化，並且證明呢個起始倉庫運作到。測試涵蓋清單結構描述驗證、轉接器快照、同步試行、偏移偵測、一次安全掃描嘅冒煙測試、Windows 路徑處理、檢查有冇重複嘅技能識別碼，同埋檢查有冇生成檔案漏咗檔頭。持續整合管線會檢查 JavaScript 嘅程式碼風格、驗證 JSON、喺可行嘅地方檢查 Markdown 連結、跑測試、跑同步檢查、跑安全掃描，並且上載審查同安全嘅產物。成功嘅意思係管線係綠色、一個全新克隆可以通過安裝同測試，而同步檢查喺生成之後通過。

跟住係完成嘅定義。當初始化可以整出一個運作到嘅起始倉庫、同步可以更新所有受支援嘅代理配置，而 Claude Code、Cursor，連同 Codex、Gemini 或者 OpenCode 至少其中一個都核實過，呢個項目就算完成。E C C 基礎必須安裝好同有記錄、Karpathy 規則生效中、自我審查工作流程運作到，而技能建議要受人手批准把關。安全掃描必須運作到並且攔截關鍵問題、文件必須足夠俾一個新用戶用、每一個生成檔案都必須可重現、來源審計同決策都要完整，而 Project Starter 呢個名要處處一致咁使用。

最後做個總結，呢度係即時嘅下一步行動：起好倉庫骨架、跑第零階段嘅來源審計、實作主腳本、doctor 同一個空嘅同步框架、加入中央規則同清單、先起 Claude 同 Cursor 嘅轉接器、再加 Codex、Gemini 同 OpenCode、安裝同改造 E C C 核心配置檔、跑一次安全掃描、寫文件同範例，並且將呢一切作為 Project Starter 嘅初始化去提交。而整體嘅實作優先次序都跟住同一條軌跡：基礎行先，跟住係同步，再係 E C C，跟住係規則、精選技能、安全、記憶、評論員、生命週期，最後就係範例、持續整合同文件嘅打磨。呢個就係完整嘅圖景，一個精簡、安全、可審計嘅起始倉庫，由單一可信來源出發，令每一個編碼代理都保持同步。
