撐起呢一切嘅，係一套輔助工具。一個處理檔案系統嘅工作，例如檢查檔案同目錄、確保目錄存在、讀寫 JSON 同文字。另一個處理 git：檢查係咪用得到、行安全指令、複製、攞目前嘅提交、驗證地址。第三個處理記錄，分資訊、警告、錯誤同成功幾個級別，仲有步驟標題。第四個處理項目相關嘅嘢，例如找出項目根目錄、驗證路徑有無留喺項目之內或者外部來源之內、將項目名轉成套件名、以及載入項目資料。

份規格亦都定義咗一系列規則檔。通用規則講好習慣：編輯前先讀任務、改之前先讀現有檔案、傾向細小清晰嘅改動、唔好憑空捏造需求、唔好移除安全檢查、喺行為改變嗰陣更新文件同測試、匯報前先驗證、並且匯報埋改咗乜同有乜未做完。安全規則重申嗰幾條核心禁令，圍繞執行落載返嚟嘅程式碼、喺入面安裝相依套件、改全域設定、洩露機密、寫去項目以外、覆寫使用者嘅成果、行破壞性指令，以及未經審查就將參考來源變成運行時相依套件，最後以一句簡單嘅指示作結：唔肯定嘅時候，就停低問清楚。編程規則傾向簡單嘅程式碼、標準函式庫、模組語法、清晰嘅命名、輸入驗證、有用嘅錯誤、可重複執行嘅腳本同確定性嘅輸出。git 規則話唔好改寫歷史或者強推、唔好提交機密、令外部來源保持隔離、除非有要求否則唔好建立提交，以及喺收尾前展示改咗嘅檔案。測試規則堅持行為改變就要更新測試、匯報前先行、永遠唔造假結果、令煙霧測試快速，以及測試結構、清單同保安。而來源規則就再次強調外部來源純粹係參考、淨係落載入外部資料夾、永遠唔執行、永遠唔自動引入，並且要記錄提交同做審核。

跟住係技能檔，佢哋讀落去就好似一份份聚焦嘅操作手冊。規劃技能帶你行一次：重述目標、找出要查睇同要改嘅檔案、辨識風險、整一張短清單、用細小步驟執行，跟住驗證。除錯技能講重現問題、細心讀錯誤、隔離出最細嘅失敗單元、檢視最近嘅改動、一次淨係測一個假設，跟住記低個修正。重構技能強調保留行為、改之前同之後都令測試通過、一次改一個概念、傾向清晰、並且淨係喺你肯定嗰陣先移除死碼。測試技能講先行現有測試、加煙霧測試、測成功又測失敗、保持確定性、永遠唔造假輸出。文件技能講先解釋目的、展示快速上手指令、寫清楚輸入、輸出同安全，以及令例子可以即貼即用。而保安審查技能就檢查機密外洩、有風險嘅殼層指令、路徑穿越、落載來源嘅處理、權限同相依套件嘅改動，跟住匯報任何未解決嘅風險。

亦都有掛鈎檔，配合工作嘅節奏。任務前掛鈎話，開始之前，先讀任務同狀態檔以及相關規則、辨識打算做嘅改動、辨識驗證指令。任務後掛鈎話，做完之後，行驗證、更新狀態、有需要就更新文件，並且總結改咗嘅檔案、行過嘅測試同任何已知問題。提交前掛鈎話，提交之前，行嗰個合併嘅檢查指令、確保無機密被暫存、確保除非有意否則無落載返嚟嘅外部程式碼被暫存，並且確認文件同狀態都準確。

個項目仲包括模型情境協定嘅設定檔。有個簡短嘅說明檔解釋呢個目錄淨係放範例設定檔、你永遠唔應該將機密擺喺度，以及全域設定永遠唔可以自動被改動。仲有一個範例設定，一開始基本上係空嘅。

記憶檔俾將來嘅代理一啲持久嘅情境。項目記憶檔解釋自己嘅用途、總結個項目係由合約而嚟、列出最重要嘅檔案，並且留位俾長期嘅筆記。決策檔係一個地方，俾你隨時間記錄架構同流程上嘅選擇。而詞彙檔就定義啲關鍵詞：project starter 係嗰份樣板規格、外部來源係嗰啲落載返嚟嘅第三方倉庫、來源鎖定就係落載提交嘅紀錄。

一套豐富嘅文件檔將呢一切收尾。架構文件將個項目描述成一個由真理來源合約砌出嚟、可執行嘅樣板，用簡單嘅 Node 腳本、無任何必要嘅運行時相依套件、隔離嘅外部來源同一致嘅代理指示。設定文件列出需求同安裝、初始化嘅步驟。用法文件展示點樣初始化、行體檢、落載同審核來源、行保安、同步代理設定、行測試同建立新項目。來源審核文件一開始顯示為未行過，之後先填返入去。代理文件列出支援嘅代理、通用嘅閱讀清單同安全方面嘅期望。疑難排解文件帶你行一次：當體檢失敗、當來源落載失敗、當保安檢查失敗、以及當同步試行報告有改動嗰陣，分別應該點做。而變更紀錄就記低最初產生出嚟嘅結構。

至於測試方面，有個煙霧測試由頭到尾檢查項目結構：根目錄檔案存在、需要嘅目錄存在、JSON 解析得到、套件腳本齊備、來源清單有效、安全規則存在、腳本存在。佢可以由測試腳本嚟行。仲有個細細嘅 fixtures 資料夾放測試資料。

初始化流程本身係一條清晰嘅管線。佢開始、行體檢、再落載來源、再審核來源、再行保安檢查、再用試行模式做同步、再行測試、再更新狀態檔、再印出總結，然後結束。如果來源落載因為網絡問題而失敗，個流程就淨係繼續行本地檢查，並將落載標記為被阻塞或者失敗。

咁項目幾時先算有效呢？當體檢過、保安過、來源審核過、同步試行過、測試過、初始化以一個準確嘅總結完成、無違反任何安全規則、需要嘅檔案同目錄都存在，以及所有 JSON 都解析成功嗰陣。

份規格跟住描述代理設定同步應該點樣運作。來源檔係通用指示同幾個規則檔。目標係 Claude 檔、Gemini 檔同代理文件，將來可能加埋 Cursor、Copilot 同 OpenCode 嘅目標。同步腳本讀啲來源、檢查目標檔有無需要嘅安全字句、喺試行模式下匯報缺失或者過時嘅段落、喺寫入模式下更新受管段落，並且小心避免刪除嗰啲受管段落以外、由使用者寫嘅內容。嗰啲受管段落會用清晰嘅開始同結束標記包住，等大家都知邊啲係產生出嚟、邊啲唔係。

講到來源落載政策，背後嘅諗法係：外部倉庫作為例子同文件參考，確實可以好有用，但佢哋唔係可信嘅項目程式碼。所以個項目可以複製已啟用嘅倉庫、讀佢哋嘅文件、記錄佢哋嘅提交雜湊值、喺審核入面引用佢哋、比較佢哋嘅設定模式。但佢唔可以執行佢哋嘅腳本、安裝佢哋嘅相依套件、自動將佢哋嘅程式碼複製去運行路徑、將佢哋當成可信嘅相依套件、預設咁提交大型嘅落載倉庫、用喺入面搵到嘅任何機密，或者喺未獲明確要求之下改動佢哋。而每一個落載返嚟嘅來源都一定要住喺外部來源資料夾之內，無嘢可以落載到佢以外。

保安政策同樣咁堅定。任何機密都唔可以提交，而呢個明確包括 API 金鑰、權杖、私鑰、工作階段權杖、資料庫密碼同雲端憑證。環境檔會被忽略，不過可以建立一個淨係得佔位嘅範例檔。腳本必須避免危險嘅殼層行為，例如將遠端腳本導入殼層、遞迴刪除、提升權限刪除，或者喺成個系統開放權限。而腳本必須驗證路徑，除非使用者明確提供一個建立項目嘅目標，否則永遠唔寫去項目根目錄以外，而落載就永遠留喺外部來源之內。

喺重大操作之後，狀態檔必須更新，用啲清晰嘅值，例如過、唔過、被阻塞、已略過或者未知。佢應該包括項目名同目的、上次更新嘅時間戳、初始化狀態、已知問題同下一步。

當個代理完成實作或者產生一個項目之後，佢必須交出一份誠實嘅最終報告。嗰份報告點出項目嘅名同佢嘅路徑、列出產生咗啲乜、並講出每一個驗證步驟嘅結果，究竟係過、唔過、被阻塞，定係根本無行過。佢列出已知問題，或者話無問題，並俾出下一條要行嘅指令。仲有一條嚴格嘅誠實規則：除非一條指令真係成功行過，否則個代理永遠唔可以聲稱佢過咗。

份規格定咗一個接受嘅最低門檻。一個產生出嚟嘅項目，淨係喺包括以下嘢嘅時候先算最低限度可接受：合約、說明檔、套件檔、代理指示檔、任務同狀態檔、啲清單同鎖定檔、所有核心腳本，以及放規則、技能、掛鈎、設定、記憶、文件、測試同外部來源連同佔位檔嘅資料夾。如果呢啲有任何一樣缺咗，個實作就淨係未完成。

亦都有一個建議嘅工作次序：先建立目錄，再整根目錄資料檔、清單、規則、技能、掛鈎、記憶檔、文件、套件檔、輔助工具腳本、驗證腳本、初始化腳本、建立項目腳本同測試。跟住行體檢、行保安、行測試、行初始化、更新狀態，並匯報最終結果。

呢個樣板亦都好清楚講明佢唔係乜。佢唔需要一個網頁應用、一個後端伺服器、一個資料庫、一個前端框架、TypeScript、Docker、Kubernetes、雲端部署、運行時第三方相依套件，或者自動咁全域設定 AI 工具。下游項目大可以自由加返呢啲，但樣板本身唔會要求佢哋。而且有啲明顯嘅延伸點，俾團隊將個項目發展開去，例如加原始碼資料夾、應用資料夾、套件、服務、網頁同 api 層、命令列工具、設定，以及好似型別檢查、程式碼檢查、格式化、測試執行器、容器同持續整合與部署流程嗰類工具。每當加咗呢啲嘢，文件同測試都應該跟住一齊成長。

最後，份檔案以兩段範例提示語作結，一段用嚟建立一個全新項目，好似一個叫做 abc 嘅樣本，另一段用嚟為樣板本身做自我初始化，仲有一句俾任何讀緊佢嘅編程代理嘅堅定收尾指示。唔好用總結呢份檔案嚟代替實作佢。唔好淨係整 markdown。唔好略過啲可執行腳本、測試或者安全檢查。唔好執行落載返嚟嘅第三方程式碼。唔好改動項目以外嘅檔案。如果你行唔到指令，就清清楚楚講出嚟。如果無網絡，就將來源落載標記為被阻塞。永遠都要更新狀態檔。並且永遠以一份誠實、完整、講真實驗證結果嘅報告作結。呢個，基本上就係 project starter 嘅整個承諾：一份你讀一次，就可以變成一個真正可運行、安全、為代理而設嘅項目嘅合約。
