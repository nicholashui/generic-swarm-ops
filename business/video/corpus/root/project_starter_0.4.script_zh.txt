咱们今天来聊一个叫做 project starter 的东西。简单来说，project starter 是一份可以重复使用的蓝图，专门用来启动全新的软件项目，而且它本身就内置了对 AI 编程代理的支持，还包括下载外部原始素材、初始化项目、做验证、写文档、执行安全规则，以及在需要的时候自动重复这一连串的工序。

最重要的一点是要明白，这份文件是要充当唯一的真理来源，也就是一份实现合约。一个编程代理应该能够从头到尾把它读完，仅仅靠这一次阅读，就能搭建出一个完整的、可以运行的起始项目。而这里有个关键：成果不能只是文档那么简单。它一定要包含真正能跑得起来的脚本、清单文件、验证命令、代理配置文件、文档，以及安全控制。换句话说，是一些你真的可以执行的东西，而不仅仅是拿来看的。

那么 project starter 是怎么用的呢？它有两种主要模式。

第一种是自我初始化模式。当你在搭建样板仓库本身的时候，就会用这种模式。思路很简单。你让编程代理在现有仓库里实现这份规范，把规范当成真理来源，创建每一个需要的文件、目录、脚本、清单、文档、测试和代理配置文件，然后再运行初始化命令。做完之后，你应该会得到一个整齐的项目文件夹，里面有合约本身、一个说明文档、一个包文件、给不同代理用的指示文件、任务和状态文件，还有一整套文件夹，分别存放原始素材、脚本、规则、技能、钩子、配置、记忆、文档、测试和外部素材。

第二种是创建新项目模式。当你想根据这个样板开一个全新项目的时候，就用这种。你给它一个项目名，指向那份规范，让它搭建一个能运行、能执行的项目，而不是只是一堆 markdown。你还会让它下载所有已启用的来源，然后再运行初始化。最终结果的结构是一模一样的，只是换成了你新项目的名字而已。

那么一个人究竟应该怎么让编程代理使用这份文件呢？建议的做法，是给它一段清晰的提示语。你起好新项目的名字，指向规范所在的位置，讲清楚新项目要在哪里创建。你描述项目的目的和主要技术组合。你列出哪些编程代理需要支持，而通常那一套包括 Claude Code、Codex、Gemini CLI、Cursor、OpenCode 和 GitHub Copilot。

接着，你就要一步一步讲清楚这个代理要做什么。读完整份规范。创建目标目录。创建规范要求的每一个文件和文件夹。把占位用的资料换成你真正的项目名和目的。把规范复制到新项目的根目录。创建包文件。创建来源清单。创建可执行的脚本。创建规则文件、文档和测试。创建代理配置文件。把所有已启用的上游来源下载到外部来源文件夹里。生成来源锁定记录和来源审计文档。运行初始化验证。用真实的结果更新状态文件。最后，汇报究竟一切是成功还是失败。

伴随这些步骤的，是一套硬性的安全要求。代理不可以执行下载回来的第三方代码。它不可以在那些下载回来的仓库里安装包。它不可以自动把那些下载回来的代码引入进来。它不可以动全局的用户配置。除非有人明确提供，否则它不可以使用任何机密资料。它也不可以在未经许可的情况下覆盖现有文件。万一刚好没有网络或者没有命令行访问，代理也仍然要把所有项目文件和脚本都创建好，只是在状态文件里标明来源下载被阻塞，然后让用户自己进入项目文件夹，亲自运行初始化。

我们讲一下代理可能会收到的输入。它有几个变量。项目名，如果没有提供就会有个合理的默认值。输出路径，也就是生成出来的项目放在哪里。项目目的的简短描述。主要技术组合，默认是 Node 二十版或者更新的版本，用纯 JavaScript，而且初始化脚本不需要任何运行时依赖包。一个用来决定要不要下载已配置上游来源的标志。支持的代理清单。包管理器，默认是 npm。还有许可协议，默认是 MIT。每当有个值缺了，代理就直接退回去用默认值。

那么一个实现这份规范的编程代理，预期应该有哪些行为呢？首先是一般规则。把这份文件当成合约。搭建一个完整可运行的项目。倾向于用简单、可审计的脚本。除非真的需要依赖包，否则靠 Node 标准库来做自动化。避免不必要的第三方包。让生成出来的文件保持易读和易维护。在实际可行的情况下让命令可以重复执行。把状态信息记在状态文件里，把下载来源的细节记在来源锁定文件里，把审计信息记在来源审计文档里。

接着是安全规则，用一张代理绝对不可以做的事来表达。它不可以执行下载回来的第三方代码。它不可以在下载回来的仓库里运行安装或者构建命令，无论是 npm、pnpm、yarn、pip、cargo、go 还是任何类似的东西。它不可以自动从下载回来的仓库把源文件拿进主项目。它不可以改动目标目录以外的文件。它不可以改全局代理配置。它不可以把机密资料存在已提交的文件里。它不可以在未获明确许可的情况下删除用户文件。它不可以在未获许可的情况下覆盖一个已经有内容、非空的输出目录。它不可以运行有破坏性的命令。而它也永远不应该假设网络是可以用的。

有个特别情况值得提一下。如果输出路径已经存在而且不是空的，代理就一定要停下来，在继续之前请求许可。唯一的例外，是用户明确说要覆盖或者更新那个目录，而即使是这样，代理也应该尽量保留重要的、由用户创建的文件。

接着规范就详细列出了需要的项目结构。在根目录，你会找到合约本身、一个说明文档、一个包文件、代理指示文件、任务和状态文件，以及用来忽略文件和配置编辑器的配置文件。里面，来源文件夹存放清单和生成出来的锁定文件。脚本文件夹存放所有自动化的东西，包括初始化、体检、创建项目、下载来源、审计来源、安全检查、同步代理配置和测试，还有一套辅助工具，分别处理文件系统工作、git、日志和项目数据。也有文件夹存放规则、技能、钩子、模型上下文协议配置、记忆、文档和测试。最后，还有个外部来源文件夹，给下载回来的素材住，刻意跟其他东西分开。

包文件值得特别讲一下。它把项目名定义成一个包安全的版本，一段配合你目的的描述，并且把项目标记为模块。它声明需要 Node 二十版或者更新。它定义了一套跟自动化干净对应的脚本：初始化、体检、创建、下载和审计来源、安全、同步、测试，还有一个合并的检查命令，一次性运行体检、安全、来源审计和测试。默认情况下它没有任何运行时和开发依赖包，而它的脚本在实际可行的情况下，应该在 macOS、Linux 和 Windows 上都能运行。

有几个根目录文件有指定的内容。说明文档介绍这个项目，展示快速上手、列出常用命令、指回合约去看完整结构、讲出支持的 AI 代理，还解释下载回来的来源住在哪里，以及那条规则：除非经过明确审查，否则不要执行第三方代码。

通用代理指示文件描述这个项目、指出真理来源，并列出代理规则：在做结构改动之前先读合约、遵守通用和安全规则、永远不执行下载回来的代码、永远不改全局配置、让改动细小而易审、在重大改动之后更新状态文件、在行为改变的时候更新文档、在汇报完成之前运行测试。它还提醒大家，那些外部仓库只是参考素材，不可以自动引入或者执行。

还有针对个别代理的指示文件。一个给 Claude Code，一个给 Gemini CLI。两个都告诉代理要先读哪些文件，也就是合约、通用指示、任务文件和状态文件。两个都坚持改动要细小而明确、不执行下载回来的代码、倾向于用简单的标准库脚本，并且在说完成之前先运行验证。

任务文件用一张简单的清单追踪当前的目标：读合约、核实结构、核实脚本、核实清单、运行初始化、运行测试、更新状态。状态文件记录项目当前的状况，包括它的名字和目的、上次更新的时间，还有一个总结初始化各项检查的表格，加上已知问题和下一步。

还有两个配置文件补齐这个根目录。一个告诉版本控制要忽略哪些文件和文件夹，比如依赖包文件夹、环境文件、构建产物、缓存，尤其重要的是外部来源的内容，但同时保留一个占位文件，让这个文件夹还在。另一个定义一致的编辑器配置：UTF 八编码、换行符用 line feed、文件末尾加一个换行、用两个空格缩进，对 markdown 文件则有一个小小的例外。

接着是来源清单。主清单列出可以纯粹当参考素材下载的上游仓库。每一项都有一个独一无二的标识符、一个表示有没有启用的标志、仓库地址、分支、在外部来源里的目的地、一个写明的用途，以及一个明确标记说它永远不可以被执行。默认情况下可能有一个来源启用了，其他的停用，而停用的来源除非用户亲自开启，否则永远不会下载。也有一个文档清单，指向本地文件，比如合约、通用指示和安全规则。还有来源锁定文件，它一开始基本上是空的，由下载脚本填回去，记录每个来源的标识符、地址、分支、目的地、状态、提交和下载时间。允许的状态有：已下载、已跳过、失败和被阻塞。

现在讲回脚本本身，全部用现代 Node 模块语法写，并且用标准库。初始化脚本按顺序运行整个流程：先体检，再下载来源，再审计来源，再安全检查，再用试运行模式做同步，再运行测试，然后更新状态文件并打印最终总结。如果有一步失败，它会记下那个失败、在安全的情况下继续，而如果有任何必要的步骤失败了，就用错误码结束。如果没有网络，它会把来源下载标记为被阻塞，但仍然继续运行本地检查。

体检脚本验证本地的健康状况。它检查 Node 是不是至少二十版、需要的目录和文件是不是存在、包文件和两份清单是不是有效、需要的脚本是不是齐备、外部来源文件夹是不是存在。然后它就只是打印体检是过还是不过。

创建项目脚本从样板搭建一个新项目。它接受名字、路径、目的、技术组合的选项，还有一个跳过下载的选项和一个强制选项。它解析那些参数、验证名字和路径、除非给了强制选项否则拒绝覆盖非空目录、创建目录结构、复制合约、生成所有文件并且换好占位、视情况运行来源下载，然后打印给用户的下一步。

下载来源脚本读主清单，然后对每一个已启用的来源，验证地址、确认目的地坐落在外部来源之内、拒绝任何未经明确标记为不可执行的东西、如果仓库还不在就克隆回来、如果已经在就取最新的、检出已配置的分支，并且记录提交哈希值。然后它就写出来源锁定文件。它永远不会执行下载回来的仓库的代码，也永远不会在里面运行安装命令。如果有单一来源失败它会继续，但如果有必要的下载失败就用错误结束。它唯一允许用的命令行命令是安全的 git 操作，比如浅克隆、读当前的提交、抓取、检出和快进式拉取。至于在那些来源里安装包或者构建那一类命令，则是严格禁止的。

审计来源脚本检视那些清单和下载回来的素材。它确认清单存在、所有标识符独一无二、已启用的目的地坐落在外部来源之下、每个来源都标记为不可执行、锁定文件存在、下载有被记录下来，以及没有可疑的目的地路径，比如指向上层目录、根目录或者主目录的东西。然后它就创建或者更新来源审计文档，并打印审计是过还是不过。

安全检查脚本运行一些基本的仓库安全检查。它会找有没有提交了任何环境文件、有没有机密占位被填上真实那样的值、有没有来源目的地走出了外部来源文件夹、有没有任何清单来源被标记为可执行、有没有下载回来的来源被当成项目代码引用、有没有全局配置路径被当成目标，以及包文件里有没有危险命令。危险的模式包括像从根目录递归删除、用提升权限去删除、把下载回来的脚本直接导入到 shell，或者把整个系统的权限开到大开那样的东西。它会打印安全是过还是不过。

同步代理配置脚本，让通用指示和个别代理的文件保持一致。它读通用指示和规则文件，然后更新或者核实 Claude 文件、Gemini 文件和代理文档。它支持试运行模式，在那个模式下什么都不写，而是汇报会有什么改动。将来的目标可能包括 Cursor、Copilot 和 OpenCode 的配置。重要的是，它永远不可以改动全局代理配置。

测试脚本运行项目的测试。它运行冒烟测试、核实需要的文件、确认 JSON 文件能解析、检查重要的脚本和根目录文档是不是存在，并打印一个测试总结，说测试是过还是不过。

撑起这一切的，是一套辅助工具。一个处理文件系统的工作，比如检查文件和目录、确保目录存在、读写 JSON 和文本。另一个处理 git：检查能不能用、运行安全命令、克隆、取当前的提交、验证地址。第三个处理日志，分信息、警告、错误和成功几个级别，还有步骤标题。第四个处理项目相关的东西，比如找出项目根目录、验证路径有没有留在项目之内或者外部来源之内、把项目名转成包名、以及加载项目数据。

规范还定义了一系列规则文件。通用规则讲好习惯：编辑前先读任务、改之前先读现有文件、倾向于细小清晰的改动、不要凭空捏造需求、不要移除安全检查、在行为改变的时候更新文档和测试、汇报前先验证、并且汇报清楚改了什么和有什么没做完。安全规则重申那几条核心禁令，围绕执行下载回来的代码、在里面安装依赖包、改全局配置、泄露机密、写到项目以外、覆盖用户的成果、运行破坏性命令，以及未经审查就把参考来源变成运行时依赖包，最后以一句简单的指示作结：不确定的时候，就停下来问清楚。编码规则倾向于简单的代码、标准库、模块语法、清晰的命名、输入验证、有用的错误、可重复执行的脚本和确定性的输出。git 规则说不要改写历史或者强推、不要提交机密、让外部来源保持隔离、除非有要求否则不要创建提交，以及在收尾前展示改了的文件。测试规则坚持行为改变就要更新测试、汇报前先运行、永远不造假结果、让冒烟测试快速，以及测试结构、清单和安全。而来源规则就再次强调外部来源纯粹是参考、只下载到外部文件夹、永远不执行、永远不自动引入，并且要记录提交和做审计。

接着是技能文件，它们读起来就像一份份聚焦的操作手册。规划技能带你走一遍：重述目标、找出要查看和要改的文件、辨识风险、做一张短清单、用细小步骤执行，然后验证。调试技能讲重现问题、细心读错误、隔离出最小的失败单元、检视最近的改动、一次只测一个假设，然后记下那个修正。重构技能强调保留行为、改之前和之后都让测试通过、一次改一个概念、倾向于清晰、并且只在你确定的时候才移除死代码。测试技能讲先运行现有测试、加冒烟测试、测成功又测失败、保持确定性、永远不造假输出。文档技能讲先解释目的、展示快速上手命令、写清楚输入、输出和安全，以及让例子可以即贴即用。而安全审查技能就检查机密外泄、有风险的 shell 命令、路径穿越、下载来源的处理、权限和依赖包的改动，然后汇报任何未解决的风险。

也有钩子文件，配合工作的节奏。任务前钩子说，开始之前，先读任务和状态文件以及相关规则、辨识打算做的改动、辨识验证命令。任务后钩子说，做完之后，运行验证、更新状态、有需要就更新文档，并且总结改了的文件、运行过的测试和任何已知问题。提交前钩子说，提交之前，运行那个合并的检查命令、确保没有机密被暂存、确保除非有意否则没有下载回来的外部代码被暂存，并且确认文档和状态都准确。

这个项目还包括模型上下文协议的配置文件。有个简短的说明文档解释这个目录只放示例配置文件、你永远不应该把机密放在这里，以及全局配置永远不可以自动被改动。还有一个示例配置，一开始基本上是空的。

记忆文件给将来的代理一些持久的上下文。项目记忆文件解释自己的用途、总结这个项目是由合约而来、列出最重要的文件，并且留位给长期的笔记。决策文件是一个地方，给你随时间记录架构和流程上的选择。而词汇文件就定义那些关键词：project starter 是那份样板规范、外部来源是那些下载回来的第三方仓库、来源锁定就是下载提交的记录。

一套丰富的文档文件把这一切收尾。架构文档把这个项目描述成一个由真理来源合约搭建出来、可执行的样板，用简单的 Node 脚本、没有任何必要的运行时依赖包、隔离的外部来源和一致的代理指示。配置文档列出需求和安装、初始化的步骤。用法文档展示怎么初始化、运行体检、下载和审计来源、运行安全、同步代理配置、运行测试和创建新项目。来源审计文档一开始显示为还没运行过，之后才填回去。代理文档列出支持的代理、通用的阅读清单和安全方面的期望。疑难排解文档带你走一遍：当体检失败、当来源下载失败、当安全检查失败、以及当同步试运行报告有改动的时候，分别应该怎么做。而变更记录就记下最初生成出来的结构。

至于测试方面，有个冒烟测试从头到尾检查项目结构：根目录文件存在、需要的目录存在、JSON 能解析、包脚本齐备、来源清单有效、安全规则存在、脚本存在。它可以由测试脚本来运行。还有个小小的 fixtures 文件夹存放测试数据。

初始化流程本身是一条清晰的流水线。它开始、运行体检、再下载来源、再审计来源、再运行安全检查、再用试运行模式做同步、再运行测试、再更新状态文件、再打印总结，然后结束。如果来源下载因为网络问题而失败，这个流程就只是继续运行本地检查，并把下载标记为被阻塞或者失败。

那么项目什么时候才算有效呢？当体检过、安全过、来源审计过、同步试运行过、测试过、初始化以一个准确的总结完成、没有违反任何安全规则、需要的文件和目录都存在，以及所有 JSON 都解析成功的时候。

规范接着描述代理配置同步应该怎么运作。来源文件是通用指示和几个规则文件。目标是 Claude 文件、Gemini 文件和代理文档，将来可能加上 Cursor、Copilot 和 OpenCode 的目标。同步脚本读那些来源、检查目标文件有没有需要的安全字句、在试运行模式下汇报缺失或者过时的段落、在写入模式下更新受管段落，并且小心避免删除那些受管段落以外、由用户写的内容。那些受管段落会用清晰的开始和结束标记包住，让大家都知道哪些是生成出来的、哪些不是。

讲到来源下载政策，背后的想法是：外部仓库作为例子和文档参考，确实可以很有用，但它们不是可信的项目代码。所以这个项目可以克隆已启用的仓库、读它们的文档、记录它们的提交哈希值、在审计里引用它们、比较它们的配置模式。但它不可以执行它们的脚本、安装它们的依赖包、自动把它们的代码复制到运行路径、把它们当成可信的依赖包、默认地提交大型的下载仓库、用在里面找到的任何机密，或者在未获明确要求的情况下改动它们。而每一个下载回来的来源都一定要住在外部来源文件夹之内，没有东西可以下载到它以外。

安全政策同样坚定。任何机密都不可以提交，而这明确包括 API 密钥、令牌、私钥、会话令牌、数据库密码和云端凭证。环境文件会被忽略，不过可以创建一个只有占位的示例文件。脚本必须避免危险的 shell 行为，比如把远程脚本导入到 shell、递归删除、提升权限删除，或者在整个系统开放权限。而脚本必须验证路径，除非用户明确提供一个创建项目的目标，否则永远不写到项目根目录以外，而下载就永远留在外部来源之内。

在重大操作之后，状态文件必须更新，用一些清晰的值，比如过、不过、被阻塞、已跳过或者未知。它应该包括项目名和目的、上次更新的时间戳、初始化状态、已知问题和下一步。

当代理完成实现或者生成一个项目之后，它必须交出一份诚实的最终报告。那份报告指出项目的名字和它的路径、列出生成了什么、并讲出每一个验证步骤的结果，究竟是过、不过、被阻塞，还是根本没运行过。它列出已知问题，或者说没有问题，并给出下一条要运行的命令。还有一条严格的诚实规则：除非一条命令真的成功运行过，否则代理永远不可以声称它过了。

规范定了一个接受的最低门槛。一个生成出来的项目，只有在包括以下东西的时候才算最低限度可接受：合约、说明文档、包文件、代理指示文件、任务和状态文件、那些清单和锁定文件、所有核心脚本，以及存放规则、技能、钩子、配置、记忆、文档、测试和外部来源连同占位文件的文件夹。如果这些有任何一样缺了，这个实现就只是没完成。

也有一个建议的工作顺序：先创建目录，再做根目录数据文件、清单、规则、技能、钩子、记忆文件、文档、包文件、辅助工具脚本、验证脚本、初始化脚本、创建项目脚本和测试。然后运行体检、运行安全、运行测试、运行初始化、更新状态，并汇报最终结果。

这个样板也很清楚地讲明它不是什么。它不需要一个网页应用、一个后端服务器、一个数据库、一个前端框架、TypeScript、Docker、Kubernetes、云端部署、运行时第三方依赖包，或者自动地全局配置 AI 工具。下游项目大可以自由加上这些，但样板本身不会要求它们。而且有一些明显的扩展点，让团队把这个项目发展开去，比如加源代码文件夹、应用文件夹、包、服务、网页和 api 层、命令行工具、配置，以及像类型检查、代码检查、格式化、测试运行器、容器和持续集成与部署流程那一类工具。每当加了这些东西，文档和测试都应该跟着一起成长。

最后，这份文件以两段示例提示语作结，一段用来创建一个全新项目，比如一个叫做 abc 的样本，另一段用来为样板本身做自我初始化，还有一句给任何正在读它的编程代理的坚定收尾指示。不要用总结这份文件来代替实现它。不要只做 markdown。不要跳过那些可执行脚本、测试或者安全检查。不要执行下载回来的第三方代码。不要改动项目以外的文件。如果你运行不了命令，就清清楚楚讲出来。如果没有网络，就把来源下载标记为被阻塞。永远都要更新状态文件。并且永远以一份诚实、完整、讲真实验证结果的报告作结。这，基本上就是 project starter 的整个承诺：一份你读一次，就可以变成一个真正能运行、安全、为代理而设的项目的合约。
