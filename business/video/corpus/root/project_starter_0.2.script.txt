Let's walk through an improved living task specification for a project called Project Starter. Think of it as a blueprint for building a production-ready starter repository, one that installs, synchronizes, audits, and evolves a shared coding harness across a whole family of AI coding tools. We're talking about Claude Code, Cursor, Codex, OpenCode, the Gemini command line tool, Grok Build, and similar assistants. This is version zero point two, the rethink edition, and it's ready for what the team calls Phase Zero execution. It was created in June of twenty twenty-six, and its primary goal is clear: make one repo that brings consistency and safety to a messy landscape of agents whose configuration systems all work differently.

So why a rethink? The original plan was strong, but it was also broad, a little repetitive, and slightly risky. It assumed you could do full installs and achieve exact parity across tools that simply don't store their settings the same way. This improved version makes the whole project more executable, and there are ten big ideas behind it.

First, it makes a foundation harness, referred to as E C C, the bedrock of everything, but instead of blindly copying everything, it installs carefully curated profiles first. Second, it drops the idea of leading with shell scripts and instead uses a cross-platform command line tool built in Node. Third, it defines one single source of truth, with generated adapters and automatic checks for drift. Fourth, it reframes the goal of having the same skills and rules everywhere as semantic parity, meaning the behavior is the same, rather than insisting every file look byte-for-byte identical. Fifth, it adds proper manifests that track licenses, sources, checksums, and versions. Sixth, it cleanly separates instructions, skills, memory, hooks, the model context protocol configs, and the generated tool configurations. Seventh, it introduces an explicit security threat model. Eighth, instead of logging every reasoning trace, it stores auditable summaries: the evidence, the decisions, the diffs, the commands, the test results, and the review outputs. It deliberately does not request or store hidden chains of thought. Ninth, self-improvement stays proposal-only until a human approves it. And tenth, it adds measurable quality gates, sync tests, and time budgets for installation.

Before any of that gets built, there's a source verification snapshot. Phase Zero must re-check every source first. The intended priority order goes like this: the E C C harness is the primary foundation, then a set of concise behavioral rules in the style associated with Andrej Karpathy, then a persistent memory tool such as claude-mem or an equivalent, if it's compatible and safe. After that come selected planning and workflow patterns from well-regarded Claude Code repositories, then curated skill libraries imported selectively rather than in bulk, and finally the official documentation for each agent. For every source, Phase Zero must verify the latest version or release, the license, how it's installed, which config paths it supports, the security implications, and exactly what gets included, adapted, or rejected.

Next come the non-negotiable principles. Specifications drive implementation, always. When components overlap, prefer the E C C choices unless a source audit proves something fits better. The Karpathy behavior layer means thinking before coding, favoring simplicity, making surgical changes, and staying goal-driven. There's a single source of truth: the central folders for skills, rules, hooks, and configs are authoritative. Everything else, the per-tool files, are generated from that truth. The system is safe by default, with no destructive automation, remote connections, or skill changes without explicit human approval. It's local-first, leaning on local scripts, local memory, and local audit logs, with external services kept optional. The core stays minimal while larger bundles remain optional and curated. It aims to work across macOS, Linux, and Windows wherever possible. And it's auditable, not opaque, storing concise rationales and evidence rather than demanding hidden reasoning.

Now let's talk about what actually ships. The deliverables include the new repository itself, one main command line entry point run through Node, and a set of convenient package scripts for initializing, syncing, checking the sync, running a doctor diagnostic, scanning security, running reviews, running tests, and formatting. On top of that there's the E C C-based starter profile, a cross-agent sync engine, curated manifests for skills, rules, hooks, and configs, and adapters for each major tool. There are optional adapters for GitHub Copilot, Zed, and Windsurf too. Then there's a self-review and critic routine, a skill lifecycle proposal and approval workflow, a security baseline, and full documentation with examples, tests, and acceptance checks.

The repository structure is organized around that source-of-truth idea. At the top sit the agent-facing instruction files, the readme, the package definition, and living task and status files. Then there are hidden folders for each tool's generated config, a central skills folder broken down by category like planning, implementation, testing, review, security, memory, and lifecycle. A rules folder holds numbered files: a constitution, the Karpathy behavior rules, the spec-driven development rules, security, testing, token efficiency, and human approval. There are folders for hooks, for model context protocol configs split into minimal and optional, and a memory folder holding project context, handoff notes, and reflections. There are folders for reviews, for suggestions sorted into pending, approved, and rejected with an audit log, and a scripts folder containing the main router and the per-tool adapters. Finally there's a docs folder, an examples folder, and a tests folder with fixtures and test files for sync, manifests, and adapters.

There's also a strict generated file policy. Every generated or adapted file carries a header that announces it was auto-generated by Project Starter, names its source folders, and reminds you to run the sync command rather than editing it directly. The rules are firm: central files win, generated files are overwritten only after conflict checks, and local user files are backed up before any overwrite. A dry-run mode must show the exact writes and deletes it would perform. A check mode must fail if generated files have gone stale. Symlinks are preferred only where they're safe and supported, and on Windows the system falls back to copy mode unless it detects proper symlink support.

Phase Zero is the research, scope-lock, and source-audit phase. The goal is to confirm the latest sources, install commands, licenses, and compatibility before generating a single file. The tasks include verifying the latest E C C release and its installer, confirming the Karpathy rules and a Cursor variant, checking the persistent-memory candidate, reviewing best-practice workflow repositories while selecting only the non-duplicative patterns, and choosing only high-value skill bundles. A source audit document captures, for each source, its name, location, version, license, install command, the components selected, the components rejected, and the rationale. A decisions document records architectural decisions in a structured format. And three starter profiles are defined: a core profile that's minimal, safe, and initializes in under five minutes; a power profile with a broader install plus memory and curated skills; and an experimental profile for Grok Build, extra connectors, and multi-agent demos. To pass, no source may be used without its license and version recorded, no conflicting duplicates may be accepted, a clear priority order must exist, and install commands must be verified before any scripting.

Phase One builds the repo skeleton and the command line foundation. The team initializes the Git repository, adds the package definition with all those scripts, and implements the main command router. A doctor script checks the Node version, whether Git is available, the operating system, symlink capability, which agents are installed, and whether the required directories exist. Base configuration files, a readme, and starter docs get added, along with empty manifests backed by JSON schemas, and the first task and status files. Success here means the doctor command works on a clean machine, placeholder tests pass, the skeleton builds in under a minute, and everything consistently uses the Project Starter name.

Phase Two installs and adapts the E C C foundation as the primary harness layer. An adapter or installer wrapper supports the core, minimal, power, and experimental profiles. It installs selected components: core rules, planning, review, and security skills, the security scanner, token and context optimization, memory and instinct learning where safe, parallel orchestration patterns, and connector conventions. Crucially, it avoids a blind full copy. The E C C source version is recorded, attribution is added to derived files, and a security scan runs afterward. The acceptance bar: the core profile installs reproducibly, the scan runs, no oversized skill dump loads by default, and E C C is the default winner when conflicts arise.

Phase Three creates the behavioral rules and the spec-driven development constitution. These are the concise, high-impact rules every agent receives. The behavioral layer is simple to state: think before coding, prefer simple solutions, make surgical changes, execute against the stated goal, ask only when blocked by meaningful ambiguity, plan before editing many files, run tests or explain why you didn't, update the status file after major work, and never auto-apply changes to skills, rules, or hooks without human approval. To pass, all the agent-facing files must receive semantically identical behavior, the rules must stay concise and non-duplicative, and any conflicts must be documented.

Phase Four builds the curated skill set, lean enough to avoid bloat but covering the high-value workflows: planning, implementation, test-driven development, code review, security review, debugging, refactoring, documentation, memory and handoff, context compaction, cross-agent sync, self-review, and the skill suggestion lifecycle. Each skill entry in the manifest carries a rich set of fields: an identifier, name, version, description, source, source priority, license, checksum, the compatible agents, side effects, whether it requires human approval, tags, paths, an owner, and the date it was last reviewed. The target is roughly fifteen to thirty skills in the core profile, not hundreds, with optional bundles selectable, every skill attributed and versioned, and all of them tested through the sync adapters.

Phase Five is the cross-agent synchronization layer, and this is really the heart of the project. The goal is for every supported agent to receive the same curated skills, rules, hooks, and connector configuration, as far as each tool can support it. The command line lets you sync everything at once, or target a single agent like Claude, Cursor, Codex, Gemini, OpenCode, or Grok Build. You can preview changes with a dry run, verify freshness with a check, choose between automatic, copy, or symlink modes, and even select just a subset of categories such as planning, review, and security. Each adapter writes to the right place for its tool: Claude Code gets its skills, rules, settings, command shims, and root instruction file; Cursor gets its rule files and connector config; Codex and OpenCode get their agent instruction files and configs, with a generated skills index where native skills aren't available; Gemini gets its instruction file and settings; Grok Build and Copilot get their compatible references and instruction files. Sync safety is built in: modified files get backed up, manual edits are detected, a sync report is produced, manifest or schema errors cause a failure, and a continuous integration drift check is supported. The proof is satisfying: change a central skill, run sync, and every supported agent updates, with no manual copying anywhere.

Phase Six handles memory and handoff, providing persistent, privacy-aware memory without bloating the context. There are layers: the status file for current progress, the task file as the living spec, a project memory file for stable context, a handoff file for compact continuation summaries, and a reflections folder for lessons and repeated patterns. Optionally a dedicated memory layer can be plugged in. The rules protect privacy: never store secrets, redact credentials and personal data, summarize instead of dumping logs, load only relevant memory, prefer handoff files for continuation, and keep the auto-injected context small. A new session should be able to resume from the task, status, and handoff files, self-review can generate reflection entries, and memory can be turned off entirely.

Phase Seven adds the self-evaluation and critic routine, a structured loop of plan, implement, review, and refine. There are roles: a solver who implements, a critic who reviews correctness, simplicity, spec adherence, security, performance, and maintainability, an optional security critic, an optional test critic, and a judge who summarizes what's blocking versus what isn't. A rubric scores nine dimensions from one to five, covering correctness, spec adherence, simplicity, security, test coverage, maintainability, performance and token efficiency, auditability, and self-improvement value. The outputs are review files in both readable and structured formats, listing blocking findings, non-blocking suggestions, tests run, files changed, and recommended next steps. There are commands to trigger self-review and critique, both as agent commands and through the command line. The guardrails matter here: no hidden chain-of-thought logging, concise rationale and evidence only, a default cap of two refine loops, and human confirmation for high-impact changes.

Phase Eight is the skill lifecycle, allowing safe self-evolution of skills, rules, and hooks through suggestions only. The workflow analyzes usage, reviews, failures, gaps, and duplication, then generates a suggestion file. A human reviews it and approves, rejects, or modifies it. Only an approved change applies to the central source, after which sync propagates it and an audit log records the decision. Each suggestion includes an identifier, the action to add, update, or remove, the target, a rationale, evidence, the proposed change, a risk assessment, a rollback plan, a confidence level, and a human approval field. The strict rule: no mutation happens without explicit approval, approved changes are logged, rejected suggestions are kept for audit, and a rollback path always exists.

Phase Nine establishes the security baseline, making the harness safe for real-world local use. The threat model protects against prompt injection through documents, issues, web pages, or connector output; secret exfiltration; destructive shell commands; overly permissive hooks; unsafe connector servers; supply-chain risk from skill libraries; generated file drift; hidden remote telemetry; and accidental overwrites of global config. The controls include a security scan, a secret scanner, allowlists for connectors and hook commands, protections for sensitive files like environment files, keys, and tokens, human approval for destructive commands, a continuous integration security check, dependency auditing, no default remote credentials, and a firm ban on piping downloads straight into a shell unless explicitly approved and documented. To pass, the security command runs locally, critical findings fail the build, sensitive files are protected, and the docs explain the model.

Phase Ten covers docs and examples, with the goal of making the starter usable in under ten minutes. The documentation spans an overview and quick start, exact install commands, day-to-day workflows, the architecture of source-of-truth and adapters, sync behavior and troubleshooting, the security model, the decision records, and the source audit. The examples cover a spec-driven feature workflow, cross-agent sync, self-review, skill suggestion, and a security scan. A new user should be able to initialize and sync in under ten minutes, the docs should explain how the tools differ, and the examples should actually run.

Phase Eleven is validation and continuous integration, there to prevent regressions and prove the starter works. The tests cover manifest schema validation, adapter snapshots, sync dry runs, drift detection, a security scan smoke test, Windows path handling, a check for duplicate skill identifiers, and a check that no generated file is missing its header. The continuous integration pipeline lints the JavaScript, validates the JSON, checks Markdown links where possible, runs the tests, runs the sync check, runs the security scan, and uploads the review and security artifacts. Success means the pipeline is green, a fresh clone passes install and test, and the sync check passes after generation.

Then there's the definition of done. The project is complete when initialization creates a working starter, sync updates all supported agent configs, and Claude Code, Cursor, and at least one of Codex, Gemini, or OpenCode are verified. The E C C foundation must be installed and documented, the Karpathy rules active, the self-review workflow working, and skill suggestions gated behind human approval. The security scan must work and block critical issues, the docs must be enough for a new user, every generated file must be reproducible, the source audit and decisions must be complete, and the Project Starter name must be used consistently everywhere.

To close, here are the immediate next actions: create the repo skeleton, run the Phase Zero source audit, implement the main script, the doctor, and an empty sync framework, add the central rules and manifests, build the Claude and Cursor adapters first, then add Codex, Gemini, and OpenCode, install and adapt the E C C core profile, run a security scan, write the docs and examples, and commit it all as the initialization of Project Starter. And the overall implementation priority follows the same arc: foundation first, then sync, then E C C, then the rules, the curated skills, security, memory, the critic, the lifecycle, and finally the polish of examples, continuous integration, and documentation. That's the full picture, a lean, safe, auditable starter that keeps every coding agent in sync from a single source of truth.
