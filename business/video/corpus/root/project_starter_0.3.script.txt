Let's talk about a project called project starter. At its heart, this is a blueprint for building an executable starter repository, one that downloads, audits, curates, and keeps in sync the configuration sources for a whole family of AI coding agents. Think Claude Code, Cursor, Codex, Gemini's command line tool, OpenCode, Grok Build, GitHub Copilot, and the other agent harnesses that live alongside them. The version we're looking at is the actionable bootstrap release, last updated in the summer of twenty twenty six.

The first thing to understand is what kind of document this really is. It's a boilerplate specification. A person can hand this file to a coding agent and say, in effect, create a brand new project from this. The typical request goes something like this. Create a new project with a chosen name, based on this starter file, and treat the starter file as the single source of truth, the implementation contract. Place the new project in a chosen output directory. Define its purpose. Define its primary technology stack. And then carry out a long list of concrete steps.

Those steps are worth walking through, because they set the tone for everything else. The agent must read the starter file completely. It must create the new project directory, and then build out every folder and file the specification calls for. It replaces the placeholder details with the real project name and purpose. It copies the starter file into the new project. It creates the package definition, the source manifests, the executable scripts, the rule files, and the documentation. It downloads all of the enabled upstream repositories into an external sources folder. It generates a source lock file and a source audit document. It runs a bootstrap validation. It updates a status file. And finally, it reports whether everything passed or failed.

And here's the crucial instruction. Do not merely summarize. Do not only write documents. Create a working, executable project. If, for some reason, network or shell access isn't available, the agent should still create every project file, and simply mark the source download as blocked in the status file.

This point is hammered home in what the spec calls its critical instruction to coding agents. This file is not only documentation. When an agent receives it and is asked to implement it, the agent must create a real, executable project. It must not stop after writing some text. It has to create the repository structure, the package definition, the manifests that list sources and documentation, and a whole set of scripts. There's a main project starter script, a source downloader, a source auditor, a doctor script that checks the environment, a sync script, and a security scanner. The agent must download every enabled repository listed in the manifest, save them under the external sources folder, record the download details in a lock file, generate the audit document, run the validation commands, and report either success or the exact blockers. If the network or the shell simply isn't available, the agent is told to say so plainly, that it's blocked because network or shell execution is unavailable, and to still create all the scripts and manifests so a human can run the bootstrap later.

That leads us to the non negotiable outcome. There's one command, bootstrap, and it must perform a specific flow. First it runs the doctor. Then it creates the required directories. It validates the source manifest. It clones or updates every enabled source from the web. It writes the source lock file. It generates the audit. It runs security smoke checks. It runs a sync dry run. And it runs the tests. The project is simply not considered complete unless the source repositories have actually been downloaded into that external sources folder. No pretending.

Next, the spec lays out the required repository structure, and it's extensive. At the top level you have the starter file itself, a readme, and agent instruction files for the various tools. There's a package definition, a task file, and a status file, along with the usual editor and ignore settings. A sources folder holds the manifest, a docs manifest, the lock file, and its own readme. An external folder holds the downloaded sources, kept out of version control. A scripts folder contains all the executable pieces, including a set of adapters, one each for Claude, Cursor, Codex, Gemini, OpenCode, Grok Build, and Copilot, plus a library of shared helpers for git operations, safe file handling, manifest parsing, and reporting.

Beyond that, there's a rules folder with a numbered set of rule documents, covering a constitution, Karpathy style behavior, spec driven development, security, testing, token efficiency, and human approval. There's a skills folder organized by area, like planning, implementation, testing, review, security, memory, and lifecycle. There are folders for hooks, for model context protocol configurations, for persistent memory, for reviews, and for suggestions, where pending, approved, and rejected ideas each get their own place alongside an audit log. There's a documentation folder covering installation, usage, architecture, the source audit, security, sync, and troubleshooting. There's an examples folder showing several end to end workflows. And finally a tests folder with fixtures and test files for the downloader, the auditor, the sync, the manifest, and the adapters.

Then comes the package definition. It names the project, marks it as private, and sets it up as a modern module based Node project. It defines a whole set of commands, the scripts a developer actually runs. Bootstrap and init to set things up. Doctor to check health. A family of source commands to download, update, check, and audit. Sync and a sync check. Security. Review. Test. And format. It requires Node version twenty or newer. And notably, for this first implementation, no external dependencies are needed at all. Everything runs on Node's built in capabilities.

Now we reach the most important data file, the GitHub source download manifest. The agent must create this manifest exactly, and it must download every source marked as enabled. It must never silently skip an enabled source, and if a required source fails, the command has to exit with an error. Each entry in this manifest describes a repository with a unique identifier, a friendly name, its web address, where it should be saved locally, its type, whether it's enabled, its priority, its tier, whether it should be quarantined, an import policy, and a short statement of purpose.

Let me describe the sources in plain terms rather than reading out web addresses. The primary one is called Everything Claude Code, the core cross agent source for skills, agents, commands, hooks, rules, and security scanner references, and it's the only one set up for curated import rather than reference only. Then there's a cluster of official Anthropic repositories, the Claude Code project itself, its GitHub action for automation and pull request review, the official agent skills examples, and the official plugin marketplace. There are official tools from other vendors, OpenAI's Codex command line tool, Google's Gemini command line tool, and OpenCode. There's a group around the model context protocol, including its server examples, its registry, and the official GitHub server for issues, pull requests, repositories, and code search. There's the agents standard, a reference for cross agent repository instructions.

After that come the behavior and community sources. There are two versions of Karpathy style behavioral rules, one general and one adapted for Cursor and VS Code, emphasizing thinking before coding, simplicity, and surgical changes. There's a persistent memory architecture reference, which the spec is careful to say must never install automatically. There's a composable skill methodology project, a community best practices collection, several curated discovery lists for the Claude Code ecosystem, a set of community subagent definitions, frontend oriented skills from Vercel Labs, a collection of Cursor rule examples, and a security focused set of Cursor rules. A few additional discovery indexes are marked optional rather than required. And one source, an archived collection of older protocol servers, is deliberately disabled, set never to import, kept only as historical reference.

There's also a second, lighter manifest for documentation sources. Some related tools have official docs online but no repository that needs cloning. So this docs manifest points at the Claude Code feature documentation, the Cursor rules documentation, the Grok Build overview, the GitHub Copilot custom instructions, and the model context protocol website. The first version of the project can simply create this file without actually downloading the docs, while a later version would implement a docs download step.

The external directory has its own strict policy. The downloaded upstream repositories are intentionally kept out of version control. They must not be committed into this project. Only curated, audited, and properly attributed files may be copied into the project's own first party folders, like rules, skills, hooks, configurations, and docs.

The download behavior itself is carefully defined. The downloader script supports several modes. You can download everything, or filter by profile, such as core, official, or discovery. You can do a dry run, or run in strict mode. By default, downloading reads the manifest, selects every enabled source, creates the external sources folder, and then for each source does one of three things. If the target doesn't exist yet, it does a shallow clone, just the latest snapshot. If the target already exists and is a proper git repository, it fetches and fast forwards to the newest version. And if the target exists but isn't a git repository, it fails. Along the way it records metadata, things like the resolved remote address, the current commit, the branch, and the date and subject of the latest commit. Then it produces a human readable summary.

The failure rules depend on priority. For required sources, any failure is fatal, a failed clone, a failed update, a missing git folder, or an invalid address. For optional sources, a clone failure is tolerated unless strict mode is on, but the failure must still be recorded. And anything disabled simply must not download by default.

The security rules around downloading are strict and clear. The downloader must never run install scripts from downloaded repositories. It must never run a package install inside them. It must never pipe a remote script straight into a shell. It must not execute any hooks from those repositories. It must not copy their content into the active agent configuration automatically. And it must never write anything outside the project root. The downloader is allowed to do exactly two things, clone or update, and inspect metadata. Nothing more.

The source lock file captures the result. For each source it records the identifier, name, the requested and the resolved address, the target location, the status, the commit, the branch, the time and subject of the last commit, any license files found, any package files found, the quarantine flag, and the import policy. It also keeps a list of failures.

The audit script reads both the manifest and the lock file and produces a human readable audit document, with one section per source. Each section lays out the name, address, target, status, commit, branch, the license and package files, the priority, tier, quarantine flag, import policy, purpose, which components were selected, which were rejected, and any security notes. The audit must flag a source as unsafe for automatic import under several conditions, if it has no license file, if it contains install scripts that run remote commands, if it has suspicious post install scripts, if it's archived, if it requires credentials, if it modifies global agent configuration, or if it ships protocol servers that demand broad filesystem or network access. On the very first pass, the audit should mark almost everything as not yet selected, with bulk import rejected until a human reviews it. The one exception is that small, rule only repositories, like the Karpathy skills, may be selected for careful extraction once their license is verified.

The bootstrap router ties the commands together. It supports bootstrap, init, and format. And bootstrap, in order, runs the doctor, downloads the sources, audits them, runs security, runs sync as a dry run, and runs the tests. If any required step fails, the whole bootstrap fails.

The doctor script checks the environment before anything else happens. It confirms the Node version is twenty or higher, that git is available, where the working directory is, whether there are write permissions, that the source manifest exists, that the external folder exists or can be created, that the scripts folder is present, what operating system platform is in use, and whether symbolic links are supported if sync wants to use them. Then it prints a clean status report, each check marked OK, and an overall result, or it fails with clear messages.

The sync layer is the part that generates configuration for each agent. Even if the adapters start out minimal, there must be a working sync framework. Running sync as a dry run previews the files it would create, the cross agent instruction files, the Claude, Gemini, and Copilot instructions, the Cursor rules, and the settings folders for Claude, Gemini, Codex, and OpenCode. Every generated file must carry a header that marks it as auto generated by project starter, tells the reader not to edit it directly, points back to the source folders it came from, and reminds them to run sync to regenerate it.

The security script is a static scanner. In its first version it scans the downloaded repositories for suspicious top level scripts. It detects environment files, private keys, tokens, or certificates that may have been committed into the project by accident. It reports any protocol configurations that grant broad filesystem access. It warns about remote scripts piped into a shell and about post install scripts in the downloaded sources. And it never executes any downloaded script. A simple static scanner is enough to start.

Then there are the initial rule files, and a few are spelled out. The constitution says to follow the user's goal exactly, prefer simple and maintainable solutions, make surgical changes, avoid destructive actions without approval, never install, execute, or import downloaded third party code until it's been audited, run the relevant tests or explain why they weren't run, and update the status after major work. The Karpathy rules say to think before coding, not to assume hidden requirements, to prefer the simplest thing that works, to avoid speculative abstractions, to change only what's necessary, to keep the goal visible, and to verify the result. And the human approval rules list the actions that always require a person's sign off first, installing global packages, running remote install scripts, enabling protocol servers that use credentials, copying third party code into active configs, modifying hooks, deleting files, and applying any self generated changes to skills or rules.

The import policy deserves emphasis, because it's the philosophical core of the whole design. Downloading sources is mandatory. Importing them is not automatic. The project keeps four distinct stages cleanly separated. There's the downloaded source, which lives in the external folder and is treated as untrusted, ignored by version control, and never executed. There's the audited source, whose metadata has been reviewed and recorded in the audit and lock files, but which is not yet imported. There's the curated import, where a specific file is brought into the project's own folders, only after its license is verified, its source commit is recorded, its checksum is recorded, a human approves anything high impact, and there's no hidden install behavior. And finally there's the generated adapter output, the per agent configuration files, which must always be reproducible just by running sync.

For testing, the agent must create Node test files at minimum for the manifest, the downloader, the auditor, and the sync. The tests verify that the manifest parses, that every enabled source has all its required fields, that there are no duplicate identifiers, that every target sits under the external sources folder, that disabled and archived sources aren't selected by default, that the lock file has a valid shape after a run, and that generated files include that auto generated header.

The readme gives a quick start. Run bootstrap. That downloads the approved upstream repositories into the external sources folder, then writes the lock file and the audit. And it carries an important reminder, downloaded repositories are untrusted until audited, the project does not execute downloaded code, and it does not import third party skills automatically.

The task file lays out the first milestone as a checklist, create the package definition, create the source manifests, create the downloader, download the enabled sources, generate the lock, generate the audit, run the doctor, and run the tests. The status file starts in the bootstrap implementation phase, with the latest update marked as not started and no blockers yet.

The acceptance criteria are concrete. The implementation is only accepted when the doctor, the download, the audit, the security scan, the sync dry run, and the tests all pass. After downloading, a specific list of source directories must exist, unless an optional source failed and that failure was recorded. The lock file must exist. The audit must exist. And the downloaded repositories must not be committed to version control.

The spec then gives the exact order of implementation, twenty two steps in all, beginning with reading the file fully and creating the folder structure, then the package definition and the ignore files, then the manifests, then the library helpers for safe file handling, git, and manifests, then the downloader, the auditor, the doctor, the security scanner, the sync, and the main router, then the rule files and documentation, then the tests, then running the full validation suite, and finally updating the status and reporting a summary.

There's also a clear list of things the agent must never do. It must not merely rewrite this document, pretend downloads happened, skip the download silently, execute downloaded scripts, install global packages, run remote installers, copy all downloaded skills into the active config, overwrite user files without a backup, mutate the global configuration of any of the agents, put secrets into memory, log hidden reasoning, or auto approve its own self improvement suggestions.

A couple of edge cases round things out. If a repository has moved or redirects, the downloader should let git clone it, record the resolved address, note a warning in the lock file, and continue if the clone succeeded. If a repository no longer exists, a required one fails the bootstrap and the failure is written down and reported precisely, while an optional one lets the process continue, with the failure recorded and mentioned in the audit.

Finally, the definition of done. The project is finished when the repository is named project starter, when the bootstrap command exists, when the manifest contains the exact set of repositories, when downloading actually pulls down the enabled ones, when the external sources folder holds them, when the lock file records their commits, when the audit is generated, when the downloaded repositories are ignored by version control, when no downloaded code is executed, when the sync dry run works, when the tests pass, and when the status file is updated with the real results. And when all of that is true, the agent reports a clean human facing summary, how many sources succeeded, failed, or were skipped, what was generated, the pass or fail result of each validation step, and a final reminder that the downloaded repositories live in the external folder and are not imported or executed until they've been audited.

And that's the whole design, a starter that treats the wider world of agent configuration as something to gather carefully, inspect honestly, and adopt only on purpose.
