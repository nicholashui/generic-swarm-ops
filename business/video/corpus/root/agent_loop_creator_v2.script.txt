Let's walk through a detailed, build-ready blueprint for a cognitive-enhanced agent loop. Think of this as the executable companion to the version three design. Where the earlier document described the philosophy, this one is the implementation guide, the thing you hand to a coding agent so it can actually start writing the system. It takes the original first version and extends it by treating a set of ranked human thinking models as first-class, configurable mechanisms. We're talking about frameworks like Cynefin, premortem analysis, the after-action review, double-loop learning, recognition-primed decision making paired with dual process thinking, metacognition, the five whys with the Ishikawa fishbone, red teaming, and the Paul-Elder standards. All of these become real, switchable parts of the machine, not just ideas in a slideshow.

This spec is meant to be parsed by a coding agent, whether that's a build-focused assistant, a code editor with a powerful model behind it, or a custom harness. It synthesizes deep research from several sources: a taxonomy of why multi-agent systems fail, an orchestration protocol that treats tools, environments, and agents as first-class entities, enhancements to the core reasoning loop, production patterns from frontier model providers, and a systematic analysis of cognitive frameworks.

So what's genuinely new in this version? A few things. First, explicit Cynefin context classification, which lets the system route between a fast loop and a full loop depending on the situation. Second, a mandatory premortem step right at the start. Third, recognition-primed decision making, dual process thinking, and metacognition driving a fast recognition path early in the loop. And fourth, structured reflection at the end, combining the after-action review, double-loop learning, the five whys and fishbone diagrams, and multiple critic perspectives. Importantly, everything from the first version is preserved and extended, not thrown away. The skeletons, the failure-mode mappings, the phased roadmap, all of it carries forward.

Let me share the core insight that drives the whole design. When researchers studied why multi-agent systems fail, they found that roughly forty-two percent of failures come down to specification and design issues. Another twenty-one percent or so come from weak verification and bad termination. The lesson is blunt: just adding more agents, without strong upfront spec validation, structured observations, an explicit and verified finish condition, and real progress tracking, often gives you diminishing or even negative returns. On the other hand, hierarchical organization combined with versioning and self-evolution delivers outsized gains on long, complex tasks, pushing benchmark scores up near ninety percent. And the new layer here, the human cognitive frameworks, provides the highest-leverage missing piece: adaptive intelligence, proactive risk handling, fast and slow deliberation, and multi-level learning.

On the model side, the design aligns with frontier multi-agent systems. It uses a server-side multi-agent capability, with a leader synthesizing the work of several sub-agents, for deep research sub-tasks. And in the client loop, it emulates the patterns of build-oriented tools: generate an explicit plan first, run parallel isolated sub-agents, and track state in a todo-style document. The hybrid approach maximizes power, control, and cost efficiency all at once, and the Cynefin routing helps decide when to delegate outward versus stay on a local fast path.

Several architectural tradeoffs got resolved along the way. The core is a custom build in Python, using strict data schemas, strict JSON output, and a full tracer, chosen over a heavier framework for the sake of transparency, auditability, and educational clarity. Memory is hybrid: a structured todo document, a long-term vector store, aggressive summarization, and a dedicated pattern store that feeds recognition-primed decisions. Self-evolution is deliberately bounded, validated against held-out traces, and structured around the after-action review and double-loop learning so the system doesn't drift. A cognitive profile lives in each task spec, with flags for enabling the fast path, choosing a reflection style, and selecting critic modes, so simple tasks don't get buried under unnecessary complexity. The principle is minimal dependencies first, optional adapters later. Local-first, observable, sandboxed, and production-hardened with circuit breakers, retries, and budgets. And it should help build and improve itself, including its own cognitive components.

Every known failure mode is explicitly mapped to a mitigation in a specific phase or component, and each one also gets a cognitive countermeasure. Premortem handles specification and design risks. Metacognition plus cycle detection handles repetition. The after-action review and double-loop learning close the gaps in verification and termination. And recognition-primed decisions handle context and history issues through pattern matching.

Now let's talk about goals and how success is measured. The primary objective is a controlled, hierarchical, reasoning-and-acting agent loop that is reliable against known failure modes and enhanced with human cognitive best practices. It should be evolvable through versioning, tracing, and self-reflection. It should be hybrid, combining full client-side control with optional delegation to a server-side multi-agent system, and it should choose its cognitive mode intelligently. It should be production-grade: observable, cost-aware, secure, testable, and extensible. And it should match the engineering style of its intended user, which means spec-driven with a living task specification, iterative critic loops, careful harness engineering, local and minimal infrastructure, Python first with strict data contracts, and clean integration points for existing tools.

The success criteria are concrete. On reliability, in synthetic failure-injection tests covering all fourteen known failure modes, the residual failure rate after mitigation should stay under five percent, with early detection of spec and role violations, cycle detection that triggers a replan or termination, and a verifier that rejects incomplete or incorrect finish attempts. On performance, base success on held-out research and coding tasks should be at least seventy percent, climbing to at least eighty-five percent after two or three rounds of self-evolution, with fewer steps and tokens than a plain baseline. On observability, every execution should produce a complete, replayable trace, with provenance, versions, timings, token counts, and full thought-action-observation tuples, and you should be able to visualize it and replay from any step. On evolvability, a version manager should support registering, rolling back, and selecting the best variants of prompts, tool code, and agent configs, while a self-evolver proposes and validates improvements with demonstrable gains after three bounded rounds. On the hybrid integration, research sub-tasks should delegate seamlessly to the server-side multi-agent system and come back with synthesized results and citations woven into the main trajectory. On production hardening, you want proper circuit breakers with closed, open, and half-open states, exponential backoff retries, per-phase budgets, structured error observations, sandboxed tool execution, input sanitization, and least privilege. On usability, you want a clean Python package with a command-line interface, an optional server mode, thorough examples, full type hints and docstrings, a passing test suite, and good documentation. And on integration, it should work with common client libraries and optional adapters, and export structured plans for build tools to consume.

There's a second tier of success criteria specific to the cognitive layer. On adaptive routing, the system should classify task context with at least eighty-five percent agreement against a human baseline, trigger the fast recognition path on at least seventy percent of repeated or routine sub-tasks, and cut token usage by twenty to thirty percent versus an always-full baseline, without losing quality on simple cases. On proactive risk, the premortem step should identify at least three to five plausible failure modes per complex task and fold mitigations into the spec and quality gates, measurably reducing downstream issues. On structured reflection, the after-action reviews should produce complete artifacts answering all four questions with evidence, double-loop learning should question governing assumptions and propose meta-level changes that pass validation in at least sixty percent of bounded rounds, and the five whys with fishbone categorization should be used on at least eighty percent of diagnosed failures. And on multi-perspective critique, the verifier's red-team and Paul-Elder modes should catch at least fifteen percent more issues, especially around assumptions, bias, and edge cases, than the standard mode alone.

There are constraints and explicit non-goals too. The stack is Python three eleven or newer, with strict data validation, asynchronous calls, and modern HTTP clients. Optional pieces include a vector store, a server framework, and a state-machine adapter, but the core loop avoids heavy framework lock-in. Minimalism comes first: the core loop, state, reliability, and basic hierarchy should fit in under two thousand lines initially, with evolution and the hybrid integration added later. No uncontrolled loops are allowed, ever, so there are hard step limits, cycle detection by state hash, progress-based exits, and circuit breakers, and every model call uses a strict output schema. Security means sandboxing code execution, never trusting model-generated arguments blindly, and watching for anomalous patterns. Cost control means token budgets, parallelism only for independent branches, summarization under pressure, and early termination once criteria are met. And the non-goals for the early phases include full distributed execution, a graphical dashboard, native multimodal handling, and production multi-tenancy. The whole spec is itself a living document that the built system can eventually update.

Let me ground this in the research synthesis. The failure taxonomy is the primary map. It came from analyzing over sixteen hundred traces from seven frameworks, with strong agreement between human annotators, and it identifies fourteen failure modes across three categories. The first category, system design issues, is the largest, at nearly forty-two percent. It includes disobeying the task specification, disobeying role specification, step repetition, loss of conversation history, and being unaware of termination conditions. The second category, inter-agent misalignment, accounts for about thirty-seven percent. It covers conversation resets, failing to ask for clarification, task derailment, withholding information, ignoring another agent's input, and reasoning that doesn't match the action taken. The third category, verification and termination, is about twenty-one percent, and it captures premature termination, incomplete verification, and incorrect verification.

The key findings translate directly into mitigations. Spec quality is the number one return on investment, so phase zero is mandatory: a structured task spec with explicit objective, success criteria, constraints, output format, and quality gates, plus an automated validator and critic before the loop even begins, plus role contracts during delegation. Verification is a weak spot even in runs that look successful, so there's a dedicated verifier that outputs strict structured judgments at multiple levels, an explicit finish action that must pass verification with evidence, and progress tracking inside the thought step. Context loss and repetition are common, so there's aggressive summarization, structured state that keeps key facts rather than dumping full history, cycle detection via hashing of recent actions and observations, and a hard step cap with progress-based early exit. Inter-agent problems are handled by a strong central orchestrator with explicit decomposition, routing, and contracts, structured handoff observations, circuit breakers per tool and role, and versioned shared state. The coding agent is told to reference these failure modes explicitly in its prompts and verifier, and to build a failure-injection test suite that simulates each one and asserts the mitigation works.

The second research pillar is an orchestration protocol that treats tools, environments, and agents as first-class abstractions. A system built on it reached around eighty-nine percent on a hard benchmark, with self-evolution pushing it higher. Tools are versioned and lifecycle-managed, registered with a name, description, schema, implementation, and a semantic embedding for retrieval, and they can even be generated dynamically. The environment defines the observation and action spaces and keeps state coherent, which here means a working directory, a sandboxed file system, and code runtime state. Agents carry roles, competencies, and metadata, support hierarchy and coordination contracts, and can transform between being an agent and being a tool. The mechanisms to implement include a version manager that tracks lineage and metrics and supports registration, rollback, and selecting the best variant; a tracer that records the full trajectory for replay and optimization; a self-evolution module inspired by gradient-style textual optimization that collects a trace, diagnoses the root cause, applies an edit, validates it on held-out examples, and registers the new version if it's better; careful context management that slices and provenances context for sub-agents instead of dumping everything; and hierarchical orchestration where a central planner decomposes the objective, routes to specialists, lets them run their own controlled loops, and consolidates the results.

The third pillar is the foundational reasoning-and-acting paradigm and its enhancements. The core interleaves a thought, which is a reasoning trace, with an action, like a tool call or a delegation or a finish, followed by a grounded observation. On top of that base, the design folds in reflection-style verbal self-critique, optional generation and ranking of multiple candidate trajectories, a technique that strengthens grounding inside the reasoning step itself by re-anchoring against the latest observation and the original objective, and explicit plan-and-execute patterns with optional tree-search elements. From frontier providers it borrows the server-side reasoning loop, where the model decides on tools, executes them internally, and iterates until it has a final answer, and the parallel-specialists-with-a-leader pattern. The hybrid here keeps a client orchestrator in charge of global state, tracing, and verification, while delegating deep research sub-problems to the server-side multi-agent system, and running coding sub-tasks either through local specialists or through isolated parallel processes with copied state slices.

After all that rethinking, the final architectural decisions are clear. The loop style is a controlled custom reasoning-and-acting loop with hash-based cycle detection and circuit breakers as the foundation, with hierarchy layered on top, where an orchestrator decides whether to delegate, call a tool, synthesize, or finish. Central control beats coordination chaos. The state object holds the task spec, the history, the todo list, the plan, short and long-term memory, the version registry, the tracer, the budgets, and the set of seen hashes for cycle detection. Memory leans on the structured todo and key facts as the low-token primary, with summarization under pressure and an optional vector store for semantic retrieval, and sub-agents only ever get sliced context with provenance. Model calls go through a unified client supporting multiple providers, always with a system prompt, appropriate few-shot examples, and a strict output schema. Tools live in a registry with validation and a safe execution wrapper. The hybrid integration has a dedicated wrapper that sends a narrow sub-objective with success criteria and enabled tools, and logs the result as a special observation. Self-evolution expands in scope across phases, from prompts to tool code to agent configs. And the system dogfoods itself, using the harness to improve its own prompts and verifier on held-out traces during development.

Now to the system architecture and the high-level flow, which runs through six phases, hardened with the cognitive layer. Phase zero is initialization, spec-driven with cognitive setup. It parses the instruction into a validated task spec, including the cognitive profile, then runs a spec validator and critic to check for completeness, ambiguity, role clarity, and termination conditions. Then comes the mandatory premortem: assume the spec and plan fail spectacularly, identify the top causes and mitigations, and merge them back into the living spec, the success criteria, the todo, and the quality gates. Then Cynefin classification tags the task as simple, complicated, complex, or chaotic, and uses that to auto-configure the loop, favoring the fast path for simple and complicated work and the full deliberative path with heavy reflection for complex and chaotic work. The agent state is created with the pattern store for recognition-primed decisions and a current mode of fast or full. Optionally, a planner generates a high-level plan and a todo document, validated against the spec and the premortem risks. Finally, it decides the architecture, flat or hierarchical or hybrid, and sets the initial mode.

Phase one is the core controlled loop with cognitive mode selection. At the start of each iteration, or after a major observation, the system decides its operating mode. If Cynefin allows it, and there's a high-similarity match in the pattern store, and metacognitive confidence is high, it takes the fast recognition path: a lightweight thought that references the matched trace, minimal tokens, straight to action. Otherwise it goes into full deliberative mode, with a detailed thought that re-grounds against the latest observation, and full gates, while a lightweight parallel metacognition monitor scans for bias, checks progress against criteria, watches for context drift, and takes a confidence pulse, ready to force a mode switch or an early replan. It builds context from summarized history, key facts, the todo, the spec, the latest observation, and relevant pattern-store entries. The model produces a strict decision: a thought with a metacognitive overlay first, then an action type, then a payload. There's a cycle check that hashes the recent action and observation and forces a replan or termination if it's been seen before. Execution goes through safe tool calls or safe delegation. Every result becomes a structured observation that now also records which mode was used. Each event is appended to history, the todo and progress are updated, and the pattern store is updated on high-quality outcomes. Light reflection happens every so often or on error, with a note on how effective the mode was. Circuit breakers run per tool and role, and termination can come from meeting the criteria with a verifier pass, hitting the step or budget limits, an explicit verified finish, an irrecoverable situation that escalates, or a strong fast-path success on sub-criteria.

Phase two is hierarchical delegation. The orchestrator decomposes the work and selects or instantiates a specialist from the registry, things like a researcher, coder, verifier, reporter, tool generator, browser agent, or analyzer. It creates a narrow sub-task spec with a subset objective, success criteria, a context slice, and provenance, then invokes the sub-agent, either as its own controlled loop or as a server-side multi-agent delegation. The sub-agent returns a structured observation that bubbles up with a summary of its trace for auditing, and the parent records it, validates and integrates it, updates the global plan, and decides what's next.

Phase three is consolidation and quality gates. An aggregator collects the observations and the plan progress. A harmonizer and reporter merge, deduplicate, cross-reference, and resolve conflicts by citing sources and versions, producing a unified draft. The verifier scores it against the success criteria, checks for hallucinations and gaps and verification failures, and suggests fixes, all as structured output. If it fails the gate, refinement is triggered, re-planning a branch, re-delegating, or self-editing. If it passes, it moves on to polish or final output.

Phase four is reflection and self-evolution, the advanced part, combining the after-action review, double-loop learning, the five whys, and multi-perspective critics. At milestones or at the end, the full trace and mode history go to the self-evolver. The mandatory after-action review answers four questions: what was supposed to happen, what actually happened including the fast-versus-full mode stats and pattern matches, why it happened with a deep five-whys pass and fishbone categorization across dimensions like prompts and methods, models and tools, data and observations, context and environment, state and memory, verification, and the human spec, and finally what to do next as concrete edits. Then the double-loop layer questions the governing assumptions: was the definition of quality too loose, did role boundaries allow drift, should the success criteria or the cognitive defaults themselves evolve? Only meta-approved changes proceed. Root causes are diagnosed with awareness of the failure modes and the cognitive frameworks, targeted edits are proposed to prompts, tool code, role definitions, and even cognitive defaults or new critic modes, using Paul-Elder standards or a red-team attack where configured. Improvements are validated on held-out examples or by replay, and better versions are registered with lineage and an after-action justification, while the pattern store gets updated for future recognition. It's all bounded to three rounds, with a lighter review for fast-path traces.

Phase five is termination and output: a final synthesis in the format the spec requires, persisting the full trace, versions, and metrics, an optional post-hoc reflection summary, and human-in-the-loop hooks at high-stakes gates or when the budget is exhausted.

The core data models are meant to be generated first, because they are the contract. There's a task spec carrying the objective, the list of success criteria, constraints, output format, quality gates, an optional initial plan, metadata, and the cognitive profile with its flags for the fast path, reflection style, critic modes, Cynefin classification, and maximum reflection rounds. There's a structured observation carrying status, data, a summary, a confidence between zero and one, issues, next suggestions, provenance, a trace id, and the versions used. There's a trace event carrying the step number, timestamp, thought, action type, payload, the observation, token usage, and versions. And there's the agent state, tying together the task spec, the history, the todo, the plan, short and long-term memory, the pattern store reference, the set of seen hashes, budgets, versions, the tracer, the current mode, the Cynefin context, and the history of after-action review artifacts.

The key modules to implement include the core loop, an enhanced circuit breaker with metrics and per-tool instances, a verifier tuned to catch the specific failure modes, an orchestrator with planning, routing, a sub-agent factory, and a consolidator, a self-evolver that diagnoses from a trace and proposes and validates edits, a minimal protocol module for the tool, environment, and agent abstractions plus the version manager, an integration wrapper for the server-side multi-agent calls, memory and tracing modules, and a set of versioned prompt files that reference the failure modes in the critic prompts. For the hybrid integration specifically, when the action calls for research delegation or the sub-spec complexity is high, the system calls the multi-agent system instead of a local sub-loop, with a configurable threshold, always capturing the source and agent count, and falling back to a local sub-agent or direct tools if the call fails or the circuit is open. It can also emulate build-tool patterns by exporting the plan and running parallel sub-agents on independent branches with isolated state copies.

The phased roadmap gives the coding agent a clear path. Phase zero builds the scaffold, the data models, a basic model client, the strict schema helper, and a simple loop that runs without crashing on mock tools; its gate is that the loop executes ten steps on a toy task, always produces valid observations, detects injected repetition, and catches obvious spec issues. Phase one builds the full controlled loop with cycle detection, the full circuit breaker, safe execution, progress tracking, the fast recognition path with pattern matching, the lightweight metacognition monitor, light reflection, an explicit finish with a verifier gate, structured state, and aggressive summarization, with failure-injection tests passing for repetition, history loss, and premature or incomplete termination, and the fast path triggering correctly. Phase two adds the orchestrator and a pluggable sub-agent registry, the local delegation path, the consolidator and reporter, and a multi-level verifier, proven end-to-end on a multi-step task with delegation. Phase three adds the version manager, the minimal protocol schemas, the self-evolver, and the server-side multi-agent client, with self-evolution showing measurable improvement across three similar tasks and delegation working end-to-end. Phase four is polish, examples, production hardening, and dogfooding: a deep research agent with cognitive routing, a coding project agent with planning and sub-agents, and a self-improving harness that evolves its own verifier, prompts, and cognitive profile, plus a sandboxed code tool, a command-line interface and optional server, a full test suite including cognitive feature tests, observability exports with mode and review overlays, cost dashboards, and documentation. The mandate throughout: after each module, generate code, run an internal critic, fix the issues, re-validate against the gate, and only then proceed.

On production hardening, security, observability, and extensibility, the themes carry through. Reliability comes from circuit breakers, retries with backoff, structured error observations, budget enforcement, and progress-based early exit. Security comes from sandboxing tool execution, validating and sanitizing all model-generated arguments, least-privilege tool access, and anomaly detection on loop patterns. Observability makes the tracer first-class, recording every event with optional full context snapshots, versions, token counts, timings, and sub-calls, exportable for replay from any step, with optional integration into existing logging. Cost and scalability come from per-phase budgets, parallelism only on independent branches, smart summarization signals, and session isolation. And extensibility comes from pluggable model backends, a simple tool registry with schema validation, a sub-agent factory with role prompts, pluggable memory backends, swappable evolution strategies, and adapters for state-machine frameworks and remote orchestration.

The testing strategy is detailed and treated as critical. Unit tests cover schema validation, cycle-hash correctness, the circuit breaker's full state machine including half-open recovery, and the verifier's parsing and logic on edge cases. Integration tests run the full loop on toy tasks and exercise hierarchical delegation and the hybrid path. Failure-injection tests simulate each major failure mode and assert early detection, correct mitigation, and no silent failure. A mini benchmark measures success, efficiency, and the evolution delta before and after three rounds. Property-based tests throw random valid and invalid specs and observations at the system and assert invariants like always-structured output, sane confidence values, respected budgets, and consistent versions. And a dogfood evolution test runs self-evolution on the verifier or planner prompt using real traces and verifies the improved version scores higher on held-out data without regressing on base tasks. The mandate is simple: don't mark a phase complete until its tests pass, and generate tests alongside the code.
