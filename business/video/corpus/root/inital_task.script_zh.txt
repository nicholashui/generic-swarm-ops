我们一起来看看这份笔记的核心任务。目标是建立一个全新的项目，而我们会从头到尾用一个占位名称来称呼它。这个项目是直接根据一份起始蓝图来建立的，那份文件叫做 project starter，就放在这份笔记的旁边。新项目会放在它自己的文件夹里，而文件夹就以项目本身的名字来命名。

关于那份起始文件，有个重点要讲清楚。它不是一份粗略的草图，也不是一堆松散的建议。它应该被当成一份完整的实现合同。也就是说，它清清楚楚地讲明了这个样板应该包含什么，又应该怎样运作，而我们会从头到尾忠实地完全照着它来做。

那么我们究竟是在建立一个什么样的项目呢？它是一个 AI 编程代理的起始仓库。主要技术是 Node，版本要二十或者更新，而其中有一个刻意定下的限制很值得留意。负责启动的 bootstrap 脚本，被设计成完全不需要任何运行时的依赖项。这样就可以让基础保持精简，也更让人信得过。

最有意思的地方就在于它的主要目的。我们要建立一个可以实际运行的样板项目，一个真正跑得起来、能够下载、审核以及同步 AI 编程代理配置文件的项目，而且要横跨一整个系列的工具。这些工具包括 Claude Code、Codex、Gemini 命令行界面、Cursor、OpenCode、Grok Build，以及 GitHub Copilot。这个构思就是把所有这些代理的配置集中在同一个地方，让它们能够保持一致并且同步。

说到必须支持的范围，要讲清楚一点，这个项目需要处理 Claude Code、Codex、Gemini 命令行界面、Cursor、OpenCode，以及 GitHub Copilot。这些都不是可有可无的附加功能，而是成品必须兼顾的一部分。

而这一切还有一个很明确的要求。你必须建立一个真正能够运作的项目，而不是只交一堆描述它的 Markdown 文档。最终交付的，是真实、能够运行的代码。

当一切准备就绪之后，只需要一个命令就可以让它运作起来。你通过项目的 run 命令去执行 bootstrap 这个步骤，这样就会启动整个下载、审核以及同步的流程。

最后，这份笔记也预先想到了一个很实际的问题。如果因为根本没有网络，导致下载失败，那该怎么办呢？在这种情况下，指示很清楚。尽量完成所有你在本机能够完成的文件，然后把这个障碍，也就是让你停下来的那件事，记录在一个状态文件 status 里，让整件事都有记录，不会有东西遗失。这样即使网络帮不上忙，这份工作也已经做到尽可能完整，而下一个接手的人也会清楚知道，现在的进度到了哪里。
