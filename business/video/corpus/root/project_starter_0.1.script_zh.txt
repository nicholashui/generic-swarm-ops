想象一下，你可以在短短几分钟之内，为一个 AI 智能体设置好一个完美的编程环境，一个预先载入了最佳工具、最聪明规则，还有真正记得住东西的记忆的环境。这就是整个项目的核心想法。它叫做 ultimate agent harness starter，目标是建立一个干净、可以直接投入生产的起始项目，一个全新的代码仓库，连同设置脚本和配置文件，能够快速启动一个强大的编程智能体环境，让你在很多不同的工具上使用。

这里的想法，是要结合市面上排名最高的编程智能体项目的最佳元素，也就是在二零二六年年中那会儿领先的那些。每当其中两个项目出现重叠或者冲突的时候，就有一个清晰的决胜准则。一个叫做 ECC 的 harness 会被视为主要基础。最终成果应该是你用一个命令，或者几个命令就能装好，立即得到一个生产力极高的智能体工作流程，可以配合 Claude Code、Cursor、Codex、OpenCode、Gemini 的命令行工具，以及类似的系统一起运行。

我们来讲讲指导这一切的原则。首先最重要的，是规格驱动开发。这个想法很简单但很有力，就是清晰的规格带动其他所有的事情。当功能、技能或者规则之间出现重叠的时候，你就优先采用排名最高的来源。ECC 排在最前，接着是受 Andrej Karpathy 启发的一套行为规则，之后是一个持久记忆工具，再之后是一系列最佳实践，然后是一个更广泛的技能库。整件事应该保持可移植、精简、本地优先，并且尽可能避免过度依赖 Docker。安全、记忆，以及高效地运用 token，全都很重要。

输出必须对智能体本身友好。也就是说要有清晰的阶段、清单、验收标准，以及为评审智能体而设的内置钩子。它应该支持一个迭代节奏，规划、实现、评审，然后改进，按需要循环回去。那这是为谁而设的呢？是为独立开发者、小型团队，以及想自己打造定制化 AI 编程工作流程的高级用户。

那么成功实际上是什么样的呢？一个新项目文件夹应该在五分钟之内、由一个主要脚本初始化好。ECC 应该完整安装和配置好，作为核心 harness。Karpathy 行为规则应该默认启用。持久记忆应该开启。最有价值的技能应该从顶尖技能库当中有选择地合并进来，不要有重复。最佳实践配置、钩子、规则和示例工作流程都应该包括在内。安全基线应该启用。应该有清晰的文档和一个任务文件，方便日后扩充。它应该跨平台运行，在 Mac、Linux 和 Windows 上尽可能地使用，也可以配合多个智能体。还要包含质量关卡，比如代码检查、测试和评审提示。最重要的是，每一个受支持的编程智能体，都应该共用完全相同的精选技能、规则和钩子，并由单一可信来源保持同步。

这就引出了其中一个核心新功能，跨智能体的技能和规则同步层。这里的目标，是要确保每一个编程智能体都知道相同的高质量技能和规则。要做到这一点，方法就是从一个中央位置，让每个智能体的配置文件夹保持同步。这解决了一个很常见的麻烦，就是不同智能体最终会出现零碎或者过时的技能。这个做法首先倚重 ECC 的跨 harness 设计，然后加上轻量级的适配器和同步脚本，去覆盖其余所有的东西。

这方面的设计原则很直接。有一个单一可信来源，也就是在仓库根目录的中央文件夹，分别存放技能、规则、钩子和 MCP 配置。当一个智能体支持的时候，你会优先用符号链接，因为它快而且永远保持最新。当一个智能体有不同的文件夹结构或者文件格式的时候，你就退而求其次，用智能复制加上轻量转换。你会尽量倚靠 ECC 的内置兼容性。而整个同步过程保持简单、可脚本化和安全，意思是你可以重复运行而不会出问题，并且内置备份和还原。实际上，文件夹布局会在智能体专属文件夹之外，比如一个给 Claude Code、一个给 Cursor，再加上一些中央来源文件夹，以及一个脚本文件夹，里面有主同步脚本和各个智能体的辅助脚本。

要让同步成真，有一套清晰的任务。首先，建立中央文件夹作为权威来源。然后，建立或者改编一个同步脚本。对 Claude Code 来说，就是复制或者链接技能、命令、钩子和规则，并尽量用 ECC 的插件和市集模式。对 Cursor 来说，你从中央规则去生成或者更新它的规则和配置。对其他智能体，比如 Codex、OpenCode 和 Gemini 的工具，你就结合中央内容和 ECC 适配器，去建立合适的根目录文件。这个脚本应该支持完整同步，也支持有选择的同步，比如只同步规划和评审技能。它要安全而且可重复，有试运行模式和冲突检测。你首先整合 ECC 现有的功能，加一个简单的同步命令，让任何人在技能更新之后都可以触发，清晰地记录整个过程，甚至包含一个命令，让智能体可以自己触发同步。你还通过一个清单去做版本锁定，让到处都用相同的技能版本，并且至少在 Claude Code、Cursor 和另一个智能体之间测试同步。

那你怎么知道它有没有用呢？当你在中央文件夹更新一个技能，然后运行同步脚本，那个技能就会立即在每一个受支持的智能体那里可以用到。不用在文件夹之间手动复制。所有智能体都表现一致，因为它们全都是在参考相同的精选内容。而同步本身又快、又安全、又有良好的文档。这一层，就是让整个 starter 在你的整套工具之间真正做到可移植和一致。

接下来的，是一个自我评估和评审例程，本质上是一个方法，让智能体可以评估自己工作的质量。这建立了一个闭环，规划、实现、自我评估、然后改进。这一节是根据深入研究而大幅强化的，其中借鉴了 xAI 在多智能体能力以及透明、可审计推理方面的成果，再加上二零二五和二零二六年一系列高质量的研究，涵盖反思、自我精进、多智能体自我演化、结构化的评审和执行者循环、人在回路中的改进，以及记忆架构。

这些有研究支持的设计原则值得逐一讲解。有一个多智能体评审模式，采用专门的角色，一个执行者或者求解者，一个评审者或者挑战者，以及一个可选的裁判，它们可以并行运行，而每一个子智能体都会展示自己的推理，让所有的东西都保持可审计。有一种结构化的自我批评，超越含糊的意见，走向明确的前置条件验证、状态追踪、评分准则打分，以及对过往反思的记忆。它很强调人在回路中的安全，每一个高影响的改动都需要人工确认。它有细致的记忆和上下文管理，支持分层摘要和所谓的上下文折叠，去处理长任务。它以透明和可审计为核心，每一个推理步骤、评审和决定都会被记录，并且可以被查看。而再说一次，ECC 排在最前，研究层是建立在它现有的评审能力之上的。

这个例程会在一套更丰富的维度上评估工作。正确性和功能性，连同明确的前置条件和状态验证。简洁，以及与 Karpathy 理念的一致性。对规格的忠实度。安全和安全性。性能、效率和 token 用量。可维护性和清晰度。推理质量和可审计性，这关乎思考过程有多透明和可验证。还有自我改进潜力，意思是那个批评实际上有多可以付诸行动。

核心流程是这样运行的。执行者用 ECC 技能和 Karpathy 规则去生成解决方案。评审者，它本身可以是多个智能体，会用那套评分准则去做结构化批评，也可以派生出并行的子智能体去深入钻研不同维度，一个看安全，另一个看简洁，从而产生分数、具体问题，以及具体的改进建议。那些批评和学到的经验会存储在情节记忆当中。执行者接着会用那些批评和已存储的反思去改进输出，支持多次迭代，并有智能的上下文管理。高影响的建议要经过一个人工确认关卡。而一个完整的审计记录会记下所有推理、批评和决定。

为什么这件事重要呢？它防止了人们所说的凭感觉写代码的偏移。它强制执行对规格的遵从。它及早发现问题，无论是正确性、安全、复杂度、浪费掉的 token，还是可维护性。也让智能体随着时间变得更自主和可靠。在设计上，有中央评审技能，有一个结构化的自我评估模板，输出一个整体质量分数、各个维度的细项拆解、发现的具体问题，以及以新任务或者差异建议形式呈现的具体改进建议。它可以由一个完成后钩子自动触发，由一个自我评审或者评审命令手动触发，或者作为一个更大的多步骤工作流程的一部分。输出会和工作一起存储，并反馈回这个循环。

要建立这件事的任务，包括建立一个核心的评审和自我评审技能，定义一个可重用的自我评估模板，加一个触发结构化评估的斜杠命令，建立一个完成后钩子，让输出结构化，让其他智能体或者脚本可以解析，整合现有的编排，加入示例用法和一个样本工作流程，允许评分准则可以定制，比如项目专属的权重或者额外维度像无障碍和国际化，并确保评审例程本身也可以被评估，做到持续改进。成功就是说智能体可以对自己最近的工作运行一次自我评估，并产生一个清晰、可付诸行动的批评，可以手动或者自动触发，附带分数和具体的改进任务，反馈回工作流程。这就是让智能体从一个一次性的生成器，变成一个真正自我改进的系统。

建立在这个之上的，是技能生命周期，也就是自动建议加入新技能、更新现有技能，或者移除低价值或者过时技能的能力，但是从来不会在没有明确人工确认的情况下实施任何改动。这建立了一个安全、受控的方式，让技能集可以演化。它直接建立在评审例程和中央可信来源之上。

这个工作流程，人永远在回路之中。首先，是一个分析阶段，由评审者、定期审视，或者完成了重要工作之后触发，智能体会看看这些技能怎样被使用、它们的质量分数、它们的相关性、有没有重复，以及有没有缺口。然后是建议生成，产生清晰、结构化的提案。要加入一个新技能，它会给出名称、目的、来源或者草稿内容，以及为什么它有价值。要更新一个，它会给出具体的改进，连同前后对照的摘要。要移除一个，它会给出理由和一个影响评估。这些建议会连同独一无二的标识符一起存储。然后是人工审视和确认，由一个人通过文件、仪表盘或者命令去查看这些建议，并确认、拒绝或者修改它们。只有经确认的项目才会继续。然后是安全应用，经确认的改动会应用到中央技能，而同步层会把它传播到每一个智能体。最后是一个审计和还原步骤，所有改动都会连同时间戳、理由，以及批准的人的名字一起记下，并通过版本控制做到容易还原。

指导原则很坚定。永远不要自动应用，人工确认是必须的。首先倚靠 ECC 的持续学习模式。保持所有的东西透明和可审计，每一个建议都有清晰的理据和风险。保持非阻塞，让这些建议在方便的时候才收集和查看，而不是打断工作。并且保持可扩充，让相同的模式日后可以套用到规则、钩子，甚至整个项目。这里的成功就是说智能体可以生成清晰、可付诸行动的建议，没有任何技能会在没有明确确认的情况下被改动，经确认的改动会被安全地应用和传播，存在一个完整的审计轨迹，而整件事感觉自然而不碍手。智能体提出建议，人去决定，系统去应用。这样加在一起，就完成了一个强大而安全的自我改进循环，同时让人牢牢掌握控制权。

现在我们来逐一讲讲真正建立它的各个阶段。阶段零是研究和最终选择，做一次就行。目标是确认最新版本，并用排名优先次序去解决任何重叠。你会核实当前顶尖的仓库，ECC 作为主要 harness，Karpathy 行为规则，持久记忆工具，一个最佳实践合集，以及一个大型技能库，其中你只会安装高价值的套装。你会识别重叠并作出决定。核心 harness、规则、钩子、安全和 MCP 排 ECC 在前。行为指引来自 Karpathy 规则。记忆来自专用的记忆工具，或者 ECC 的内置记忆，如果那个已经足够。规划和最佳实践工作流程，由最佳实践来源和 ECC 的规划技能合并而成。至于大型技能，你会用那个库的安装程序，但只精选头二十到五十个最有用的，比如规划、测试驱动开发、评审、安全和前端工作，而不是全部安装。你还会检查有没有任何官方技能或者新的高排名新增项目，并记录每一个决定。成功就是一个清晰的决策日志，没有任何冲突的重复规则或者技能。

阶段一是项目脚手架和 ECC 基础，最高优先次序。目标是建立仓库骨架，并安装 ECC 作为完整的基础。你会初始化新仓库，建立标准结构，包含给每个智能体的文件夹，以及给技能、规则、钩子、MCP 配置、文档、脚本、示例和任务文件的文件夹。然后你安装 ECC 作为基础，跟着它的官方安装，复制和改编它的核心智能体、它的精选技能、规则、钩子和安全层，并启用它的关键功能，比如 token 优化、会话持久化、本能学习和 MCP。你加上它的并行编排支持，它会生成任务和交接文件，并建立一个初始的项目规则文件，参考 ECC 并合并入 Karpathy 原则。成功就是一个单命令的启动可以运行，基本的智能体命令比如规划、评审和一次安全扫描立即可以用，而且仓库通过基本的检查和安全检查。

阶段二加入 Karpathy 行为规则和最佳实践。目标是注入有纪律的护栏和生产模式。你会整合四项核心 Karpathy 原则作为基础行为层，写代码之前先思考、简洁优先、外科手术式的改动，以及目标驱动的执行，并把它们干净地合并入 ECC 的规则而没有冲突，任何重叠都是 ECC 排在前。你从最佳实践来源拉入高价值模式，比如规划工作流程、子智能体使用、上下文管理、斜杠命令和 MCP 模式，只挑选那些还没覆盖的。你建立或者强化根目录的规则文件，加入合并后的指引，并加入一个规格驱动风格的示例项目规格模板。成功就是智能体一开箱就持续遵循这些模式，没有规则冲突，并且有清晰的文档讲解规则怎样加载。

阶段三涵盖记忆、技能和有选择的技能库整合。目标是加入持久记忆和精选的高影响技能。你会安装记忆工具去做跨会话的上下文，配置它去捕捉工具使用、做摘要，并注入相关上下文。你会有选择地用那个大型技能库，只安装顶尖套装，规划、测试驱动开发、代码评审、安全、前端和后端模式，以及编排，同时避免一个会造成臃肿的庞大安装。你会把任何独特的高价值技能合并入中央文件夹，并附上清晰的出处，建立一个清单或者索引方便发现，并加入规格驱动的技能，如果它们还不存在，比如规格分析、功能规格生成、路线图任务和验证关卡。成功就是记忆跨会话保持，并为长任务改进上下文，技能集保持精简但强大，并且日后有一个容易的方法去加入或者移除技能。

阶段四是安全、钩子、MCP、token 优化和打磨，这是生产加固阶段，发挥 ECC 的强项。你会启用安全层，连同密钥检测和漏洞扫描。你会配置关键钩子，提交前验证、完成后评审、上下文压缩和成本追踪。你会为常用工具设置 MCP 配置，比如 GitHub 和文件系统，从最精简和安全开始。你会应用 ECC 的 token 优化设置。你会加入标准的仓库文件，一个 gitignore、一个 MIT 许可证和贡献者指南。并建立启动脚本，运行 ECC 安装、记忆、精选技能和配置复制，并有给不同智能体的标志。成功就是安全基线启用并有文档记录，钩子正确触发，token 用量明显被优化，而一个主要启动命令可靠地运行。

阶段五是文档、示例、验证和质量关卡，全都关乎让这个 starter 易用和可扩充。你会写优秀的文档，一个有快速开始、架构概览和排名理据的 readme，一个有确切命令的安装指南，以及一个有示例工作流程的使用指南。你会加入一两个示例迷你项目或者工作流程演示。你会实现质量关卡，在改动上自动做检查和安全扫描，一个给改动用的评审提示或者命令，以及一个验证核心命令运行的自我测试脚本。你会加入任务文件作为活的规格，并建立一个评审智能体，去持续改进 harness 本身。成功就是一个新用户可以在十分钟之内，从零到一个有生产力的智能体工作流程，文档清晰而且和规格一致，仓库干净、安全，并且准备好可以发布。

阶段六关乎未来的可扩充性和迭代钩子，为持续演化而设计。你会加入一个自我改进循环技能，它会研究新技能、提出新增、经过评审审视，然后合并。你会通过脚本支持从上游来源做容易的更新。你会包含给领域专属扩展的占位，比如交易技能、前端设计或者特定的技术栈。并计划用 ECC 的并行模式和任务文件去做多智能体编排示例。成功就是有一条清晰的路径去演化这个 starter 而不会破坏现有设置，并且支持一个偏好的节奏，也就是配合评审智能体做迭代精进。

几个整体执行笔记把所有的东西扣在一起。永远从规划和规格阶段开始。任何重叠都永远先用排名最高的来源。为复杂的子任务生成任务和状态文件。在重大阶段之后运行评审步骤。全程追踪成本和 token 用量。在任何合并或者发布之前运行一次安全扫描。并一路记录决定。实现的初始优先次序是，ECC 基础排在最前，然后是 Karpathy 规则和最佳实践，再然后是记忆和精选技能，接着是安全、钩子和 MCP，最后是文档和验证。
