我们一起来过一遍一份改良版的实时任务规格，项目叫做 Project Starter。你可以把它当成一张蓝图，用来搭建一个能够直接投入生产的起始仓库，这个仓库负责安装、同步、审计，并且持续演化一套跨代理的编码框架，覆盖一整个系列的 AI 编码工具。我们说的是 Claude Code、Cursor、Codex、OpenCode、Gemini 命令行工具、Grok Build，还有类似的助手。这是零点二版本，也就是重新思考的版本，已经准备好去做团队所说的第零阶段执行。它在二零二六年六月创建，主要目标很清晰：做一个仓库，在一个各自为政、配置系统各不相同的工具世界里，带来一致性和安全性。

那为什么要重新思考呢？原本的计划其实很扎实，但同时也太宽泛、有些重复，还有一点风险。它假设你可以做完整安装，又可以在一些根本不会用同一种方式存储设置的工具之间做到完全一致。这个改良版让整个项目更具可执行性，背后有十个重点。

第一，它把一个基础框架，我们叫做 E C C，定为一切的根基，但不是盲目地复制所有东西，而是先安装经过精心挑选的配置档。第二，它不再以 shell 脚本为先，而是改用一个用 Node 编写、跨平台的命令行工具。第三，它定义一个单一可信来源，配上生成出来的适配器，以及自动检查是否有偏移。第四，它把"处处都有相同技能和规则"这个目标重新理解为语义上的一致，也就是行为一样就行，而不是要求每个文件逐个字节都完全相同。第五，它加入正式的清单，记录授权、来源、校验码和版本。第六，它清晰地分开指令、技能、记忆、钩子、模型上下文协议的配置，以及生成出来的工具配置。第七，它引入一个明确的安全威胁模型。第八，它不再记录每一条推理轨迹，而是存储可审计的摘要：证据、决定、差异、命令、测试结果和审查输出。它刻意不会要求或者存储隐藏的思维链。第九，自我改进在人工批准之前只能停留在建议阶段。第十，它加入可衡量的质量关卡、同步测试，以及安装时间的预算。

在搭建任何东西之前，会有一个来源核实快照。第零阶段必须先重新检查每一个来源。预定的优先次序是这样：E C C 框架是主要根基，接着是一套精简的行为规则，风格和 Andrej Karpathy 相关，再接着是一个持久记忆工具，例如 claude-mem 或者同等的选择，前提是它兼容又安全。之后就是从口碑好的 Claude Code 仓库挑选出来的规划和工作流程模式，再接着是有选择性地引入、而不是大批安装的精选技能库，最后就是每个代理的官方文档。对每一个来源，第零阶段都必须核实最新版本或者发布、授权、安装方式、支持哪些配置路径、安全影响，还有具体有什么会被纳入、改造或者拒绝。

接着是一些不可妥协的原则。规格永远驱动实现。当组件有重叠的时候，优先采用 E C C 的选择，除非来源审计证明另一个更加合适。Karpathy 行为层的意思是：写程序之前先想清楚、偏好简单、做精准的改动、以目标为导向。系统有一个单一可信来源：放技能、规则、钩子和配置的中央文件夹才是权威，其他每个工具的文件都是由这个来源生成的。系统默认安全，没有人工明确批准就不会有破坏性的自动化、远程连接，或者技能改动。它以本地为先，依靠本地脚本、本地记忆和本地审计记录，外部服务保持可选。核心保持精简，较大的组合包保持可选并经过挑选。它尽可能在 macOS、Linux 和 Windows 上都能运行。它是可审计而不是不透明的，存储简洁的理据和证据，而不是要求隐藏的推理。

现在说说实际会交付些什么。交付物包括这个新仓库本身、一个通过 Node 运行的主要命令行入口，以及一组方便的软件包脚本，用来初始化、同步、检查同步、做 doctor 诊断、扫描安全、做审查、跑测试和格式化。在这些之上还有以 E C C 为基础的起始配置档、一个跨代理的同步引擎、技能、规则、钩子和配置的精选清单，以及每个主要工具的适配器。也有为 GitHub Copilot、Zed 和 Windsurf 而设的可选适配器。再有一个自我审查和评论员的例行程序、一个技能生命周期的建议和批准工作流程、一个安全基线，还有齐全的文档，配上示例、测试和验收检查。

仓库结构是围绕着可信来源这个概念来组织的。最顶层放的是面向代理的指令文件、说明文档、软件包定义，以及实时的任务和状态文件。接着有为每个工具而设、放生成配置的隐藏文件夹，还有一个中央技能文件夹，按类别细分，例如规划、实现、测试、审查、安全、记忆和生命周期。一个规则文件夹放着一些带编号的文件：一份章程、Karpathy 行为规则、规格驱动开发的规则、安全、测试、令牌效率和人工批准。还有放钩子的文件夹、放模型上下文协议配置的文件夹，分为最小和可选，以及一个记忆文件夹，放着项目上下文、交接笔记和反思。也有放审查的文件夹、放建议的文件夹，按待处理、已批准和已拒绝分类，配一份审计记录，还有一个脚本文件夹，里面放主路由器和每个工具的适配器。最后有文档文件夹、示例文件夹，以及一个测试文件夹，里面有测试夹具，以及针对同步、清单和适配器的测试文件。

还有一套严格的生成文件政策。每一个生成或者改造出来的文件都会带着一个文件头，声明它是由 Project Starter 自动生成、写明它的来源文件夹，并且提醒你去执行同步命令，而不要直接编辑它。规则很硬性：中央文件占上风，生成文件只会在冲突检查之后才被覆写，本地用户文件在任何覆写之前都会先备份。一个试运行模式必须显示它会执行的确切写入和删除。一个检查模式必须在生成文件过时的时候报错。符号链接只会在安全又支持的情况下优先采用，在 Windows 上，除非检测到妥当的符号链接支持，否则系统会退而采用复制模式。

第零阶段是研究、范围锁定和来源审计的阶段。目标是在生成任何一个文件之前，先确认最新的来源、安装命令、授权和兼容性。任务包括核实最新的 E C C 发布和它的安装器、确认 Karpathy 规则和一个 Cursor 变体、检查持久记忆的候选方案、审视最佳实践的工作流程仓库但只挑选不重复的模式，并且只选高价值的技能组合包。一份来源审计文档会就每一个来源记录它的名称、位置、版本、授权、安装命令、选中的组件、拒绝的组件，以及理据。一份决策文档会用结构化的格式记录架构决定。还会定义三个起始配置档：一个核心配置档，精简、安全，可以在五分钟内初始化好；一个强力配置档，安装范围更广，加上记忆和精选技能；以及一个实验配置档，用来做 Grok Build、额外连接器和多代理演示。要过关，没有一个来源可以在没有记录授权和版本的情况下使用，不可以接受有冲突的重复，必须有清晰的优先次序，而安装命令在写任何脚本之前都必须核实过。

第一阶段搭建仓库骨架和命令行基础。团队初始化 Git 仓库、加入带齐所有脚本的软件包定义，并且实现主命令路由器。一个 doctor 脚本会检查 Node 版本、Git 是否可用、操作系统、符号链接能力、安装了哪些代理，以及必要的目录是否存在。基本配置文件、一个说明文档和起始文档都会加入，连同有 JSON 结构描述支撑的空清单，以及第一份任务和状态文件。这里的成功标准是：doctor 命令在一台干净的机器上能运行、占位测试通过、骨架可以在一分钟内搭好，而且所有东西都一致地使用 Project Starter 这个名字。

第二阶段安装和改造 E C C 基础，作为主要的框架层。一个适配器或者安装器封装会支持核心、最小、强力和实验这几个配置档。它会安装选好的组件：核心规则、规划、审查和安全技能、安全扫描器、令牌和上下文优化、在安全前提下的记忆和本能学习、并行编排模式，以及连接器惯例。最关键的是，它避免盲目地做完整复制。E C C 的来源版本会被记录、衍生文件会加上署名，之后还会跑一次安全扫描。验收门槛是：核心配置档可重现地安装好、扫描有跑、默认不会加载过大的技能堆，而当有冲突的时候，E C C 就是默认的赢家。

第三阶段建立行为规则和规格驱动开发的章程。这些是每个代理都会收到、精简而高影响力的规则。行为层很简单就能讲清楚：写程序之前先想、偏好简单方案、做精准改动、以既定目标为依归、只在被有意义的含糊卡住的时候才发问、改动多个文件之前先规划、跑测试或者解释为什么没跑、做完重大工作之后更新状态文件，并且在没有人工批准之前，永远不自动套用对技能、规则或者钩子的改动。要过关，所有面向代理的文件都必须收到语义上一致的行为，规则必须保持精简和不重复，而任何冲突都必须有记录。

第四阶段建立精选的技能集，要够精简避免臃肿，但又要覆盖高价值的工作流程：规划、实现、测试驱动开发、代码审查、安全审查、调试、重构、文档、记忆和交接、上下文压缩、跨代理同步、自我审查，以及技能建议的生命周期。清单里每一个技能条目都带着一组丰富的字段：一个标识符、名称、版本、描述、来源、来源优先级、授权、校验码、兼容的代理、副作用、是否需要人工批准、标签、路径、一个负责人，以及最后审查的日期。目标是核心配置档大约有十五到三十个技能，而不是几百个，可选的组合包可以选用，每个技能都有署名和版本，而且全部都通过同步适配器测试过。

第五阶段是跨代理同步层，而这个其实就是整个项目的核心。目标是让每个受支持的代理，在各自工具支持得到的范围内，都收到相同的精选技能、规则、钩子和连接器配置。命令行允许你一次性同步所有东西，或者针对单一代理，例如 Claude、Cursor、Codex、Gemini、OpenCode 或者 Grok Build。你可以用试运行模式预览改动、用检查模式核实新鲜度、在自动、复制或者符号链接模式之间选择，甚至只选择一个子集的类别，例如规划、审查和安全。每个适配器都会写入适合自己工具的位置：Claude Code 收到它的技能、规则、设置、命令垫片和根指令文件；Cursor 收到它的规则文件和连接器配置；Codex 和 OpenCode 收到它们的代理指令文件和配置，在没有原生技能的地方就用一个生成出来的技能索引；Gemini 收到它的指令文件和设置；Grok Build 和 Copilot 收到它们兼容的引用和指令文件。同步安全是内置的：被修改的文件会备份、人工的编辑会被检测到、会产生一份同步报告、清单或者结构描述出错会导致失败，还支持持续集成的偏移检查。证明过程很令人满意：改一个中央技能、跑一次同步，每个受支持的代理就会更新，全程没有任何地方需要人工复制。

第六阶段处理记忆和交接，提供持久、注重隐私的记忆，但不会让上下文臃肿。它有几层：状态文件记录当前进度、任务文件作为实时规格、一个项目记忆文件放稳定的上下文、一个交接文件放精简的延续摘要，以及一个反思文件夹放经验教训和重复出现的模式。可选择加入一个专用的记忆层。规则保护隐私：永远不存储秘密、遮蔽凭证和个人资料、用摘要代替倒出整堆日志、只加载相关的记忆、延续工作时优先用交接文件，并且让自动注入的上下文保持小巧。一个新的工作会话应该可以从任务、状态和交接文件恢复，自我审查可以生成反思条目，而记忆也可以完全关闭。

第七阶段加入自我评估和评论员的例行程序，是一个规划、实现、审查和精炼的结构化循环。它有角色：一个解题者负责实现、一个评论员审查正确性、简洁度、规格遵从、安全、性能和可维护性、一个可选的安全评论员、一个可选的测试评论员，以及一个裁判去总结什么是阻塞性、什么不是。一份评分表会就九个维度从一分到五分评分，涵盖正确性、规格遵从、简洁度、安全、测试覆盖率、可维护性、性能和令牌效率、可审计性，以及自我改进的价值。输出是可读和结构化两种格式的审查文件，列出阻塞性发现、非阻塞性建议、跑过的测试、改过的文件，以及建议的下一步。还有命令去触发自我审查和评论，可以是代理命令，也可以通过命令行。这里的护栏很重要：不记录隐藏的思维链、只存储简洁的理据和证据、默认最多两个精炼循环，并且对高影响的改动要人工确认。

第八阶段是技能生命周期，允许技能、规则和钩子只通过建议去做安全的自我演化。工作流程会分析使用情况、审查、失败、缺口和重复，然后生成一个建议文件。一个人会审视它，然后批准、拒绝或者修改。只有获批准的改动才会套用到中央来源，之后同步就会传播它，而审计记录会记下这个决定。每一个建议都包含一个标识符、要做的动作即新增、更新或者移除、目标、理据、证据、提议的改动、风险评估、回滚计划、信心程度，以及一个人工批准字段。严格规则是：没有明确批准就不会有任何变动、获批准的改动会被记录、被拒绝的建议会保留来做审计，而且永远有一条回滚路径。

第九阶段建立安全基线，让这套框架可以安全地在现实世界本地使用。威胁模型防范以下情况：通过文档、议题、网页或者连接器输出而来的提示注入；秘密外泄；破坏性的 shell 命令；过于宽松的钩子；不安全的连接器服务器；来自技能库的供应链风险；生成文件偏移；隐藏的远程遥测；以及意外覆写全局配置。控制措施包括一次安全扫描、一个秘密扫描器、针对连接器和钩子命令的允许清单、对敏感文件例如环境文件、密钥和令牌的保护、对破坏性命令要人工批准、一个持续集成的安全检查、依赖审计、没有默认的远程凭证，以及一条硬性规定：除非明确批准并记录，否则禁止把下载内容直接灌入 shell。要过关，安全命令在本地能运行、关键发现会让构建失败、敏感文件受保护，而文档也解释了这个模型。

第十阶段涵盖文档和示例，目标是让这个起始仓库可以在十分钟内用起来。文档涵盖概览和快速入门、确切的安装命令、日常工作流程、可信来源和适配器的架构、同步行为和疑难排解、安全模型、决策记录，以及来源审计。示例涵盖一个规格驱动的功能工作流程、跨代理同步、自我审查、技能建议，以及一次安全扫描。一个新用户应该可以在十分钟内初始化和同步好、文档应该解释清楚这些工具有什么不同，而示例应该真的能跑起来。

第十一阶段是验证和持续集成，存在的目的是防止退化，并且证明这个起始仓库能运行。测试涵盖清单结构描述验证、适配器快照、同步试运行、偏移检测、一次安全扫描的冒烟测试、Windows 路径处理、检查有没有重复的技能标识符，以及检查有没有生成文件漏了文件头。持续集成管线会检查 JavaScript 的代码风格、验证 JSON、在可行的地方检查 Markdown 链接、跑测试、跑同步检查、跑安全扫描，并且上传审查和安全的产物。成功的意思是管线是绿色、一个全新克隆可以通过安装和测试，而同步检查在生成之后通过。

接着是完成的定义。当初始化可以做出一个能运行的起始仓库、同步可以更新所有受支持的代理配置，而 Claude Code、Cursor，连同 Codex、Gemini 或者 OpenCode 至少其中一个都核实过，这个项目就算完成。E C C 基础必须安装好并有记录、Karpathy 规则生效中、自我审查工作流程能运行，而技能建议要受人工批准把关。安全扫描必须能运行并且拦截关键问题、文档必须足够给一个新用户用、每一个生成文件都必须可重现、来源审计和决策都要完整，而 Project Starter 这个名字要处处一致地使用。

最后做个总结，这里是即时的下一步行动：搭好仓库骨架、跑第零阶段的来源审计、实现主脚本、doctor 和一个空的同步框架、加入中央规则和清单、先搭 Claude 和 Cursor 的适配器、再加 Codex、Gemini 和 OpenCode、安装和改造 E C C 核心配置档、跑一次安全扫描、写文档和示例，并且把这一切作为 Project Starter 的初始化去提交。而整体的实现优先次序也沿着同一条轨迹：基础先行，接着是同步，再是 E C C，接着是规则、精选技能、安全、记忆、评论员、生命周期，最后就是示例、持续集成和文档的打磨。这就是完整的图景，一个精简、安全、可审计的起始仓库，从单一可信来源出发，让每一个编码代理都保持同步。
