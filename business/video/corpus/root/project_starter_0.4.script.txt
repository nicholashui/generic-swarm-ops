Let's talk about something called project starter. At its heart, project starter is a reusable blueprint for spinning up brand new software projects, and it comes with built-in support for AI coding agents, downloading external source material, bootstrapping the project, validating everything, writing documentation, enforcing safety rules, and repeating all of this automatically whenever you need to.

The most important thing to understand is that this file is meant to be the single source of truth. It's an implementation contract. A coding agent should be able to read it from top to bottom and, from that reading alone, produce a complete, working starter project. And here's the key point: the result must not be just documentation. It has to include real, runnable scripts, manifests, validation commands, agent configuration files, documentation, and safety controls. In other words, something you can actually execute, not just read.

So how is project starter used? There are two main modes.

The first is self bootstrap mode. You reach for this when you're building the boilerplate repository itself. The idea is simple. You tell your coding agent to implement the specification in the current repository, treat the spec as the source of truth, create every required file, directory, script, manifest, document, test, and agent configuration file, and then run the bootstrap command. When it's done, you should end up with a tidy project folder containing the contract itself, a readme, a package file, instruction files for the different agents, task and status files, and a whole set of folders for sources, scripts, rules, skills, hooks, configuration, memory, documentation, tests, and external material.

The second mode is create new project mode. This is what you use when you want to start a fresh project based on the boilerplate. You give the agent a project name, point it at the spec, and ask it to build a working, executable project rather than just a pile of markdown. You also tell it to download all the enabled sources and then run bootstrap. The end result mirrors the same structure, just under your new project's name.

Now, how should a human actually ask a coding agent to use this file? The recommended approach is to give the agent a clear prompt. You name the new project, point to the location of the specification, and specify where the new project should be created. You describe the project's purpose and its primary technology stack. You list which coding agents need to be supported, and the usual set includes Claude Code, Codex, Gemini CLI, Cursor, OpenCode, and GitHub Copilot.

From there, you spell out what the agent must do, step by step. Read the entire specification. Create the target directory. Create every file and folder the spec requires. Swap out the placeholder details for your real project name and purpose. Copy the specification into the new project's root. Create the package file. Create the source manifests. Create the executable scripts. Create the rule files, the documentation, and the tests. Create the agent configuration files. Download all the enabled upstream sources into the external sources folder. Generate the source lock record and the source audit document. Run the bootstrap validation. Update the status file with the real results. And finally, report whether everything passed or failed.

Alongside those steps come firm security requirements. The agent must not execute downloaded third party code. It must not install packages inside those downloaded repositories. It must not automatically import that downloaded code. It must not touch global user configuration. It must not use secrets unless they're explicitly provided. And it must not overwrite existing files without permission. If network or shell access happens to be unavailable, the agent should still create all the project files and scripts, simply mark source downloading as blocked in the status file, and then tell the user to go into the project folder and run bootstrap themselves.

Let's talk about the inputs the agent might receive. There are several variables. The project name, with a sensible default if none is given. The output path, which is where the generated project lands. A short description of the project's purpose. The main technology stack, which by default is Node version twenty or newer using plain JavaScript with no runtime dependencies for the bootstrap scripts. A flag for whether to download the configured upstream sources. The list of supported agents. The package manager, which defaults to npm. And the license, which defaults to MIT. Whenever a value is missing, the agent simply falls back to the default.

Now, what behavior is expected from a coding agent that implements this? First, the general rules. Treat the file as the contract. Build a complete working project. Favor simple, auditable scripts. Lean on the Node standard library for automation unless dependencies are truly required. Avoid unnecessary third party packages. Keep the generated files readable and maintainable. Make commands repeatable wherever that's practical. Record status information in the status file, downloaded source details in the source lock file, and audit information in the source audit document.

Then there are the safety rules, framed as a list of things the agent must never do. It must not execute downloaded third party code. It must not run install or build commands inside downloaded repositories, whether that's npm, pnpm, yarn, pip, cargo, go, or anything similar. It must not automatically pull source files from downloaded repositories into the main project. It must not modify files outside the target directory. It must not change global agent configuration. It must not store secrets in committed files. It must not delete user files without explicit permission. It must not overwrite an existing, non empty output directory without permission. It must not run destructive commands. And it must never simply assume that network access is available.

There's a special case worth highlighting. If the output path already exists and isn't empty, the agent must stop and ask for permission before continuing. The only exception is when the user explicitly says to overwrite or update that directory, and even then the agent should try to preserve important user created files.

The specification then lays out the required project structure in detail. At the root you'll find the contract itself, a readme, a package file, the agent instruction files, the task and status files, and configuration files for ignoring files and for editor settings. Inside, the sources folder holds the manifests and the generated lock file. The scripts folder holds all the automation, including bootstrap, doctor, create project, download sources, audit sources, security check, sync agent configs, and test, plus a set of utility helpers for file system work, git, logging, and project metadata. There are folders for rules, skills, hooks, model context protocol configurations, memory, documentation, and tests. And finally, an external sources folder where downloaded material lives, kept deliberately separate.

The package file deserves its own attention. It defines the project name as a package safe version of your chosen name, a description matching your purpose, and marks the project as a module. It declares that Node twenty or newer is required. It defines a set of scripts that map cleanly onto the automation: bootstrap, doctor, create, download and audit sources, security, sync, test, and a combined check command that runs doctor, security, the source audit, and the tests together. By default it carries no runtime and no development dependencies, and its scripts should work across macOS, Linux, and Windows wherever that's practical.

Several root files have required content. The readme introduces the project, shows the quick start, lists the common commands, points to the contract for the full structure, names the supported AI agents, and explains where downloaded sources live and the rule that you don't execute third party code unless it's been explicitly reviewed.

The universal agent instruction file describes the project, names the source of truth, and lays out the agent rules: read the contract before structural changes, follow the universal and safety rules, never execute downloaded code, never modify global configuration, keep changes small and reviewable, update the status file after major changes, update documentation when behavior changes, and run tests before reporting completion. It also reminds everyone that the external repositories are reference material only and must not be imported or executed automatically.

There are agent specific instruction files too. One for Claude Code and one for Gemini CLI. Both tell the agent which files to read first, namely the contract, the universal instructions, the task file, and the status file. Both insist on minimal, explicit edits, no execution of downloaded code, a preference for simple standard library scripts, and running validation before claiming completion.

The task file tracks the current goal with a simple checklist: read the contract, verify the structure, verify the scripts, verify the manifests, run bootstrap, run tests, and update status. The status file records the current state of the project, including its name and purpose, when it was last updated, and a table summarizing the bootstrap checks, plus known issues and next steps.

Two more configuration files round out the root. One tells version control which files and folders to ignore, things like dependency folders, environment files, build output, caches, and crucially the contents of external sources, while keeping a single placeholder file so the folder still exists. The other defines consistent editor settings: UTF eight encoding, line feed endings, a final newline, and two space indentation, with a small exception for markdown files.

Next come the source manifests. The main manifest lists upstream repositories that may be downloaded purely as reference material. Each entry has a unique identifier, a flag for whether it's enabled, the repository address, the branch, the destination inside external sources, a stated purpose, and an explicit marker that it must never be executed. By default one source might be enabled and others disabled, and disabled sources are never downloaded unless the user turns them on. There's also a documentation manifest that points to local files like the contract, the universal instructions, and the safety rules. And there's the source lock file, which starts essentially empty and gets filled in by the download script, recording each source's identifier, address, branch, destination, status, commit, and download time. The allowed statuses are downloaded, skipped, failed, and blocked.

Now to the scripts themselves, all written in modern Node module syntax using the standard library. The bootstrap script runs the whole flow in order: doctor, then download sources, then audit sources, then the security check, then sync in dry run mode, then the tests, then it updates the status file and prints a final summary. If a step fails, it records the failure, continues where it's safe to do so, and exits with an error code if any required step failed. If the network is down, it marks source download as blocked but keeps running the local checks.

The doctor script validates local health. It checks that Node is at least version twenty, that the required directories and files exist, that the package file and both manifests are valid, that the required scripts are present, and that the external sources folder exists. It then simply prints whether doctor passed or failed.

The create project script builds a new project from the boilerplate. It accepts options for the name, the path, the purpose, the stack, an option to skip downloading, and a force option. It parses those arguments, validates the name and path, refuses to overwrite a non empty directory unless force is given, creates the directory structure, copies the contract, generates all the files with placeholders replaced, optionally runs the source download, and prints the next steps for the user.

The download sources script reads the main manifest and, for each enabled source, validates the address, confirms the destination sits inside external sources, refuses anything not explicitly marked as non executable, clones the repository if it isn't there yet or fetches the latest if it is, checks out the configured branch, and records the commit hash. It then writes the source lock file. It never executes code from downloaded repositories and never runs install commands inside them. It continues if a single source fails but exits with an error if a required download fails. The only shell commands it's allowed to use are safe git operations like shallow clone, reading the current commit, fetching, checking out, and fast forward pulls. Commands like installing packages or building inside those sources are strictly off limits.

The audit sources script reviews the manifests and the downloaded material. It confirms the manifest exists, that all identifiers are unique, that enabled destinations sit under external sources, that every source is marked non executable, that the lock file exists, that downloads are recorded, and that there are no suspicious destination paths like parent directory references, root, or home directories. It then creates or updates the source audit document and prints whether the audit passed or failed.

The security check script runs basic repository safety checks. It looks for any committed environment file, for secret placeholders filled with real looking values, for source destinations escaping the external sources folder, for any manifest source marked executable, for downloaded sources being referenced as project code, for global config paths being targeted, and for dangerous commands in the package file. Dangerous patterns include things like recursively deleting from the root, using elevated privileges to delete, piping downloaded scripts straight into a shell, or opening permissions wide on the whole system. It prints whether security passed or failed.

The sync agent configs script keeps the universal instructions in step with the agent specific files. It reads the universal instructions and the rule files, then updates or verifies the Claude file, the Gemini file, and the agents documentation. It supports a dry run mode where it writes nothing and instead reports what would change. Future targets might include configuration for Cursor, Copilot, and OpenCode. Importantly, it must never modify global agent configuration.

The test script runs the project tests. It runs the smoke tests, verifies required files, confirms the JSON files parse, checks that key scripts and root documents exist, and prints a test summary saying whether tests passed or failed.

Underpinning all of this is a set of utility helpers. One handles file system tasks like checking for files and directories, ensuring directories exist, and reading and writing JSON and text. Another handles git: checking availability, running safe commands, cloning, getting the current commit, and validating addresses. A third handles logging at the info, warning, error, and success levels, along with step headers. And a fourth handles project concerns like resolving the project root, validating that paths stay inside the project or inside external sources, converting a project name into a package name, and loading project metadata.

The specification also defines a family of rule files. The universal rules cover good habits: read the task before editing, read existing files before changing them, prefer small clear changes, don't invent requirements, don't remove safety checks, update documentation and tests when behavior changes, validate before reporting, and report both what changed and what didn't get done. The safety rules restate the core prohibitions around executing downloaded code, installing dependencies inside it, modifying global configuration, exposing secrets, writing outside the project, overwriting user work, running destructive commands, and turning reference sources into runtime dependencies without review, ending with the simple instruction that when you're unsure, you stop and ask. The coding rules favor simple code, the standard library, module syntax, clear names, input validation, useful errors, idempotent scripts, and deterministic output. The git rules say don't rewrite history or force push, don't commit secrets, keep external sources isolated, don't create commits unless asked, and show changed files before finishing. The testing rules insist on updating tests with behavior, running them before reporting, never faking results, keeping smoke tests fast, and testing structure, manifests, and security. And the source rules reinforce that external sources are reference only, downloaded only into the external folder, never executed, never auto imported, with commits recorded and audits performed.

Then there are skill files, which read like focused playbooks. The planning skill walks through restating the goal, identifying files to inspect and change, spotting risks, building a short checklist, executing in small steps, and validating. The debugging skill is about reproducing the issue, reading the error carefully, isolating the smallest failing unit, inspecting recent changes, testing one hypothesis at a time, and recording the fix. The refactoring skill stresses preserving behavior, keeping tests green before and after, changing one concept at a time, favoring clarity, and removing dead code only when you're certain. The testing skill covers running existing tests first, adding smoke tests, testing both success and failure, staying deterministic, and never faking outputs. The documentation skill is about explaining purpose first, showing quick start commands, documenting inputs, outputs, and safety, and keeping examples copy and paste ready. And the security review skill checks for secret exposure, risky shell commands, path traversal, downloaded source handling, permissions, and dependency changes, then reports anything unresolved.

There are also hook files for the rhythm of work. The pre task hook says, before you start, read the task and status files and the relevant rules, identify the intended change, and identify the validation commands. The post task hook says, after you finish, run validation, update status, update documentation if needed, and summarize the changed files, the tests you ran, and any known issues. The pre commit hook says, before committing, run the combined check command, make sure no secrets are staged, make sure no downloaded external code is staged unless intended, and confirm that documentation and status are accurate.

The project also includes model context protocol configuration files. There's a short readme explaining that this directory holds example configuration files only, that you should never put secrets there, and that global configuration must never be modified automatically. And there's an example configuration that starts essentially empty.

Memory files give future agents durable context. The project memory file explains its own purpose, summarizes that the project came from the contract, lists the most important files, and leaves room for long lived notes. The decisions file is a place to record architectural and process choices over time. And the glossary defines the key terms: project starter as the boilerplate specification, external sources as the downloaded third party repositories, and the source lock as the record of downloaded commits.

A rich set of documentation files rounds things out. The architecture document describes the project as an executable boilerplate built from a source of truth contract, using simple Node scripts, with no required runtime dependencies, isolated external sources, and consistent agent instructions. The setup document lists the requirements and the install and bootstrap steps. The usage document shows how to bootstrap, run doctor, download and audit sources, run security, sync the agent configs, run tests, and create a new project. The source audit document starts as not yet run and gets filled in later. The agents document lists the supported agents, the universal reading list, and the safety expectations. The troubleshooting document walks through what to do when doctor fails, when source download fails, when the security check fails, and when the sync dry run reports changes. And the changelog records the initial generated structure.

For testing, there's a smoke test that checks the project structure end to end: root files exist, required directories exist, JSON parses, package scripts are present, the source manifest is valid, safety rules exist, and scripts exist. It can be run by the test script. And there's a small fixtures folder for test data.

The bootstrap flow itself is a clear pipeline. It starts, runs doctor, then downloads sources, then audits sources, then runs the security check, then sync in dry run mode, then the tests, then updates the status file, then prints a summary, and ends. If source downloading fails because of network problems, the flow simply continues with the local checks and marks the download as blocked or failed.

So when is the project considered valid? When doctor passes, security passes, the source audit passes, the sync dry run passes, the tests pass, bootstrap completes with an accurate summary, no safety rule is violated, the required files and directories exist, and all the JSON parses successfully.

The specification then describes how agent configuration sync should work. The source files are the universal instructions and several rule files. The targets are the Claude file, the Gemini file, and the agents documentation, with possible future targets for Cursor, Copilot, and OpenCode. The sync script reads the sources, checks the targets for the required safety language, reports missing or outdated sections in dry run mode, updates the managed sections in write mode, and carefully avoids deleting any user written content outside those managed sections. Those managed sections are wrapped in clear begin and end markers so everyone knows what's generated and what isn't.

On source download policy, the thinking is that external repositories can be genuinely useful as examples and documentation references, but they are not trusted project code. So the project may clone enabled repositories, read their documentation, record their commit hashes, reference them in audits, and compare their configuration patterns. But it must not execute their scripts, install their dependencies, copy their code into runtime paths automatically, treat them as trusted dependencies, commit large downloaded repositories by default, use any secrets found inside them, or modify them unless explicitly asked. And every downloaded source must live inside the external sources folder, with nothing downloaded outside it.

The security policy is equally firm. No secrets may be committed, and that explicitly includes API keys, tokens, private keys, session tokens, database passwords, and cloud credentials. Environment files are ignored, though an example file with placeholders only may be created. Scripts must avoid dangerous shell behavior like piping remote scripts into a shell, recursive deletion, elevated deletion, or opening up permissions across the system. And scripts must validate paths, never writing outside the project root unless the user explicitly provides a target for project creation, with downloads always staying inside external sources.

After major operations, the status file must be updated, using clear values like pass, fail, blocked, skipped, or unknown. It should include the project name and purpose, the last updated timestamp, the bootstrap status, known issues, and next steps.

When the agent finishes implementing or generating a project, it must give an honest final report. That report names the project and its path, lists what was generated, and states the result of each validation step, whether it passed, failed, was blocked, or simply wasn't run. It lists known issues, or says there are none, and gives the next command to run. And there's a strict honesty rule: the agent must never claim a command passed unless it actually ran successfully.

The specification sets a minimal bar for acceptance. A generated project is only minimally acceptable if it includes the contract, the readme, the package file, the agent instruction files, the task and status files, the manifests and the lock file, all the core scripts, and the folders for rules, skills, hooks, configurations, memory, documentation, tests, and external sources with their placeholder. If any of these are missing, the implementation is simply incomplete.

There's a recommended order of work as well: create the directories, then the root metadata files, the manifests, the rules, the skills, the hooks, the memory files, the documentation, the package file, the utility scripts, the validation scripts, the bootstrap script, the create project script, and the tests. Then run doctor, run security, run the tests, run bootstrap, update status, and report the final result.

The boilerplate is also clear about what it is not. It does not require a web app, a backend server, a database, a frontend framework, TypeScript, Docker, Kubernetes, cloud deployment, runtime third party dependencies, or automatic global configuration of AI tools. Downstream projects are free to add those, but the boilerplate itself doesn't demand them. And there are obvious extension points where teams can grow the project, adding source folders, app folders, packages, services, web and api layers, command line tools, configuration, and tooling like type checking, linting, formatting, test runners, containers, and continuous integration and deployment workflows. Whenever those are added, the documentation and tests should grow with them.

Finally, the file closes with two example prompts, one for creating a fresh project like a sample called abc and one for self bootstrapping the boilerplate itself, and a firm closing instruction to any coding agent reading it. Don't summarize this file instead of implementing it. Don't create only markdown. Don't skip the executable scripts, the tests, or the safety checks. Don't execute downloaded third party code. Don't modify files outside the project. If you can't run commands, say so clearly. If the network is unavailable, mark source downloading as blocked. Always update the status file. And always finish with an honest, complete report of the real validation results. That, in essence, is the entire promise of project starter: a contract you can read once and turn into a real, working, safe, agent ready project.
