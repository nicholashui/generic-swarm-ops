设想一下，你为一个软件项目积累了极其庞大的知识。功能规格、需求文档、用户故事、架构笔记、API 合约，还有每一个界面的描述。体量随时跨越几百个文件。现在你想让一个 AI 编程代理真正根据这一切去构建出来，问题就在这里爆发了。如果你直接把一大堆东西丢给代理，它会顶不住。即使是最大型的模型，那些一次能吸纳远超过十万个 token 的模型，也会开始失去头绪。它们会把关键细节淹没在噪声里，失去连贯性，结果也跟你最初要求的内容越走越远。

这正是 SIPA 要解决的问题。SIPA 就是软件实现规划代理，今天我们一起走过的，是它第二版的完整设计。你可以把它当成一份权威而详尽的蓝图，详细到你可以交给一位工程师，或者交给另一个 AI 系统，然后告诉它，照着这个去构建。

让我们从为什么那些显而易见的做法行不通讲起。第一个问题就是上下文长度的硬性限制，这一点我们刚才已经提过。但还有一个更深层、更微妙的问题，而这才是整个设计的核心。软件项目的不同部分，需要的是本质上完全不同种类的细节。一份高层次的架构计划，需要的是策略性的摘要、关键的决策、宏观的视图。一个具体的用户界面界面，需要的恰恰相反，它要深入而狭窄聚焦的细节，每一条验收标准、每一个流程、每一个边界情况。一个共用程序库或者组件，需要的又不一样。它需要清晰的接口合约、扩展点，以及从众多文件之中综合出来的共通行为。如果你用一种通用的摘要方式去对待所有这些东西，你就会摧毁它的价值。你要么把界面埋葬在含糊的泛论之中，要么就把架构淹没在无关痛痒的琐碎细节里。

除此之外，粗糙的做法还会失去可追溯性。代理会出现幻觉、漏掉交叉引用，慢慢偏离最初的意图。而且当你想让多个编程代理或者多个人并行工作的时候，这些做法根本无法扩展。

那么 SIPA 的答案是什么呢？它是一个分层式的多代理规划器。让我拆开来讲。它会用一个混合式的检索系统，智能地吸纳你的整套文件，那个系统结合了语义搜索、关键词搜索，以及可选的知识图谱。接着它会按照类型把每一个组件分类，而这个分类就是关键。它会触发专门的子规划器，每一个都会为那种组件产出恰到好处的抽象层次和细节程度。在整个过程之中，一个内嵌的评审员会检查工作成果，为可追溯性评分、检查一致性、搜寻反模式，并且推动计划经过一轮又一轮的修订循环，直到通过一系列的质量关卡为止。最终的输出是干净、有版本管理、对 git 友好的 Markdown。主计划、按类型划分的规格，以及细致的任务清单，每一份都是自足的，而且大小恰到好处，让下游的编程代理可以直接使用。

最大的差异化之处，就是那个组件分类法。SIPA 不是做通用的分解，而是用一套明确的分类，架构与策略类、功能与界面与战术类、共用与共享与运维类等等，去驱动完全不同的检索、摘要、综合和输出结构。这是一个简单但有力的洞见的实际体现。软件开发确确实实在不同的地方，需要不同程度的实现细节。

那么回报是什么呢？这个设计的目标，是把编程代理需要处理的有效上下文大小减少五倍到二十倍，同时实际上提升保真度。减少返工的循环、提升成功率，并且即使最初是来自一大堆规格，也能够保留原有的意图。

SIPA 并不是凭空发明出来的。它是对二零二五和二零二六年最顶尖的研究和工具的一次审慎综合。让我给你感受一下它的根基。其中之一是规格驱动开发，也就是规格应该成为核心而具生命力的唯一真相来源这个理念。GitHub 的 Spec Kit 推广了一套分四个阶段、设有关卡的工作流程。首先你明确高层次的意图。接着你规划技术栈和架构。然后你把它拆成细小、可审阅的任务。最后编程代理逐一实现这些任务，并设有人类的检查点。SIPA 正是为了预先已经存在的庞大文件集合这种情况，去实现和延伸规划与任务这两个阶段。

接着是关于分层式多代理系统的研究。一个叫 AgentOrchestra 的框架，演示了一个顶层的规划代理，把复杂的目标分解成子目标，再委派给专门的子代理。Self-Organized Agents 则展示了母代理管理高层次的抽象，而子代理就处理细致的子任务。其他系统，例如 GoalAct、ALMAS 和好几个，探讨了持续更新的全局计划配合分层式执行，以及敏捷式的故事拆解。SIPA 采用了这种顶层规划器加专门子规划器的模式。

影响最直接的一份研究，是一个叫 MAAD 的东西，也就是多代理架构设计。它把软件需求转化为全面而可追溯的架构蓝图，而且是靠四个专门代理协同合作做到的。一个分析员会解析需求，抽取出功能性、非功能性，以及在架构上具重大意义的需求，并且一路标注可追溯性。一个建模员会把这些需求对应到经典的四加一架构视图，并生成图表，过程中靠检索去引入经过验证的模式和标准。一个设计员会综合出可投产的文档，可追溯性矩阵、API 规格、部署配置，以及每一个抉择背后的依据。还有一个评估员，也就是评审员，坐镇每一个阶段，验证完整性和一致性、捕捉反模式、进行权衡分析，然后产出附有严重程度和补救建议的报告，去触发迭代式的修订。MAAD 还引入了一套三层记忆系统，工作记忆、情节记忆和语义记忆，并且证明了注入外部架构知识可以明显减少幻觉。SIPA 把整条管线改用在它的架构子规划器上，并把评审员、记忆和检索的模式推广到每一种组件类型。

根基里面还有更多东西。针对软件工程的多代理系统的广泛综述，从粗糙检索走向如今所谓的上下文工程，也就是写入、挑选、压缩和隔离上下文这些动态策略。还有来自 xAI 的 Grok Build 的启发，那个编程代理用多代理协调，配合一个规划、搜索、构建的工作流程，还有一个竞技场模式，在人类审阅之前自动评估和排名互相竞争的输出。SIPA 的计划就是设计得可以直接喂入这类工具的。

让我列出把一切串联起来的设计原则。第一，分层式分解结合按类型的专门化。第二，把上下文工程视为头等大事，让每一份产物都使用最小而信号最强的上下文，永远不会用上整个语料库。第三，由组件类型分类法去驱动一切。第四，一个内嵌的评审员，配备真正的质量关卡和迭代式修订，在未通过之前没有东西算定稿。第五，具备完整可追溯性的生命力规格，每一个决策都连回它的来源，而实现的反馈会反流回去更新计划。第六，真真正正可投产的工程，配备结构化输出、错误处理、日志记录，以及在关键之处设有人类介入的关卡。第七，对现有框架原生友好的输出，可以直接放入你的工作流程。第八，一个能够自我改善并可扩展的系统，可以反思自己的表现，并且轻易接纳新的组件类型。

现在让我们更贴近地看看那个分类法，因为它真的是整件事的引擎。SIPA 维护着一份明确而可扩展的组件类型清单，每一种都带着它自己的规则。以架构或者策略类型为例。它在一个高抽象层次运作。检索会在整个语料库之中广泛扫描横切面的关注点和在架构上具重大意义的需求，并引入模式和标准。重点是全局综合和关键决策，输出则强调视图、决策记录、接口、质量属性和依据。现在对照功能或者界面类型。它在一个中高层次运作，细致但范围紧凑。检索很狭窄，按功能名称和相关需求标识符去搜索，抽取相关的完整段落。输出强调用户故事、详尽的验收标准、流程、状态机、API 合约和边界情况。

共用或者共享组件类型聚焦在接口和可重用性上。它会搜索接口定义和使用模式，综合共通行为和扩展点，并输出公开合约、配置纲要、不变量和使用范例。接着是数据和领域模型类型，聚焦在实体、它们之间的关系和生命周期规则上。集成和外部类型，以 API 合约、错误处理和兼容性为中心。还有基础设施和运维类型，涵盖部署、可观测性和扩展。在这些主要类型之上，还有次要标签，例如安全关键、性能敏感或者面向用户，会调整评审员的评分标准，并在需要的地方加入额外章节。分类是靠一个 AI 分类器配合简单规则来做，根据文件里面的措辞和结构去判断。

让我们走一走运行中的系统架构。坐在最顶端的是主协调器，也就是元规划器。它接收高层次的目标和语料库索引，把项目分解成阶段、史诗和模块，为每一个组件分类，构建依赖图，并生成附有工作量、优先级和风险估算的主路线图。接着它会孵化并协调那些专门的子规划器，管理全局状态和记忆，运行一次高层次的评审，并且可以选择暂停去做人类关卡。

在它下面，在可以并行的时候就并行运作的，是子规划器。架构子规划器，仿照 MAAD，运行它的分析、建模和设计阶段，并检索模式。功能与界面子规划器做深入而范围紧凑的检索，并扩展故事、验收标准、流程和合约。共用与共享组件子规划器综合合约、配置和扩展点，并撰写使用范例。它们的草稿全部流入评估员，也就是评审员，由它为可追溯性评分、检查一致性、对照重大需求衡量完整性、检测反模式、进行权衡分析、建议补丁，并作出关卡决策。反馈会循环回去，直到工作通过为止。一旦通过，任务分解器就接手，把经过验证的计划拆成细小、可审阅的单元，每一个都有自己的验收标准、上下文摘录和排序。

在这一切之下，坐镇着共用的基础设施。一个语料库索引器和检索引擎，处理语义分块、嵌入、混合搜索和元数据标注。一个分层记忆管理器，配备它的三层。工作记忆用来存放当前的产物和活跃的上下文。情节记忆用来存放每个模块的历史、它的决策、修订轨迹和汲取的教训。而语义记忆用来存放通用化的模式、术语表条目和可重用的依据。还有一个可选但建议采用的知识图谱，其中节点是需求、组件和决策，而边就捕捉各种关系，例如追溯到、依赖于、精炼、冲突于和实现。那个图谱让影响分析和更丰富的检索成为可能。还有一个结构化输出强制器，保证每一个代理都产出经过验证的、带类型的产物。

让我更具体地描述每一个代理的角色。主协调器吸纳目标和已索引的语料库，做分解和分类，生成附有优先级、风险和工作量估算的主计划，协调子规划器，维护全局可追溯性矩阵，运行一次一致性和完整性检查，并就策略性产物提供一个人类审阅关卡。它的提示人格，基本上就是一个专家级的软件项目经理兼架构师，被要求去分解项目、为每一个模块分类、为最小可行产品排优先级，并以严格的结构化格式输出。

架构子规划器会走过三个阶段。在分析阶段，它抽取并验证相关需求，并确保可追溯性。在建模阶段，它生成多个架构视图，逻辑、流程、开发和实体，并通过检索引入模式。在设计阶段，它综合出文档、依据、接口、部署考量和权衡分析。它在每一个子阶段都跟评审员协作，并产出一份附有结构化前置数据和清晰划分章节的架构文档。

功能与界面子规划器会对单一功能做深入而狭窄的检索，从原始素材之中精炼并扩展用户故事和验收标准，综合交互流程和状态机，并抽取精确的 API 合约、载荷、验证规则和错误情况。它刻意把全局上下文保持在最低限度，并聚焦在这一个界面或者功能的深度上。它的标志就是详尽的验收标准，写到可以直接转化为测试。

共用与共享组件子规划器会从整个语料库之中收集散落的提及，并综合成连贯的接口合约、类型、方法和事件。它定义配置点和默认值，记录扩展点和钩子，捕捉不变量、保证和错误处理，并提供使用范例和可测试性笔记。这往往是所有类型之中最需要综合的一种，因为它要把来自很多地方的碎片拼合在一起。

评估员，也就是评审员，是内嵌而多阶段的。它为可追溯性评分，确保每一个重大元素都连回附有证据的来源章节。它检查计划之内和计划之间的一致性。它对照相关需求衡量完整性。它检测反模式，例如万能对象、紧密耦合、缺失的边界情况或者含糊的合约。它在相关的地方进行权衡分析，并产出附有严重程度和建议补丁的结构化反馈，然后作出关卡决策，通过、修订，或者升级给人类处理。它可以是一个配备评分标准的强大模型，或者是一小队专门的评审员，并且结合了以模型作裁判的评分和以规则为本的检查。

而任务分解器把经过验证的计划拆成细小、聚焦、可审阅的任务，每一个都大小恰好可以放入单一上下文窗口，并代表一个有清晰验收标准的逻辑改动。它嵌入或者链接相关的摘录，为任务排序，标注依赖关系，并建议测试和审阅重点。

现在让我们从头到尾追踪一次完整的工作流程。它从第零阶段开始，吸纳和索引，设计得对增量友好。系统扫描规格目录，沿着标题层级和语义边界智能地分块，把所有东西连同丰富的元数据一起嵌入，可选地抽取实体去构建知识图谱，并生成语料库层级的摘要和术语表。输出是一个可搜索的索引，而且每当文件有变动，它都可以自动重新运行。

第一阶段是高层次分解和主计划。协调器接收目标和索引，分解项目，为模块分类，构建依赖和风险视图，并生成主计划，连同它的史诗、阶段、优先级、关键路径，以及一个可追溯性矩阵的骨架。评审员审阅它，接着有一个可选的人类关卡，然后就为优先的模块孵化子规划器，在它们互相独立的时候就并行进行。设想一个教育平台的主计划。它可能把第一阶段定为核心领域和身份验证，也就是基础，然后把第二阶段定为面向用户的仪表盘，并列明依赖关系，核心领域喂入身份验证服务，再由它喂入所有界面功能。

第二阶段是核心循环，按类型而有范围的规划，配合评审循环。对于每一个模块，系统会组装一个最小的上下文套件。分类器确认类型，混合检索以一个按类型的查询策略去运行，并引入相关的记忆。合适的子规划器用它的模板去执行。接着内嵌的评审员运行它的各个阶段，可追溯性、一致性、完整性、反模式，并发出结构化反馈。如果工作低于关卡，子规划器就会收到针对性的补丁指示，并只重新运行受影响的章节。一旦通过，产物就会连同完整的前置数据和可追溯性写入磁盘，而全局矩阵和情节记忆都会更新。让我给你感受一下评审交互是怎么进行的。一份草稿生成了。评审员指出，比如说事件驱动通知的可追溯性只链接到一个章节，漏掉了一条可靠性需求，而部署视图又欠缺可观测性的战术。它发出一条补丁指示，加回那条漏掉的链接，并检索可观测性模式，用几个具体的战术去更新部署章节。经过修订的草稿之后就通过关卡。

第三阶段是细致任务生成。任务分解器吸纳经过验证的计划，并产出任务文件，内含细小的工作单元，每一个都引用父规格的具体章节，嵌入相关的摘录或者精确的链接，列明直接从计划抽取的清晰验收标准，并建议测试和审阅重点，全部都经过排序并考虑依赖关系。一个单一的任务可能写着，构建学生仪表盘的页首组件，依据功能规格的这几个具体章节，以及架构文档里面的合约，验收标准例如要呈现头像和通知铃铛、在本地处理下拉菜单而不需要额外的 API 调用，并在手机上保持响应式，另外还有建议的单元测试和端到端测试。

第四阶段是实现、反馈和生命力更新。任务和计划喂入编程代理。它们的成果，差异、测试结果、审阅意见，以及计划原来有些含糊的地方，都会流回 SIPA。情节记忆会更新，在需要的地方触发针对性的重新规划，而语义记忆会随着发现新模式而演进。评审员会在有变动的产物上重新运行，而可追溯性会保持生命力。在增量模式之下，当有新规格到来，系统会通过图谱检测出究竟哪些模块受影响，并只重新规划那些。

讲一点数据模型。每一份产物都使用一致的结构化前置数据，后面跟着 Markdown 章节，并通过纲要验证去强制执行。前置数据记录着例如产物类型、组件、抽象层次、版本、可追溯性链接、依赖关系、评审分数、最后修订日期、状态和任何标签。每一种产物类型都有它自己的完整纲要。主计划载有路线图、阶段拆解、模块清单、依赖图、风险登记册和可追溯性骨架。架构规格载有执行摘要、视图、决策记录、接口目录、质量属性场景、部署笔记和依据。功能规格载有经过精炼的故事、详尽的验收标准、交互流程、界面状态、附有范例的 API 合约、验证和错误矩阵，以及线框图提示。共用组件规格载有目的和范围、公开接口、配置纲要、扩展点、不变量、错误处理、性能笔记、使用范例和迁移笔记。任务清单载有排好优先级和顺序的任务。而评审则载有结构化的分数和问题。一切都设计得既方便人类阅读，也方便机器使用。

让我们在检索和记忆架构上讲得深入一些，因为上下文工程是如此核心。吸纳时，系统主要按标题层级和语义边界去把文件分块，其次靠抽取需求标识符和标注组件提及，并为每一块盖上元数据。检索是混合而按类型的。它结合了用来捕捉含义的稠密向量相似度、用来捕捉精确标识符和技术词语的关键词搜索，以及用来捕捉关系的可选知识图谱遍历，然后在组装之前重新排名和压缩排在前面的段落。查询本身会按类型而有不同的扩展。对于架构，系统会加入关于重大需求、模式和战术的词语。对于界面，它会加入关于验收标准、流程和边界情况的词语。上下文组装以最小而充足的套件为目标，也就是检索到的段落，加上一个压缩过的摘要，加上相关的记忆，再加上必要的主计划摘录，而每一次子规划器的调用都会得到它自己的、隔离而聚焦的上下文。三层记忆各司其职，工作记忆处理进行中的东西，情节记忆处理每个模块的结构化历史，让你可以问自上一个版本以来有什么变了，而语义记忆则处理关于模式和陷阱的通用化、可检索知识。而建议采用的知识图谱把需求、组件、决策和计划绑在一起，让你可以问，如果我改这条需求，会影响哪些计划？

评审员子系统也值得看深入一些。它运行多个阶段，可以并行或者顺序进行。一个可追溯性评审员检查每一个重大主张、决策、视图和 API，是否都有附带证据地明确链接到来源章节。一个一致性评审员检查内部矛盾、与主计划及兄弟计划的对齐，以及除非有正当理由改动，否则与先前版本的一致性。一个完整性评审员确认已涵盖所有相关需求，并搜寻有没有遗漏的故事或者边界情况。一个反模式和质量评审员搜寻万能对象、紧密耦合、缺失的错误处理、含糊的扩展点和单薄的验收标准。一个可实现性和规模评审员问，计划的范围是不是划得让任务可以放入编码者的上下文窗口，以及验收标准是不是可测试。而一个权衡评审员，主要用在架构上，识别质量属性场景、权衡和风险。反馈以一种结构化的形式回来，附有总体裁决、分数、一份问题清单，每一条都附有严重程度和建议补丁，还有一份优点清单和一个建议的下一步行动。修订循环刻意保持最小化，子规划器只施加针对性的补丁，重新生成受影响的章节，而评审员再检查，理想情况下只检查有变动的部分。当关卡通过，例如可追溯性分数至少达到九成而又没有高严重程度的问题，或者当人类推翻决策时，就达成收敛。关卡本身很具体。架构必须有至少三个视图，以及几份附有依据和对主要需求的可追溯性的决策记录。每一个用户故事必须有至少三条验收标准，涵盖正常路径和至少两个边界或者错误情况，而所有被引用的合约都必须真实存在。而一个共用组件必须有一个完整指定的公开接口、至少一个有记录的扩展点，以及一个使用范例。

讲到提示工程，系统提示确立角色、原则、分类法和输出纲要，把每一个代理塑造成一个专家级的专门人员，遵循严格的可追溯性，只产出被要求的抽象层次，并且绝不杜撰没有支持的细节。每一个代理都内建一条重新思考的指示，要求它在产出最终输出之前，先在内部对照评分标准去批判自己的草稿。少样本范例，每种组件类型两到四个高质量的，要么放在提示里面，要么从语义记忆之中动态检索。输出靠纲要验证去强制执行，而对于以 Markdown 为主的产物，通常先生成结构再渲染成模板会更可靠。温度设置按任务调校，创意规划和综合用中等温度，评审员用接近零的温度让它的评分保持确定性，而修订补丁就用低温度。推理能力最强的模型留给协调器、架构子规划器和顶层评审员，而较轻、较快的模型就处理检索、简单分类和大量的任务分解。

还有关于输出模板的指引，一套共用的章节会横跨各种类型出现，前置数据、一个量身定制的摘要、可追溯性和来源引用、详细的主体、依赖和风险、留给人类审阅的未决问题和假设，以及一个自动追加的变更记录。还有一个分阶段的实现路线图。最小可行产品，目标一到两周，构建索引器和基本的混合检索、配备一个子规划器的协调器、一个配备简单循环的基本评审员，以及结构化输出，然后在一个细小的模块上做端到端测试，并把输出接入一个现有的编程工作流程。核心完整性阶段加入所有子规划器、完整的分类法、三层记忆、进阶评审员、一个轻量的知识图谱、增量模式和完整的模板库。投产加固阶段加入可观测性、人类介入的关卡、通过缓存和模型路由去做的成本和 token 优化、并行执行和重试逻辑、一个指标仪表盘，以及打包成一个可重用的组件。

建议的技术栈配合一个务实而现代的环境。一个有状态的图式协调框架去处理分层循环、一个配备本地向量数据库的进阶索引和检索程序库、一个可选的图式数据库去做知识图谱、用来做规划和评审的强推理模型配合用来应付大量工作的较廉价本地模型、经纲要验证的结构化输出、Markdown 原生的图表，以及一个 git 仓库作为所有计划和产物的家，让你免费就有版本管理和差异比对。建议的代码结构把索引器、各个代理、记忆管理器、检索层、纲要、模板、提示和工具区分开，并建议先以单一文件的原型起步，等它行得通之后再重构成一个软件包。

集成是设计得无摩擦的。你把系统指向你现有的规格文件夹，它会尊重你的包含和排除清单，并把主计划、按类型的规格和任务文件直接放入你的项目仓库，在那里编程代理会像对待手写的任务文件一样去使用它们。编码者的会话日志、测试结果和拉取请求的意见，都可以喂回情节记忆。这些计划是规划搜索构建代理和它的竞技场式评估的绝佳输入。一切都可以用本地模型和本地向量数据库在本地运行，只在值得的地方才保留重量级推理给外部 API。而可选的持续集成钩子可以在规格变动时重新索引、重新规划受影响的模块，或者对照当前的计划验证新的代码。

你怎么知道它行得通呢？在量化方面，你追踪可追溯性覆盖率、计划通过关卡之前平均的修订迭代次数、相比起倾倒整个语料库所减少的上下文 token、编程代理的下游成功率、从规格变动到计划更新所需的时间，以及评审分数在项目生命周期之中的趋势。在质化方面，真正的问题是，一个计划是不是让人感觉它完整而准确地捕捉了一个模块的原始规格意图，而又不逼你重新读完所有东西，以及编程代理或者初级开发者是不是可以比看原始规格更快、更少澄清地从这些计划去实现。对于最小可行产品，目标是已核准计划上至少八成半的可追溯性、平均少于三次的修订迭代、上下文相关失败的可衡量下降，以及来自第一个真实模块的正面反馈。

没有一个设计是完整到不用面对它自己的风险。在杂乱语料库上的糟糕检索，靠在分块、元数据和混合搜索上的前期投入去应对，并从结构良好的子集入手。过度或者不足的分解，靠规模评审员和人类关卡去捕捉，目标是让任务恰好可以放入单一个聚焦的工作环节。模块之间的不一致，靠强大的语义记忆、全局一致性检查和知识图谱去抗衡。幻觉靠结构化纲要、评审员关卡、低温度评估，以及只准使用提供或者检索到的上下文这条严格指示去遏制，而版本管理让任何偏移都显而易见。大型项目上的高成本靠有范围的检索、压缩、模型路由和缓存去管理。人类审阅的瓶颈靠让关卡可配置、为高评审分数之上的低风险模块自动核准，以及提供干净的差异和摘要去缓解。而即使是遗留或者结构欠佳的规格也受益，因为系统强制做明确的分类和可追溯性，虽然它们一开始可能需要更多人类的种子整理。至于边界情况，很小的项目仍然会在一致的结构和生命力可追溯性上得益。高度视觉化的规格会得到一个视觉步骤，为功能规划器描述那些模型图。互相冲突的需求会明确地浮现出来给人类去解决。而急速演变的规格就依靠增量模式和影响分析。

展望将来，路线图构想出一个完全自动化的知识图谱检索，配备实时影响分析、能够读取界面模型图和图表的多模态吸纳、直接从验收标准生成测试和验证产物的自动化、能够自动升级最棘手计划的风险和复杂度评分、一个分析历史运行并演进它自己模板和策略的元规划器、一道通往形式化方法的桥梁以服务高保证的模块、团队和企业功能例如基于角色的访问和导出到项目工具，以及配备预制分类法的领域特定扩展，去服务教育科技、交易界面和嵌入式硬件等领域。

这个设计取材自一大批参考文献，关于桥接需求和架构的 MAAD 论文、关于分层式多代理框架的 AgentOrchestra、针对软件工程的多代理系统的广泛综述，以及一整个家族的分层式规划论文，旁及上下文工程文献、规格驱动开发工具、Grok Build 文档，以及基础性的架构著作例如四加一视图和权衡分析方法。

最后总结，这份规格刻意做到详尽无遗，让它可以充当唯一真相来源，无论是从零开始构建 SIPA，还是演进一个现有的元代理系统。它直接应对核心挑战，把大型规格拆成可管理、细节恰当的部分，同时通过可追溯性、评审驱动的质量和生命力更新去保留意图。从这里开始，你可以把这份文档本身喂入一个代理框架去开始构建、为最小可行产品做原型，或者要求提供具体的提示模板、纲要或者起步代码。它是工程化到去复合你在 AI 代理工程、规格驱动开发和严谨框架构建上的现有强项，并且是以同样的哲学去打造，深入思考、强制执行质量关卡、维持可追溯性，并且始终保持务实。
