Imagine you've built up an enormous body of knowledge about a software project. Functional specifications, requirements documents, user stories, architecture notes, API contracts, descriptions of every screen. It might span hundreds of files. Now you want an AI coding agent to actually build from all of that. And here's where things fall apart. If you just hand the whole pile to the agent, it chokes. Even the largest models, the ones that can take in well over a hundred thousand tokens at once, start losing the thread. They drop critical details into the noise, they lose coherence, and the result drifts away from what you actually asked for.

This is the problem that SIPA was designed to solve. SIPA stands for the Software Implementation Planner Agent, and what we're walking through today is version two of its complete design. Think of it as a definitive, exhaustive blueprint. It's detailed enough that you could hand it to an engineer, or to another AI system, and say, build this.

Let's start with why the obvious approaches don't work. The first issue is the hard limit on context length, which we just touched on. But there's a deeper, more subtle problem, and it's the real heart of this design. Different parts of a software project need fundamentally different kinds of detail. A high-level architecture plan needs strategic summarization, the key decisions, the big-picture views. A specific user interface screen needs the opposite. It needs deep, narrowly focused detail, every acceptance criterion, every flow, every edge case. A shared library or component needs something different again. It needs clear interface contracts, the extension points, the common behaviors pulled together from across the documents. If you treat all of these the same way, with one generic style of summarization, you destroy the value. You either bury the screen in vague generalities or you drown the architecture in irrelevant minutiae.

On top of that, naive approaches lose traceability. The agent hallucinates, misses cross-references, and slowly drifts from the original intent. And they simply don't scale when you want multiple coding agents or multiple humans working in parallel.

So what is SIPA's answer? It's a hierarchical, multi-agent planner. Let me unpack what that means. It ingests your entire collection of documents intelligently, using a hybrid retrieval system that combines semantic search, keyword search, and optionally a knowledge graph. It then classifies each component by its type, and that classification is the key. It triggers specialized sub-planners, each producing exactly the right level of abstraction and detail for that kind of component. Throughout the process, an embedded critic checks the work, scoring traceability, checking consistency, hunting for anti-patterns, and pushing the plan through revision loops until it passes a set of quality gates. The final output is clean, versioned, git-friendly Markdown. Master plans, type-specific specs, and granular task lists, each one self-contained and sized so that a downstream coding agent can consume it directly.

The single biggest differentiator is that component taxonomy. Instead of generic decomposition, SIPA uses an explicit classification, architecture and strategic, feature and interface and tactical, common and shared and operational, and so on, to drive completely different retrieval, summarization, synthesis, and output structures. This is the practical realization of a simple but powerful insight. Software development genuinely needs different levels of implementation detail in different places.

What's the payoff? The design targets a five to twenty times reduction in the effective context size that coding agents have to deal with, while actually increasing fidelity. Fewer rework cycles, higher success rates, and the original intent preserved even when it came from a massive pile of specs.

Now, SIPA wasn't invented from nothing. It's a careful synthesis of the best research and tooling from twenty twenty-five and twenty twenty-six. Let me give you a sense of the foundations. There's spec-driven development, the idea that specifications should be the central, living source of truth. GitHub's Spec Kit popularized a four-phase gated workflow. First you specify the high-level intent. Then you plan the stack and architecture. Then you break it into small, reviewable tasks. Then coding agents implement those tasks one by one with human checkpoints. SIPA implements and extends the plan and tasks phases specifically for the case of pre-existing, massive document collections.

Then there's the research on hierarchical multi-agent systems. A framework called AgentOrchestra demonstrated a top-level planning agent that decomposes complex goals into sub-goals and delegates to specialized sub-agents. Self-Organized Agents showed mother agents managing high-level abstractions while child agents handle the detailed subtasks. Other systems, GoalAct, ALMAS, and several more, explored continuously updated global plans paired with hierarchical execution and agile-style story breakdown. SIPA adopts this top-planner plus specialized sub-planner pattern.

The most directly influential piece of research is something called MAAD, short for Multi-Agent Architecture Design. It transforms software requirements into comprehensive, traceable architectural blueprints, and it does so with four specialized agents working together. An Analyst parses the requirements and extracts the functional, non-functional, and architecturally significant ones, annotating traceability as it goes. A Modeler maps those requirements to the classic four-plus-one architectural views and generates diagrams, using retrieval to pull in proven patterns and standards. A Designer synthesizes production-ready documentation, traceability matrices, API specs, deployment configs, and the rationale behind every choice. And an Evaluator, the critic, sits at every stage, validating completeness and consistency, catching anti-patterns, and running tradeoff analysis, then producing reports with severity levels and remediation suggestions that trigger iterative refinement. MAAD also introduced a three-layer memory system, working, episodic, and semantic, and showed that injecting external architecture knowledge measurably reduced hallucinations. SIPA adapts this whole pipeline for its architecture sub-planner and generalizes the critic, memory, and retrieval patterns across every component type.

There's more in the foundations. Broad surveys of multi-agent systems for software engineering, the move from naive retrieval toward what's now called context engineering, the dynamic strategies of writing, selecting, compressing, and isolating context. And there's inspiration from xAI's Grok Build, a coding agent that uses multi-agent orchestration with a plan, search, build workflow and an arena mode that automatically evaluates and ranks competing outputs before a human reviews them. SIPA's plans are designed to feed directly into tools like that.

Let me lay out the design principles that hold it all together. First, hierarchical decomposition combined with type-aware specialization. Second, treating context engineering as a first-class citizen, so every artifact uses the smallest possible high-signal context and never the full corpus. Third, the component-type taxonomy driving everything. Fourth, an embedded critic with real quality gates and iterative refinement, where nothing is final until it passes. Fifth, living specs with full traceability, where every decision links back to its source and implementation feedback flows back to update the plans. Sixth, genuinely production-ready engineering, with structured outputs, error handling, logging, and human-in-the-loop gates where they matter. Seventh, outputs that are native to existing harnesses, ready to drop straight into your workflow. And eighth, a system that's self-improving and extensible, able to reflect on its own performance and easily accept new component types.

Now let's look more closely at that taxonomy, because it really is the engine of the whole thing. SIPA maintains an explicit, extensible list of component types, and each one carries its own rules. Take the architecture or strategic type. It operates at a high abstraction level. Retrieval scans broadly across the corpus for cross-cutting concerns and architecturally significant requirements, and pulls in patterns and standards. The focus is global synthesis and key decisions, and the output emphasizes views, decision records, interfaces, quality attributes, and rationale. Now contrast that with the feature or interface type. It works at a medium-high level, detailed but tightly scoped. Retrieval is narrow, searching on the feature name and related requirement identifiers, pulling full relevant paragraphs. The output emphasizes user stories, exhaustive acceptance criteria, flows, state machines, API contracts, and edge cases.

The common or shared component type focuses on interfaces and reusability. It searches for interface definitions and usage patterns, synthesizes common behaviors and extension points, and outputs public contracts, configuration schemas, invariants, and usage examples. Then there's the data and domain model type, focused on entities and their relationships and lifecycle rules. The integration and external type, centered on API contracts and error handling and compatibility. And the infrastructure and operations type, covering deployment, observability, and scaling. On top of these primary types, there are secondary tags, things like security-critical, performance-sensitive, or user-facing, that adjust the critic's rubrics and add extra sections where needed. Classification happens through a mix of an AI classifier and simple rules, keying off phrases and structures in the documents.

Let's walk through the architecture of the running system. At the top sits the Master Orchestrator, the meta-planner. It takes in the high-level goal and the corpus index, decomposes the project into phases, epics, and modules, classifies every component, builds a dependency graph, and generates the master roadmap with effort, priority, and risk estimates. It then spawns and coordinates the specialized sub-planners, manages the global state and memory, runs a high-level critic pass, and optionally pauses for a human gate.

Below it, working in parallel where they can, are the sub-planners. The Architecture sub-planner, modeled on MAAD, runs its analyst, modeler, and designer phases with retrieval of patterns. The Feature and interface sub-planner does deep, scoped retrieval and expands stories, acceptance criteria, flows, and contracts. The Common and shared component sub-planner synthesizes contracts, configuration, and extension points, and writes usage examples. Their drafts all flow into the Evaluator, the critic, which scores traceability, checks consistency, measures completeness against the significant requirements, detects anti-patterns, runs tradeoff analysis, suggests patches, and makes the gate decision. Feedback loops back until the work passes. Once it does, the Task Decomposer takes over, breaking validated plans into small, reviewable units with their own acceptance criteria, context excerpts, and sequencing.

Underneath all of this sits shared infrastructure. A corpus indexer and retrieval engine that handles semantic chunking, embeddings, hybrid search, and metadata tagging. A hierarchical memory manager with its three layers. Working memory for current artifacts and active context. Episodic memory for the history of each module, its decisions and refinement traces and lessons learned. And semantic memory for generalized patterns, glossary terms, and reusable rationale. There's an optional but recommended knowledge graph, where nodes are requirements, components, and decisions, and edges capture relationships like traces-to, depends-on, refines, conflicts-with, and implements. That graph enables impact analysis and richer retrieval. And there's a structured output enforcer that guarantees every agent produces validated, typed artifacts.

Let me describe each agent's role a little more concretely. The Master Orchestrator ingests the goal and the indexed corpus, decomposes and classifies, generates the master plan with priorities, risks, and effort estimates, coordinates the sub-planners, maintains the global traceability matrix, runs a consistency and completeness pass, and offers a human review gate on the strategic artifacts. Its prompt persona is essentially an expert software program manager and architect, asked to decompose the project, classify each module, prioritize for a minimum viable product, and output in a strict structured format.

The Architecture sub-planner moves through three phases. In the analyst phase it extracts and validates the relevant requirements and ensures traceability. In the modeler phase it generates the multiple architectural views, logical, process, development, and physical, pulling in patterns through retrieval. In the designer phase it synthesizes the documentation, the rationale, the interfaces, the deployment considerations, and the tradeoff analysis. It collaborates with the critic at each sub-stage and produces an architecture document with structured frontmatter and clearly delineated sections.

The Feature and interface sub-planner performs deep, narrow retrieval on a single feature, refines and expands the user stories and acceptance criteria from the source material, synthesizes interaction flows and state machines, and extracts precise API contracts, payloads, validation rules, and error cases. It deliberately keeps global context minimal and focuses on this one screen or feature in depth. Its hallmark is exhaustive acceptance criteria, written so they can be turned directly into tests.

The Common and shared component sub-planner gathers scattered mentions from across the corpus and synthesizes them into coherent interface contracts, types, methods, and events. It defines configuration points and defaults, documents extension points and hooks, captures invariants and guarantees and error handling, and provides usage examples and testability notes. This is often the most synthesized type of all, because it requires pulling together fragments from many places.

The Evaluator, the critic, is embedded and multi-stage. It scores traceability, making sure every major element links back to source sections with evidence. It checks consistency within a plan and across plans. It measures completeness against the relevant requirements. It detects anti-patterns like god objects, tight coupling, missing edge cases, or unclear contracts. It runs tradeoff analysis where relevant, and it produces structured feedback with severity levels and suggested patches, then makes a gate decision, pass, refine, or escalate to a human. It can be a single powerful model with a rubric, or a small crew of specialized critics, and it combines model-as-judge scoring with rule-based checks.

And the Task Decomposer breaks validated plans into small, focused, reviewable tasks, each sized to fit a single context window and represent one logical change with clear acceptance criteria. It embeds or links the relevant excerpts, sequences the tasks, notes dependencies, and suggests tests and review focus.

Now let's trace the complete workflow from end to end. It begins with phase zero, ingestion and indexing, designed to be incremental-friendly. The system scans the spec directory, chunks the content intelligently along heading hierarchies and semantic boundaries, embeds everything with rich metadata, optionally extracts entities to build a knowledge graph, and generates corpus-level summaries and a glossary. The output is a searchable index, and it can re-run automatically whenever the documents change.

Phase one is high-level decomposition and the master plan. The orchestrator receives the goal and the index, decomposes the project, classifies the modules, builds a dependency and risk view, and generates the master plan with its epics, phases, priorities, critical path, and the skeleton of a traceability matrix. The critic reviews it, there's an optional human gate, and then sub-planners are spawned for the priority modules, in parallel where they're independent. Picture a master plan for, say, an education platform. It might lay out phase one as the core domain and authentication, the foundation, then phase two as the user-facing dashboards, with explicit dependencies, the core domain feeding the auth service, which in turn feeds all the interface features.

Phase two is the core loop, scoped type-aware planning with critic loops. For each module, the system assembles a minimal context package. The classifier confirms the type, hybrid retrieval runs with a type-specific query strategy, and relevant memory is pulled in. The appropriate sub-planner executes using its template. Then the embedded critic runs its stages, traceability, consistency, completeness, anti-patterns, and emits structured feedback. If the work falls below the gate, the sub-planner receives targeted patch instructions and re-runs only the affected sections. Once it passes, the artifact is written to disk with full frontmatter and traceability, and the global matrix and episodic memory are updated. Here's a flavor of how a critic interaction goes. A draft is generated. The critic notes that the traceability for, say, event-driven notifications links to only one section and misses a reliability requirement, and that the deployment view lacks observability tactics. It issues a patch instruction, add the missing link and retrieve observability patterns, update the deployment section with a couple of concrete tactics. The refined draft then passes the gate.

Phase three is granular task generation. The task decomposer consumes the validated plans and produces task files with small units of work, each referencing specific sections of the parent specs, embedding the relevant excerpts or precise links, stating clear acceptance criteria drawn directly from the plan, and suggesting tests and review focus, all sequenced and dependency-aware. A single task might read, build the student dashboard header component, drawing on these specific sections of the feature spec and the contracts in the architecture document, with acceptance criteria like rendering the avatar and notification bell, handling the dropdown locally without an extra API call, and staying responsive on mobile, plus suggested unit and end-to-end tests.

Phase four is implementation, feedback, and living updates. The tasks and plans feed into the coding agents. Their outcomes, the diffs, the test results, the review comments, the places where a plan turned out to be unclear, flow back into SIPA. Episodic memory is updated, targeted re-planning is triggered where needed, and semantic memory evolves as new patterns are discovered. The critic re-runs on changed artifacts, and traceability stays alive. In incremental mode, when new specs arrive, the system detects exactly which modules are affected through the graph and re-plans only those.

A word on the data models. Every artifact uses consistent structured frontmatter followed by Markdown sections, enforced through schema validation. The frontmatter records things like the artifact type, the component, the abstraction level, the version, the traceability links, the dependencies, the critic score, the last refinement date, the status, and any tags. Each artifact type has its own full schema. The master plan carries the roadmap, phase breakdowns, module inventory, dependency graph, risk register, and traceability skeleton. The architecture spec carries an executive summary, the views, the decision records, an interface catalog, quality attribute scenarios, deployment notes, and rationale. The feature spec carries refined stories, exhaustive acceptance criteria, interaction flows, interface states, API contracts with examples, a validation and error matrix, and wireframe prompts. The common component spec carries purpose and scope, the public interface, the configuration schema, extension points, invariants, error handling, performance notes, usage examples, and migration notes. The task list carries prioritized, sequenced tasks. And the critique carries structured scores and issues. Everything is designed to be both human-readable and machine-consumable.

Let's go a little deeper on the retrieval and memory architecture, since context engineering is so central. Ingestion chunks the documents primarily by heading hierarchy and semantic boundaries, secondarily by extracting requirement identifiers and tagging component mentions, and stamps each chunk with metadata. Retrieval is hybrid and type-aware. It combines dense vector similarity for meaning, keyword search for exact identifiers and technical terms, and optional knowledge-graph traversal for relationships, then re-ranks and compresses the top passages before assembly. The queries themselves expand differently by type. For architecture, the system adds terms about significant requirements, patterns, and tactics. For interfaces, it adds terms about acceptance criteria, flows, and edge cases. Context assembly aims for the minimal sufficient package, the retrieved passages plus a compressed summary plus relevant memory plus the necessary master plan excerpts, and each sub-planner invocation gets its own isolated, focused context. The three memory layers do their distinct jobs, working memory for what's in flight, episodic memory for the structured history of each module that lets you ask what changed since the last version, and semantic memory for generalized, retrievable knowledge about patterns and pitfalls. And the recommended knowledge graph ties requirements, components, decisions, and plans together so you can ask, if I change this requirement, which plans does it affect?

The critic subsystem deserves a closer look too. It runs multiple stages, which can be parallel or sequential. A traceability critic checks that every major claim, decision, view, and API has explicit links to source sections with evidence. A consistency critic checks for internal contradictions, alignment with the master and sibling plans, and consistency with prior versions unless a change is justified. A completeness critic confirms coverage of all the relevant requirements and looks for missing stories or edge cases. An anti-pattern and quality critic hunts for god objects, tight coupling, missing error handling, unclear extension points, and thin acceptance criteria. An implementability and sizing critic asks whether the plan is scoped so that tasks will fit a coder's context window and whether the acceptance criteria are testable. And a tradeoff critic, used mainly for architecture, identifies quality attribute scenarios, tradeoffs, and risks. Feedback comes back in a structured form with an overall verdict, scores, a list of issues each with severity and a suggested patch, a list of strengths, and a recommended next action. The refinement loop is deliberately minimal, the sub-planner applies only targeted patches, regenerates the affected sections, and the critic re-checks, ideally just the changed parts. Convergence happens when the gate is passed, with thresholds like a traceability score of at least ninety percent and no high-severity issues, or when a human overrides. The gates themselves are concrete. Architecture must have at least three views and a couple of decision records with rationale and traceability to the major requirements. Every user story must have at least three acceptance criteria covering the happy path and at least two edge or error cases, with all referenced contracts actually existing. And a common component must have a fully specified public interface, at least one documented extension point, and a usage example.

On prompt engineering, the system prompts establish the role, the principles, the taxonomy, and the output schema, casting each agent as an expert specialist who follows strict traceability, produces only the requested abstraction level, and never hallucinates unsupported details. A rethink instruction is built into every agent, telling it to internally critique its own draft against the rubric before producing the final output. Few-shot examples, two to four high-quality ones per component type, are either placed in the prompt or retrieved dynamically from semantic memory. Output is enforced through schema validation, and for Markdown-heavy artifacts it's often more reliable to generate the structure first and then render it into a template. Temperature settings are tuned by task, moderate for creative planning and synthesis, near zero for the critic so its scoring stays deterministic, and low for refinement patches. The strongest reasoning models are reserved for the orchestrator, the architecture sub-planner, and the top-level critic, while lighter, faster models handle retrieval, simple classification, and high-volume task decomposition.

There's guidance on output templates, with a shared set of sections appearing across types, frontmatter, a tailored summary, traceability and source references, the detailed body, dependencies and risks, open questions and assumptions for human review, and an auto-appended changelog. And there's a phased implementation roadmap. The minimum viable product, targeted at one to two weeks, builds the indexer and basic hybrid retrieval, the orchestrator with one sub-planner, a basic critic with a simple loop, and structured output, then tests end to end on a small module and wires the output into one existing coding workflow. The core completeness phase adds all the sub-planners, the full taxonomy, the three-layer memory, the advanced critic, a lightweight knowledge graph, incremental mode, and the full template library. The production hardening phase adds observability, human-in-the-loop gates, cost and token optimization through caching and model routing, parallel execution and retry logic, a metrics dashboard, and packaging as a reusable component.

The recommended tech stack matches a practical, modern environment. A stateful graph orchestration framework for the hierarchical loops, an advanced indexing and retrieval library with a local vector database, an optional graph database for the knowledge graph, strong reasoning models for planning and critique with cheaper local models for volume, schema-validated structured outputs, Markdown-native diagrams, and a git repository as the home for all plans and artifacts so you get versioning and diffing for free. The suggested code structure separates the indexer, the agents, the memory managers, the retrieval layer, the schemas, the templates, the prompts, and the utilities, with a recommendation to start as a single-file prototype and refactor into a package once it works.

Integration is meant to be frictionless. You point the system at your existing spec folder, it respects your include and exclude lists, and it drops the master plan, the type-specific specs, and the task files straight into your project repository, where coding agents consume them exactly as they would hand-written task files. Coder session logs, test results, and pull request comments can be fed back into episodic memory. The plans make excellent input to a plan-search-build agent and its arena-style evaluation. Everything can run locally with local models and a local vector database, reserving heavyweight reasoning for an external API only where it pays off. And optional continuous integration hooks can re-index on spec changes, re-plan affected modules, or validate new code against the current plan.

How do you know it's working? On the quantitative side, you track traceability coverage, the average number of refinement iterations before a plan passes the gate, the reduction in context tokens compared to dumping the whole corpus, the downstream success rate of coding agents, the time from a spec change to updated plans, and the trend of critic scores over the life of the project. On the qualitative side, the real question is whether a plan feels like it fully and accurately captured the intent of the original specs for a module without forcing you to re-read everything, and whether coding agents or junior developers can implement from these plans faster and with fewer clarifications than from the raw specs. For the minimum viable product, the targets are at least eighty-five percent traceability on approved plans, an average of fewer than three refinement iterations, a measurable drop in context-related failures, and positive feedback from the first real module.

No design is complete without confronting its risks. Poor retrieval on a messy corpus is met with upfront investment in chunking, metadata, and hybrid search, starting on well-structured subsets. Over- or under-decomposition is caught by the sizing critic and the human gate, aiming for tasks that fit a single focused session. Inconsistency across modules is countered by strong semantic memory, global consistency passes, and the knowledge graph. Hallucination is held in check by structured schemas, critic gates, low-temperature evaluation, and strict instructions to use only provided or retrieved context, with versioning making any drift visible. High cost on large projects is managed through scoped retrieval, compression, model routing, and caching. The human review bottleneck is eased by making gates configurable, auto-approving low-risk modules above a high critic score, and providing clean diffs and summaries. And even legacy or poorly structured specs benefit, because the system forces explicit classification and traceability, though they may need more human seed curation at first. As for edge cases, very small projects still gain consistent structure and living traceability. Highly visual specs get a vision step that describes the mockups for the feature planner. Conflicting requirements are surfaced explicitly for human resolution. And rapidly evolving specs lean on incremental mode and impact analysis.

Looking ahead, the roadmap imagines a fully automated knowledge-graph retrieval with real-time impact analysis, multi-modal ingestion that reads interface mockups and diagrams, automated generation of tests and validation artifacts straight from acceptance criteria, risk and complexity scoring that auto-escalates the hardest plans, a meta-planner that analyzes historical runs and evolves its own templates and strategies, a bridge to formal methods for high-assurance modules, team and enterprise features like role-based access and export to project tools, and domain-specific extensions with pre-built taxonomies for fields like education technology, trading interfaces, and embedded hardware.

The design draws on a deep body of references, the MAAD paper on bridging requirements and architecture, AgentOrchestra on hierarchical multi-agent frameworks, broad surveys of multi-agent systems for software engineering, and a whole family of hierarchical planning papers, alongside the context engineering literature, the spec-driven development tooling, the Grok Build documentation, and foundational architecture work like the four-plus-one views and tradeoff analysis methods.

To close, this specification is intentionally exhaustive so it can stand as a single source of truth, either for building SIPA from scratch or for evolving an existing meta-agent system. It directly addresses the core challenge, breaking large specifications into manageable, appropriately detailed parts while preserving intent through traceability, critic-driven quality, and living updates. From here you could feed this very document into an agent harness to start building, prototype the minimum viable product, or ask for concrete prompt templates, schemas, or starter code. It's engineered to compound existing strengths in AI agent engineering, spec-driven development, and rigorous harness building, built with the same philosophy of thinking deeply, enforcing quality gates, maintaining traceability, and staying relentlessly practical.
