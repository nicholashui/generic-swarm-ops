Let's talk about SIPA, which stands for the Software Implementation Planner Agent. This is version two point one, a research-hardened, security-operationalized design. Think of it as a production-ready blueprint with one central goal: to take very large bodies of software specifications and turn them into high-fidelity, traceable, and adaptable implementation plans, along with granular tasks that both AI coding agents and human developers can actually act on.

This revision isn't just a bigger version of what came before. It's a deep-research pass, an adversarial rethink that tightens the whole system into something implementable, governable, and genuinely security-aware. Several things changed in meaningful ways. The team went back and verified the research the design leans on, confirming the multi-agent architecture work that inspires the planning approach, and updating references to orchestration research that now emphasizes treating agents, tools, environments, prompts, memory, and outputs as versioned, lifecycle-managed resources. The alignment with spec-driven development tooling was refreshed to match the current flow of moving from specification, to plan, to tasks, to implementation, including newer practical steps like establishing a project constitution, clarifying, analyzing, and turning tasks into issues. And claims about external vendor tools were carefully qualified, so the design no longer depends on any unofficial feature and instead defines its own optional local evaluator.

A big addition in this version is a real security and trust model. It covers prompt injection, memory poisoning, the safety of tools and external connections, least-privilege execution, clear boundaries around untrusted content, and human approval controls. The guiding insight here comes from security research warning that large language models do not enforce a hard line between data and instructions. In other words, they're inherently confusable, so prompt injection should be treated as a residual risk you reduce through good system design, not something you can fully eliminate.

The context engineering was also strengthened, organized around four verbs: write, select, compress, and isolate. On top of that, the design added retrieval budgets, evidence ledgers, source confidence levels, evaluation of retrieval quality, and detection of stale sources. It even allows an optional graph-based retrieval mode for understanding the whole corpus and analyzing how dependencies ripple through it.

So why does SIPA exist at all? Large software specs tend to break AI coding workflows in predictable ways. There's context overload, where too much raw documentation causes distraction, contradiction, and missed details. There's the failure of uniform summarization, because architecture needs synthesis, user interface features need exhaustive local detail, shared components need clear contracts, data models need invariants, and integrations need error semantics and compatibility rules. There's loss of traceability, where coding agents implement plausible-sounding behavior that isn't actually grounded in the source. There's weak translation from plan to task, where big plans become vague tasks that produce bad code and hidden assumptions. And finally, there's security and tool-risk amplification, because agentic coding tools can read files, run commands, call interfaces, and modify repositories. SIPA is meant to solve these problems before any coding begins.

The core idea is simple but powerful: different software components require different levels and types of detail. A strategic architecture plan should not be generated with the same retrieval scope, summarization style, or output format as a single screen, a shared library, a data model, or a migration adapter. So SIPA uses component-type classification, scoped retrieval, evidence-based synthesis, hierarchical memory, embedded critic loops, traceability-first artifacts, granular task generation, and security-aware execution. The payoff is a planner that shrinks the context handed to downstream coding agents while improving fidelity, traceability, and the odds of successful implementation.

The research foundation draws on several streams. Spec-driven development treats the specification as the center of AI-assisted work, and SIPA extends that idea to pre-existing massive corpora, not just brand-new features, adding corpus indexing, type-aware planning, traceability enforcement, and critic loops. From multi-agent architecture research, SIPA borrows the pattern of specialized agents, an analyst, a modeler, a designer, and an evaluator, that convert requirements into multi-view architectural blueprints with quality assessments. From orchestration research, it takes the practical lesson that good coordination is not just agents chatting; it's explicit state, typed handoffs, versioned artifacts, and lifecycle-aware execution. It also leans on findings that the interface between an agent and its computer materially affects performance, and that self-evaluation and iterative refinement genuinely help on repository-level tasks.

For architecture quality, SIPA aligns its outputs with well-known standards: the formal concepts of stakeholders, concerns, viewpoints, and views; the layered model that describes systems at the levels of context, container, component, and code while only drawing diagrams that add value; the tradeoff analysis method for reasoning about quality attributes and risks; and the classic four-plus-one approach to describing architecture from multiple perspectives.

Now let's walk through the component taxonomy, because it's the engine that drives almost everything. The type of a component controls its retrieval strategy, its context budget, which planner handles it, the output template, the critic rubric, whether a human must review it, and how tasks get broken down. The primary types include architecture or strategic components, which need high-level synthesis and broad cross-corpus retrieval, producing views, decision records, tradeoffs, and quality attributes. Feature and user-interface components need deep, scoped detail and full local excerpts, producing user stories, acceptance criteria, flows, state, and validation. Shared components are contract-focused, emphasizing interfaces, types, invariants, and extension points. Data and domain models focus on entities, relationships, invariants, lifecycle rules, schemas, and migrations. Integration components handle external contracts, mapping, retries, idempotency, and compatibility. Infrastructure and operations components are highly operational, covering deployment, continuous integration and delivery, observability, and scaling. Security and compliance components demand high rigor, producing threat models, controls, and audit evidence. Testing and quality components produce test matrices, fixtures, and validation strategies. Migration and legacy components produce strangler plans, compatibility notes, and rollback strategies. AI and agent-workflow components are high-rigor, covering agent state machines, tool policies, and evaluation harnesses. And documentation and developer-experience components produce quickstarts, conventions, and contributor guides.

On top of the primary type, secondary tags fine-tune the gates. A component might be tagged security-critical, privacy-sensitive, performance-sensitive, high-availability, user-facing, internal-only, regulated, legacy-modernization, agentic-tool-use, destructive-actions, requires-human-review, experimental, or mvp-critical. Each tag can change retrieval depth, critic strictness, and review requirements. Classification itself happens in layers: explicit metadata first, then requirement identifiers and section names, then keyword heuristics, then a language-model classifier with structured output, then human override, and finally critic validation. The classification result records a component identifier, a name, a primary type, secondary tags, a confidence score, the supporting evidence, and whether human review is needed.

The system architecture flows like a pipeline. The spec corpus is ingested and indexed, feeding a hybrid retriever, an optional knowledge graph, and a corpus summary with a glossary. All of that feeds a master orchestrator. The orchestrator classifies components, builds a dependency graph, and produces a master plan. From that master plan, work fans out to specialized sub-planners for architecture, feature and interface, shared components, data and domain, integration, security and compliance, and testing and quality. Every sub-planner's draft passes through a critic and evaluator, which sends work back for refinement until it's good enough. Approved specs then flow to a task decomposer, which produces task lists and agent work packs for coder agents or humans. Their implementation feedback flows back into memory and impact analysis, which updates the orchestrator, closing the loop.

Let's look at the agent roles. The master orchestrator owns the global plan, the decomposition, the dependencies, and coordination. It builds corpus-level understanding, identifies epics and modules and components and phases, classifies components, generates the master plan, maintains the dependency graph, triggers the sub-planners, coordinates critic feedback, and escalates high-risk ambiguity to humans. The architecture sub-planner follows that analyst, modeler, designer, evaluator pattern to extract requirements and architecturally significant ones, produce useful views, generate decision records, identify tradeoffs, define interface boundaries, and run a lightweight quality analysis. The feature and interface sub-planner extracts user stories, expands acceptance criteria, defines interface states and user flows, identifies validations and errors, links data dependencies, and generates implementation-sized tasks. The shared-component sub-planner aggregates scattered mentions into a single contract, defining public interfaces, invariants, extension points, configuration, failure modes, and usage examples. The data and domain sub-planner identifies entities, aggregates, value objects, and relationships, defines lifecycle states, captures invariants and constraints, and links domain objects to features and interfaces. The integration sub-planner defines external contracts, maps source and target data, and specifies retries, timeouts, idempotency, backoff, authentication, secret handling, version compatibility, and reconciliation. The security and compliance sub-planner identifies assets, trust boundaries, actors, and data flows, threat-models the agent, tool, and data paths, maps requirements to controls, defines least-privilege policies and audit evidence, and requires human approval for high-risk actions.

The critic and evaluator exists to prevent bad plans from becoming implementation tasks. It judges along many dimensions: traceability, source fidelity, completeness, consistency, implementability, testability, security, risk, sizing, and downstream usability for coding agents. It emits a structured critique, patch instructions, and a quality-gate report. The task decomposer then converts approved specs into small, executable tasks. Each task represents one logical change with clear acceptance criteria, source links, a relevant context excerpt, dependencies, suggested files, test expectations, and rollback notes when needed.

The end-to-end workflow runs in phases. Phase zero establishes a project constitution: architecture principles, coding and testing standards, security rules, stack preferences, user-experience rules, naming conventions, and the human review policy, all captured in one governing document. Phase one is ingestion and indexing, where SIPA scans directories, respects include and exclude rules, parses the heading hierarchy, extracts requirement identifiers, detects components and entities and interfaces and screens and decisions, chunks the text semantically, generates embeddings, builds a keyword index, optionally builds a graph index, and produces a corpus summary and glossary. Phase two builds the master plan, where the orchestrator reads summaries, retrieves cross-cutting requirements, builds and classifies the component inventory, generates the dependency graph, creates implementation phases, identifies the minimum viable slice, flags risks and unknowns, and runs a critic over the plan. The master plan must contain a project summary, scope, non-goals, the component inventory, phases, the dependency graph, a risk register, a traceability skeleton, open questions, and a human approval checklist.

Phase three is type-aware sub-planning. For each component, SIPA confirms its type, assembles a context package, runs the right sub-planner, generates a draft, runs the critic, patches only the failing sections, re-runs the critic, and persists the approved artifact. Phase four is task generation, where the decomposer turns approved plans into implementation tasks. A typical task names the component, links the source section, lists what it depends on, states acceptance criteria such as rendering the right states and meeting responsive behavior, specifies the tests, and points to the relevant source excerpt rather than restating everything. Phase five is implementation feedback. After agents or humans do the work, SIPA ingests diffs, test results, review comments, blockers, changed assumptions, failed tasks, and new requirements. It then updates episodic memory, performs impact analysis, updates affected plans, re-runs the critic gates, and regenerates the affected tasks.

Context engineering deserves a closer look. Every sub-planner receives a context package with a token budget that's deliberately split across source evidence, memory, master context, and the template and instructions. The package lists retrieved chunks with their source, relevance, and trust level, and it specifies a compression method that preserves quotes and requirement identifiers. Retrieval comes in several modes: exact identifier lookup for requirements and interface names, keyword search for precise technical terms, vector search for semantic similarity, graph traversal for dependencies and trace links, global summarization for corpus-wide themes, local neighborhood retrieval for closely related chunks, and a reranked hybrid mode that serves as the default in production.

Memory is organized into several types. Working memory holds the current run, like the draft and active critic notes. Episodic memory holds past runs, including refinements and implementation feedback. Semantic memory holds generalized knowledge such as patterns, the glossary, and conventions. Procedural memory holds agent behavior like prompts, rubrics, and templates. Evidence memory holds the source grounding, the chunk identifiers, citations, and excerpts. And security memory holds trust and permissions, the approved tools and blocked actions. Tied to this is the evidence ledger, where every major claim maps to specific sources with quotes, a confidence score, and a support status, so you can always trace an assertion back to where it came from.

The critic subsystem enforces quality gates with explicit thresholds. Traceability, source fidelity, internal consistency, cross-artifact consistency, implementability, and testability all have numeric pass bars in the high eighties to low nineties. Security review requires no high-severity findings on tagged artifacts. And strategic or high-risk work requires explicit human approval. Critiques are structured: an overall verdict, scores per dimension, a list of issues with severity and location and a suggested patch, and a recommended next action. There's also an optional arena evaluator that runs several competing planner outputs and scores them against the same rubric, comparing traceability, clarity, completeness, risk handling, taskability, security posture, and source fidelity. Importantly, this evaluator stays vendor-neutral, independent of any external product.

The security and trust model rests on some sober assumptions. Retrieved documents may contain malicious instructions. Tool descriptions may be untrusted. Memory can be poisoned. Agent outputs can be wrong even when they sound confident. Human approval is only useful if the interface shows the actual action, not just the agent's summary of it. And high-impact operations require real gates. Sources are labeled by trust level, ranging from canonical approved truth, to trusted-derived approved output, to unreviewed drafts, to external web docs, to untrusted content that may carry injection, all the way down to deliberately hostile red-team fixtures. Tool permissions follow least privilege: reading specs and writing plans to the output directory are allowed, but modifying source code is blocked by the planner, running shell commands is blocked unless an explicit tool mode is enabled, deleting files and calling external interfaces and installing packages all require human approval, accessing secrets is prohibited outright, and memory can only be updated through a sanitizer.

To control prompt injection, SIPA wraps retrieved content as untrusted evidence, never lets that content override system or developer instructions, strips or flags suspicious instructions in source chunks, separates evidence from planner instructions, uses deterministic policy checks for tool calls, logs how sources map to claims, and requires human approval for high-impact actions. If SIPA exposes external tools, every tool gets a narrow schema, destructive tools require explicit approval, tool descriptions are treated as untrusted unless they come from trusted servers, resources are never forwarded without consent, and every tool call is logged with its inputs, outputs, requester, and artifact identifier.

The design also defines clear data models. Every artifact carries front matter recording its identifier, type, component, version, status, timestamps, the source corpus version, traceability and critic scores, dependencies, whether review is required, and who approved it. Requirements are recorded with their text, source, type, priority, the components they touch, and status. Decisions are recorded with context, the choice made, the alternatives considered, the positive and negative consequences, and supporting evidence. Tasks are recorded with their component, source artifacts, dependencies, whether they can run in parallel, risk level, acceptance criteria, the tests required, and whether human review is needed.

On outputs, the master plan has a fixed set of required sections, from executive summary through scope and non-goals, corpus summary, component inventory and classification, dependency graph, phases, the minimum viable slice, risk register, traceability skeleton, open questions, approval checklist, and changelog. The architecture spec covers scope, stakeholders and concerns, the significant requirements, the layered views, runtime scenarios, deployment, decision records, an interface catalog, quality attribute scenarios, tradeoff analysis, risks, and a traceability matrix. The feature spec covers the feature summary, user stories, flows, interface states, acceptance criteria, a validation matrix, error and edge cases, data dependencies, accessibility notes, analytics events, test scenarios, and traceability. The shared-component spec covers purpose, scope, public interface, data contracts, invariants, configuration, extension points, error handling, performance expectations, security considerations, usage examples, test strategy, and traceability. The security spec covers assets, actors, trust boundaries, data flows, threats, controls, residual risks, approval requirements, monitoring, audit evidence, and traceability. And the task list covers an overview, execution order, parallelizable tasks, the critical path, the task checklist, test and review checklists, rollback notes, and source links.

The implementation roadmap is staged. Phase one builds a safe minimum viable product as a vertical slice: a corpus loader, a markdown chunker, hybrid search, a component classifier, the master planner, one sub-planner, a basic critic, a markdown renderer, a task decomposer, and a command-line interface. The recommended first sub-planner is either feature and interface, if you're validating the handoff to implementation, or architecture, if you're validating strategic planning. Phase two builds the core system: all the primary sub-planners, the knowledge graph, the evidence ledger, hierarchical memory, a multi-stage critic, human approval gates, incremental re-planning, evaluation metrics, and a template library. Phase three is production hardening: an observability dashboard, cost and token tracking, model routing, a prompt and version registry, an external tool server, a sandboxed tool mode, a security red-team suite, continuous-integration validation, plan-drift detection, and issue-tracker export. And phase four is self-improvement: planner performance analytics, learning from failed tasks, retrieval strategy optimization, evolving prompts and templates, domain-specific taxonomies, and automatic benchmark generation.

On harness integration, SIPA writes plan and task directories along with an agents file, and rule or command directories for popular coding tools. The agents file captures module conventions, links to relevant plans, forbidden actions, test expectations, and the order of sources to trust. For a tool like Grok Build, which officially supports plan mode, headless scripting, custom models, skills and plugins, external tool servers, and parallel subagents, the integration pattern is clean: SIPA generates an approved plan and task pack, the tool runs in plan mode, a human compares its execution plan to SIPA's task criteria, implementation proceeds only after approval, and then SIPA ingests the diffs and test results. SIPA can also expose its own interface for indexing a corpus, fetching the master plan, planning a component, critiquing an artifact, generating tasks, tracing a requirement, and analyzing impact, with every write or destructive operation requiring authorization.

Success is measured carefully. Retrieval metrics include context precision and recall, source coverage, duplicate chunk rate, stale source rate, evidence sufficiency, and the rate of unsupported claims. Artifact metrics include traceability, source fidelity, consistency, completeness, testability, implementability, security, and human review time. Downstream metrics include task completion rate, rework rate, test pass rate, spec deviation rate, clarification count, the average task context size, and the time from a spec change to updated tasks. For the minimum viable product, the bar is traceability of at least eighty-five percent, rising to ninety percent for production, no high-severity critic findings on approved artifacts, fewer than three refinement loops on average, at least a thirty percent reduction in downstream clarification, and at least a fifty percent reduction in the context passed to coder agents compared with raw-spec prompting.

The design closes by naming its risks and mitigations. Bad retrieval is countered with hybrid search, reranking, the evidence ledger, and retrieval evaluation. Hallucinated plans are caught by the source-fidelity critic and an unsupported-claim detector. Over-decomposition and under-decomposition are both checked by sizing and dependency rules. Conflicting specs trigger a conflict detector and human resolution. Prompt injection is contained by untrusted-source boundaries and deterministic controls. Memory poisoning is handled by the sanitizer and trust levels. Tool abuse is limited by least privilege and approval gates. Stale plans are addressed by incremental indexing and impact analysis. High token cost is managed with compression, caching, and model routing. The human bottleneck is eased by risk-based approvals. And vendor lock-in is avoided by keeping artifacts in markdown first and exposing a portable interface.

Looking further ahead, the future roadmap imagines full graph-based impact analysis, multimodal ingestion of interface mockups and architecture diagrams, automatic test generation from acceptance criteria, a bridge to formal methods for critical invariants, domain packs for fields like education technology, finance, embedded systems, and legacy modernization, issue export to common trackers, continuous detection of drift between plan and code, a multi-model evaluator arena, a self-optimizing prompt and template registry, and a secure enterprise deployment bundle. Taken together, SIPA is a planner built to think before the coding starts, so that what gets built is faithful to what was actually specified.
