Two sibling directories with distinct purposes:
- `tests/`: flat collection of Node.js test files (`.test.mjs`) using the built-in `node:test` runner and `node:assert/strict`. Each file targets one subsystem — adapters (`adapters.test.mjs`), business governance/risk tiers/schemas/security/structure (`business-*.test.mjs`), manifest parsing (`manifest.test.mjs`), source download/audit/sync (`source-*.test.mjs`, `sync.test.mjs`), workflow DNA (`workflow-dna.test.mjs`), and an evolution sandbox harness (`evolution-sandbox.test.mjs`). Tests import production code from `../scripts/**` and resolve repo roots via `fileURLToPath(import.meta.url)` + `path.resolve(__dirname, "..")`; fixtures live under `tests/fixtures/` for shared inputs like `business-secret.txt` and `sample-source-lock.json`.
- `reviews/`: human-readable audit trail. `e1_operator_checklist.md` records a full end-to-end operator run (health → login → flagship run → human gate → improve pipeline) including the fix applied during execution. `mark100-logs/` stores per-subsystem log outputs (backend-unit, business-eval, frontend-test, etc.) produced by the Mark100 review process.
Dependency direction is one-way: tests depend on `scripts/` and `business/` sources; reviews are pure artifacts consumed by operators, not by code.
