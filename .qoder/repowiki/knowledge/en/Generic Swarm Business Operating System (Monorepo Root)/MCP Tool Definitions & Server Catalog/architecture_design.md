Two cooperating sub-trees with a clear producer/consumer split:
- `mcps/<domain>/tools/*.json` — one file per tool, each declaring an MCP-compatible schema with `name`, `description`, and a JSON-Schema `inputSchema`. Domains are grouped as flat namespaces (`github`, `memory`, `tasks`, `sequential-thinking`, `pencil`) so the runtime can enumerate them without code.
- `mcp-configs/` — server-level configuration. `manifest.json` is the index listing active config files; `minimal.json` is a skeleton `{"mcpServers": {}}`; `optional/grok-mcp-catalog.json` is the canonical reference that maps logical server names to stdio transports running via `npx @modelcontextprotocol/server-*` (sequential-thinking, memory, github, filesystem), with optional `env` entries like `GITHUB_PERSONAL_ACCESS_TOKEN` and a `requires_secrets` flag gating activation.
Dependency direction: configs reference tool definitions only through naming conventions (server name → `mcps/<name>/tools/*`); there is no cross-file JSON pointer between them, keeping both sides independently editable.
