## 0. What This Is

A governed, self-improving multi-agent system that:

1. Learns how a business actually operates (from documents, experts, **and** real event logs).
2. Distills that knowledge into reusable rules, skills, workflows, and playbooks.
3. Executes work through **stateful multi-agent orchestration graphs** (LangGraph under a FastAPI control plane)—bounded, auditable, interruptible.
4. Evolves those workflows in a sandbox — **never** directly in production.

Design priorities, in order: **Safety → Auditability → Correctness → Efficiency → Autonomy.**  
Autonomy is earned per workflow through evidence, not granted by default.

**Orchestration posture (v3.1):** Linear DNA step lists alone are insufficient for real multi-agent work. The normative execution engine is **[LangGraph](https://github.com/langchain-ai/langgraph)** (StateGraph, checkpointers, interrupts, multi-agent patterns). Workflow DNA remains the **portable definition / IR** compiled into graphs. FastAPI remains the **only public control plane**—LangGraph is an in-process engine, not a second product API.

## 1. Core Principles

1. **Evidence over opinion.** Learn from real traces, not only from what people say they do.
2. **Bounded autonomy.** Every action has a risk tier, a permission scope, and (where needed) a human gate.
3. **Everything is testable.** No agent, prompt, or workflow reaches production without passing an eval.
4. **Sandbox evolution.** The evolution engine proposes; it never mutates production directly.
5. **Provenance always.** Every rule, decision, and memory traces back to a source.
6. **Reversibility first.** Prefer reversible actions; require rollback plans for the rest.
7. **Human-centered.** Show confidence, show evidence, make correction easy.

**Critical execution facts:**

1. Starting a run **queues** it unless the engine auto-continues after start (product should still support explicit `POST /api/v1/workflow-runs/dispatch`).  
2. **LangGraph** runs use durable `thread_id` (typically `{organization_id}:{run_id}`) and may **interrupt** for human gates; resume goes through host approvals (`decision` → graph resume).  
3. **Legacy** runs use status `waiting_for_approval` and re-enter the linear runner after approve.  
4. Run record should carry `engine`, and for LangGraph also `graph_thread_id`, `graph_id`, `orchestration_pattern` when available.

### 2.3 Process Intelligence Layer

The swarm must not learn only from documents and interviews. It should learn from **actual
operational traces**: tickets, CRM/ERP actions, calendar events, emails, approvals, file edits,
API calls, and completion records. Process mining turns these logs into discoverable workflow
models, conformance checks, and bottleneck analysis — the empirical version of "Shadow Mode."

**Logical roles (architecture):**

| Role | Purpose |
|------|---------|
| Process Miner | Discovers real workflows from event logs |
| Task Mining | Observes UI/human-level steps where permitted |
| Conformance | Compares actual work against documented SOPs |
| Bottleneck Analyzer | Finds delays, loops, rework, handoff failures |
| Causal Improvement | Proposes interventions likely to improve outcomes |

**As-built note:** These roles are realized as **PI services + disk artifacts** under `business/process-intelligence/` (discovered / conformance / bottlenecks / causal-hypotheses), not necessarily five independent always-on LLM agents.

## 3. Knowledge Acquisition, Distillation & Memory

### 3.1 Elicitation Methods

Expert knowledge is largely tacit — which cues matter, when to override the rule, when something
"feels wrong." Cognitive Task Analysis and the Critical Decision Method are established ways to
surface this. Upgrade a single "Expert Shadow" into a multi-method capture system.

| Method | Best For | Output |
|---|---|---|
| Shadow Mode | Real actions | Event logs, action traces |
| Critical Decision Interview | Rare / high-stakes calls | Decision requirement cards |
| Think-Aloud Session | Routine expert work | Step-by-step heuristics |
| Exception Interview | Edge cases | Exception library |
| Retrospective Review | Completed cases | Lessons learned |
| Apprentice Mode | Expert teaches swarm | Skills, playbooks |

### 3.3 Hybrid Memory

One generic "knowledge base" is not enough. Long-running agents need differentiated memory —
raw observations, higher-level reflections, and retrievable long-term stores.

| Memory Type | Stores | Example |
|---|---|---|
| Event | Raw operational logs | "Agent sent invoice at 9:42 AM." |
| Episodic | Case narratives | "This renewal almost failed — legal was pulled in late." |
| Semantic | Facts / rules | "Enterprise contracts over 250k need legal review." |
| Procedural | Skills / workflows | "How to onboard a new client." |
| Decision | Decisions + reasons | "We approved exception X because Y." |
| Exception | Edge cases | "If supplier in region Z, use alternate process." |
| Evaluation | Test results | "Workflow v12 failed privacy test." |
| Provenance | Source attribution | "Rule came from SOP v4 and expert Alice." |

**As-built:** Memory is scoped (`workflow_memory`, `organization_memory`, agent scopes for ALC). Workflow `memory_reads` / `memory_writes` are enforced at execution. High-impact writes carry provenance; rejection of suggestions can feed lesson libraries.

### 3.4 Retrieval: Tiered Hybrid (LightRAG core)

GraphRAG-style community summarization is deliberately avoided — its per-chunk
extraction plus community-report generation makes both initial indexing and
(critically) re-indexing on every new document prohibitively expensive for a
system that ingests event logs and documents continuously.

Instead, use a cost-tiered retrieval stack. Most queries never touch the
expensive tiers.

**Tier 0 — Vector / semantic search (default, cheapest).**  
Semantic similarity over chunked documents. Handles the majority of
"find the relevant passage" queries. No graph, no extra LLM calls.

**Tier 1 — LightRAG graph layer (relational reasoning).**  
- Graph-based text index with dual-level retrieval:
  low-level (entity-specific) + high-level (thematic).
- **Incremental updates** — new documents/events are added without
  rebuilding the graph.
- Answers relational questions: "which obligations depend on this contract?",
  "who touched this case and in what order?"

**Tier 2 — Hierarchical summaries (RAPTOR-style, optional, on demand).**  
Recursive cluster-and-summarize tree, built only for corpora that get
frequent corpus-wide questions. Built lazily, not for the whole knowledge base.

**Always-on — Provenance layer.**  
Every answer cites its source documents, experts, event logs, or decisions,
regardless of which tier served it.

**Escalation rule:** start at Tier 0 → escalate to Tier 1 only when the query
needs relationships/multi-hop → escalate to Tier 2 only for global synthesis.
This keeps 80%+ of traffic on the cheapest tier.

**As-built (product bar):**  
- Tier 0: keyword + hash-style embedding + **always provenance**  
- Tier 1: entity-link multi-hop **lite** (not full commercial LightRAG mesh)  
- Tier 2 + full vendor LightRAG / Neo4j production mesh = **non-goal** for product bar (§12.4)

**Evaluation of retrieval:** score context relevance, answer relevance, and faithfulness separately —
a weak retriever silently poisons every agent.

### 4.1 Workflow DNA schema (portable IR)

DNA is **not** discarded. It remains the human-readable, pack-portable definition. The LangGraph compiler turns DNA (+ optional `orchestration` / pack `graphs/*.graph.json`) into a compiled StateGraph.

  domain: "operations"

  owner: "business_orchestrator"
  version: "12.0"
  engine: "langgraph"   # optional; host default may apply
  orchestration:
    pattern: pipeline   # pipeline | supervisor | router | critique | map_reduce | pack_spine
    config: {}
  inputs: ["signed_contract", "customer_profile", "billing_details"]

    - "contract_status == signed"

    - id: "create_customer_record"

    - id: "activate_billing"

      tools: ["billing_system"]
    - id: "send_welcome_packet"
      agent: "communications_agent"
      tools: ["email"]
    - id: "audit_and_close"
      agent: "business_orchestrator"

### 4.2 LangGraph execution pattern

Use a **bounded StateGraph** (LangGraph), not a free-form ungoverned swarm. Reasoning/acting loops (ReAct-style) may run *inside* a node; the graph enforces topology, permissions, budgets, and human-in-the-loop **interrupts**.

```text
Event → Intake Router → Risk Classifier
   → Compile DNA/graph IR → LangGraph StateGraph
   → Nodes: memory_read · agent · host_tools · gate(interrupt) · memory_write
   → Patterns: pipeline | supervisor ⇄ specialists | router | critique | pack_spine
   → Checkpoint (Postgres) · stream events (node/handoff/tool/interrupt)
   → Audit + Evaluation → Evolution Sandbox (orchestration variants only)
```

#### 4.2.1 Multi-orchestration patterns (required product capability)

| Pattern | Shape | Use when |
|---------|-------|----------|
| **pipeline** | Linear nodes (DNA steps) | Classic ops workflows, onboarding |
| **supervisor** | Supervisor ⇄ specialists via handoffs | Multi-agent teams (video craft) |
| **router** | Classify → branches → join | Intent / risk based routing |
| **critique** | Produce → critique → revise/accept loop | Quality gates with max iterations |
| **map_reduce** | Fan-out over list → aggregate | Parallel subcases |
| **pack_spine** | Host entry → domain subgraph → host audit | Domain packs (video spine) |

Circuit breakers: `max_handoffs`, `max_node_visits`, wall-clock and tool-call budgets (denial-of-wallet).

#### 4.2.2 Host graph state (conceptual channels)

Serializable shared state for all host-compiled graphs (names illustrative):

| Channel | Purpose |
|---------|---------|
| `case` | Input payload + evolving case fields |
| `risk` | Risk tier, gate flags |
| `messages` | Agent/supervisor dialogue (append-only) |
| `route` | Next agent / pattern control / done flag |
| `artifacts` | Named outputs (scripts, QC reports, …) |
| `tool_effects` | Durable side-effect summaries |
| `memory_hits` | Pre-act retrieval / ALC lessons |
| `interrupt` | Preview payload when paused for human |
| `metrics` | Tokens, cost, latency, visit counts |

Project graph state back to the existing run API (`steps[]`, `status`, `output`, `approval_request_id`) so operators and tests stay stable.

#### 4.2.3 Tool nodes (security)

Every tool call goes through **HostToolNode** → allow-list intersection  
`agent.allowed_tools ∩ DNA step tools ∩ RBAC ∩ risk policy` → host adapters → `tool_effects` + audit.  
LangChain free-tools that bypass the broker are **forbidden** in production graphs.

#### 4.2.4 HITL interrupts

| Mechanism | Behavior |
|-----------|----------|
| LangGraph interrupt | Gate node pauses; checkpoint stored |
| Host bridge | Creates approval row; run `waiting_for_approval` |
| Resume | `POST /api/v1/approvals/{id}/decision` with `{ "decision", "reason" }` → graph resume (optional overrides) |
| Reject | Route to fail/compensate path per DNA |

## 5. Evolution Engine (Sandboxed)

### 5.1 The One Non-Negotiable Rule

> **The Evolution Manager must never mutate production directly.**  
> It may only: propose variants → test in sandbox → compare to baseline → request approval →  
> canary deploy → auto-rollback on failure.

This single constraint converts a risky autonomous swarm into a controlled, auditable system.

### 5.2 Fitness Function

Do not evolve on subjective preference. Score every variant:

\[
F = w_q Q + w_s S + w_c C + w_e E + w_h H - w_r R - w_l L - w_k K
\]

Where \(Q\)=quality, \(S\)=safety, \(C\)=compliance, \(E\)=efficiency, \(H\)=human satisfaction,
\(R\)=risk penalty, \(L\)=latency penalty, \(K\)=cost penalty. For conflicting objectives, use
**Pareto selection** rather than collapsing everything into one scalar.

**As-built coevolution fitness (deterministic composite example):**  
`0.6 * suite_pass_rate + 0.2 * knowledge_growth_norm + 0.2 * lesson_reuse_norm`.

### 5.3 Pipeline

```text
1.  Observe production / shadow traces
2.  Detect failures, bottlenecks, or opportunities
3.  Generate variants (prompt / workflow / tool-use / role / expert-pattern crossover)
4.  Test offline against golden tasks
5.  Run security + adversarial tests
6.  Run compliance checks
7.  Replay on historical cases (simulation)
8.  Human review if risk tier requires
9.  Canary deploy to small scope
10. Monitor metrics
11. Promote / rollback / retire
12. Store lessons in evolution memory
```

Natural-language reflection is a powerful optimizer: a *few* trajectories, diagnosed in language,
can beat many rounds of scalar-reward RL in data-scarce business settings. Keep these loops
**inside the sandbox gates above.**

### 5.4 Promotion Rule

A variant is promoted only if it (1) improves target metrics, (2) does not regress safety or
compliance, (3) passes regression + adversarial tests, (4) has a rollback plan, (5) has complete
audit logs, and (6) has human sign-off when the risk tier requires it.

**As-built / target realization:**

| Capability | Behavior |
|------------|----------|
| Propose | Variants marked `sandbox_only` (DNA and/or **orchestration / graph config**) |
| Evaluate | Disk corpus eval against golden/regression sets; **trajectory** metrics for graphs (handoffs, node visits, tool denials) when LangGraph engine is used |
| Canary | Versioned promote to canary status only after pass |
| Rollback | Supported API/runtime path |
| Auto-promote to production | **Blocked** |
| Self-improvement | Auto-reflect, lesson library, auto-propose, loop runner, FE Improve pipeline |
| Host code self-rewrite (DGM-style) | **Non-goal** |
| LangGraph topology rewrite in prod | Only via sandbox propose → evaluate → canary (LG-N5) |

## 6. Governance, Risk & Compliance

Anchor to established public frameworks rather than inventing an ad-hoc taxonomy.

- **NIST AI RMF (AI 100-1)** — map / measure / manage risk backbone for trustworthy AI.
- **ISO/IEC 42001** — AI management system (Plan-Do-Check-Act): risk management, impact assessment, lifecycle, supplier oversight.
- **EU AI Act** — applies if you touch EU users, workers, or regulated decisions. Employment-related AI (recruitment, performance evaluation, monitoring, promotion/termination) is high-risk class — expect risk management, data governance, documentation, record-keeping, transparency, human oversight, robustness, cybersecurity, and incident reporting obligations where applicable.

### 6.1 Autonomy Risk Tiers

| Tier | Autonomy | Allowed Behavior |
|---|---|---|
| 0 | Observe | Log and summarize only. |
| 1 | Recommend | Suggest; human acts. |
| 2 | Draft | Prepare artifacts; human approves before send/execute. |
| 3 | Execute (reversible) | Act if rollback exists and risk is low. |
| 4 | Execute + gate | Act, but human approves the critical step. |
| 5 | Restricted | No autonomous action until an assurance case exists. |

**As-built:** Flagship onboarding runs as tier-4 style execute-with-gate. Risk tier is carried on runs and events.

### 6.2 Mandatory Artifacts

AI inventory · use-case risk tiering · human-approval policy · audit logs · incident-response plan ·
rollback plans · data-retention policy · vendor/model register · tool-permission register ·
assurance cases · model cards.

These live under `business/governance/` and related trees; runtime also appends operational audit logs on mutations.

|-------|------|

## 7. Security

Agentic systems widen the attack surface far beyond classic AppSec. Two public OWASP frameworks apply:
the **Top 10 for LLM Applications (2025)** for the model layer and the **Top 10 for Agentic
Applications (2026)** for the autonomy layer (behavior hijacking, tool misuse, identity/privilege abuse).

Design facts:

1. **Indirect prompt injection** is the key threat; after alignment and filtering, **assume injection can still happen** — control blast radius.
2. **System prompts are not security controls**; if a secret is in the prompt, it is already gone. Enforce security **deterministically outside the LLM**.

### 7.1 Control Matrix (mapped to OWASP LLM 2025)

| Risk | OWASP | Control |
|---|---|---|
| Prompt injection (esp. indirect) | LLM01 | Treat all retrieved/user content as untrusted; separate instructions from data; blast-radius limits. |
| Sensitive info disclosure | LLM02 | DLP on outputs/logs; secrets never in prompts; retention limits. |
| Supply chain | LLM03 | Model/tool/adapter registry; provenance; dependency + SBOM scanning. |
| Data & model poisoning | LLM04 | Vet fine-tunes/LoRAs; validate retrieval sources; write-filter memory. |
| Improper output handling | LLM05 | Treat model output as untrusted input; sanitize before any execution. |
| Excessive agency | LLM06 | Risk-tiered autonomy; least privilege; approval gates. |
| System prompt leakage | LLM07 | No secrets/roles in prompts; enforce authz externally. |
| Vector/embedding weaknesses | LLM08 | Tenant isolation on vector stores; poisoned-content detection. |
| Misinformation | LLM09 | Grounding + citations; confidence display; human review on high stakes. |
| Unbounded consumption | LLM10 | Rate limits, timeouts, budget caps ("denial-of-wallet" defense). |

For excessive agency specifically: restrict tools the model can access, require human approval for sensitive or irreversible actions, apply least privilege to data/APIs/tools — maps cleanly onto §6.1 tiers.

### 7.2 Additional Agentic Controls

- **Tool Permission Broker** — narrow, temporary, scoped credentials per task.
- **Allow-list intersection** — agent allowed tools ∩ DNA step tools ∩ RBAC ∩ gates (fail closed).
- **Memory-poisoning defense** — provenance + human review for high-impact memory writes.
- **Skill/plugin vetting** — third-party agent skills are a supply-chain vector; scan and pin them.
- **Full observability** — one audit trail across model calls, tool calls, and agent-to-agent traffic.
- **AI incident response** — defined runbook for GenAI-specific incidents.
- **Multi-pack isolation** — domain tools must not silently expand allow-lists across namespaces.

**As-built tool adapters:** local adapters for CRM/billing/email/audit/contract/policy with durable side-effect records. Live external SaaS OAuth brokers are **non-goals** for the product bar.

## 8. Evaluation System

The biggest gap in most "swarm" designs. Every agent, skill, workflow, and prompt must own:

1. Golden task set  
2. Regression tests  
3. Adversarial tests  
4. Human-review set  
5. Historical-replay set  
6. Cost/latency benchmark  
7. Business-outcome metric  
8. Safety/compliance score  

Evaluate in realistic, multi-step, tool-using environments — isolated prompt tests do not predict real task performance.

**Evaluation card:**

```yaml
evaluation:
  target: "wf_customer_onboarding_v12"
  eval_type: "workflow_regression"
  test_set: "historical_onboarding_cases_q2"
  metrics:
    quality_score: 0.94
    compliance_pass_rate: 0.99
    average_cycle_time_minutes: 38
    escalation_rate: 0.12
    hallucination_rate: 0.01
    unauthorized_tool_attempts: 0
    cost_per_case_usd: 0.42
  result: "pass"
  promotion_decision: "canary_only"
  reviewer: "ops_lead"
```

|-------------|---------|

| Control plane | API + durable state | FastAPI + Postgres `runtime_state` | unchanged public API; richer run fields |
| Orchestration | Multi-agent graphs | **Linear** `_execute_run` | **LangGraph** StateGraph + patterns |
| DNA | Portable definition | Executed as step list | **Compiled IR** → graph |

| Tool adapters | Real execution | Local adapters + `tool_effects` | Same via HostToolNode only |
