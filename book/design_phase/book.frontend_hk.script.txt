呢集我想同大家由頭到尾拆清楚呢套 Frontend Server 說明書。前端 as-built 係一套 Next.js 營運主控台 (ops console)，產品標竿大約一百分已經關閉，包括 live ops 要將公開演示模式 (NEXT_PUBLIC_DEMO_MODE) 設做 false、真實建立表單 (create forms)、OpenAPI 型別 (types)、即刻執行 (Run now)、改進管線 (Improve pipeline) 由反思 (Reflect) 到提案 (Propose) 到評估 (Evaluate) 再到金絲雀 (Canary)、演化歸檔 (evolution archive) 喺 app 下面 evolution、接受邀請 (accept-invite) 會打後端接受邀請應用程式介面、使用者邀請同停用、組織用局部更新 (PATCH)、同埋執行生命週期取消重試暫停恢復同逾期。規格驅動開發包 (SDD) 喺 planning 下面 frontend，有需求 (requirements)、設計 (design)、任務 (tasks)，由 FE-01 去到 FE-20，任務係第二點三版，每項都有交付程式路徑 (Deliverable code paths)。證據可以睇 status.md、frontend 下面 README、gap_analysis_for_frontend 落差報告一百分、同埋任務對程式索引 TASK_TO_CODE_TRACEABILITY。後端契約就係營運台只可以經版本化應用程式介面路徑 (API /api/v1) 去打 backend，細節喺 backend_hk 第二十四節同 frontend_hk 第三十三節三 a。好，跟住我哋正式開始。

呢本書係將 frontend_hk.md 講到人哋聽得明，英文對照係 frontend.md，適合產品負責人、設計師、營運團隊、審批人員、前端同全端工程師。佢主要答八條問題。第一，呢個前端係平台邊一部分，同後端點分工。第二，產品願景同範圍，應該做咩、絕對唔好做咩。第三，技術棧、渲染策略同執行期邊界點樣保證前端唔執行業務。第四，資訊架構、導航、角色同應用殼層 (App Shell) 點組織營運體驗。第五，核心頁面例如認證、儀表板 (dashboard)、代理 (agents)、工作流程 (workflows)、執行 (run)、審批、知識、設定點運作。第六，點解改進 (Improve) 同演化 (Evolution) 只可以經後端沙盒應用程式介面。第七，應用程式介面整合、即時更新、安全、測試點支撐營運設定檔 (ops profile)。第八，planning 下面 frontend 嘅規格包同 as-built 程式點樣對返 frontend_hk.md。內容順序係先定位前端角色同願景，再講邊界技術棧設計方法，再拆導航頁面同元件，最後補安全測試完成定義，同埋第十四章實作對應，來源係計劃文件第三十三節。

而家講下第一章，前端係咩。最簡單講，前端伺服器 (Frontend Server) 係通用蜂群 (Generic Swarm) 業務作業系統嘅營運主控台 (ops console)。佢係使用者入口，負責登入、查看、管理、觸發同監察，但係唔做最終授權，亦都唔執行工作流程 (workflow)、代理 (agent) 或者工具 (tool)。呢度有段分層角色樣板，我哋唔讀原文，只講重點。前端係使用者體驗層，負責呈現同互動；後端應用程式介面 (Backend API) 係受治理智能同控制層；代理係專門工作者；工作流程係結構化業務執行路徑；治理負責風險同審批；審計負責信任同可追溯。

跟住前端負責咩。佢負責呈現、互動、路由 (routing)、版面、使用者介面狀態 (UI state)、前端可用性驗證、使用者體驗、客戶端即時更新顯示，同埋設計系統 (design system) 實作。再嚟，前端唔負責咩。工作流程代理工具嘅執行唔好放瀏覽器；唔可以將客戶端 (client) 當唯一授權真相；唔好直接寫資料庫 (database)、背景工作 (background job)、嵌入 (embedding) 或者索引；唔好喺瀏覽器存密鑰或者供應商應用程式介面金鑰 (provider API key)；唔好由前端建立系統審計紀錄或者計費計算；唔好靜默改生產工作流程基因 (production DNA)，亦都唔好做主機自我改寫介面。重點嚟啦，邊界規則係咁：前端可以請求動作，後端決定允唔允許，隱藏掣唔係安全機制。同其他文件關係方面，structure_hk.md 係整體架構願景權威，backend_hk.md 講應用程式介面控制平面點落地，frontend_hk 同呢本書講營運台點樣只經應用程式介面操作系統，planning 下面 frontend 就係 FE-01 到 FE-20 子功能規格驅動開發 (SDD)。

好啦，第二章，產品願景同範圍。前端應該似嚴肅企業軟件即服務營運平台 (enterprise SaaS)，而唔係通用人工智能示範 (generic AI demo)。介面要傳達信任、可靠、營運清晰、安全、專業、速度、控制同可觀測。使用者要成日知到有邊啲代理同工作流程、咩嘢喺度跑、咩嘢失敗、咩嘢等審批、知識同資料邊度嚟、邊個做過咩、邊度要關注。

接落嚟範圍內。包括公開根路徑同認證，連接受邀請 (accept-invite)；已驗證應用殼層 (app shell) 同儀表板 (dashboard)；代理、工具、工作流程、執行詳情、審批；知識、記憶、評估、流程、審計；演化歸檔 (Evolution archive)；執行詳情上面嘅改進管線 (Improve)；組織、使用者、應用程式介面金鑰、安全；帳單 (billing) 可以佔位；回應式版面；載入 (loading)、空白 (empty)、錯誤 (error) 三態；權限感知導航；型別化應用程式介面 (typed API)；即時執行介面；同埋營運設定檔 (ops profile) 要演示模式 (DEMO_MODE) 做 false，對住即時後端加 Postgres。範圍外同第一章唔負責嗰啲一致。後端先係認證 (authn)、授權 (authz)、執行、知識、記憶、評估、治理、審計同組織安全嘅唯一真相源。

而家嚟到第三章，技術棧同執行期邊界。呢度有段技術棧樣板，我哋唔讀碼，只講用途。用 Next.js、React、TypeScript 同 Tailwind CSS。資料流就係瀏覽器入 Next.js 前端，再入後端應用程式介面 FastAPI，再入各服務。混合渲染方面，殼同元資料可以用伺服器元件 (server components)，即時時間軸、表單、指令面板 (command palette)、模態框 (modal) 用客戶端元件 (client components)。套件管理 as-built 用 pnpm。表單用 Zod 加 react-hook-form 做真實建立代理同工作流程。型別化客戶端係必要，用套件指令 api generate 產生 OpenAPI 型別。

再睇下執行期邊界。前端做渲染、路由、體驗層路由保護、呼叫應用程式介面、介面工作階段 (UI session)、顯示資料錯誤同即時事件、角色感知導航。後端做認證驗證、授權、角色、持久化同執行、審批狀態、審計寫入、知識索引、記憶、評估同密鑰。演示對營運方面，演示模式未設或者 true 用嚟介面預覽，唔依賴即時變更；營運即產品真相就係 false 加即時應用程式介面加 Postgres，用嚟產品標竿同 E1 操作路徑。

講完技術，第四章應用架構同設計流程。呢度有段高層架構樣板，我哋只講流程。使用者瀏覽器經超文本傳輸安全協定 (HTTPS) 入 Next.js，再經應用程式介面版本一路由 (API /api/v1) 入後端，涵蓋認證、代理、工具、工作流程、執行、審批、知識、記憶、評估、審計、組織、演化同改進。受保護路由係 app 下面星號。未登入去 login，無權限可以出四百零三拒絕存取 (403 Access Denied)，但呢個只係體驗層，後端四百零三仍然係最終依據。

跟住 OpenDesign 模型上下文協定 (OpenDesign MCP)。重大頁面版面應先走搜尋參考、提取設計標記 (tokens)、出版面計劃、再實作。as-built 如果協定唔用得，就用文件化後備 (documented fallback)，檔案喺 frontend docs design 下面 open-design-reference，呢個係規格允許，唔係偷懶。設計系統方面，有色彩字型間距狀態語義嘅標記 (tokens)，狀態徽章 (badge) 包括運行中、成功、失敗、等審批、暫停、取消、逾期，風格要企業營運密度，表格好掃，動效克制。

好，第五章資訊架構、導航同應用殼層。呢度有段主要路由樣板，我哋唔逐個讀路徑，只講地圖用途。公開有根路徑、登入、註冊、忘記同重設密碼、接受邀請。已登入有 app 下面代理、工具、工作流程、工作流程執行、審批、知識、記憶、評估、流程、審計日誌、演化、同設定下面組織使用者金鑰安全等。as-built 好多已驗證區可以用動態 app 萬用字元路由提供，但呢啲路徑仍然係深層連結 (deep link) 同導航嘅資訊架構。

接落嚟側邊欄分組樣板，我哋只講分組。主區有儀表板、代理、工作流程、審批；資料區有知識同記憶；品質區有評估、流程、演化；安全區有審計日誌；管理區有設定。頁首 (header) 有麵包屑 (breadcrumbs)、指令、環境指示、組織切換、通知、使用者選單。指令面板用命令鍵或者控制鍵加 K，可以建立、搜尋、最近執行、審批、邀請、應用程式介面金鑰等。

而家講下第六章，角色、權限同視覺系統。建議角色有擁有者 (Owner)、管理員 (Admin)、開發者 (Developer)、操作者 (Operator)、審核者 (Reviewer)、檢視者 (Viewer)、帳單管理員 (Billing Manager)、安全審計員 (Security Auditor)。權限顯示規則係前端按工作階段或者角色隱藏或者停用控制，呢個只係體驗層；後端最終允許或者四百零三；權限資料缺席就失敗封閉 (fail closed)。應用殼層就係側邊欄加頁首加內容區，細螢幕可以收成流動導航 (mobile nav)。

講完殼，第七章核心頁面由認證去儀表板。登入會對後端認證登入路徑做 POST，成功就按契約建立工作階段，例如權杖 cookie 或者 session 策略，再入 app；失敗顯示後端訊息，有請求識別碼 (request_id) 就一齊顯示。接受邀請已經出貨，路由係 accept-invite，可以帶查詢參數 token；對後端使用者邀請接受路徑做 POST，本文有 token、password、可選 name；呢個係公開端點，成功後寫工作階段入 app。as-built 用 AcceptInviteForm 加後端應用程式介面外觀 acceptInvitation。儀表板喺 app 根，有指標卡 (metric cards)、待審批、失敗執行、上手清單 (onboarding checklist)、快捷動作，同埋載入空白錯誤三態。

好啦，第八章代理、工具同工作流程。代理同工具有列表建立詳情，真實表單用 Zod 加 RHF，錯誤要有後端訊息同請求識別碼，記住唔好喺瀏覽器執行代理或者工具。工作流程同樣列表建立詳情，即刻執行 (Run Now) 會組合法載荷 (payload)，叫後端開執行，再導去執行詳情。as-built 用 run-workflow-button 同 workflow-run-payload。

重點嚟啦，第九章執行詳情，即時、關卡、改進同演化，呢個係營運最重要畫面之一。必備區塊有執行頁首、狀態、時間軸 (timeline)、步驟詳情、即時日誌 (live logs)、工具呼叫、輸入輸出、審批狀態、錯誤、審計元資料、同改進面板。狀態對齊後端生命週期，有排隊、運行中、等審批、暫停、成功、失敗、取消、逾期等。

as-built 動作全部經後端應用程式介面，呢度有段對照樣板，我哋唔讀路徑符號，只講用途。取消、重試、暫停、恢復、逾期分別對應執行底下 cancel、retry、pause、resume、expire。介面係 WorkflowRunConsole，有伺服器推送事件 (SSE) 連線狀態指示。即時更新優先用 SSE，斷線顯示重連中，事件要去重，唔好喺客戶端執行步驟。

人機閘門方面，審批有列表詳情，要明確批准或者拒絕，禁止靜默自動批准，執行上面待審提示要連去審批流。改進管線順序係反思、提案、評估、金絲雀，可以逐步或者全日管線，每一步都要人按加後端成功，禁止客戶端寫生產基因。演化歸檔路由係 app 下面 evolution，列沙盒變體適應度 (fitness)，評估金絲雀晉升只經後端，冇主機自我改寫介面。

跟住落嚟第十章，知識、記憶、評估、審計同設定。知識記憶評估流程以讀後端為主，搜尋要防抖 (debounce)，唔好做瀏覽器內向量庫或者索引引擎。審計係唯讀列表同篩選，前端唔建立系統審計紀錄。設定核心已出貨，使用者列表邀請停用用 UserAdminPanel，打邀請同局部更新使用者；組織儲存用 OrganizationSettingsForm，局部更新組織；應用程式介面金鑰有列表遮罩，建立後一次性密鑰規則；帳單可以佔位，唔好虛構收費。邀請體驗就係管理員建立邀請，顯示或者複製一次性權杖，受邀者去 accept-invite 帶 token。

而家嚟到第十一章，元件、資料夾、環境同應用程式介面。可重用元件包括按鈕、輸入、卡片、資料表 (DataTable)、狀態徽章、指標卡、空白狀態、錯誤狀態、骨架屏 (Skeleton)、時間軸、日誌檢視、區塊 (Section)、指令面板，同埋領域面板例如執行主控台、改進、審批、演化、使用者管理、組織表單。

呢度有段資料夾樹樣板，我哋唔逐行讀，只講用途。frontend 下面 src app 放路由，components 分介面版面領域認證，lib 放應用程式介面認證設定即時表單，design 放設計標記，tests 有單元同端到端，docs 有應用程式介面同設計，仲有 openapi.json。環境變數方面，公開應用程式介面基底網址指後端版本一路由，演示模式 false 係營運、true 係演示，密鑰絕對唔好用公開前綴暴露。應用程式介面整合規則係只打版本化版本一路由星號，用 backendApi 外觀加 OpenAPI 產生型別，錯誤用應用錯誤 (AppError) 加請求識別碼，綱要變咗要跑 api generate。主要域包括認證、使用者、邀請、組織、代理、工具、工作流程、執行連生命週期、審批、知識、記憶、評估、流程、審計、演化、改進。

講完整合，第十二章狀態、無障礙、安全同測試。主要資料頁一定要載入空白錯誤三態齊，錯誤有請求識別碼就顯示。無障礙要鍵盤可操作、表單標籤、焦點管理，狀態唔可以淨靠顏色，目標係無障礙內容指引二點二 AA 級 (WCAG 2.2 AA) 實務。安全上客戶端捆包 (bundle) 唔好有供應商密鑰，渲染要防跨站指令碼 (XSS-safe)，停用使用者撤銷金鑰等破壞性動作要確認，權限介面永遠唔取代後端。測試同品質閘有單元用 vitest、靜態檢查 (lint)、型別檢查 (typecheck)、建置 (build)；端到端煙霧測試伺服器未開可以跳過，唔係常駐持續整合硬性要求；as-built 產品標竿係單元綠燈、建置通過。

structure 能力閘前端份額方面，甲階段基礎有認證、殼、權限導航、空錯態；乙階段影子學習有知識記憶流程視圖；丙階段可控副駕駛有即時營運表單、即刻執行、執行詳情、人機閘、OpenAPI 客戶端；丁階段演化沙盒有改進、演化歸檔、評估金絲雀只經後端。關鍵 as-built 模組包括應用程式介面外觀 client.ts、即時營運清單 live-ops-surfaces、登入同接受邀請 auth 元件、執行主控台 workflow-run-console、改進 improve-run-button、演化 evolution-archive-panel、使用者同組織 user-admin-panel 同 organization-settings-form、建立表單 form-route-actions。

最終前端規則總結有六點。第一，後端係授權同執行真相源。第二，前端唔執行代理工作流程工具。第三，唔好靜默改生產基因。第四，重大版面用 OpenDesign 或者文件化後備。第五，應用程式介面變更後要跑 api generate。第六，營運設定檔演示模式 false 先係產品真相路徑。
